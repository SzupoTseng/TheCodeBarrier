# 第六篇　現實校準

# 第6章　那條線有多紅：法律、倫理與一筆算到最後的帳

> 矽谷，律師事務所的玻璃會議室。一份「用蒸餾模型加速產品」的提案被推回到桌子中央，法務只問了一句：「你們訓練數據裡，有沒有一條來自別人家的付費 API？」會議室安靜了三秒——那三秒的沉默，比這一整章任何一句話都值錢。

> 這是全書最重要的一章。前五章拆「能做什麼」，這一章談「該不該做」與「划不划算」。它不是道德說教，而是一個工程師在按下 Enter 之前，必須替自己和公司算清的那筆帳。

## 情境 1　法律與條款：四條你必須先知道的事實

**背景**：把坊間流傳的那些零散警告——「違反 ToS」「推動定為刑事犯罪」「指控阿里巴巴」——收攏成能用的判斷依據。

**四條事實（基於公開資料與常識，非法律意見）**：

1. **ToS 幾乎必然禁止蒸餾**。主流商業 LLM 廠商的條款，普遍明文禁止「用我家模型的輸出去訓練競品模型」。大規模採集 API 響應拿去訓練，**違約近乎確定**，後果從封號、追償到訴訟。
2. **「繞過防護」可能讓事情從「違約」升級為「違法」**。單純違反 ToS 通常是民事問題；但若你**主動繞過技術防護措施**（第 5 章 M1 那套代理輪換、對抗提示詞、降級偵測），在某些法域可能觸及「規避技術保護措施」「未授權存取電腦系統」這類**刑事或準刑事**條款。坊間流傳的「推動將大規模蒸餾攻擊定為刑事犯罪」，正是這個方向。
3. **數據集本身帶「法律放射性」**。下載一份來路不明的 `claude-traces`，即使不是你採集的，**使用它訓練商業模型，仍可能讓你成為侵權鏈條的一環**（第 2 章）。
4. **企業場景風險被放大 N 倍**。個人玩票被抓可能就封號；公司被認定靠蒸餾競品建立商業模型，面對的是**商業訴訟、賠償、品牌信任崩塌**——量級完全不同。

> ⚠️ 免責
> 以上不是法律意見。真要在商業場景碰這條線，**請諮詢專業律師**。本書立場很簡單：**對閉源商業模型的未授權蒸餾，技術可行，法律與倫理不可取。**

> 🔍 進階點評──為什麼「繞過防護」是整件事裡法律性質最嚴重的一步
> 第 2 條值得單獨深掘，因為它是多數工程師最容易低估的法律斷層。**違反 ToS 與繞過技術防護，是兩種完全不同量級的事。** 前者本質是「違約」——你和廠商有合約，你違反了使用條款，這通常落在民事範疇（封號、求償）。但當你部署代理 IP 池去規避限流、用對抗提示詞去騙過防護分類器、用降級偵測去對抗廠商的污染機制時，你做的已經不是「違反一份合約」，而是「**主動破解一個為阻止你而設的技術控制系統**」——這在許多法域（如美國 CFAA「未授權存取」、DMCA「規避技術保護措施」的相關討論，及各國類似立法）會把行為的法律性質從民事推向刑事。**這就是 M1 為什麼是整條流水線裡最危險的一塊**：它不只讓你「違約」，它讓你的每一次採集都成為「規避控制」的證據。一個關鍵的判斷啟示：**當你發現自己需要「騙過」或「繞過」對方專門設來阻止你的機制時，你大概率已經從「灰色地帶」邁進了「黑色地帶」——技術上的「能繞過」，恰恰是法律上「故意性（intent）」最有力的證明。** 防護盾的存在本身，就是一塊寫著「禁止進入」的牌子；繞過它，你就再也無法主張「我不知道這不被允許」。

### 條款的結構：三大廠商「禁止什麼」其實高度一致

別把「ToS 禁止蒸餾」當成一句模糊的口號——它在每一家主流廠商的合約裡，都對應到一條**結構幾乎相同的「禁止以輸出訓練競品」條款**。把三家擺在一起看，你會發現這不是某一家的脾氣，而是整個產業的共識防線：

| 廠商（條款所在）| 普遍包含的限制類型 | 對蒸餾者的意義 |
|---|---|---|
| OpenAI（Usage Policies / Business Terms）| 不得用「輸出」開發與其競爭的模型 | 拿輸出訓練學生 = 直接命中 |
| Anthropic（Usage Policy / Commercial Terms）| 不得用服務建立競品，**含訓練競爭性 AI 模型** | 蒸餾 Claude 系列 = 直接命中 |
| Google（Gemini API 附加條款 / 禁用政策）| 限制以服務輸出開發競爭性 ML 模型 | 蒸餾 Gemini = 直接命中 |

> ⚠️ 真偽提醒
> 上表是**結構性歸納，不是逐字引用**，且各家條款**版本更迭頻繁**（近年三家都調整過相關措辭，部分產品線放寬、部分收緊）。真要判斷，**以你簽約當下、該產品線的最新條款原文為準**，並請法務確認。本書要你記住的不是某一句話，而是——**那條防線一直在那裡，三家都在。**

**一個多數工程師沒想清楚的法律斷層：這主要是「合約問題」，不是「著作權問題」。** 美國著作權局已多次表明：**缺乏人類創作的純 AI 生成內容，不具可註冊的著作權**——也就是說，教師模型吐出的原始 token，本身**多半不受著作權保護**。這很容易讓人誤判成「那拿來用就沒事」，真相卻恰恰相反：**正因為著作權這道牆可能不存在，廠商的保護主要建立在你按下「同意」時生效的那紙合約（ToS）之上**，再疊加可能的**營業秘密**與前述**未授權存取／規避防護**等電腦犯罪法理。換句話說，蒸餾閉源模型的法律風險，**不取決於輸出有沒有著作權，而取決於你有沒有違約、有沒有繞過控制**——這也是為什麼「我只是用了它公開吐出來的文字」這種辯解，在合約與 CFAA 一類法理面前幾乎沒有意義。

> ⚠️ 真偽提醒
> 「AI 輸出能否受著作權保護」「拿模型輸出去訓練另一個模型是否構成侵權」目前**全球都沒有定論**——各國立場分歧、訴訟正在進行，而且「訓練資料端的爭議」與「模型輸出端的爭議」是**兩個不同問題**，別混為一談。本段陳述的是**目前的開放問題與主流方向**，不是已定案的法律。

### 兩道正在升起的新牆：監管與出口管制

紅線不只來自廠商合約，還來自國家。兩個方向值得每個做模型的人放進雷達：

- **EU AI Act（通用型 AI／GPAI 義務）**：若你蒸餾後**對外發布**一個通用型模型，在歐盟法下你很可能就是「GPAI 提供者」，需承擔技術文件、**著作權合規政策**、訓練資料摘要揭露等義務；達到系統性風險門檻（以訓練算力衡量）者義務更重。蒸餾不會讓你「繞過」這些義務——反而可能讓你**憑空繼承一個提供者的全套責任**。
- **出口管制（美國 EAR 等）**：先進 AI **模型權重本身**，正逐步被納入出口管制的討論與規則框架。你蒸出並散布的權重，在某些情境下可能就是受管制的標的——這跟「軟體開源就自由」是兩回事。

> ⚠️ 真偽提醒
> EU AI Act 的 GPAI 細則、與美國對模型權重的出口管制框架，**都處於高速演變、甚至反覆的階段**（相關出口管制規則在 2025 年曾數度調整與重估）。具體門檻、生效時點、適用範圍，請以**最新官方法規原文**為準；本書只負責標出「**這兩道牆正在升起**」這個方向，不為任何具體數字背書。

## 情境 2　經濟賬：偷，可能比買還貴

**背景**：很多人以為蒸餾是「省錢」——不付昂貴 API 費，自己蒸一個放本地。但把第 1 章那道「污染之盾」的成本算進去，這筆帳未必划算。

**一筆被嚴重低估的帳**：

| 成本項 | 內容 | 為何被低估 |
|---|---|---|
| **採集成本** | 多帳號、代理 IP 池、API 費；帳號被封要重買 | 以為「採一次就夠」，實則防護盾逼你反覆採 |
| **除污成本** | 教師降轉污染數據（第 1 章），採十萬條能用的可能剩三萬 | 還要再燒一輪算力 + 另一個可信模型判斷哪條被污染 |
| **訓練成本** | 全參數微調 35B 的 GPU 時數 | 一次調不好就要重來，迭代成本疊加 |
| **維護成本** | 教師一更新/下架，你的數據就過時，得重採重訓 | 你的模型「保鮮期」綁在別人的發布節奏上 |
| **法律期望成本** | 被追究機率 × 後果 | 對企業而言，這項可能是**最大的一項**，卻最難被算進 spreadsheet |

把這些加總，**和「乾脆合法付費用官方 API」相比，蒸餾的省錢神話往往不成立**。這正是第 1 章那道盾最聰明之處——**它不阻止你，它只是讓「偷」變得不划算。**

**把「合法蒸自己的」與「偷別人的閉源」並排，帳就一目了然。** 上面那張表算的是「偷」的隱性成本；下面這張，是把兩條路放在同一把尺上量——你會發現省錢神話不只是「未必成立」，而是**在企業尺度上幾乎必然倒掛**：

| 維度 | 合法路：蒸自有／開放權重 | 非法路：偷閉源 API |
|---|---|---|
| 採集合法性 | 清白，授權白紙黑字 | 違約，可能觸刑 |
| 供給穩定性 | 權重在你手上，永不下架 | 綁死他人發布節奏，一改版／下架即斷供 |
| 可追溯風險 | 無 | 學生可能**繼承教師的統計指紋／水印**，事後可被驗出 |
| 可商用／可審計 | 可商用、可過審計、可寫進盡職調查 | 寫不進任何合規文件，盡職調查一查就爆 |
| 監管義務 | 正常承擔 GPAI 等義務即可 | 義務照樣有，外加一筆違法事實 |
| 總成本曲線 | 前期投入高、之後攤平、可預期 | 看似前期省，後期被「持續採集＋持續除污＋法律期望成本」拖成無底洞 |

**「可追溯風險」這一列，值得單獨拉警報。** 學術界早有「放射性數據（radioactive data）」的研究：訓練資料會在模型權重裡留下**可被統計檢測的痕跡**；近年的文本浮水印（統計式 watermark 一類）與成員推斷攻擊，更把「這個學生是不是蒸了我的輸出」從玄學變成**可舉證的技術問題**。這意味著一件讓人脊背發涼的事：偷來的能力**不會因為你關掉採集腳本就變乾淨**——它已經焊進了權重，成為對方未來任何一天都能拿出來的呈堂證據。**你以為偷完船過水無痕，實際上你親手把指紋烤進了模型，而且烤進去就拿不出來了。**

> 💡 君之一席話
> 與君一席話：**最好的防盜，不是更高的牆，而是讓偷竊的總成本高過購買——當「偷」比「買」還貴，誠實就成了最理性的選擇，而非最高尚的選擇。** 這是安全經濟學最深的一課：你無法阻止所有攻擊者，但你可以重塑他們的損益函數，讓攻擊在「期望值」上變成虧本生意。Anthropic 那道污染之盾的天才，不在於技術多難破，而在於它把蒸餾者的成本結構從「一次性採集」改寫成「持續採集 + 持續除污 + 持續維護 + 持續承擔法律風險」——四個「持續」疊加，再對比官方 API 那條清晰、合法、有 SLA 的定價，理性的人自己就會放棄。**真正成熟的防禦，是讓對手在計算機前算著算著，自己嘆口氣關掉了終端機。**

## 情境 3　六條鐵律：讓「想蒸餾」這件事安全落地

**背景**：把「想要本地強模型」這個完全正當的需求，導向安全合法的軌道。

1. **先問教師是誰**。蒸餾**開放權重模型**（Qwen、Llama、Mistral、GPT-OSS，遵守其授權）——綠燈。蒸餾**閉源商業 API**——紅燈。這一條過濾掉九成麻煩。
2. **讀懂授權，不只讀懂 ToS**。開放權重 ≠ 隨便用。有些禁商用、有些要求標註、有些限制衍生模型再授權。**用之前把 LICENSE 讀完**（Llama 的可接受使用政策、Qwen 的授權、Apache-2.0/MIT 的差異都要分清）。
3. **數據集要查身世**。任何 `claude-traces`、`gpt-dump` 類數據集，使用前確認來源合法、授權清楚、上傳者可信。**來路不明的原料，再香也別下鍋。**
4. **絕不主動繞過技術防護**。速率限制、防護分類器、存取控制——這是廠商劃的界。繞過它們，是把「違約」升級成「違法」的那一步，**永遠不要踏出去**（見情境 1）。
5. **去自我宣稱 ≠ 抹除來源**。在自有模型上做品牌中性化是正常工程；在他人模型數據上抹除來源痕跡，是湮滅證據。**同一技術，認清你站在哪一邊。**
6. **企業場景，法務先行**。任何牽涉商業模型輸出的訓練計劃，**先過法務、再寫程式**。工程師的浪漫，不該變成公司的官司。

**第 1、2 條值得展開：開放權重不等於「隨便用」，授權地景比想像中崎嶇。** 把幾種主流授權擺在一起，你就知道為什麼「讀 LICENSE」不是官腔，是保命：

| 授權類型（代表）| 商用 | 關鍵限制／義務 |
|---|---|---|
| Apache-2.0 / MIT（多數 Qwen、部分 Mistral）| ✅ | 最寬鬆；保留版權聲明即可，Apache 另含專利授權 |
| Llama 社群授權（Meta Llama 系）| ✅ 附條件 | 自訂授權＋可接受使用政策；超大用戶量（MAU 門檻）需另談授權；衍生模型有命名／標註要求 |
| Gemma 授權（Google）| ✅ 附條件 | 受 Gemma 條款＋禁用政策約束 |
| 部分廠商自訂授權（某些大尺寸權重）| 視情況 | 可能含 MAU 門檻、再授權限制或特定領域禁用 |

> ⚠️ 真偽提醒
> 上表是**結構性導覽**。各家授權的細節——尤其「能否用輸出去改進其他模型」「MAU 門檻的具體數字」「衍生模型命名規則」——**版本間會變動**，本書成稿與你閱讀之間可能又改過。**任何一次商用前，以該模型該版本的 LICENSE 原文為唯一依據**，別靠記憶、別靠本表。

**第 3 條的正面解法，是合成數據。** 與其去下載來路不明的 `claude-traces`，不如用**一個授權清楚的強開放模型**，搭配 `distilabel` 這類框架**自己生成**高品質合成 CoT 與指令資料——原料、過程、產出三段全在陽光下，可商用、可審計、可重現。這條路的品質正逐年逼近「偷來的軌跡」，而它最大的優勢甚至不是品質，是**乾淨**：你能對任何人、任何審計、任何法庭，解釋每一條數據從哪來。

**而把這六條鐵律走到極致、最具商業價值的那個用法，叫「蒸餾你自己的模型」。** 企業手上往往已有一個又大又貴、跑在雲端的內部大模型；把它的能力蒸進一個輕量學生，部署到邊緣、端側或本地內網——**延遲更低、成本砍掉一個數量級、資料不出私網、IP 100% 屬於自己、過得了任何盡職調查**。這正是第 5 章那套 M2–M5 流水線最正當、最賺錢的落點：教師是你自己的模型，全程零紅線。**注意這裡的對稱之美：同一套蒸餾技術，只要把教師換成「你有權榨取的那一個」，整件事就從『攻擊』瞬間變回『工程』——分水嶺從來不在程式碼，在你對那個教師，有沒有授權。**

> 🔍 進階點評──合法的路不是「次優選擇」，它正在變成「最優選擇」
> 這六條鐵律讀起來像「約束」，但換個角度看，它們其實在指向一條越走越寬的陽關道，而這條道的回報正在快速超過那條紅線。**第一，開放權重生態已經夠強。** 2026 年的 Qwen、Llama、Mistral、GPT-OSS 系列，能力已強到配上第 4 章的蒸餾方法論 + 第 3 章的 DGX Spark，你完全能在零紅線的前提下蒸出一個跑在自己機器上、可商用、可微調、能睡得安穩的強模型。**第二，合成數據正在崛起。** 用強開放模型 + `distilabel` 自動生成高品質合成 CoT，已是完全合法、且品質逐年逼近「偷來的軌跡」的路線——你蒸的是「開源模型的影子」，但這個影子本身合法、可控、且越來越聰明。**第三，蒸餾自有模型是企業剛需。** 把公司內部的大模型能力蒸給輕量模型部署到邊緣/端側，這是第 5 章那套 M2-M5 流水線最正當、最有商業價值的用法——沒有任何紅線。**所以真正成熟的姿態是：把追求『本地、私有、可控的強模型』這個理想，與『繞過別人的盾去偷』這個手段徹底解綁。** 理想完全正當，手段大有選擇。繞過別人的盾，從來不是唯一的路，甚至——把帳算到最後——不是最划算的那條路。

## 情境 4　回到那個最初的問題
