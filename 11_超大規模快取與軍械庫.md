## 情境 1　三級異構儲存金字塔：把 KV Cache 當作業系統的虛擬記憶體來管

**背景**：在動輒支援百萬級 token 上下文的模型裡，如果把所有用戶的 KV Cache 全都死鎖在 GPU/TPU 的 HBM（高頻寬顯存）裡，集群會在併發稍微一漲時就因 OOM 瞬間癱瘓。HBM 是整個系統裡最快、也最貴、最稀缺的資源——拿它去存「一個三天沒回來的用戶的對話歷史」，是奢侈到犯罪的浪費。原始素材主張的解法，本質是把作業系統管了五十年的老把戲——**虛擬記憶體與多級快取**——原封不動搬到 KV Cache 上：建一座三級異構儲存金字塔（Hierarchical KV Cache Management）。

三級金字塔，由快到慢、由貴到便宜、由小到大：

```
            ┌───────────────────────────────────────────┐
            │  L1：TPU/GPU HBM  (High-Bandwidth Memory)  │  ← 最快·最貴·最小
            │  存什麼：當前正在 Decode 的 Active Segments  │
            │  容量量級：數十 GB / 卡   頻寬：~TB/s         │
            └───────────────────────────────────────────┘
                          ▲   100 GB/s ~ TB/s
                          │   PCIe Gen5 / NVLink / ICI
                          ▼
            ┌───────────────────────────────────────────┐
            │  L2：Host Memory  (CPU DDR5)                │  ← 中速·中價·中量
            │  存什麼：暫不活躍、但可能馬上用到的多輪歷史    │
            │  容量量級：數百 GB ~ TB / 節點                │
            └───────────────────────────────────────────┘
                          ▲   Direct I/O
                          │   NVMe-over-Fabrics / RDMA
                          ▼
            ┌───────────────────────────────────────────┐
            │  L3：Distributed NVMe SSD Pool             │  ← 慢·便宜·海量
            │  存什麼：冷數據（長視頻多模態、冷凍 Session）  │
            │  容量量級：數十 TB ~ PB / 池                  │
            └───────────────────────────────────────────┘
```

對應關係一目了然：**L1 之於 KV Cache，正如 CPU L1 快取之於主記憶體；L3 之於 L1，正如硬碟之於 RAM**。當一個用戶正在打字、模型正在逐 token 生成，他的 KV 必須在 L1；當他發完一句、進入「人類在思考下一句」的幾秒空檔，把他的 KV 從 HBM 換出（Offload）到 L2 的 DDR5，HBM 立刻騰出來服務別人；當他半小時沒回來，再把整段 Session 沉到 L3 的 SSD 池裡冷凍。

**關鍵技術：異步預取流水線（Prefetching Pipeline）**。金字塔本身不難，難的是「搬運」這件事——HBM 和 Host Memory 之間隔著 PCIe，一次搬幾 GB 的 KV 是要時間的。如果搬運期間計算停下來等，那這套分層就成了負優化：你省了顯存，卻賠了延遲。原始素材的解法是在編譯器（XLA）與執行期裡，把**計算線程與 I/O 線程徹底解耦**，並用一個古老而強大的詞掩蓋住搬運成本——**重疊（Overlapping）**：

```python
# 概念示意：用「別人的 decode 時間」掩蓋「我的 KV 搬運延遲」
# 真實系統在 XLA runtime / CUDA stream 層用異步 DMA 實現，此處僅示意調度語義

while serving:
    batch = scheduler.next_batch()          # 取一批正在 decode 的請求

    # ── I/O 線程（與計算並行，不阻塞）──────────────────
    if detect_user_typing_next_turn(user_X):
        # 偵測到用戶 X 正在輸入下一句 → 提前把他的歷史 KV
        # 從 L2 DDR5 異步預取回 L1 HBM。此刻計算線程根本不知道、也不用等
        io_thread.async_prefetch(user_X.kv, src="DDR5", dst="HBM")

    # ── 計算線程（持續滿載跑當前 batch）────────────────
    compute_thread.decode_step(batch)        # 這一步的耗時，正好「蓋住」上面的搬運

    # 等用戶 X 真正按下送出時，他的 KV 早已躺在 HBM 裡 → 零感延遲
```

精髓在那句註解：**用其他請求正在執行的 decode 計算時間，去「掩蓋」當前請求 KV Cache 的搬運延遲**。在大批次推理裡，永遠有一堆請求正在 decode、把 TPU 的 Tensor Core 餵得飽飽的；I/O 引擎就躲在這片計算的陰影下，悄悄把下一個要用的 KV 從 DDR5 搬回 HBM。等用戶真的按下 Enter，他的歷史早就在 HBM 裡等著了——延遲被藏進了硬體本來就忙著的那段時間裡，硬體利用率（MFU）始終高位運行。

> 💡 君之一席話
> 與君一席話：**所有「讓系統變快」的高招，拆到底層，幾乎都是同一句話——別讓昂貴的資源等便宜的資源；當搬運不可避免，就把它藏進計算的陰影裡。** 三級金字塔不是什麼 AI 新發明，它是作業系統教科書第三章的虛擬記憶體、是 CPU 的 L1/L2/L3、是資料庫的 buffer pool，換了一身「KV Cache」的衣服又走了一遍。真正值錢的不是「分三層」這個結構——那是常識；值錢的是**異步預取那一手「用別人的計算時間掩蓋我的搬運時間」**。新手做分層快取，做出來的是「換出換入都要停下來等」的同步版本，結果省了顯存、賠了延遲，得不償失；老手做分層，第一件想的事就是「搬運的延遲，能不能躲到某段反正也要花的時間後面去」。**這就是 overlapping 的哲學：系統工程的最高境界，不是讓每件事都變快，而是讓慢的那件事，發生在沒人等它的時候。** 你能不能在腦子裡同時轉動「計算線程」和「I/O 線程」兩條獨立的時間軸，並讓它們在正確的點交錯——這一念之差，分開了「會用快取的人」和「會調度快取的人」。

> 🔍 進階點評 — 為什麼「KV Cache 換頁」比「OS 換頁」更兇險，也更值得
> 把 KV Cache 當虛擬記憶體換頁，方向絕對正確，但要清醒地看到它和 OS 換頁的**三個關鍵差異**，否則照搬必翻車。**其一，粒度與語義不同**：OS 換頁的單位是固定的 4KB page，內容無語義；KV Cache 的單位是「一個 token 序列的注意力狀態」，它有強烈的**時序前綴依賴**——你不能只換出一段歷史的中間，因為後面的 token 算的時候要看見前面的。所以 KV 的換出單位通常是「整段 prefix」或 vLLM 那種固定 block，且必須前綴連續。**其二，代價結構是反的**：OS 換頁換出的是「資料」，換回來就是原樣；KV Cache 如果不換、選擇丟掉，要還原它得**重算 prefill**——而 prefill 是 compute-bound 的重活。這意味著「換出到 DDR5」和「直接丟棄再重算」之間有一道精算題：搬一趟 PCIe 的 I/O 成本，對比重跑一遍 prefill 的算力成本，誰便宜？序列越長，重算越貴，越該留著搬；序列越短，重算越便宜，丟了反而省事。這道精算，正是情境 3「投機式快取遷移」的微觀版本。**其三，這是 vLLM 已經產品化的方向**：vLLM 的 CPU offloading、`--swap-space`、以及社群在做的 KV 分層，正是這套金字塔的 L1↔L2 落地；再往 L3 SSD 走，則進入 Mooncake（Kimi 背後那套以 KVCache 為中心的分離式架構）這類前沿系統的領域。**所以這一節不是空中樓閣，它是把一個已在生產線跑著的方向，講清楚它的物理。** 唯一要對素材打的折是：「零感延遲」是理想態——只要用戶回得比預取快，或者預取猜錯了人，就會露出搬運的尾巴。沒有真正的零延遲，只有「藏得夠不夠好」的延遲。

## 情境 2　給快取做減法：從 MHA 到 MLA，模型結構層的瘦身術

**背景**：情境 1 是純工程——不改模型，只在外面建金字塔搬來搬去。但原始素材一句話點破了它的天花板：**「純粹靠工程優化是有上限的，必須和算法團隊聯手（Co-design），從模型結構上給 Cache 瘦身。」** 這是全章最重要的一句方法論。KV Cache 之所以大，根子在 Attention 的結構：每一層、每一個注意力頭，都要為每個 token 緩存一份 K 和一份 V。要從源頭砍，就得動 Attention 本身。這條演進路線——MHA → MQA → GQA → MLA——是過去幾年大模型推理效率最重要的一條暗線。

先講清楚 KV Cache 的體積公式。在經典 **Multi-Head Attention（MHA）**裡，每個 token 要緩存的 KV 大小正比於：`2（K和V）× L（層數）× H（頭數）× D（每頭維度）`。其中那個 **H（頭數）**，就是被歷代改進反覆開刀的對象：

```
MHA  ：每個 Q 頭配自己的 K 頭、V 頭          KV 頭數 = H      （基準·最大）
        Q1 Q2 Q3 Q4 ... QH
        K1 K2 K3 K4 ... KH
        V1 V2 V3 V4 ... VH

MQA  ：所有 Q 頭共享 唯一 一組 K、V          KV 頭數 = 1      （省 H 倍·傷能力）
        Q1 Q2 Q3 Q4 ... QH
        K（只有一個，大家共用）
        V（只有一個，大家共用）

GQA  ：Q 頭 分組，每組共享一對 K、V          KV 頭數 = G（組數，如 8）  （業界主流）
        [Q1 Q2 Q3 Q4][Q5 Q6 Q7 Q8]...
         K_grp1        K_grp2
         V_grp1        V_grp2

MLA  ：不存高維 K/V，存低秩壓縮隱向量 c_t     緩存 = 一個極小的 latent 向量（解碼時線上解壓）
        K, V ──低秩投影──► c_t（維度極小）   解碼時 ──線性投影──► 還原出 K, V
```

逐個拆解這條瘦身路線的取捨：

| 機制 | KV Cache 相對大小 | 核心做法 | 代價 / 現狀 |
|---|---|---|---|
| **MHA** | 1×（基準） | 每個 Q 頭獨享一對 KV 頭 | 表達力最強，但 KV 體積最大、長上下文最吃顯存 |
| **MQA** | ~1/H | 所有 Q 頭**共享唯一一組 KV** | 顯存驟降 H 倍，但表達力下降太多，大模型能力受損嚴重 |
| **GQA** | ~G/H（如 8/H） | Q 頭**分組**，每組共享一對 KV（Llama 系常用 8 組） | 速度與效果的折中，**當前業界主流**，被 Llama 3 等廣泛採用 |
| **MLA** | 宣稱省 90%+ | **低秩壓縮**：把 K/V 投影進低維隱空間 c_t，解碼時線上解壓 | DeepSeek 提出（V2/V3 代表作），在維持表徵力下大幅縮 KV；算法-基建協同設計的典範 |

MQA 和 GQA 是「減少 KV 頭的數量」——MQA 一刀砍到只剩一組，太狠，模型變笨；GQA 學乖了，砍到剩幾組（比如 8 組），在「省顯存」和「保能力」之間找了個甜蜜點，於是成了 Llama 3 一類主流開源模型的標配。

**MLA（Multi-head Latent Attention）** 走的是完全不同的一條路。它不在「頭數」上做減法，而是在「維度」上做壓縮。它的洞察是：高維的 K 和 V 裡藏著大量冗餘，可以用**低秩矩陣分解（Low-rank Compression）**把它們投影壓進一個維度極小的隱空間向量 `c_t`——緩存的時候，只存這個小小的 `c_t`，而不是原來那一大坨高維 K、V；到了解碼階段，再用線性投影矩陣**線上（on-the-fly）把 `c_t` 解壓**回真正的 K 和 V 去算注意力。本來每個 token 要存 `2 × L × H × D`，現在只要存一個小向量。

> ⚠️ 真偽提醒
> 「MLA 把 KV Cache 砍掉 90% 以上、並發量提升數倍」這個數字來自 DeepSeek 的論文與技術報告，是**特定模型、特定配置下的測量值**，不是普適常數——壓縮比強烈依賴隱空間維度的選取、模型規模與序列長度。MLA 由 DeepSeek 提出並在其開源模型上驗證，是可查證的真實工作；但「DeepMind 也在深度跟進類似方向」這句是原始素材的內部口吻**宣稱**，本書無法獨立查證，當方向性參考即可。可平實陳述的事實是：MQA / GQA / MLA 都是已發表、已被生產級模型採用的真實技術，這條「給 Attention 的 KV 瘦身」的演進線是真的。

> 💡 君之一席話
> 與君一席話：**最深的性能優化，從來不發生在「工程」和「算法」各自的圍牆裡，而發生在那道牆被拆掉、兩邊的人坐到同一張桌子上的時候——MLA 就是這道牆倒下時長出來的東西。** 情境 1 的金字塔，是 infra 團隊一個人能做完的事：模型不動，我在外面搬。但它有天花板——KV 本身有多大，金字塔就得搬多大，你只是把「貴的搬運」變成「便宜的搬運」，沒讓需要搬的東西變少。MLA 不一樣，它要改的是注意力的數學形式，這是算法團隊的領地；可決定「壓到多少維才不掉點、解壓的算子在 TPU 上跑不跑得動」的，又是 infra 的領地。**任何一邊單獨都做不出 MLA：算法團隊不懂解壓算子在硬體上的代價，會壓出一個理論漂亮、實際慢的東西；infra 團隊不敢動注意力公式，只會在外面搬箱子。** 這就是 co-design 的真義——不是「算法先設計好，丟給工程去實現」這種接力賽，而是兩邊在白板前同時被對方的約束逼著讓步、長出一個誰都想不到的折中。**一個團隊能不能做出結構性的突破，不看它招了多強的算法科學家或多硬的系統工程師，看它有沒有勇氣讓這兩種人，為了同一個 KV 數字吵到同一塊白板上去。**

## 情境 3　集群分布式：當快取有了「位置」，路由就成了戰爭

**背景**：單機把 PagedAttention、Prefix Caching 做到極致還不夠。一旦你管的是由成千上萬顆 TPU/GPU 組成的超大推理集群，網關（Gateway）與調度器就必須升級出一種新能力——**快取感知（Cache-Aware）**。因為在集群裡，快取不再是一塊抽象的「顯存空間」，它**長在某一台具體的物理機器上**。誰持有用戶 A 的歷史 KV，是節點 7 還是節點 42，這件事決定了下一個請求該去哪——這正是本章開頭那場「電費燒成灰」災難的根源。

**傳統調度（Round-Robin / Least-Connection）的災難**：

```
用戶 A 連發 5 句，傳統負載均衡按「最閒」或「輪詢」分發：

  msg#1 ──► 節點1   (重算 A 的全部歷史 prefill)
  msg#2 ──► 節點2   (節點2 沒有 A 的快取 → 又重算一遍全部歷史)
  msg#3 ──► 節點3   (再重算一遍)
  msg#4 ──► 節點1   (節點1 的快取可能已被別人擠掉 → 還是重算)
  msg#5 ──► 節點4   (重算)

結果：同一段「五萬字歷史」的 prefill，被算了 5 遍。
      你在每台機器上做的 Prefix Cache，命中率全部歸零——
      因為下一句話根本不回到上一句所在的那台機器。
```

負載均衡器的本能是「把活分給最閒的機器」——這在無狀態的 Web 服務裡是金科玉律。但 LLM 推理**不是無狀態的**：請求 N 的便宜，建立在「請求 N-1 的 KV 還在這台機器上」之上。把有狀態的快取，交給一個假設無狀態的調度器，就是在拿全集群的算力，為「快取放錯了位置」這個盲區陪葬。

**正解：前綴哈希 + 親和性路由 + 投機式快取遷移**。網關必須持有一張全局的分布式快取索引（通常用內存資料庫或高效 RPC 維護），三步走：

| 步驟 | 機制 | 做法 |
|---|---|---|
| ① 識別 | **Prefix Hashing（前綴哈希）** | 網關對請求的 `System Prompt + 歷史對話` 算一個前綴哈希（如 **MurmurHash3**，快、分布均勻）。相同前綴 → 相同哈希 |
| ② 路由 | **Affinity Routing（親和性路由）** | 不再投給「最閒」的節點，而是優先投給**已經緩存了該前綴 KV 的那台物理節點**，最大化集群整體的 Prompt Cache 命中率 |
| ③ 權衡 | **Speculative Cache Transfer（投機式快取遷移）** | 若持有快取的節點 A 已過載：精算「在閒置的節點 B 重算 prefill」vs「用集群高速網路把 KV 從 A 直接拉到 B」，誰快選誰 |

第三步是整套設計裡最精密的一手，值得拆開看那道精算題：

```
情況：用戶請求命中節點 A 的快取，但節點 A 此刻過載（佇列很長）。
調度器面前兩條路：

  路徑甲【重算】：把請求發給閒置的節點 B，讓 B 從零重算整段歷史 prefill
       成本 ≈ Prefill_FLOPs(歷史長度) / B 的算力
       —— 序列越長，這條路越貴（prefill 是 compute-bound）

  路徑乙【遷移】：把 A 上那段已算好的 KV，經高速網路 RDMA 直接搬到 B
       成本 ≈ KV_Bytes(歷史長度) / 集群網路頻寬
       —— 頻寬夠猛時，搬一份現成的，比從頭算一份快得多

  決策：if  搬運成本(乙) < 重算成本(甲):  走 RDMA 遷移，省下整段 prefill 算力
        else:                          認賠，在 B 重算
```

注意這道題和情境 1「KV 換頁 vs 重算」是**同一道題在不同尺度上的重演**：情境 1 是「搬到隔壁的 DDR5」對比「丟了重算」，這裡是「跨機搬到節點 B」對比「在 B 重算」。物理直覺一致：**prefill 的算力成本隨序列長度超線性增長，而搬一份現成 KV 的成本只隨它的位元組數線性增長**。所以序列越長、上下文越大，「搬現成的」就越划算——這正是為什麼長上下文時代，集群內的快取遷移網路會變得和算力一樣關鍵。

> ⚠️ 真偽提醒
> 「Google Jupiter 網路 1.6 Tbps」「直接傳輸快取帶來顯著 TTI 優勢」這類具體頻寬數字與內部網路代號，是原始素材以 DeepMind 內部口吻給出的**宣稱**，本書無法獨立查證，當量級示意看待。可平實陳述的真實技術是：**親和性路由 + 前綴哈希是 SGLang 的 RadixAttention、以及各家推理網關（如 vLLM 的 prefix-aware 調度、Mooncake 的全局 KV 索引）正在做的真方向**；MurmurHash3 是真實的非加密哈希；RDMA 跨節點搬 KV 也是 PD 分離（情境 4）架構裡的真實機制。要打折的只是「某廠某網路某頻寬」這類具體到產品的數字。

> 🔍 進階點評 — 親和性路由的暗礁：快取親和與負載均衡，是一對天生反目的孿生子
> 親和性路由聽起來完美，但生產線上它會把你拖進一個經典的**兩難**，沒看清這點就會從一個坑跳進另一個坑。**負載均衡要的是「把活分散開」，快取親和要的是「把活集中到持有快取的那台」——這兩個目標在物理上直接對沖。** 你越是嚴格地把用戶 A 的所有請求釘在節點 7 上（快取命中率拉滿），就越容易製造**熱點**：某個被瘋狂調用的長 system prompt（比如一個爆紅的 Agent 模板），會把所有命中它的流量全灌到最早緩存它的那兩三台機器上，把它們烤糊，而集群裡別的機器在睡覺。這就是為什麼成熟的 cache-aware 調度器，從來不是「無腦親和」，而是一套**帶負載感知的親和**——SGLang 的調度、以及各家網關的做法，本質都是在「快取命中收益」和「節點負載代價」之間動態打分：命中當然好，但若持有者已經過載，就觸發情境 3 第③步那道「遷移 vs 重算」的精算，主動把流量導開或把快取複製到第二台。**更深一層，這逼出一個架構哲學：在有狀態的推理集群裡，根本不存在「最優路由」這種靜態答案，只存在「在快取親和、節點負載、遷移成本三者之間持續再平衡」的動態過程。** 把它當成一個一次算完的哈希函數，你會死於熱點；把它當成一個持續調節的控制迴路，你才真正擁有了一座能呼吸的集群。這也是分布式系統永恆的母題：一致性（把狀態釘在固定位置）與可用性／負載（把流量攤開）之間，永遠在拔河——快取感知路由，只是這場拔河在 LLM 時代的最新戰場。

## 情境 4　PD 分離：當 Prefill 和 Decode 的硬體需求徹底相反

**背景**：把快取感知路由推到極致，會撞上一個更深的結構性矛盾，它的解法是當前超大規模推理基建的「終極演進方向」——**PD 分離（Disaggregated Prefill & Decode）**。要理解它為什麼是終局，得先看清一個被很多人忽略的事實：LLM 推理的兩個階段，**Prefill 和 Decode，是兩種物理性質完全相反的負載**，把它們塞在同一張卡、同一個 batch 裡，是在讓兩個八字不合的人合住一間房，互相拖累。

兩個階段的對照：

| 維度 | Prefill（預填：消化 prompt） | Decode（解碼：逐 token 生成） |
|---|---|---|
| 計算特性 | **Compute-bound（算力受限）** | **Memory-bound（訪存受限）** |
| 在幹嘛 | 把整段 prompt 一次性過一遍，產出 KV Cache | 每次只生成 1 個 token，反覆讀取龐大的 KV |
| 吃什麼資源 | 吃**算力**（Tensor Core 滿載）、吃頻寬 | 吃**顯存容量**（裝下所有並發的 KV）、吃訪存頻寬 |
| 理想硬體 | 高算力、高頻寬晶片（如 HBM3e） | 大容量顯存 |
| 痛點 | 一個長 prompt 進來會**霸佔算力**，餓死正在 decode 的請求 | 算力大量閒置，只是在不停搬運 KV |

矛盾就在這：**Prefill 缺的是算力，Decode 缺的是顯存容量；Prefill 把算力榨乾的時候，Decode 的算力在閒置；Decode 把顯存佔滿的時候，Prefill 的顯存需求被擠壓。** 它倆混在同一個 GPU batch 裡，會發生最難看的事——一個五萬 token 的長 prompt 進入 prefill，瞬間霸佔全部算力，正在逐字回答的幾十個短請求集體卡住，它們的 **TTFT（首 token 延遲）和 TPOT（token 間延遲）一起飆升**，用戶眼裡就是「打字打到一半突然頓住」。

業界對這個矛盾有兩層解法，一輕一重：

```
─── 同機緩解：Chunked Prefill（分塊預填）───────────────────
  把長 prompt 的 prefill 切成固定大小的 chunk（如每次 512 token），
  算完一塊 → 讓出算力給 decode 跑一步 → 再算下一塊。
  把「一口吞下長 prefill」變成「小口流水線化」，避免長請求餓死短請求。
  ↑ vLLM / DeepSpeed-FastGen 的 Dynamic SplitFuse 都在做。屬「同一張卡上的時間切片」。

─── 跨機根治：PD Disaggregation（PD 分離）─────────────────
  乾脆把集群劈成兩種專用節點：

   ┌────────────────────┐                      ┌─────────────────────┐
   │  Prefill 節點 (P)   │   RDMA 直推 KV       │  Decode 節點 (D)     │
   │  高算力·高頻寬       │ ───────────────────► │  大容量顯存          │
   │  (如 HBM3e)         │   把算好的 KV Cache   │  專心逐 token 生成   │
   │  專心消化 prompt    │   經高速網路推給 D    │  不被 prefill 打斷   │
   └────────────────────┘                      └─────────────────────┘

  P 節點算完 KV，透過超高速 RDMA 網路把整份 KV「推」給 D 節點，
  D 節點接過來專心 decode。兩種硬體各司其職，互不干擾。
```

Chunked Prefill 是**同一張卡上的時間切片**——緩解，治標；PD 分離是**跨機器的空間隔離**——根治，治本。因為兩個階段的硬體需求是相反的，最徹底的解法不是讓它們在同一張卡上輪流忍讓，而是給它們配**各自最適合的硬體**：prefill 節點堆算力，decode 節點堆顯存容量，中間用 RDMA 把 KV 一推了事。

還有一層編譯器底層的功夫，與 PD 分離互補——**XLA 的算子融合（Kernel Fusion）**：

```
未融合（通用框架的天真做法）：
  PagedAttention 尋址 ──寫回HBM──► RoPE 旋轉位置編碼 ──寫回HBM──► Softmax
  └ 每一步的中間結果都寫回 HBM 再讀出來，HBM 往返是延遲與頻寬的殺手

融合後（手工優化的單一 TPU Kernel）：
  ┌─────────────────────────────────────────────────┐
  │  PagedAttention + RoPE + Softmax  融合成一個算子   │
  │  中間變量留在寄存器，不寫回 HBM，寄存器級數據復用    │
  └─────────────────────────────────────────────────┘

另一手：固定 block size（如 16 / 32 token 一塊）→ 把「動態序列長度」
        轉成「靜態 block 數組組合」→ 避免動態形狀觸發 XLA 頻繁重編譯
```

兩個底層細節都指向同一件事：**消滅不必要的 HBM 往返、消滅動態形狀**。算子融合把 attention 尋址、RoPE、Softmax 三步揉成一個 kernel，中間結果留在寄存器裡，不來回寫 HBM；固定 block size 則把「序列長度動態變化」這個編譯器最討厭的東西，轉成「固定大小 block 的數組」這種靜態、可預測的形狀，讓編譯出來的機器碼跑在最高效率上，不會因為每來一個新長度就觸發一次代價高昂的重新編譯。

> ⚠️ 真偽提醒
> 「PD 分離是 Google DeepMind / OpenAI 在基建層的終極演進方向」是原始素材的內部視角宣稱；可查證的事實是 PD 分離是真實且火熱的研究與工程方向（DistServe、Splitwise、Mooncake 等公開工作都在做），vLLM 也已支持 disaggregated serving。「Prefill compute-bound、Decode memory-bound」是被廣泛測量驗證的真實特性。要打折的是「某某頂尖公司內部已全面採用、某某具體頻寬」這類無法外部核實的斷言——把它當成「業界公認的前沿方向」，而非「某廠既成現實」。

> 💡 君之一席話
> 與君一席話：**當你發現一個系統怎麼調都調不好，十有八九不是參數沒調對，而是你硬把兩個本性相反的東西，塞進了同一個盒子裡——真正的解法往往不是更聰明地共處，而是承認它們合不來，然後把盒子拆成兩個。** Prefill 和 Decode 的故事，是這條智慧最乾淨的範例。多少團隊在「同一張卡上讓 prefill 和 decode 和平共處」這件事上耗盡心力——調 batch、調優先級、做 chunked prefill 切得越來越細——本質都是在一段不幸福的婚姻裡做夫妻調解。Chunked Prefill 是高明的調解，但它改變不了「一方要算力、一方要顯存」這個根本不合。PD 分離的洞察樸素到近乎粗暴：**既然你倆要的東西相反，那就別住一起了。** 這在分布式系統裡有個莊嚴的名字叫「關注點分離」，可它最難的從來不是「分」這個動作，而是**敢於承認「合在一起本身就是錯的」這個判斷**——因為「合在一起」通常是歷史包袱、是「一直以來都這麼做」、是省事。**一個架構師的成熟度，不在他能把多複雜的東西揉到一起，而在他能不能在所有人都習慣了某種耦合之後，站出來說『這兩件事根本不該在一起』，然後扛著遷移成本把它們劈開。** 看穿不合，比優化共處，難得多，也值錢得多。

## 情境 5　二十大前沿軍械庫：一張按攻堅小組分群的作戰地圖

**背景**：前四個情境，是把素材裡幾條主線講透。但原始素材最後攤開的，是一份更大的東西——大模型快取領域的**二十個前沿方向**，橫跨頂級會議（NeurIPS / ICML / ICLR / MLSys）的最新論文、工業界生產環境的開源神兵、以及頂尖架構方法論。它把這二十個方向，按「該由哪個攻堅小組去打」分成了三層：**1–6 演算法層**（給演算法工程師魔改模型）、**7–12 基礎設施層**（給 infra 團隊做引擎選型壓測）、**13–20 架構方法論層**（給架構師寫系統 HLD 時通盤佈局）。下面把這份軍械庫整理成一張作戰地圖——它正是本書附錄 G 那份更完整的快取技術速查的種子。

**【演算法層 1–6】——歸演算法 / 研究工程師：在模型結構與 KV 本身上動刀**

| # | 方向 | 一句話技術核心 | 代表 |
|---|---|---|---|
| 1 | **MLA** | 低秩壓縮把 K/V 投影進低維隱空間 c_t，解碼時線上解壓，KV 體積大砍（見情境 2） | DeepSeek-V2/V3 |
| 2 | **H2O（Heavy-Hitter Oracle）** | Attention 矩陣呈**冪律分佈**，少數 Token 貢獻絕大部分權重；動態識別並只保留這些 Heavy-Hitter 的 KV，即時剔除其餘 → 定長快取裝無限長文 | 《H2O》論文 |
| 3 | **StreamingLLM** | 發現 **Attention Sink** 效應（最初 2–4 個 Token 鎖住巨大注意力）；保留「初始幾個 token + 最新滑動窗口」，不重算 prefill 即可流式無限解碼 | 《Attention Sinks》論文 |
| 4 | **Speculative Decoding + Tree-KV** | 投機解碼（小模型出草稿、大模型校驗）會產生多分支；用**樹狀 KV Cache** 並行校驗多分支，避免多路重複計算 | Medusa 系列 |
| 5 | **Infinite-LLM / Mamba-2 混合** | 把超出視窗的舊 KV 壓進一個**固定大小的循環狀態（Recurrent State）**，空間複雜度 O(N) → O(1) | Griffin / Hawk / Mamba-2 |
| 6 | **KV-Quant（低位元量化）** | Per-Channel/Per-Token 混合量化 + 非均勻量化 + 離群值保留，把 KV 壓到 **2-bit/3-bit 幾乎不掉點** | 《KV-Quant》論文 |

這六個方向有個共同點：它們都在問「**KV Cache 本身，能不能更小或更聰明**」——MLA 壓維度、H2O 和動態淘汰丟無用 token、StreamingLLM 只留頭和尾、Mamba 系把歷史揉進固定狀態、KV-Quant 降位元。注意 1、2、3、5 之間有一條暗線：**它們都在跟「注意力的長尾」較勁**——H2O 說「大部分 token 不重要，丟」，StreamingLLM 說「頭幾個和最近的最重要，留」，Mamba 說「乾脆把過去壓成一個狀態」。三種哲學，回答同一個問題：**面對無限長的歷史，到底該記住什麼、可以忘掉什麼**。

**【基礎設施層 7–12】——歸 Infra 團隊：推理引擎的選型與壓測**

| # | 工具 | 殺手鐧 | 適用場景 |
|---|---|---|---|
| 7 | **vLLM** | **PagedAttention** 鼻祖，解決顯存碎片化；Chunked Prefill / Prefix Caching / 張量並行 | 工業界黃金標準、通用首選 |
| 8 | **SGLang** | **RadixAttention**——把 Prompt 快取做成自動化**基數樹（Radix Tree）** | 複雜控制流、Few-shot、Tool-use 的多輪 Agent，命中率與吞吐強 |
| 9 | **LMDeploy** | KV Cache 量化（W4A16、KV INT4/INT8）支持極完美 + Persistent RPC | 邊緣端 / 私有化顯存受限場景榨極限 |

| 11 | **LightLLM** | **Token Attention**——Token 級細粒度顯存調度 | 超大並發、請求長度極不均勻的惡劣環境，抗 OOM |

這六個是「現成能用的兵器」——選型的關鍵不是「哪個最強」，而是「**你的負載長什麼樣**」：通用就 vLLM；多輪 Agent、prompt 高度復用就上 SGLang 的 RadixAttention（它正是情境 3 親和路由的單機版）；顯存受限要極致量化就 LMDeploy；綁死 NVIDIA 全家桶就 TensorRT-LLM。

**【架構方法論層 13–20】——歸架構師 / Tech Lead：寫 HLD 時的戰略佈局**

| # | 方法論 | 落地要點 |
|---|---|---|
| 13 | **Cache-Aware Routing** | 網關用 MurmurHash3 算前綴哈希，把同前綴請求路由到持有該 KV 的節點，最大化集群命中率（見情境 3） |
| 14 | **Tiered Cache Orchestration** | HBM(L1)/DDR5(L2)/NVMe(L3) 三級，背景線程 Offload/Prefetch（見情境 1） |
| 15 | **Chunked Prefill** | 長 prefill 切成固定 chunk，與 decode 流水線化，防長請求餓死短請求（見情境 4） |
| 16 | **Dynamic KV Eviction** | 不再 LRU 粗暴丟整段歷史；按**注意力重要度**打分，**優先丟標點、虛詞、過渡詞**，保留名詞與核心語意實體——「有損但高智慧」的回收 |
| 17 | **Deterministic Prompt Formatting** | 「**靜態在前、動態在後**」；Tools 定義 / System Prompt / Few-shot 按固定順序序列化；**絕對禁止在 prompt 開頭塞 UUID、隨機鹽、時間戳**（否則前綴永遠不命中） |
| 18 | **Exact Prefix vs Semantic Cache** | Exact：token 完全一致走 RadixTree 復用 KV；Semantic：用 **GPTCache + Milvus/Qdrant** 算向量相似度，**>0.98 直接攔截返回上次答案，成本歸零** |
| 19 | **PD Disaggregation** | Prefill 節點算完 KV，RDMA 直推 Decode 節點（見情境 4） |
| 20 | **Context Cache Monetization** | 參考 **Anthropic Claude 的 `cache_control`**：代碼裡顯式聲明快取錨點，海量長文（法律合約、代碼庫）任務輸入費用大降 |

這八個是架構師的「全局棋」。其中第 16、17 兩個值得單獨點名，因為它們最便宜、最常被忽略、收益卻最高。**第 17 確定性提示詞格式化，幾乎是零成本的純紀律**——你不需要改一行模型、不需要買一張卡，只要在團隊開發規範裡寫死一條「prompt 開頭永遠是靜態內容，動態的東西放最後」，前綴快取的命中率就能從「形同虛設」變成「常態命中」。反過來，只要有一個粗心的工程師在 system prompt 開頭塞了個 `request_id=<uuid>` 或 `當前時間：...`，就足以讓**整個團隊精心建設的 prefix cache 全線失效**——因為前綴第一個 token 就變了，後面再相同也沒用。**第 18 的語意快取則是另一個維度的降本核武**：exact cache 省的是「相同前綴的重算」，semantic cache 省的是「意思相近的問題，根本不調模型」——GPTCache 用向量資料庫算相似度，超過 0.98 直接把上次的答案返回去，這一次推理的成本是零。

> ⚠️ 真偽提醒
> 表中數字需分級看待：**Claude `cache_control` 是 Anthropic 真實提供的 API 功能，prompt caching 確實能大幅降低重複長前綴的輸入費用**——但「降本 80%」是依賴緩存命中率與前綴佔比的場景值，不是保證值（實際取決於你重複用了多少快取、快取讀取本身也有計費）。「相似度 >0.98 直接返回」的閾值是工程經驗值，設太鬆會返回答非所問的快取——語意快取是把雙刃劍，命中率與正確率要自己權衡。SGLang「命中率與吞吐超越 vLLM 數倍」是特定 benchmark 的結果，換負載未必成立。**這二十個方向本身都是真實存在的論文、工具或方法論；要打折的永遠是那些精確到個位數的百分比。**

> 🔍 進階點評 — 這張地圖最大的價值，不是二十個技巧，而是那條「該誰去打」的分群線
> 把二十個方向背下來，是助理工程師的本事；看懂它們**為什麼被分成 1–6 / 7–12 / 13–20 三層**，才是架構師的視野。這條分群線藏著一個關於組織與技術的深刻判斷：**這三層需要的，是三種根本不同的人，動三種根本不同的東西，承擔三種根本不同的風險。** 演算法層（1–6）動的是模型的數學結構，做錯了模型會掉點、會變笨，風險在「能力」，所以必須由懂 attention 數學、會跑大規模消融實驗的研究工程師來扛——一個 infra 工程師再強，也不敢去改 MLA 的隱空間維度。基礎設施層（7–12）動的是引擎選型與壓測，做錯了系統會 OOM、會吞吐崩盤，風險在「穩定性與成本」，需要的是能讀懂 CUDA kernel、會做壓力測試、會看 P99 尾延遲的系統工程師。架構方法論層（13–20）動的是跨服務的整體佈局，做錯了整個集群會在「快取放錯位置」「prompt 格式不確定」這類**沒有單點故障、卻處處在漏電**的問題上慢性失血，需要的是能在白板上同時看見路由、分層、格式規範、商業計費的架構師。**最致命的錯配，是讓錯的層的人去打錯的仗**——讓架構師去調 MLA 的低秩維度（他不懂消融），或讓演算法工程師去設計集群路由（他不懂熱點與負載），都會出事。**而真正頂尖的技術 leader，價值恰恰在這條分群線上：他不需要親手做完二十件事，他需要的是準確地判斷『這個方向是模型問題、系統問題、還是架構問題』，然後把它交給對的小組，再讓三個小組的成果在 co-design 的白板上對齊（情境 2 那道牆）。** 知道一件事該誰做，比知道怎麼做，是更高階、也更稀缺的能力——這二十個方向的分層，與其說是技術清單，不如說是一張**組織如何分配認知的地圖**。

## 本章小結

- **核心轉變**：單機，快取是「要省的顯存」；集群，快取是「要調度的戰略資產」——它有位置、有親和性、有遷移成本，跟算力一樣是會讓 SRE 半夜被叫醒的東西。
- **情境 1　三級異構金字塔**：L1 HBM（active decode）/ L2 DDR5（暫不活躍多輪歷史）/ L3 NVMe（冷數據），把 OS 虛擬記憶體搬到 KV Cache；靈魂是**異步預取 + Overlapping**——用別人的 decode 時間掩蓋我的搬運延遲。
- **情境 2　結構層瘦身**：MHA → MQA（砍到 1 組 KV，太傷能力）→ GQA（分組共享，業界主流，Llama 3）→ MLA（DeepSeek 低秩壓縮進隱空間 c_t，線上解壓，KV 大砍）。關鍵是**算法-基建 co-design**——拆掉兩個團隊之間的牆。
- **情境 3　快取感知路由**：Round-Robin 在有狀態推理裡是災難（同用戶散落多節點、重算 prefill）；正解 = 前綴哈希（MurmurHash3）+ 親和性路由 + 投機式快取遷移（重算 vs RDMA 拉取的精算）。暗礁：**快取親和與負載均衡天生反目**，需帶負載感知的動態再平衡。
- **情境 4　PD 分離**：Prefill 是 compute-bound、Decode 是 memory-bound，硬體需求相反；Chunked Prefill 是同機時間切片（治標），**PD 分離是跨機空間隔離（治本）**——承認兩者不合，劈成兩種專用節點，RDMA 推 KV。配 XLA 算子融合（消滅 HBM 往返）+ 固定 block size（消滅動態形狀重編譯）。
- **情境 5　二十大軍械庫**：演算法層 1–6（MLA / H2O / StreamingLLM / Tree-KV / Mamba 混合 / KV-Quant）歸研究工程師；基建層 7–12（vLLM / SGLang / LMDeploy / TensorRT-LLM / LightLLM / DeepSpeed-FastGen）歸 infra；架構層 13–20（路由 / 分層 / Chunked Prefill / 動態淘汰 / 確定性格式 / 語意快取 / PD 分離 / Claude `cache_control` 商業化）歸架構師。**分群線比清單本身更值錢**——知道一件事該誰做，比知道怎麼做更稀缺。
- **真偽紀律**：MLA、H2O、StreamingLLM、KV-Quant、vLLM、SGLang、Claude `cache_control` 等全是真實工作；要打折的永遠是「砍 90%、降本 80%、超越數倍、Jupiter 1.6 Tbps」這類精確數字與「某頂尖實驗室內部如何做」的宣稱。

這二十個方向只是種子。把它們連同前十一章散落各處的快取技巧——從第 3 章 UMA 零拷貝、第 5 章 KV Cache FP8 量化、到本章的三級金字塔與 MLA——收攏成一張可隨手翻查的速查表，正是**附錄 G「快取技術速查」**要做的事：核心概念、顯存壓縮、推理引擎、架構方法論、底層算子、語意快取、命中率避坑，外加一塊把「砍 90%、降本 80%、Jupiter 1.6 Tbps」這類數字單獨釘住的真偽校準區。

## 終章：三個樂章，一根脊椎

軍械庫關上門，整本書也走到了盡頭。回望這趟旅程，它其實只問了一個問題，問了三遍。

第一樂章，我們問「**怎麼造它**」——在那道反蒸餾的盾下，合法地蒸出一個只屬於你、跑在你桌上的本地模型。第二樂章，我們問「**怎麼用它**」——讓這個又小又便宜的模型長出眼和手，去看懂、去操作一台為人類設計的電腦，再抬頭質疑這門技藝所站的整個範式。第三樂章，我們問「**怎麼養它**」——當這雙眼手要從一個 demo 放大到萬級並發，去打那場最隱形的記憶體與快取之戰，好讓「廉價」這個承諾在規模上不跳票。

造它、用它、養它。你會發現這三件事根本分不開：你在第三樂章為之廝殺的每一 GB KV Cache，省下來都是為了讓第一樂章那台 DGX Spark，能撐住第二樂章那雙永不闔眼的眼睛。**這不是三個主題釘在一起的合輯，是同一個工程理想——「一個真正屬於你、聽你的、便宜到能當器官、強到能養得起的智慧」——被從三個角度逼問到底。**

全書反覆打磨的，其實也只有兩樣東西。一把**真偽的尺**：在那一連串誘人的名字（Mythos、Qwythos、Aluminium OS、Jupiter 1.6 Tbps、78.4 分）面前，本能地分清「哪些是我能驗證的、哪些是別人要我相信的」——這條線，劃開了會思考的工程師和會複述的傳聲筒。一條**清白的路**：技術從來中性，分水嶺在意圖與授權；蒸餾你有權使用的模型、用合法的數據、把它養在自己的機器上，是工程；繞過別人的盾去偷一個影子，是攻擊。

技術會過時，這本書裡的每一個型號、每一個分數、每一個發表會的名字，明年都可能無人記得。但那把尺和那條路，會比任何模型活得更久。
