# 附錄C　爭議與認知問答

> 本附錄整理蒸餾爭議的核心問答：蒸餾為什麼有爭議、到底提取的是權重還是推理邏輯、為什麼只學答案會失敗，以及小模型如何在訓練中長出「像會推理」的行為。本附錄只做教育性整理；涉及閉源商業 API 的未授權蒸餾，仍以第 6 章紅線為準。

## C.1　蒸餾的三層爭議

有一句很準確的判斷：知識蒸餾是「技術上公開，商業上敏感」的技術。爭議不是因為 KD 本身邪惡，而是因為它同時踩到三組利益邊界。

| 爭議 | 表面問題 | 深層問題 | 工程判斷 |
|---|---|---|---|
| **ToS 與授權** | 能不能用 API 輸出訓練競品 | 合約禁止常比版權問題更直接 | 閉源商業 API 輸出不可當免費訓練料 |
| **搭便車與不公平競爭** | 小模型用低成本學到大模型能力 | 教師的算力、資料、安全成本被外部化 | 合法蒸餾應限於自有模型、開放權重或授權資料 |
| **安全能力被洗掉** | 學到能力，未必學到防護 | 學生可保留推理能力但弱化拒答與風險判斷 | 評估不能只看分數，還要看安全與拒答品質 |

> 🔍 進階點評──「大家都在做」不是免責理由
> 大公司私底下做蒸餾，常見理由有三個：降低推理成本、把能力壓到端側、用強模型生成合成數據做評估。這三件事本身都合理；問題在於**教師是否授權、資料是否可追溯、用途是否與條款衝突**。工程上真正成熟的做法，不是把灰色地帶說成產業慣例，而是把管線設計成可替換教師：同一套 M2-M5 流水線，教師可以是自家模型、開放權重模型、或有授權的商業模型。只要教師一換成「未授權閉源 API」，同一套技術就從 MLOps 變成風險事件。

## C.2　蒸餾提取的是權重，還是推理邏輯？

答案：**不是複製教師的物理權重，而是讓學生用自己的權重重建教師的行為分佈、偏好與推理軌跡。**

這點很重要。閉源教師的權重不可見；即使白盒可見，教師與學生的張量形狀、層數、注意力頭、詞表也通常不同，沒有「把 120B 權重塞進 9B」這種物理操作。蒸餾真正遷移的是三類訊號：

1. **輸出序列**：教師如何回答，學生學文字表面與格式。
2. **機率分佈**：教師在多個可能 token 間如何猶豫，學生學「暗知識」。
3. **推理軌跡**：教師如何拆題、回推、使用工具，學生學問題解法。

用生活化比喻：提取權重像把老師的大腦切下來塞給學生，既不可能也沒有意義；蒸餾像讓老師大量示範解題，學生的大腦還是自己的，但解題習慣被重新塑形。

## C.3　為什麼「答案是 42」不是好訓練資料

常被提到的「答案是 42」這個梗，重點不在科幻典故，而在它對蒸餾的提醒：**沒有問題框架與推理過程的答案，是低密度資料。**

如果訓練集只有：

```text
問題：生命、宇宙與萬物的答案是什麼？
答案：42
```

學生只學到字串對應，學不到為什麼這個回答成立、何時只是玩笑、何時應該追問「你真正想問的是什麼」。這正是硬標籤 SFT 的天花板。

更好的蒸餾樣本至少包含：

- 問題語境：使用者是在問梗、數學、還是哲學？
- 推理路徑：模型如何辨識典故與語用目的？
- 反事實邊界：哪些情況下不該回答 42？
- 最終答案：短答、長答、或追問澄清？

> 💡 君之一席話
> 與君一席話：**答案是壓縮檔，推理是解壓密碼。** 只給學生答案，就像給他一個打不開的壓縮包；他可以記住檔名，卻無法重建內容。好的蒸餾不是讓學生背更多「42」，而是讓它學會判斷什麼時候 42 是玩笑、什麼時候 42 是錯答、什麼時候問題本身需要被重寫。

## C.4　大模型已經固化，為什麼學生還能學到推理？

教師固化，指的是教師權重不再更新；這不代表教師不能產生新的訓練訊號。蒸餾時，教師像一台固定的高品質標註器：每次面對不同題目，都會產生不同的輸出、不同的中間步驟、不同的工具調用路徑。

學生學習發生在自己的梯度更新中：

1. 學生先根據目前權重產生預測。
2. 訓練系統比較學生預測與教師訊號。
3. 損失函數把差距轉成梯度。
4. 優化器微調學生權重。
5. 重複數萬到數百萬次後，學生的行為分佈向教師靠攏。

所以「推理寫入」不是把教師內心搬過去，而是**用大量帶結構的示範，改變學生在下一個 token 上的條件機率**。當這種改變跨過足夠多任務、足夠多中間步驟後，看起來就像學生學會了推理。

## C.5　小模型的學習過程：三個日常比喻

**學廚**：只看成品照片，是硬標籤；看師傅每一步火候、試味、補鹽，是 CoT 蒸餾。小廚師最後做出的菜不是師傅那鍋，但味型會靠近。

**學開車**：只背「紅燈停、綠燈行」，遇到施工、救護車、無號誌路口就慌；坐在教練旁邊聽他說「先看左後視鏡、再判斷行人速度」，才是在學決策流程。

**學除錯**：只給修好的程式碼，學生會背 patch；給錯誤訊息、排查順序、假設驗證、回滾原因，學生才會學會 debug 的節奏。

## C.6　整理正文的判斷

原始素材後半段把蒸餾講得更激進，本書整理時做了三個取捨：

1. **保留機器學習原理**：權重 vs 邏輯、logits、CoT、學生如何學推理。
2. **保留工程可用性**：如何設計資料、基準、管線、工具選型。
3. **降級高風險內容**：對未授權蒸餾、去審查、越獄測試，只保留風險分類與防護觀點，不轉錄操作步驟。

這樣處理後，原始素材的價值不再是「一份可照做的灰色操作紀錄」，而是一份可被工程師安全閱讀的反面教材與方法論補充。
