## 情境 1　第一梯隊：在窄窗裡刮下的數據，與它鑄成的模型

**背景**：承上一章，社群只有極短窗口採集 Fable 5。坊間流傳的**第一梯隊**，是用那批數據最先煉出的成品。

**這份傳聞清單與技術解剖**：

| 模型 | 基底（Base）| 蒸餾方法 | 據稱特性 |
|---|---|---|---|
| **Qwable-v1** | Qwen3.6-35B-A3B | SFT（指令微調）| 繼承 Fable 5 工具調用（Tool Surface），能輸出 `str_replace_editor` 格式 |
| **GPT-OSS 120B Fable-5 Distilled** | GPT-OSS 120B | 蒸餾 + GGUF 轉換（AutoTrust AI Lab 提供算力）| MoE，128 專家／激活 4；GGUF Q8_0、Q5_0 |
| **Jackrong 系列**（如 27B Coder）| Qwen 系 | 軌跡反轉（Trace Inversion）| 社群「最受信任」，減少過擬合 |
| **數據集 armand0e/claude-fable-5-claude-code** | —— | 去隱私化軌跡採集（與 Glint-Research 合作）| Fable 5 原始 Agent Traces，可轉 OpenAI 格式 |

**把技術內核拆開看**：

- **`Qwen3.6-35B-A3B` 的命名玄機**：`A3B` 是 Qwen MoE 的標記——**總參 35B、每次只激活約 3B**。選它當基底不是隨便挑的：MoE 的「激活參數小」特性，剛好對應第 3 章 DGX Spark 的頻寬瓶頸。**選基底是在選一具與你硬體相容的骨架，不是選參數量最大的那個。**
- **`str_replace_editor` 是什麼**：這是 Claude 系列在 Agent 場景用來改檔案的**原生工具標籤**（text editor tool）。Qwable-v1 號稱「繼承工具調用能力」，意思是它在權重層面學會了**主動吐出與官方一模一樣的工具調用 XML**——這對接 OpenHands、Aider 這類開源 Agent 時，不必再寫一層 prompt 轉譯，出錯率低。**蒸餾者這次最想偷的，不是「會聊天的 Claude」，是「會調工具的 Agent 版 Claude」。**
- **採集→數據集→模型的工具鏈**：`armand0e/...` 這種「軌跡數據集」是整條鏈的源頭。標準流程是——採集的原始 traces（常是 Claude 的 messages 格式）→ 用腳本轉成 **ShareGPT / OpenAI `messages` 格式** → 餵進 **Axolotl** 或 **LLaMA-Factory** 的 SFT 配置 → 用 `peft` 上 LoRA 或全參數微調。**「去隱私化（de-PII）」這一步**通常靠 **Microsoft Presidio** 或 spaCy NER 把金鑰、信箱、人名洗掉——這在合法數據處理裡是必修，在非法採集裡則兼具「抹除來源證據」的副作用（見第 6 章）。

> 💡 君之一席話
> 與君一席話：**模型是凝固的數據，數據是流動的模型——真正的能力轉移發生在數據層，模型只是它的一種快照。** 看懂這句，你就不會再執著於「下載哪個蒸好的模型」，而會去問「它是用什麼數據、什麼格式、什麼方法煉的」。同一批 traces，今天灌進 Qwen、明天灌進 Llama、後天灌進你自家架構，出來的是三個不同身體、同一個靈魂。誰掌握了數據與配方，誰就能無限複製影子；只拿到成品的人，永遠只是終端消費者。

> 🔍 進階點評──SFT 蒸餾的天花板，與它為何「形似神不似」
> 第一梯隊幾乎清一色用 **SFT（序列級蒸餾）**，這是被 API「只給文字、不給 logits」逼出來的選擇，但它有一道結構性天花板。SFT 教的是「**模仿教師的最終輸出序列**」，學生學到的是 Claude 的**文體與表面套路**（會吐 `<thinking>`、會用那套 XML），卻學不到 Claude 在每個 token 上「**為什麼選這個詞、其他詞差多少**」的軟標籤分佈——而那層機率分佈（暗知識）才是推理能力的真正載體（詳見第 4 章）。這就是為什麼這類蒸餾模型常被吐槽「**形似神不似**」：複雜邏輯一深就露餡，因為它在模仿「答案長什麼樣」，而非「怎麼想出答案」。要突破這層天花板，方法論上有三條進階路：① **on-policy 蒸餾**（MiniLLM／GKD，讓學生先生成、教師再評分）；② **拒絕採樣 + 過程獎勵**（rejection sampling 篩出高品質 CoT）；③ **logit 對齊**（若能拿到局部 logits，做溫度 KL）。第一梯隊之所以只到「形似」，正因為窄窗 + 防護盾讓後兩條路成本爆炸——**它們不是不懂方法論，是被經濟賬逼回了最便宜的 SFT。** 順帶給你一個「真蒸餾長什麼樣」的對照組：這條光譜上**真實存在、可下載驗證**的標竿是 `DeepSeek-R1-Distill` 系列——DeepSeek 把 R1 的長推理軌跡蒸進 Qwen2.5 與 Llama 3 的 1.5B～70B 基底，權重、model card、基底血緣全部公開，拿它的 `config.json` 跟對應基底一比就完全對得上（情境 5 教你怎麼比）；反觀傳聞中的 `Fable-5 Distilled`，你連基底的 commit SHA 都查不到——**有沒有戶口、查不查得到基底，就是「真蒸餾」與「貼牌傳說」最素樸的分界。**

## 情境 2　被神話的「平民版 Claude」：Qwythos-9B

**背景**：聲量最大的一款，是傳聞中據稱由 **Empero AI** 釋出的 **Qwythos-9B-Claude-Mythos-5-1M**。行銷話術極具煽動性：「Claude 被開源了？無審查、104 萬上下文、4GB 顯存可跑！」（出處標題見附錄 A：零度解說、Interfaze、Threads）。

**坊間流傳的規格與技術**：

- **基底**：Qwen3.5-9B，做**全參數微調（Full FT）**，餵「Claude 5 系列數據」。
- **1M（104 萬）上下文**：靠長上下文擴展技術（見下文）。
- **無審查（uncensored）**：移除拒絕機制。
- **4GB 顯存可跑**：靠 GGUF 量化（如 Q4_K_M）。

衍生與預告：

- **richardyoung/qwythos-9b-abliterated**（Ollama 上架）：第三方用 **abliteration** 做二次「徹底去審查」的變體。
- **Qwythos-27B**（Empero Roadmap 預告）：據稱下一代更大的「Mythos 級」蒸餾模型。

**把「無審查」與「長上下文」的技術拆開**：

- **abliteration 到底做了什麼**：這是真實技術（Arditi 等人的 "refusal direction" 研究）。做法是——拿一批「有害提問」與「無害提問」分別跑模型，取兩組在某層殘差流的**平均激活差，得到一個「拒絕方向」向量**；再把模型權重對這個方向做**正交化（orthogonalization）**，抹掉模型「想說不」的那條迴路。工具上常用 **TransformerLens** 做激活分析、或現成的 `refusal_direction` / `heretic` 流程。**名字裡的「heretic」就是 abliterated 基底的代號**（情境 3 的 FableVibes 卡片寫明「started as Qwen3.6-35B-A3B-heretic」，正是這條技術線）。
- **1M 上下文怎麼來**：9B 原生不可能有 1M 視窗，必然是**事後外推**——主流是 **YaRN**（NTK-aware 的 RoPE 內插，把 RoPE base 從 1 萬級拉到百萬級）配 **FlashAttention** 撐住長序列的記憶體。問題在於：**「能外推到 1M」與「在 1M 上讀得懂」是兩回事**——下一章用硬體真相戳破。想當場分辨它是真做了外推還是純吹，最快的一招是打開 `config.json` 看 `rope_theta`（RoPE base，Qwen2.5 預設 100 萬、Llama 3 為 50 萬）與 `rope_scaling`（YaRN 會在這裡寫明 `type: yarn` 與 `factor`）：**真上了長上下文，這兩個欄位一定留指紋；只改 README、config 原封不動的，幾乎都是貼牌**（情境 5 詳解這套鑑識）。

> ⚠️ 真偽提醒
> 「9B + 4GB 顯存 + 1M 上下文 + 無審查 + 約等於 Claude」這組合**違反工程常識**。abliteration、YaRN 都真實，但把它們疊在「Qwythos-9B」這個具體模型上並宣稱接近 Claude，屬未核實的行銷敘事。

> 💡 君之一席話
> 與君一席話：**「去審查」從來不是免費的，它的帳單寫在你看不到的地方。** abliteration 抹掉「拒絕方向」時，連帶會鈍化模型的判斷力與安全直覺——它不只「敢回答敏感問題」了，它可能連「這段程式有 SQL 注入、我該提醒你」這種**有益的拒絕**也一併忘了。研究已反覆證實：過度 abliterate 會拉低模型在推理與事實性基準上的表現。所以當你被「無審查」三個字吸引時，先想清楚你要的究竟是「自由」，還是「一個不再對你說不、也不再替你把關的工具」——前者是能力，後者是失能，而它們長得一模一樣。

> 🔍 進階點評──小模型「裝得下」與「扛得動」的鴻溝
> Qwythos-9B 的神話，本質是混淆了三個被行銷刻意焊在一起、其實獨立的維度，而這三者背後是三套不同的工程瓶頸。**「裝得下 1M」** 是 RoPE/YaRN 的數學外推問題——改幾個參數就能宣稱達成，幾乎零成本。**「扛得動 1M」** 是 KV Cache 的記憶體與預填充（prefill）算力問題——9B 模型在 20 萬 token 時，KV Cache 會膨脹到吃掉你以為省下的所有顯存，且 prefill 一篇長文可能要等好幾分鐘，這是頻寬與容量的硬牆。**「讀得懂 1M」** 是模型容量與訓練數據的能力問題——9B 的注意力頭數與層數，決定了它的「大海撈針（Needle-in-a-Haystack）」檢索力天生不如 35B／120B，視窗再長也只是「看得到、記不住」。**行銷只賣第一個維度，因為它最便宜、最好吹；工程師必須親手驗證後兩個，因為它們才決定這 1M 是真本事還是裝飾。** 驗證工具也很現成：跑一遍開源的 RULER 或 Needle-in-a-Haystack 測試，真假立判。

## 情境 3　不追最新：前代 Claude 的成熟蒸餾品

**背景**：坊間還有一條補充的線索——若覺得 9B 推理不夠、或不迷信「最新的 Mythos」，社群裡還有幾款基於**前代 Claude（Opus / Claude 4.6 世代）**數據蒸餾的成熟模型。

| 模型 | 基底 | 方法 | 據稱定位 |
|---|---|---|---|
| **Qwen3.6-14B-A3B-FableVibes**（tvall43 GGUF）| Qwen3.6-35B-A3B「heretic」剪枝版 | Pruning + QLoRA 補償 | 小體積保留「Claude 感」；用 Fable 5 多步推理軌跡 + Opus 推理數據 |
| **clzoro/Qwen3.5-27B-Claude-distill** | Qwen3.5-27B（密集）| 全參數微調，餵 Opus 高品質 CoT | 數學、邏輯、多輪對話接近官方 |
| **Dzluck/Qwen3.5-2B-Claude-4.6-Opus-Reasoning-Distilled**（含 GGUF）| Qwen3.5-2B | Claude 4.6 長文推理蒸餾 | 手機／樹莓派等邊緣裝置的極限嘗試 |

> ⚠️ 真偽提醒
> 上表三款（**FableVibes、clzoro、Dzluck**）的模型名、上傳者、規格與「Opus／4.6 蒸餾」血緣，本書一律無法核實。其背後的技術手法（pruning、QLoRA、密集 vs MoE、2B 邊緣蒸餾）全部真實且常見；但「某人把 Opus 的 CoT 蒸進某個 27B」這種**具體血緣聲稱**，在你按情境 5 的方法驗過 `config.json` 與基底權重之前，只能當成 README 上的一句廣告詞。

**技術看點**：

- **FableVibes 的「剪枝 + QLoRA 補償」是一套完整方法論**：先對 35B MoE 做**網路剪枝（pruning）**剃掉低活化專家/神經元縮小體積，剪枝必然損傷能力，於是再用 **QLoRA**（NF4 4-bit 量化基座 + LoRA 低秩適配 + paged optimizer，Dettmers 等人的方案）做「補償訓練」把掉的分數補回來。它的基底卡片直言「started as Qwen3.6-35B-A3B-**heretic**」——印證了 abliterated 基底（情境 2）正是這條產線的上游。
- **密集 vs MoE 的取捨**：clzoro 用的是 **27B 密集（dense）**架構，全參數微調餵 Opus 的 CoT。密集模型每 token 要讀全部權重，**慢但「思考連貫性」往往更好**（不會像 MoE 那樣在專家間跳變導致邏輯反芻）——這是它「文學創作、角色扮演強」的結構原因。
- **2B 邊緣蒸餾的意義**：把 Claude 4.6 的長文推理蒸到 2B，是「**能力密度**」的極限實驗。它注定智商降級，但證明了一件事：**推理能力可以被部分壓進極小的容器，只是壓縮比越高、損耗越大。**

> 💡 君之一席話
> 與君一席話：**在蒸餾的世界裡，「最新」往往等於「最少、最不穩、最踩線」，而「前代」常常等於「最多、最穩、最可複現」。** 最新模型的數據被防護盾污染、被窄窗掐斷、來源最敏感；前代模型（Opus／4.6）沒有那道即時偵測之盾，數據採集充分、量大、配方公開，蒸出來反而更成熟。追逐尖端是人性，但工程的成熟度往往藏在「上一代」裡——**真正的高手選型，看的不是發布日期，是數據的充分度與可複現性。**

> 🔍 進階點評──剪枝、量化、蒸餾：三把刀如何協同壓縮一個天才
> FableVibes 這條產線，示範了模型壓縮的「組合拳」，值得拆解它們的分工，因為很多人會混為一談。**蒸餾（Distillation）** 是「換靈魂」——把大模型的知識遷移到小模型，改變的是**權重學到的內容**。**剪枝（Pruning）** 是「切肉」——移除冗餘的神經元/專家/注意力頭，改變的是**模型的結構與參數量**，分結構化（整層/整頭剪，對硬體友善）與非結構化（單權重剪，壓縮率高但難加速）。**量化（Quantization）** 是「降精度」——把權重從 16-bit 壓到 4-bit，改變的是**每個參數的位元數**，不動結構也不動學到的內容。三者正交、可疊加：先蒸餾灌入能力、再剪枝縮小骨架、最後量化壓進顯存。但疊加有順序講究——**剪枝後必須微調補償（QLoRA 補回掉的分數），量化最好放最後且要用校準數據（imatrix）**，否則三刀齊下，切出來的不是輕量天才，是輕量智障。FableVibes 的「heretic 剪枝 → QLoRA 補償 → GGUF 量化」正是教科書式的正確順序。三把刀各有成熟、叫得出名字的工具：剪枝有 **SparseGPT / Wanda**（一次性結構化剪枝，不需重訓即可剃權重）、補償有 **QLoRA / LoRA**（Dettmers 等人）、量化有 **GPTQ / AWQ**（GPU 推理）與 **llama.cpp 的 k-quants + imatrix**（CPU/邊緣，靠校準數據算重要性矩陣）。**能叫得出工具名，你才分得清一張 model card 是真的做了這套組合拳，還是只把別人的 GGUF 換了個封面、貼上「Claude distill」——後者連 imatrix 來源都拿不出。**

## 情境 4　真正的硬通貨：兩個公開數據集

**背景**：最後一點往往被忽略——若你懂技術、想用自己的 Llama 或 Mistral 蒸出專屬「Claude」，**重點不是下載別人的模型，是拿原始數據集**。

| 數據集 | 內容 | 蒸餾目標 |
|---|---|---|
| **armand0e/claude-fable-5-claude-code** | Fable 5 原始 Agent 執行軌跡（去隱私化，與 Glint-Research 合作）| 程式/工具調用能力 |
| **ox-ox/mythos-character-distillation** | Claude 的「說話風格與心理特質」文風數據 | 人格/文風 |

**`ox-ox/mythos-character-distillation` 的內容耐人尋味**——傳聞中它專門蒸餾這些特質：

- **自發的元認知（spontaneous meta-awareness）**、自動發現結構陷阱；
- **拒絕使用 AI 罐頭回覆**（例如不說 "I'm just an AI" / "As an AI..."）；
- **自我修正、不找藉口**。

這說明蒸餾者想偷的東西，已經從「知識」「程式能力」細分到了**「人格與對話氣質」的層次**。技術上，這類「文風蒸餾」不靠知識正確性，而靠**偏好對齊**——常用 **DPO（Direct Preference Optimization）** 或拒絕採樣，讓模型學會「在兩種說法裡，挑更像 Claude 的那種」。（這類偏好對齊真實、公開、可查的原料，是 **UltraFeedback、HH-RLHF、Nectar** 這類偏好數據集——同樣有戶口、有授權，與來路不明的 `mythos-*` 不是一個量級的可信度。）

**為什麼數據集才是核心**：有了原料，你能蒸任意身體的學生——這正是話題總會反覆導向「我可以教你怎麼用這些數據集微調 Llama / Qwen」的原因。

> ⚠️ 真偽提醒
> 「用未授權 API 響應數據訓練商業開源模型，**嚴重違反 Anthropic 服務條款**」——這一點原始素材自己也寫明。它還提到「美國科技界正推動將特定大規模蒸餾攻擊定為刑事犯罪」。無論這些數據集是否真實存在，**取得與使用來路不明的「商業模型軌跡數據集」，本身就踩在 ToS 與法律紅線上**。詳見第 6 章。

> 💡 君之一席話
> 與君一席話：**模型可以開源，但「來源的合法性」不會因為開源而消失——它只是被藏進了權重裡，等著有一天被驗出來。** 一個 abliterated、量化、改了名的蒸餾模型，表面上乾淨得像原創，但它的數據出身、它權重裡可能殘留的水印、它訓練軌跡的 commit 歷史，都是潑不掉的指紋。下載一份「claude-traces」數據集，就像收下一塊來路不明的金條——它或許真能用，但每次動用，你都在賭沒人能證明它的出處。**開源洗不白來源，它只是讓來源更難追、卻也更值得追。**

> 🔍 進階點評──監管的戰場正從「模型」轉移到「數據」
> 這一節藏著整個蒸餾攻防最深的一層轉向，值得每個從業者警覺。封鎖一個蒸餾好的開源模型，幾乎不可能——它已是開放權重，潑出去的水收不回，改個名、做次 abliteration、重新量化，就是一個「新」模型。但**追查一個「未授權商業軌跡數據集」的來源與傳播鏈，相對可行得多**：它有上傳者帳號、有 commit 時間軸、有檔案 hash、甚至有教師輸出殘留的水印統計特徵。所以理性的監管與防守，注意力必然從「抓模型」轉向「溯數據」——這也解釋了防守方為何要在輸出端打水印（第 1 章）：水印的價值不在於即時擋住採集，而在於**事後能在某個衍生模型或數據集裡，驗出「這是從我這裡流出去的」**。對從業者的啟示極其實際：**你下載的每一個 `claude-*-traces` 數據集，都可能是一份帶法律放射性的原料；碰它之前，先查它的出身、授權與上傳者信譽——這道盡職調查，比任何技術配方都重要。**

## 情境 5　驗明正身：怎麼一眼識破「套殼改名」

**背景**：回到開篇那張記事本——你抄下一排「Fable-5」「Claude-Mythos」，現在要回答最實際的問題：**哪些是真蒸餾、哪些只是把別人的權重換個封面再喊「Claude distill」？** 好消息是，模型能造假的地方比你想的少得多。一個開放權重模型的「出身」會誠實地刻在幾個檔案裡——README 可以吹，但 `config.json`、tokenizer 與 safetensors／GGUF 的指紋騙不了人。這一節給你一套**多數步驟不必下載完整權重、幾分鐘就能跑完的鑑識流程**。

**第一刀：`config.json` 指紋——模型的出生證明。** 每個 HF 模型根目錄都有一份 `config.json`，裡面的 `model_type`、`architectures`、`vocab_size`、`num_hidden_layers`、`num_attention_heads`、`num_key_value_heads`（GQA 線索）、`intermediate_size`、`rope_theta`、`rope_scaling`、以及 MoE 的 `num_experts` / `num_experts_per_tok`，共同構成一組幾乎無法偽造的骨架。把它跟下面這張**真實開源基底家族的典型指紋**對一遍，基底是誰，藏不住：

| 基底家族 | `model_type`／`architectures` | `vocab_size` | 結構線索 | 典型 `rope_theta` | 授權性質 |
|---|---|---|---|---|---|
| **Qwen2.5 / Qwen2** | `qwen2` / `Qwen2ForCausalLM` | 151,936 | GQA（KV head < attn head）| 1,000,000 | 多數 Apache-2.0（部分大尺寸走 Qwen 研究授權）|
| **Qwen3 / Qwen3-MoE** | `qwen3` / `qwen3_moe` | 151,936 | MoE 有 `num_experts`／`num_experts_per_tok` | 1,000,000 起 | Apache-2.0 |
| **Llama 3.x** | `llama` / `LlamaForCausalLM` | 128,256 | GQA、`rope_scaling.type=llama3` | 500,000 | Llama 社群授權（**非純開源**，禁某些再分發）|
| **Mistral 7B / Mixtral** | `mistral` / `mixtral` | 32,000 或 32,768 | Mixtral 為 8×7B MoE | 1,000,000（Mixtral）| Apache-2.0 |
| **Gemma 2** | `gemma2` | 256,000 | 滑動視窗注意力交替、logit softcapping | 10,000 | Gemma 使用條款（**非純開源**）|
| **GPT-OSS** | `gpt_oss` | ~20 萬（o200k 系）| MoE、原生 MXFP4 權重 | — | Apache-2.0 |
| **Phi-3 / 3.5** | `phi3` | 32,064 | 小而密集 | 10,000 | MIT |

> ⚠️ 真偽提醒
> 上表是「**典型**指紋」，**具體數值會隨版本微調**（Mistral v0.1→v0.3 的 vocab 從 32,000 變 32,768、Llama 3.0→3.1 的 `rope_scaling` 才加上 YaRN 式外推）。用它作「縮小嫌疑範圍」的篩子，而非「精確到個位數」的判決書——對不上才該起疑，對上了再往下驗。

**第二刀：tokenizer 指紋——換不掉的口音。** 模型可以改名、可以微調，但**換 tokenizer 等於重訓**，蒸餾／改名者幾乎不會動它，於是它成了最穩的胎記。看 `tokenizer_config.json` 與 `tokenizer.json`／`tokenizer.model` 的特殊 token：Qwen 系是 **ChatML**（`<|im_start|>` / `<|im_end|>`，byte-level BPE，vocab 約 15 萬）；Llama 3 是 **tiktoken 風格**（`<|begin_of_text|>` / `<|eot_id|>` / `<|start_header_id|>`，vocab 128,256）；Mistral 是 **SentencePiece**（`[INST]` / `[/INST]`，`<s>`，vocab 3 萬出頭）；Gemma 是 **SentencePiece 256k**（`<start_of_turn>` / `<end_of_turn>`）。**一個自稱「全新 Claude 蒸餾架構」的模型，若 tokenizer 是一字不差的 Qwen ChatML，它的底就是 Qwen，沒有別的可能。**

**第三刀：權重與量化的物理憑證。** safetensors 的 `model.safetensors.index.json` 會列出 `total_size`，除以每參數位元數即可反推真實參數量——**「120B」喊得響，index 卻只有 14B 的張量，當場拆穿**；張量命名（`model.layers.N.self_attn.q_proj`、MoE 的 `mlp.experts.N`）也透露架構家族。GGUF 量化檔則用 `gguf-dump`（或 `llama.cpp` 載入時印出的 KV）攤開元數據：`general.architecture`（會老實寫 `qwen2` / `llama`，封面叫「Claude-Mythos」也藏不住底層）、`general.name`、`tokenizer.ggml.model`、`general.quantization_version`。**一個量化檔若拿不出基底 commit SHA、拿不出 imatrix 校準來源，它的品質與血緣就都是無據之詞。**

**第四刀：授權血緣與 model card 的謊言。** HF model card 的 YAML front-matter 有 `base_model:`、`license:`、`datasets:` 三個關鍵欄位——**缺哪個，哪個就是它不想讓你問的問題**。最強的紅旗是**授權自相矛盾**：拿 Llama 3 或 Gemma 改的模型卻標 `license: apache-2.0`，等於白紙黑字承認它要嘛沒讀基底授權、要嘛蓄意洗白（Llama／Gemma 都**不是**純開源，再分發有條款約束）。其餘常見破綻整理如下：

| README 怎麼說 | 該去驗的可驗證信號 | 對不上代表什麼 |
|---|---|---|
| 「全新架構 / 自研蒸餾」| `config.json` 的 `model_type` + tokenizer 特殊 token | 其實是某開源基底改名 |
| 「120B 巨型 MoE」| `index.json` 的 `total_size`、`num_experts` | 參數量灌水或根本不是 MoE |
| 「1M 上下文」| `rope_theta` / `rope_scaling` / `max_position_embeddings` | 只改 README，沒做 YaRN |
| 「Claude/Opus 蒸餾」| `datasets:` 欄位、訓練數據揭露、基底 SHA | 血緣無法佐證（多半也無法佐證**反**之）|
| 「無審查」| 任一 RULER／lm-eval／拒絕率測試數字 | 只有形容詞、沒有量測 |
| 「高品質 GGUF」| imatrix 來源、基底 commit、量化版本 | 來路不明、可能被污染 |

**最關鍵、也最反直覺的一條限制必須講白**：上面四刀能**證偽**「這是全新架構」、能**鎖定基底是誰**，卻幾乎**無法證實**「它真的是用 Claude 的輸出蒸的」。基底血緣寫在權重結構裡，**訓練數據的血緣卻不寫在權重裡**——你能斷定「底是 Qwen3」，卻很難從一堆浮點數反推「這些權重的監督訊號來自 Claude」。要回答後者，得靠**行為探針**（問它身份、用金絲雀字串、比對特定罕見 token 的機率分佈）與**輸出端水印**（第 1 章）。這正好呼應情境 4 的判斷：**監管與鑑識的重心，終究會從「模型」移到「數據」——因為模型的底騙不了人，但模型的「師承」，恰恰是最容易被洗白、也最值得追的那一層。**

> 💡 君之一席話
> 與君一席話：**權重不會說謊，會說謊的是 model card——當宣傳與 `config.json` 起衝突，永遠信 config。** 一個老練的選型者打開一個陌生模型，第一個點開的不是 README 的跑分截圖，是那份不到 1KB 的 `config.json`：`model_type` 告訴他底是誰、`vocab_size` 對不對得上、`rope_theta` 有沒有為了那句「百萬上下文」而動過。**這套習慣的本質，是把「相信別人怎麼說」換成「自己去讀它是怎麼長的」——而這恰恰是工程師與消費者最深的分野。** 消費者看封面、看下載數、看「無審查」三個字就心動；工程師看出生證明、看胎記、看授權血緣，因為他知道：**在一個誰都能上傳、誰都能改名的開放權重世界裡，唯一不會背叛你的盡職調查，是那幾個改不動的底層檔案。** 你驗證的不是某一個模型，你練的是一種「不被名字唬住」的眼力。

> 🔍 進階點評 — 開放權重的供應鏈安全：你下載的不只是模型，是一條沒人簽名的食物鏈
> 把這套鑑識往上拔一層，會看見一個被嚴重低估的領域：**開放權重的供應鏈安全（model supply-chain security）**。今天你 `from_pretrained("某人/某模型")` 一行，背後其實吞下了一整條無人背書的食物鏈——基底是誰改的、用什麼數據、誰量化的、imatrix 哪來的、有沒有人在權重裡埋了後門（是的，**權重級後門**是真實研究議題，惡意微調可以讓模型在特定觸發詞下行為異常）。成熟的做法，是把「裝模型」當成「裝一個來路不明的二進位依賴」一樣對待：**讀 `config.json` 與 tokenizer 確認基底（用 `transformers.AutoConfig`、`huggingface_hub` 拉 metadata 而不必下整包權重）、用 `gguf-dump` / `safetensors` header 驗物理規格、核對 `base_model` 與 LICENSE 的血緣是否自洽、優先選有 commit 歷史與可複現訓練配方的上傳者**（Tülu、OLMo、DeepSeek-R1-Distill 這類「全鏈公開」的模型，價值正在於此）。HF 近年推的 model-lineage（模型族譜）、safetensors 取代可任意執行程式碼的 pickle、以及社群對 imatrix 與量化來源的標註要求，都是這條供應鏈在補課。**對從業者最實際的一句話：在這個影子模型滿地跑的生態裡，你最該培養的核心能力，不是「會微調」，而是「會驗貨」——因為造一個影子只要幾小時，而一個驗不出贓物、驗不出後門的工程師，遲早會把一條帶毒的依賴，親手接進公司的生產線。**

## 本章小結

把全景收攏成一張認知地圖：

- **三條獲取路徑**：① 下載蒸好的模型（最快、品質與來源最不可控）；② 用前代成熟蒸餾品（穩定、可複現）；③ 拿原始數據集自蒸（最自由、最接近紅線）。
- **最想被偷的能力**：從「聊天」→「**工具調用 / Agent 程式力**」→「**人格與文風**」三級進化。
- **方法論光譜**：SFT（最便宜、形似神不似）→ QLoRA 補償（救剪枝）→ 全參數微調（最徹底）→ on-policy/DPO（最接近神似、最貴）。
- **工具鏈**：採集→`datasets`/Presidio 清洗→Axolotl/LLaMA-Factory/Unsloth 微調→`peft` LoRA→llama.cpp GGUF 量化→Ollama 部署；abliteration 靠 TransformerLens/heretic。
- **驗明正身四刀**：`config.json`（`model_type`／`vocab_size`／`rope_theta`／MoE 欄位）→ tokenizer 指紋（ChatML／llama3／SentencePiece，換不掉的口音）→ safetensors `index.json` 參數量與 GGUF `general.architecture` → LICENSE 血緣與 model card `base_model`。**能對上基底，戳破套殼改名；但「是否真用 Claude 輸出蒸的」單靠權重驗不出**，得靠行為探針與輸出端水印（呼應第 1 章、情境 4）。真蒸餾有對照組：`DeepSeek-R1-Distill` 全鏈公開、config 對得上基底。
- **最大陷阱**：「無審查 + 小顯存 + 百萬上下文」的神話宣傳——下一章用硬體真相拆解。

清單盤完，最實際的問題浮現：**這些影子該用什麼機器跑、哪一隻配得上那台 DGX Spark？** 翻到第 3 章，上解剖台。
