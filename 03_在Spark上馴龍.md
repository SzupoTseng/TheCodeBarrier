# 第三篇　馴龍場

# 第3章　在 DGX Spark 上馴龍：硬體 Roofline 與選型

> 矽谷，麗臺代理商的出貨單上，那台 **NVIDIA DGX Spark** 標價不菲。它小得像台 Mac mini，卻號稱「個人 AI 超級電腦」。風扇轉起來的那一刻，真正的問題不是「它能跑多大的模型」，而是更尖銳的一句——**「我為這 128GB 付了高昂溢價，到底該拿它跑什麼，才不算暴殄天物？」**

> 本章技術含金量最高、也最站得住腳。GB10、統一記憶體（UMA）、GGUF、Roofline 都是真實工程。選型結論是一種常見推論，但背後的**頻寬推理是可驗證的硬體常識**——正是戳破上一章「神話宣傳」的尺。

## 情境 1　認識這頭怪獸：用 Roofline 看穿它的脾氣

**背景**：硬體是 **NVIDIA DGX Spark（GB10 晶片，非資料中心級 GB200/GB100）**。先把規格釘死，因為**選型的每一步都是從這幾個數字推出來的**：

| 規格 | 數值 | 確信度 |
|---|---|---|
| 晶片 | GB10 Grace-Blackwell 超級晶片（20 核 Arm CPU + Blackwell iGPU）| 🟢 官方 |
| 記憶體 | **128GB LPDDR5X 統一記憶體（UMA）**，CPU 與 iGPU 實體共享同一池、不經 PCIe | 🟢 官方 |
| 記憶體頻寬 | **~273 GB/s**（LPDDR5X class）| ⚠️ 常見標稱值；由匯流排寬度推算的理論峰值另有 ~301 GB/s 之說，**勿當精確定論** |
| 算力 | ~1 PFLOP **FP4（NVFP4）** 稀疏峰值（5 代 Tensor Core）| ⚠️ 官方宣傳的「稀疏 FP4」數字，密集/實測會低不少 |
| 量化原生 | Tensor Core 原生支援 **FP4 / NVFP4 / FP8 / BF16** | 🟢 架構特性 |

**注意那個被很多二手文章寫成「600 GB/s」甚至更高的頻寬——那是錯的。** DGX Spark 用的是 LPDDR5X，不是資料中心卡的 HBM3e；它的頻寬量級在「兩、三百 GB/s」，而非 HBM 的「TB/s」。**這個數字錯一倍，下面所有 tok/s 估算就全錯一倍**，所以先校準到 ~273 GB/s class。至於 NVFP4——它是 Blackwell 引入的 4-bit 浮點格式（E2M1 + 兩級 micro-block 縮放），靠分塊 scale 把 4-bit 的品質硬拉到接近 FP8，這正是後面選型敢用 Q4 的底氣（細節留到第 4 章蒸餾數學）。

**問題**：不懂它的脾氣，就會被「4GB 顯存跑 Claude」這類說法牽著走，白白浪費它最值錢的本事。而看穿它脾氣的工具，是一個工程師必懂的模型——**Roofline（屋頂線）**。

**LLM 推理的兩個階段，撞兩道不同的牆**：

- **預填充（Prefill，處理你的輸入）**：把整段 prompt 的幾百上千個 token 一次性灌進去做大矩陣乘——同一份權重被這一整批 token 重複利用，**算力受限（compute-bound）**，吃的是 FLOPS。
- **解碼（Decode，逐 token 生成）**：自迴歸，一次只生一個 token；每生一個，就得**把參與運算的權重從記憶體完整讀一遍**，而這份權重只服務了「這一個」token——讀進來、算一下、丟掉。**記憶體頻寬受限（memory-bandwidth-bound）**，吃的是 GB/s。

**為什麼 prefill 撞算力牆、decode 撞頻寬牆？** 一句話：**算術強度（Arithmetic Intensity，每讀 1 byte 權重能換來幾次浮點運算）天差地別。** Prefill 一次處理 N 個 token，權重讀一次被 N 個 token 共用，算術強度隨 N 線性放大，輕鬆衝進「算力受限」區；Decode 的 batch=1，權重讀一次只算一個 token 的乘加——算術強度趨近理論最低（每個參數約 2 FLOP），於是 GPU 的兆級算力大半閒置，瓶頸死死卡在「把權重從記憶體搬出來」這一步。**這就是為什麼你在 DGX Spark 上跑單人對話，GPU 使用率常常只有兩三成卻已經是極速——它不是不夠力，是在等記憶體。**

**這帶出一條能直接估算速度的鐵律**：

```
解碼速度 (tok/s) ≈  記憶體頻寬 (GB/s)  ÷  每 token 需讀取的權重 (GB)
```

拿 DGX Spark 的 ~273 GB/s 套（這是頻寬「天花板」；扣掉注意力 KV 讀取與框架開銷，實測通常再打 6–8 折）：

- **70B 密集模型 @ Q4_K_M**（每 token 把**全部** ~40GB 權重讀一遍）：273 ÷ 40 ≈ **理論 ~6.8 tok/s，實測 ~5 tok/s**——一個字一個字往外擠，不堪用。
- **120B-A4 MoE @ Q4**（每 token 只讀「激活的 ~4 個專家 ~2–3GB ＋ 每層常駐的注意力/共享權重 ~3–4GB」≈ 6GB）：273 ÷ 6 ≈ **理論 ~45 tok/s，實測 35–50 tok/s**——順。

看出名堂沒有？**同一台機器，跑 70B 密集卡成 ~5 tok/s，跑「總參更大」的 120B MoE 反而快近 9 倍**——差別不在總參數，在「每 token 真正要搬多少 byte」。**這一條算式，就能讓你在按下載鍵之前先算出它跑多快**，不必等 70GB 下載完才後悔。

> 🔍 進階點評 — Roofline 的「屋脊點」：一個數字看穿機器的性格
> Roofline 模型的全部精華，可以濃縮成一個座標——**屋脊點（Ridge Point）= 峰值算力 ÷ 峰值頻寬**，單位「FLOP/byte」。它是機器的「性格分水嶺」：負載的算術強度若**高於**它，你撞的是**算力牆**（屋頂的水平段）；**低於**它，你撞的是**頻寬牆**（屋頂的斜坡段）。把 DGX Spark 代進去：~1 PFLOP(FP4) ÷ ~273 GB/s ≈ **~3600 FLOP/byte**。這意味著：**任何算術強度低於 3600 的負載，都會被頻寬卡死。** 而 decode 的算術強度是多少？batch=1 時約 **2–4 FLOP/byte**——比屋脊點低了**整整三個數量級**。換句話說，**LLM 逐 token 生成這件事，在這台機器（以及所有 UMA / Apple Silicon）上，是被頻寬牆按在地上摩擦的極端記憶體受限負載**，那 1 PFLOP 算力對它幾乎是擺設。理解這一點，三件事的根你就全懂了：①**為什麼提速要靠「減少每 token 讀取量」**（量化、MoE、KV Cache 量化）而不是「買更強算力」；②**為什麼提高 batch（同時服務多個請求）能近乎線性提升總吞吐**——多個 token 共用同一次權重讀取，把算術強度往屋脊點推（這也是 vLLM 連續批次的價值所在）；③**為什麼投機解碼（情境 3）有效**——它讓大模型「一次讀權重、平行驗證多個草稿 token」，本質也是在拉高算術強度。**屋脊點是一把尺，量出你的瓶頸到底在算力還是頻寬；而對個人 LLM 推理，答案幾乎永遠是頻寬。**

> 🧠 必須焊進腦子的硬體鐵律
> **容量決定你「能不能跑」，頻寬決定你「跑多快」，而 MoE 的「激活參數」才是頻寬賬單上真正的數字。** DGX Spark 是「容量巨人、頻寬凡人」：128GB 讓它能吞下別人吞不下的大模型，但 ~273 GB/s 的 LPDDR5X 比獨顯動輒 1TB/s（GDDR7）甚至 ~8TB/s（HBM3e）的頻寬慢了一個數量級。所以它的最優策略永遠是「**用大容量去跑別人跑不動的大 MoE**」，而不是「用它去比小模型誰更快」。

**這條鐵律如何戳破上一章的神話**：在 DGX Spark 上跑 9B 模型，你**當然**能飆到 180+ tok/s——但那是浪費。9B 一台兩萬多的 M 晶片 Mac 就跑得很好；你買 128GB，是為了容量，不是為了讓小模型更快。

> 💡 君之一席話
> 與君一席話：**選硬體與選模型，本質是同一道題——在你的瓶頸資源上求最大產出。** DGX Spark 的瓶頸不是容量（它過剩），是頻寬（它稀缺）。把稀缺資源餵給「激活參數小、總參數大」的 MoE，等於用過剩的容量去補頻寬的不足——這就是 Roofline 思維的精髓：**不要被規格表上最大的數字迷惑，要找出系統裡最小的那道牆，然後讓你的負載剛好貼著那道牆跑。** 看懂這個，你選的就不只是模型，是整個系統的最優工作點。

## 情境 2　六款影子的橫向對比：誰配得上這頭怪獸

**背景**：針對 DGX Spark GB10，把上一章的模型做橫向對比如下（Q4_K_M 量化下評估）。

| 模型 | 記憶體佔用 | DGX Spark 預期速度 | 強項 | 弱項 |
|---|---|---|---|---|
| **GPT-OSS 120B Fable-5**（MoE）| ~72GB | 35–50 tok/s | 128 專家 MoE，複雜多步程式碼與數學 | 體積大，上下文 >32K 明顯變慢 |
| **Qwable-v1**（35B）| ~24GB | ~102 tok/s | 完美繼承 Fable 5 Agent 軌跡，最適合 Coding 代理 | 知識廣度遜於 120B |
| **clzoro/Qwen3.5-27B** | ~19GB | 110+ tok/s | 餵 Opus 對話，文學/角色扮演極強 | 工具調用不如 Qwable-v1 |
| **Qwen3.6-14B-FableVibes** | ~11GB | 140+ tok/s | 剪枝 MoE，保留「Claude 思考感」 | 複雜邏輯偶有反芻（幻覺）|
| **Qwythos-9B**（含 abliterated）| ~6.5GB | 180+ tok/s | 極速、徹底無審查，爛顯卡也能跑 | 參數太小，深度推理完整度差 |
| **Dzluck/Qwen3.5-2B** | ~2.1GB | 250+ tok/s | 極致輕量，為邊緣設備設計 | 大材小用，智商降級明顯 |

> ⚠️ 真偽提醒
> 速度與記憶體數字**均為未經實測的估算/坊間流傳值**。相對關係符合 Roofline 直覺（參數越小越快、MoE 比同尺寸密集快、量化越激進佔用越小），可當「合理估算」而非「實測基準」。社群有一條真實流傳的參考點：「Solved the DGX Spark, 102 stable tok/s on Qwen3.5-35B-A3B」（見附錄 A）——與表中 Qwable-v1（35B-A3B 同級）的 102 tok/s 吻合，因為 3B 激活 @ Q4 ≈ 1.7GB，273 ÷ 1.7 ≈ 160 理論上限，實測 102 落在合理折扣內。**但同一把尺照出表中的破綻**：Qwythos-9B 標 180+ tok/s——9B 密集 @ Q4 ≈ 5.4GB，273 ÷ 5.4 ≈ **50 tok/s 才是物理上限**，180 已超出 3 倍多，**單請求 decode 不可能達到，除非是高併發批次的總吞吐或表本身灌水**。這恰恰示範了本章的核心紀律：**任何 tok/s 數字，先拿「頻寬 ÷ 每 token 讀取量」這把尺量一遍，過不了物理上限的，一律存疑。**

**別只會看表——學會自己算「裝不裝得下」**：表上的「記憶體佔用」只是權重，真正塞進 128GB 的是三樣東西疊加，少算一樣就會在長文時 OOM。先把公式背下來：

```
總佔用 = 權重 + KV Cache + 框架/激活開銷(~2–4GB)

權重 (GB) ≈ 參數量(B) × 每參數位元 ÷ 8
   BF16/FP16 : 16 bit → ×2.00      FP8 / Q8_0 : 8 bit → ×1.00 (Q8_0 實 ~8.5 bit)
   Q4_K_M    : ~4.8 bit → ×0.60     NVFP4/Q4   : 4 bit → ×0.50

KV Cache (GB) ≈ 2 × 層數 × KV頭數 × 頭維度 × 上下文長度 × 每元素位元組 ÷ 1e9
   ★ ×2 = K 和 V 各一份
   ★ GQA 讓「KV頭數」遠小於注意力頭數（如 8 vs 64），是省 KV 的命脈
   ★ 經驗值：GQA-8 + FP16 KV，每 1K token 約吃 0.3–0.5GB（依層數）
```

把四款代表性配置代進 128GB 實算（⚠️ 數字為示意，依實際層數/GQA 比例浮動）：

| 模型 @ 量化 | 權重 | KV @ 32K | KV @ 256K | 128GB 結論 |
|---|---|---|---|---|
| **70B dense Q4_K_M**（GQA）| ~40GB | ~10GB | ~80GB | 32K 從容(~50GB)；**256K=120GB 幾乎撐爆，非 KV 量化不可** |
| **120B-A4 MoE Q4** | ~72GB | ~5GB | ~40GB | 32K 輕鬆；256K≈112GB 吃緊，KV FP8 可救 |
| **35B-A3B Q8_0**（GQA）| ~37GB | ~6GB | ~48GB | 單模從容；要與大模型雙開需控長度 |
| **8B dense Q4** | ~5GB | ~2GB | ~16GB | 隨便跑，但用 128GB 跑它是大材小用 |

盯著那兩欄 KV 看——**權重是一次性的固定開銷，KV Cache 卻隨上下文長度「線性膨脹」。** 在 32K 你幾乎感覺不到它，但拉到 256K，KV 能逼近、甚至吃掉比權重更多的記憶體。**這就是為什麼「裝得下 70B」和「裝得下 70B 還能跑 256K 長文」是兩個完全不同的問題**——後者真正的記憶體殺手從來不是模型權重，是 KV Cache。（這條伏筆，第 10、11 章會用整章來還它的債。）

**一個常見的選型結論**：**首選 GPT-OSS 120B Fable-5 或 Qwable-v1（35B）**——「128GB 不跑大模型太浪費」。用 Roofline 與上面的 fit 算式驗算，這結論成立：

- **GPT-OSS 120B-A4 是「智商擔當」**：120B 總參給它知識深度，但每 token 只激活 ~4 專家（約 20–30B 實際參與），讓它規避了頻寬死穴，35–50 tok/s 可接受。**這是個人設備上突破「70B 級智商上限」的唯一務實途徑。**
- **Qwable-v1（35B）是「速度擔當」**：24GB 佔用對 128GB 毫無壓力，102 tok/s 近乎「瞬噴」，且原生吐 `str_replace_editor` XML，對接 OpenHands/Aider 零轉譯——當 Coding Agent 效率比 120B 高數倍。

> 🔍 進階點評──MoE 為何是頻寬受限機器的救星，以及它的隱藏代價
> 這是全章最值錢的洞見：**在任何頻寬受限的硬體上（DGX Spark、Apple Silicon、所有 UMA 架構），「激活參數量」比「總參數量」重要得多。** 一個 120B-A4 跑起來的速度接近 20–30B 密集模型，智商卻接近 120B——這就是 2026 年面向「個人超算」的開放權重模型幾乎全押 MoE 的原因。但 MoE 不是免費的午餐，它有三個隱藏代價，選型時要一併權衡。**其一，記憶體稅**：MoE 要把**全部 128 個專家**都常駐記憶體（你不知道下個 token 會路由到哪個），所以它「省的是頻寬、費的是容量」——這恰好和 DGX Spark「容量過剩、頻寬稀缺」的特性互補，是天作之合，但在小顯存設備上反而是災難。**其二，邏輯反芻**：專家間路由切換可能讓長鏈推理出現不連貫（表中 FableVibes 的「複雜邏輯偶有反芻」正是此因），這也是為什麼 clzoro 的**密集 27B** 在「思考連貫性」上反而更受文學/角色扮演場景青睞。**其三，批次效率**：MoE 在高併發批次下的專家負載不均衡，是生產部署的老問題。**所以選型的完整心法是：問「激活多少」（定速度）、問「總共多少」（定智商）、問「密集還是 MoE」（定連貫性與記憶體形態）——三問之後，才談得上選對。**

## 情境 3　最聰明的玩法：雙模並行，與一個進階彩蛋

**背景**：有 128GB，更聰明的玩法不是「挑一個」，而是**雙模並行**——同時載入一大一小，按任務切換。

**一個值得參考的雙模方案**：

- **同時載入**：GPT-OSS 120B（~72GB）+ Qwable-v1 35B（~24GB）= ~96GB，剩 ~32GB 給兩邊 **KV Cache**。
- **後端**：**Ollama** 內建動態多模型管理；開兩個終端或 API 並行：`ollama run gpt-oss:120b` / `ollama run qwable:35b`。
- **前端**：**Open WebUI / Cursor**，下拉一鍵切換，甚至「**雙模對決（Arena）**」——兩模型同答一題，你挑最好的。
- **熱切換（hot-swap）**：跑大模型時自動把閒置模型從 LPDDR5X 釋放，騰空間給超長上下文；靠 ~273 GB/s 的 UMA 頻寬與「零拷貝、不過 PCIe」的特性，換入換出比獨顯經 PCIe 搬運順手得多（但別誤會它有 HBM 那種 TB/s 速度）。

**黃金工作流**：

- **Qwable-v1（35B）管「日常高頻」**：邊寫邊 Debug、改碼、寫短信、翻譯、查指令——靠 102 tok/s 的噴射速度，思緒不停等 AI。
- **GPT-OSS 120B 管「重度思考」**：35B 解不掉的 Bug、規劃整套架構、丟三篇長論文做跨域分析——切過去，讓 128 專家攻堅。

> 💡 君之一席話
> 與君一席話：**真正會用工具的人，不追求一把瑞士刀解決所有事，而是讓對的刀在對的時刻無感地出現在手邊——這套「分層認知」其實是人腦的鏡像。** 心理學裡的 System 1（快、直覺）與 System 2（慢、深思），對應的正是「快模型管九成日常、大模型攻一成硬骨頭」。雙模並行的精髓，不在「有兩個模型」，而在「切換本身變得無感」——當調用深度思考的成本趨近於零，你才會真的在該深思時深思，而不是因為「切換很麻煩」就湊合用快模型硬扛。**工具的最高境界，是讓正確的選擇變成最省力的選擇。**

> 🔍 進階點評──雙模的進階彩蛋：投機解碼，讓兩個模型合體加速
> 前面只講了「按任務切換」，但既然你已經把一大一小兩個模型都載進了 128GB，有一個它沒提、卻能讓兩者**協同加速**的進階技術值得知道——**投機解碼（Speculative Decoding）**。原理是：讓**小模型（Qwable-v1 或更小的 draft model）先快速生成一串草稿 token**，再讓**大模型（GPT-OSS 120B）一次性平行驗證這串草稿**，接受對的、否決錯的。因為「驗證 K 個 token」比「逐個生成 K 個 token」在記憶體頻寬上划算得多（一次讀權重驗證多個），實測常能讓大模型的有效輸出速度提升 1.5–3 倍，**且輸出分佈與大模型完全一致（無損加速）**。這恰恰把 DGX Spark「同時有大小兩模型」的條件用到了極致——小模型不只是「快速回答日常」的備胎，更是大模型的「預言加速器」。工具上，**llama.cpp（`--draft` / draft model）、vLLM、SGLang** 都已內建支援。**另一個隱藏成本提醒**：別忘了 KV Cache 才是長上下文真正的記憶體殺手——當上下文堆到 20 萬 token，那剩下的 ~32GB 會被**隨長度線性飆升**的 KV Cache 無情吃光（回看情境 2 的 fit 算式：70B 級模型在 256K 的 KV 可達 ~80GB），這時就要靠 **FP8/INT8 的 KV Cache 量化**（vLLM、llama.cpp 已支援）把它腰斬，而不是天真地以為「卸載一個模型」就夠了。**長上下文的瓶頸從來不是權重，是 KV Cache。**

## 情境 4　部署實務：把模型釘進那 128GB

**背景**：選好了模型，最後一哩是「怎麼在 DGX Spark 上把它跑出極致速度」。以下是幾個具體建議，補上工具細節如下。

**部署工具鏈**：

- **格式與引擎**：下載 GGUF（首選 **Q4_K_M**，核心層可上 Q8_0 做混合量化）；引擎用 **llama.cpp / Ollama**（最易上手）、追求吞吐用 **vLLM / SGLang**（連續批次把算術強度往屋脊點推、KV PagedAttention）；DGX OS 內建 **NVIDIA AI 軟體堆疊 / NIM / TensorRT-LLM** 可直接拉起。
- **跨平台對照（別搬錯生態）**：DGX Spark 走 **CUDA** 生態（llama.cpp-CUDA / Ollama / vLLM / SGLang / TensorRT-LLM）；同屬 UMA 陣營的 **Apple Silicon** 則走 **MLX / llama.cpp-Metal**。兩邊**原理共通**——都在賭「統一記憶體 + 端側量化模型」這條路——但 kernel 與 binary 不互通，**Mac 上跑得好的 MLX 模型不能直接搬上 GB10**，反之亦然。認清這點，才不會抄錯部署腳本。
- **量化校準**：做 GGUF 量化時，用 **imatrix（importance matrix）**——拿一小批校準語料統計各權重重要度，讓低位元量化優先保住關鍵權重，**同樣 Q4 體積、明顯更高品質**。這是 llama.cpp 社群的標配進階手法。
- **核心級鎖定（榨乾 UMA 頻寬的關鍵）**：
  - `numactl --interleave=all`：把模型權重**交錯**配置到 LPDDR5X 所有記憶體通道，逼所有 channel 同時全載運作，避免 CPU/iGPU 搶頻寬時的快取未命中。
  - `mlock`：把權重**鎖在實體記憶體**，防止作業系統把它換頁（swap）到慢速硬碟——這是維持高 tok/s 的硬核細節。
  - **不要依賴通用 Docker 預設容器**：它的記憶體策略未必貼合 UMA，手動調 NUMA 與 mlock 才能發揮 GB10。

**戳破「104 萬上下文」神話的最後一刀**——別只看廠商規格，**自己驗**：

```
裝得下多長？   →  看 RoPE/YaRN 設定（最易宣傳的數字）
跑得動多長？   →  受 KV Cache 與頻寬限制的實際可用長度
讀得懂多長？   →  跑 RULER / Needle-in-a-Haystack，真假立判
```

**這三問裡，第三問「讀得懂多長」最常被跳過、也最致命**——它正是 **NIAH** 與 **RULER** 要量的東西：

- **NIAH（Needle-in-a-Haystack，海底撈針）**：在一篇幾十萬字的無關長文（haystack）裡，藏一句關鍵事實（needle，例如「2026 年的通關密語是 nol9981」），把上下文塞到指定長度後問模型那句話——測的是**最低門檻的純檢索**。連這都過不了，「百萬上下文」就是純廣告。
- **RULER**：NIAH 的工業強化版，除了單針檢索，再疊上**多針檢索、變數多跳追蹤（variable tracking）、聚合統計、長文 QA** 等十幾類任務，逐段加長上下文，記錄模型準確率**跌破門檻（常設 85%）的那個長度**——這個長度才叫「**有效上下文長度（effective context length）**」。殘酷現實是：**標稱 1M 的模型，RULER 實測有效長度常常只剩 64K–128K**，後面那一大段是「裝得下、卻讀不懂」的虛胖。所以你該信的不是規格表上的 RoPE base，是 RULER 曲線上那個準確率崩塌的拐點——而這條曲線，`lm-eval-harness` 配上開源 RULER 套件，你自己在 DGX Spark 上一晚就能跑出來。

> 💡 君之一席話
> 與君一席話：**規格表上的數字，是廠商想讓你看見的真相；跑一遍基準測試，才是機器願意對你說的真相。** 「1M 上下文」「180 tok/s」「無審查」這些宣傳詞，每一個都只揭露了三維問題裡最好看的那一維。成熟工程師的本能，是把任何單一數字拆回它背後的「能裝下／跑得動／讀得懂」「總參／激活／架構」這些獨立維度，再用現成的開源基準（RULER、lm-eval-harness、HumanEval）親手量一遍。**對數字的敬畏不是不信，是「信任，但驗證」——而驗證的工具，全是免費開源的；不驗，是你自己的選擇。**

> 🔍 進階點評──UMA 是一場記憶體架構的豪賭，而 DGX Spark 押對了什麼
> DGX Spark 的 **統一記憶體架構（UMA）** 不是 NVIDIA 的小聰明，是一場關於「未來個人 AI 算力長什麼樣」的架構豪賭，值得從更高處看懂它賭的是什麼。傳統 GPU 把「快但小」的 HBM/GDDR 焊在卡上，CPU 記憶體「大但慢」，兩者間隔著 PCIe 這條窄橋——跑大模型時，權重在兩塊記憶體間搬來搬去，PCIe 成為瓶頸。UMA 把這堵牆拆了：CPU 與 iGPU 共享同一池 128GB LPDDR5X，**沒有搬運、沒有 PCIe 瓶頸、沒有「顯存裝不下」**。代價是這池記憶體的頻寬（~273 GB/s）卡在 HBM3e（~8TB/s，快得發燙）與一般雙通道 DDR5（~100 GB/s 以下，慢）之間的尷尬地帶——**它賭的是「對個人/邊緣場景，容量比極致頻寬更稀缺」**。對個人/邊緣場景，這個賭注大概率正確：你更常被「模型裝不進 24GB 顯存」卡住，而非「頻寬不夠快」。Apple Silicon（M 系列）早已驗證這條路，DGX Spark 是把它推到 128GB + Blackwell 算力的版本。**這也預示了選型哲學的長期轉向：未來在這類機器上，勝出的不是「最快的小模型」，而是「最大的、激活參數可控的 MoE」——硬體架構的賭注，最終會反過來定義什麼模型值得被蒸餾、被部署。** 你今天為 DGX Spark 選型的邏輯，就是未來幾年個人 AI 算力的縮影。

## 本章小結

- **規格先校準**：DGX Spark = GB10 + 128GB LPDDR5X UMA + **~273 GB/s**（⚠️ 非二手文章常寫的 600；錯一倍 tok/s 全錯）+ ~1 PFLOP FP4/NVFP4。
- **用 Roofline 看 DGX Spark**：屋脊點 = 算力÷頻寬 ≈ **~3600 FLOP/byte**，而 decode 算術強度僅 **2–4**——深陷頻寬牆；解碼速度 ≈ 頻寬 ÷ 每 token 讀取權重。容量巨人、頻寬凡人。
- **選型三問**：激活多少（速度）、總共多少（智商）、密集還 MoE（連貫性/記憶體形態）。結論：首選大 MoE（GPT-OSS 120B）+ 速度型（Qwable-v1 35B）。
- **自己算 fit**：總佔用 = 權重 + KV Cache + 框架開銷；權重 = 參數 × bpw ÷ 8，KV 隨上下文線性膨脹。32K 看權重、256K 看 KV——**長文的記憶體殺手是 KV Cache，不是權重**（埋給第 10、11 章）。
- **雙模並行 = 分層認知**，進階可上**投機解碼**讓大小模型合體無損加速；KV Cache 量化（FP8）才是長上下文的真瓶頸解。
- **部署榨乾 UMA**：GGUF + imatrix 校準、`numactl --interleave=all` + `mlock`、避開通用 Docker。
- **長上下文拆三問**，用 RULER/Needle 親自驗，別信規格表。

硬體與選型講透，下一個問題回到本質：**這些影子到底是怎麼蒸出來的？** 離開硬體，走進蒸餾的科學——第 4 章。
