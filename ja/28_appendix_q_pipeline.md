# 付録Q　蒸留パイプライン実装ガイド：コンプライアンス工学四十ステップ（精選）

> 本付録は第5章「五大モジュールと四十ステップ」の**実装層**である。各工程ステップを「目標／技術スタックとツール／中核実装／検証（Assertions）／解説」に凝縮した。**教師モデルは一律、君が正当に利用権限をもつ対象だと仮定する**（オープンウェイトモデル、あるいは君自身が保有する社内大規模モデル）——教師を「君が搾り取る権限をもつ、あの一つ」に置き換えれば、この一連のフローは攻撃から工学へと姿を戻す（第6章の分水嶺を参照）。

> ⚠️ コンプライアンス境界（除外したステップ）
> 元素材の「モジュール一（おとりとトラジェクトリ採取）」には**ベンダー防護の回避**にあたる内容が含まれていた——ステップ1（レート制限回避のためのAPI/IPプロキシ・ローテーション）、ステップ2（敵対的アンチフィンガープリント・おとりプロンプト）、ステップ11–12（教師のデグレード保護に対する検出/対抗）、さらに各所に散見される**脱検閲（abliteration）**の成分（ステップ15、ステップ18の脱検閲加重、ステップ27の「無検閲バランス」）。これら**アクセス制御の迂回／安全アライメントの破壊**にあたる操作は実質的な危害であり、本付録では**一切収録しない**。レッドラインの判定基準は付録Eおよび第6章を参照。残したのは、「教師を合法な対象に置き換えた」あとに**汎用かつ正当**となる蒸留工学である。

### ステップ3　Student 種子モデルの取得元とアーキテクチャ初期化

**目標**：蒸留に向けて Student のベースモデルを準備し、統一メモリアーキテクチャ（128GB LPDDR5X UMA）のハードウェア上で最大のテンソルコア効率でロードし、パラメータ効率的微調整層をマウントする。
**技術スタック／ツール**：PyTorch、HuggingFace Transformers、`peft`（QLoRA）、`bitsandbytes`（NF4 量子化）、gradient checkpointing。種子モデルはオープンウェイトの Qwen3 系列や Llama-3-70B など。
**中核実装**：`model_initializer.py` を記述し、`torch.bfloat16` のフル精度でアーキテクチャをロードする。高ランクの QLoRA（rank r≥128、alpha α=256）を構成し、target を全線形層（q/k/v/o_proj、gate/up/down_proj）に設定する。フルパラメータ経路でない場合は NF4 量子化を用い、activation メモリを抑えるため gradient checkpointing を有効化する。
**検証（Assertions）**：訓練可能パラメータの総数と検証済み dtype を出力し、gradient checkpointing がモデルの計算グラフに確実にバインドされていることを assert する。
**解説**：r=128／α=256 はやや過激な高ランク設定で、VRAM を食う代わりに表現力を得る。UMA 上での bf16＋NF4 の混在は、量子化と checkpointing が互いに衝突しないことを必ず確認すること。さもなくば逆伝播がひそかにメモリを溢れさせる。

### ステップ4　超長コンテキスト KV-Cache と RoPE 拡張の構成

**目標**：UMA アーキテクチャ上で Student が最大約 1,048,576 トークン（1M）のコンテキストをネイティブに扱えるようにし、かつカーネルレベルの OOM を引き起こさないようにする。
**技術スタック／ツール**：YaRN／NTK-aware RoPE scaling、FlashAttention-3 または vLLM PagedAttention、unified-memory の事前確保。
**中核実装**：`context_extender.py` を記述し、RoPE 周波数基底 θ を 10,000 から 1,000,000 へ段階的に拡大して config に書き込む。トークン生成用に PagedAttention block を構成（block size 16 または 32、LPDDR5X のアクセスパターンに合わせてレイテンシのスパイクを抑制）する。KV-Cache の統一メモリ空間を事前確保し、明示的な制約式で 128K prefill 下でも OOM を引き起こさないことを保証する。
**検証（Assertions）**：ベンチマークを記述し、シーケンス長 32,768 と 131,072 の mock テンソルに対して順伝播を行い、メモリ確保の失敗が起きないことを確認する。
**解説**：θ を一気に 1M まで拡大することは、長さ外挿を RoPE スケーリングに丸ごと賭けることに等しい。短コンテキストが犠牲になっていないか、必ず perplexity の回帰テストで確認すること。block size のチューニングは、レイテンシとメモリ断片化の現実的なトレードオフ点である。

### ステップ5　PII 脱プライバシー洗浄サンドボックス

**目標**：訓練データから個人情報、秘密鍵、そして Student の自己認識を汚染しうるベンダーのブランドトークンを除去し、データ衛生を維持する。
**技術スタック／ツール**：プリコンパイル済み regex、軽量なローカル NER（`spaCy` または `gliner`）、ノンブロッキングな asyncio。
**中核実装**：`data_anonymizer.py` を記述し、regex とローカル NER を組み合わせて高スループットなスキャンを行う。email、IP、JWT、クラウド鍵、実在の住所を自動マスクする。さらにもう一段ブランドの中立化を行う（ベンダー／モデル名を「the assistant」「the model」などの中立トークンへ置換し、文脈書き換えルールで処理する）ことで、Student が外来のアイデンティティを継承しないようにする。
**検証（Assertions）**：機密トークンのサンプルを含む単体テスト配列を記述し、PII とブランド文字列が 100% マスクまたは書き換えられることを検証する。
**解説**：ブランドの非識別化をデータ衛生として扱うのは妥当だが、「100% 除去」は脆い約束だ——NER の取りこぼしは常態であり、自動 pass を全面的に信じるより、人手による抜き取り監査を残すことを勧める。

### ステップ6　分散ストレージ・キャッシュパイプライン

**目標**：蒸留パイプラインのストレージ基盤を構築し、上流の突発的なストリームを吸収しつつ、構造化・ゼロコピーの形で下流の訓練モジュールへ供給し、メモリとディスク I/O のブロッキングを回避する。
**技術スタック／ツール**：Redis（`redis.asyncio`）、Apache Arrow（`pyarrow`）、Parquet、worker pool。
**中核実装**：`cache_pipeline.py` を記述し、ノンブロッキングな Redis 接続マネージャを in-memory queue として用いる。worker pool がストリーム断片をバッチで取得し、構造化スキーマへ正規化のうえ Arrow でシリアライズしてゼロコピーを達成する。メモリ上の chunk が閾値（例：512MB）に達したらローリングで最適化済み `.parquet` へ落とす。スレッドセーフな読み書きロックにより、訓練モジュールが書き込みと同時に並行読み出しできるようにする。
**検証（Assertions）**：複数 worker の並行下で ≥10,000 tokens/s のスループットに耐え、パケットロスやメモリリークが起きないことをベンチマークで示す。
**解説**：Redis を queue に、Arrow をゼロコピーに使う組み合わせは実務的だ。真の落とし穴は読み書きロックとバックプレッシャの設計にある。10k tokens/s はスタートラインにすぎず、Redis のメモリ上限が落とし込みをトリガーする境界挙動を必ず負荷試験すること。

### ステップ7　Tokenizer 語彙アライメントと KL ダイバージェンス・アダプタ

**目標**：Teacher（君が利用権限をもつオープンウェイト、または自前の社内モデル）と Student が異なる tokenizer／語彙を採用する場合に、微分可能な soft target アライメントを構築し、ずれたトークンインデックス上で KL ダイバージェンスを計算して勾配が無効化されるのを防ぐ。
**技術スタック／ツール**：HuggingFace tokenizer、logits 射影、soft-alignment 行列。
**中核実装**：`vocab_aligner.py` を記述し、Teacher の top-K logits のトークンをその tokenizer で文字列へデコードし、Student の tokenizer で再分割する。特定の logit が欠ける場合は、動的マッチング／cross-attention の soft-alignment 行列によって確率質量を Student の最近接トークンインデックスへ射影する。最後に ΣP(Student)=1.0 となるよう正規化し、数学的に安定な KL 目標分布を生成する。
**検証（Assertions）**：非対称な logit 分布をアライナに通したのち、shape が完全に Student と整合し、正規化済みの有効確率テンソルが得られることを assert する。
**解説**：クロス tokenizer 蒸留の核心的難題だ。文字列の往復と再分割はアライメントノイズを持ち込む。soft-alignment 行列が品質の鍵であり、射影前後の確率質量の損失を健全性指標として監視することを勧める。

### ステップ8　自動化ベンチマーク Baseline

**目標**：訓練前に種子 Student の完全自動化された評価基準を確立し、checkpoint 検証時の客観的な比較基線とする。
**技術スタック／ツール**：lm-eval-harness 風のラッパー、MMLU-Pro、HumanEval、GSM8K、vLLM／Transformers 推論、隔離 Python サンドボックス。
**中核実装**：`eval_baseline.py` を記述し、多肢選択推論（MMLU-Pro）、コーディング能力（HumanEval）、多段数学（GSM8K）を軽くラップする。ローカル runner で種子モデルに推論を走らせる。自動採点エンジンが多肢選択の解答を解析し、コード出力を隔離サンドボックスで実行し、数学の文字列を照合して、総合正答率、Pass@1、F1 を計算する。`eval_baseline.json` を出力し、markdown のサマリ表を生成する。
**検証（Assertions）**：採点エンジンが mock の壊れたコード／誤答を正しく不正解と判定でき、スキーマが下流のトラッキング形式と完全に一致することを確認する。
**解説**：訓練前に baseline を立てるのは正しい規律だ。HumanEval のコードは必ず真に隔離されたサンドボックスで走らせること。さもなくば自動採点そのものがセキュリティホールになる。

### ステップ9　ハードウェアテレメトリと OOM 防御ウォッチドッグ

**目標**：長シーケンス蒸留は 128GB UMA を重く圧迫するため、システムの健全性をリアルタイムに監視し、カーネル OOM のハードロックが発動する前に能動的に一時停止／負荷軽減する必要がある。
**技術スタック／ツール**：`psutil`、`pynvml`、Prometheus client、Grafana、SIGUSR1 シグナル。
**中核実装**：`telemetry_watchdog.py` を記述し、500ms ごとに CPU 負荷、UMA メモリ使用量、GPU コア状態、メモリバスの帯域幅を取得する。Prometheus の `/metrics` エンドポイントでネイティブにエクスポートして Grafana から取得させる。OOM ウォッチドッグスレッドは、利用可能メモリが安全閾値（例：残り 8GB）を割り込んだ瞬間に訓練エンジンへ `SIGUSR1` を送り、一時停止・cache の dump・KV-Cache 確保量の縮小を行わせる。
**検証（Assertions）**：メモリ圧迫を加える負荷試験をシミュレートし、ウォッチドッグが <50ms 以内に閾値超過を捕捉して中断シグナルを発することを示す。
**解説**：能動的ウォッチドッグは、受動的に OOM killer を待つよりはるかに賢い。8GB 閾値は 1M コンテキストのシナリオではやや厳しいので、prefill 段階に応じて閾値を動的調整し、訓練エンジンが SIGUSR1 のグレースフルな一時停止経路を本当に実装しているか確認することを勧める。

### ステップ10　エンドツーエンドの健全性スモークテスト

**目標**：大規模 GPU 作業へ投入する前に、パイプライン全体（データ採取、フィルタリング、インフラ）が完全に動作することをエンドツーエンドで検証する。
**技術スタック／ツール**：pytest 風のオーケストレータ、Redis、Arrow、tokenizer、CI 統合。
**中核実装**：`pipeline_smoke_test.py` を記述し、完全な mock サイクルを直列につなぐ——データ生成→データソース接続プール→PII 洗浄→Redis キュー→Arrow Table 取得→語彙アライメント→1 ステップの dummy 推論。各中継 block に厳密なレイテンシ記録を仕込み、シリアライズ／ディスク I/O のボトルネックを早期に特定する。完了・失敗いずれの場合も、全テンポラリファイル、Redis key、テスト cache を自動で消去する。
**検証（Assertions）**：全モジュールが制限タイムアウト（例：60 秒）以内にエラーゼロで相互疎通すれば exit code 0 を返す。いずれかの block が失敗すれば exit code 1 を返し、完全な debug traceback を付す。
**解説**：スモークテストは ML パイプラインで最も省略されがちだが最も値打ちがある一環だ。失敗経路でも確実にリソースを片付けること。さもなくば残留した Redis key が次回テストで偽陽性を引き起こす。

### ステップ13　XML タグとツール呼び出し抽出パーサ

**目標**：Teacher の出力から `<thought>`、`<tool_use>`、`<str_replace_editor>` などの XML ノードを精確に抽出し、Student に推論と環境インタラクションの正確なトークン構造を学ばせる。
**技術スタック／ツール**：状態機械／インクリメンタルパーサ、streaming-safe な tokenizer、構造化 JSON 出力。
**中核実装**：`xml_parser.py` を記述し、状態機械（regex 不要のインクリメンタル解析）でストリーム断片中の未閉合・不正な XML を構文エラーを投げずに処理する。生テキストを `thinking_trajectory`（`<thought>` 内）、`tool_call_payload`（`<tool_use>` 内）、`final_response`（タグ外）へ切り分ける。クリーンな JSON へ正規化しつつ、tool block 内の正確な空白、コードのインデント、改行を保持する。
**検証（Assertions）**：破損・畸形・途中切断（例：タグの途中で切れたコードブロック）したストリーム入力でテストし、パーサが完全な block を正確に分離し、不完全なタグをデータ欠落なくグレースフルに処理することを検証する。
**解説**：ストリーム中で途中の XML を解析するのは本物の腕前だ。状態機械の選択は正しい——regex に後退するのは厳禁だ。コードのインデントと改行の保持はツール呼び出し忠実度の生命線であり、必ずテストの assert に含めること。

### ステップ14　思考連鎖トラジェクトリ再構築エンジン

**目標**：Student が Teacher のテキスト塊を表層模倣して過学習するのを避けるため、推論経路を分解してモデルに中間の導出ノードを理解させる。
**技術スタック／ツール**：グラフ式 trace 構築、データ拡張、多ターン誤り訂正の加重。
**中核実装**：`trace_inversion.py` を記述し、`<thought>` の実行経路を離散的な推論ステップ（assertion、仮説、コード検証、自己修正）へ解析する。再構築アルゴリズムで反事実経路（例：「第3ステップが失敗したらどうなるか」）を生成するか、独立した導出ステップを並べ替え、頑健な非線形の思考連鎖を合成する。合成トラジェクトリを Student の構造化指示へマップし、多ターンの誤り訂正ループに加重して深いデバッグ能力を強化する。
**検証（Assertions）**：1 本の線形な 5 ステップ推論 trace を多分岐ツリー構造へ変換でき、かつトークン長の制約が目標語彙の制限を満たすことを assert する。
**解説**：線形 CoT を分岐ツリーへ書き換えるのは表層模倣に抗う良い発想だが、反事実経路の正しさは保証しがたい——合成された「もし失敗したら」分岐が論理的に自己整合しなければ、かえってモデルを誤って教育してしまう。合成トラジェクトリには事実一貫性フィルタをかけることを勧める。

### ステップ16　ハードラベル交差エントロピー損失エンジン（Hard-Target Cross-Entropy Loss）
**目標**：生徒モデルが教師（君が合法に授権された対象）の生成した ground-truth トークン列に対して負の対数尤度（標準的な交差エントロピー）を最小化するようにし、蒸留損失の基礎層を構成する。
**技術スタック／ツール**：PyTorch（`torch.nn.Module`）、fused/chunked cross-entropy（Liger-Kernel／`torch.compile` など）。
**中核実装**：`HardTargetLoss` モジュールを自作する。fused または分塊（chunked）交差エントロピーを採用し、巨大な logit テンソルによる VRAM ピークの暴騰を回避する。label smoothing（ε=0.1）を加えて過信を抑制する。`ignore_index=-100` で padding トークンをマスクし、それらが勾配に寄与しないようにする（シーケンスパッキングと併用）。
**検証（Assertions）**：任意のマスクを伴う dummy トークンテンソルに対して計算し、数学的結果が正しく、かつ CUDA の確保エラーを引き起こさないことを確認する。
**解説**：分塊交差エントロピーは大語彙 LLM 訓練で標準的な VRAM 節約手法であり、label smoothing と padding マスクはいずれも必須の細部だ。実務的である。

### ステップ17　ソフトラベル KL ダイバージェンス損失エンジン（Soft-Target KL-Divergence Loss）
**目標**：生徒と教師の logit 確率分布を整合させることで、教師の推論にある「ダークナレッジ」（dark knowledge）を転移する。
**技術スタック／ツール**：PyTorch（`LogSoftmax`、`KLDivLoss`）。
**中核実装**：`KLDivergenceLoss` は生徒の生 logits と教師のアライメント済み確率テンソルを受け取る。温度スケーリング T=2.0 で分布を軟化する。前向き KL $D_{KL}(P_{Teacher}\parallel P_{Student})$ を計算し、ε=1e-7 の数値クリップで `log(0)`／NaN を回避する。混合係数 α=0.4 で総損失 $\mathcal{L}_{step}=\alpha\cdot\mathcal{L}_{soft}\cdot T^2+(1-\alpha)\cdot\mathcal{L}_{hard}$ を合成する。
**検証（Assertions）**：生徒分布が教師と完全に一致するとき KL がゼロに近づき、極端な logit 外れ値の下でも数値安定性を保つことを確認する。
**解説**：$T^2$ 補償項と α 混合は Hinton の古典的蒸留の正規の書き方であり、数値クリップさえきちんと行えば本番投入できる。

### ステップ18　思考連鎖加重損失層（CoT-Weighted Loss Layer）
**目標**：シーケンステンプレート内でトークン級の損失を動的にスケールし、生徒モデルが深層の推論ブロック（CoT）を優先的に学ぶよう強制する。
**技術スタック／ツール**：PyTorch 自作 loss wrapper、トークン級マスクテンソル。
**中核実装**：`<thought>` ブロック（モジュール二のマーク）のトークンマスクを受け取る。推論ラベル内では CE と KL 損失に重み $\omega_{CoT}=2.0$ を乗じる。汎用的な挨拶／前置きトークンは $\omega_{filler}=0.5$ に減重して情報密度を高める。正規化済みで直接 `.backward()` できるスカラー損失を返す。
**検証（Assertions）**：多様な段落アイデンティティでマークされたトークン index 構成で、逆伝播損失が確かに重み倍率どおりにスケールされることを検証する。
**解説**：推論段に加重し、フィラー語を減重するのは CoT 蒸留の効率を高める妥当なカリキュラム手法だ。重みは過学習を避けるため検証セットで調整する必要がある。

### ステップ19　DeepSpeed ZeRO-3 メモリオーケストレータ（ZeRO-3 Memory Orchestrator）
**目標**：大規模モデル（例：35B）の訓練時にモデル状態を完全にシャーディングし、統一メモリ（UMA）アーキテクチャの効率を最大限に搾り取る。
**技術スタック／ツール**：DeepSpeed ZeRO-Stage 3（または PyTorch FSDP）、bf16 混合精度、`ds_config_zero3.json`。
**中核実装**：オプティマイザ状態／勾配／パラメータの完全シャーディングを有効化する。`offload_optimizer`、`offload_param` と pin-memory を設定し、統一メモリバスを活かす。動的 loss scaling を有効化して bf16 のアンダーフローを防ぐ。通信 bucket size（例：5e7）を調整し、勾配同期時の帯域飢餓を回避する。
**検証（Assertions）**：config を解析し、すべての ZeRO-3 シャーディングフラグ、メモリパラメータ、精度変数が本番実行の要件を満たすことを assert する。
**解説**：ZeRO-3 + offload はコンシューマ級／単機大メモリで超大規模モデルを走らせる主流の方策であり、bucket size と pin-memory の調整がスループットを直接左右する。

### ステップ20　混合精度と LoRA 重み融合エンジン（Mixed Precision & Weight Fusion）
**目標**：混合精度で前後向きの更新を実行し、訓練終了時にパラメータアダプタを自動融合する。
**技術スタック／ツール**：`torch.amp.autocast`（bf16）、`torch.nn.utils.clip_grad_norm_`、peft/LoRA、safetensors。
**中核実装**：訓練ループは bf16 autocast でコアのテンソル演算を実行する。勾配ノルムを上限 1.0 でハードクリップし、トークンの急増による安定性崩壊を防ぐ。訓練成功後に base model と LoRA checkpoint をロードし、FP32 精度で $W_{final}=W_0+\Delta W$ のフルパラメータマージを実行して丸めデグレを回避する。未量子化の標準的 HuggingFace safetensors シャードを出力する。
**検証（Assertions）**：マージ後の重み辞書に孤立した adapter 層がなく、ファイル完全性のハッシュ検証が通ることを確認する。
**解説**：bf16 訓練、FP32 融合という精度の使い分けは正しい。勾配クリッピングは長シーケンス訓練の基本的な保険である。

### ステップ21　超長コンテキスト漸進的拡張スケジューラ（Progressive Context Window Scaler）
**目標**：生徒モデルが超長コンテキストをネイティブに扱えるようにしつつ、訓練初期の OOM／勾配爆発を起こさないよう、漸進的な長さ拡張を採る。
**技術スタック／ツール**：PyTorch 訓練 hook、RoPE θ 調整（NTK/YaRN の発想）。
**中核実装**：訓練 hook で global step を追跡する。多段階拡張（8K→32K→128K→さらに長く）を行う。各境界で Cosine／Linear 公式により RoPE base frequency θ（10,000 → 1,000,000）を動的調整し、モデルの max position embedding も同期更新する。長さ切り替え時には学習率を比率（例：×0.5）で引き下げ、最適化の安定性を維持する。
**検証（Assertions）**：単体テストで、RoPE base が指定 step 閾値で正しく変更され、config の最大位置埋め込みフィールドが正確に更新されることを検証する。
**解説**：漸進拡張 + RoPE 周波数の再スケーリングは業界の長コンテキスト延伸の標準路線であり、切り替え時に LR を下げるのは振動を防ぐ鍵となる細部だ。

### ステップ22　マルチタスク並列シーケンスパッキングエンジン（Multi-Task Sequence Packing）
**目標**：batch padding による演算力の浪費を解消し、不等長シーケンスを固定長 block へ高密度にパッキングする。
**技術スタック／ツール**：PyTorch `Dataset`/`DataLoader`、Best-Fit Decreasing bin-packing、FlashAttention（`cu_seqlens`）。
**中核実装**：`<eos>` 区切りで複数の短シーケンスを単一の最大長 block（例：8,192／32,768）へ詰める。Best-Fit Decreasing 箱詰めアルゴリズムで残余 padding を < 0.5% にする。1D 累積位置配列 `cu_seqlens` と attention mask メタデータを生成し、FlashAttention バックエンドへ渡して、同一 block 内の異なるシーケンスが互いに注意を跨がない（block-diagonal）ことを保証する。
**検証（Assertions）**：ランダムな可変長の 1,000 本の配列を与え、パッカーが正しく高密度 block へ平坦化し、クロス注意マスクの境界 index が完全に一致することを検証する。
**解説**：シーケンスパッキングと FlashAttention varlen の組み合わせは高スループット訓練の定番だ。cross-attention の隔離を漏らすとサンプルが汚染されるので、必ず厳密にテストすること。

### ステップ23　コサインアニーリング学習率スケジューラ（Cosine Annealing LR Scheduler）
**目標**：業界標準のスケジュールで長い訓練サイクルの学習率動態を管理し、後期の収束停滞や激しい振動を回避する。
**技術スタック／ツール**：PyTorch `torch.optim.lr_scheduler`（Linear Warmup + Cosine Annealing）。
**中核実装**：総ステップの最初の 5% を線形 warmup とし、1e-7 からピーク 2e-4 まで上げる。その後 cosine アニーリングで 100% ステップにかけて下限（1e-6、ピークの約 1%）まで滑らかに減衰する。scheduler の `state_dict` は DeepSpeed checkpoint とともに完全にシリアライズ／再ロードでき、減衰ステップ数を失わない。
**検証（Assertions）**：1,000 回の反復をシミュレートし、step 50/500/950 で期待される非線形減衰曲線の値を記録・検証する。
**解説**：Warmup + Cosine は LLM 訓練で最も汎用的な LR スケジュールであり、正しく resume できることは中断再開訓練のハード要件である。

### ステップ24　自動化チェックポイントマネージャ（Automated Checkpoint Manager）
**目標**：ノンブロッキングかつ耐障害的に中間重みを保護し、ハードウェア故障／停電／コンテナ切断による数日分の訓練の水泡化を防ぐ。
**技術スタック／ツール**：DeepSpeed `save_checkpoint()`（非同期）、NVMe ストレージ、`checkpoint_state.json`。
**中核実装**：定点ステップ（例：500 ステップごと）で検証損失を評価する。過去最小値を下回ったらバックグラウンドスレッドで保存をトリガーする。ローリングウィンドウを維持する——top-3（最低 val loss）と最新の 1 つだけを残し、古いフォルダを自動削除して NVMe を節約する。metadata tracker が epoch、global step、val loss、絶対パスを記録する。
**検証（Assertions）**：合成 checkpoint リクエストを注入し、より劣る古いフォルダが正しく削除され、最低損失と最新状態は破損なく保持されることを検証する。
**解説**：top-k + latest のローリング保持戦略は、最良と再開可能性を両立する。非同期保存は I/O が訓練をブロックするのを避ける、工業級のやり方だ。

### ステップ25　GC と訓練損失ウォッチドッグ（Garbage Collection & Train-Loss Watchdog）
**目標**：累積的な VRAM リークと突発的な勾配爆発（loss→NaN）を防ぎ、自己修復する実行時の安全防護を提供する。
**技術スタック／ツール**：Python `gc.collect()`、`torch.cuda.empty_cache()`、移動平均の異常検知。
**中核実装**：epoch 終了 hook で明示的に GC し、CUDA caching allocator を空にする。毎ステップ loss を監視し、異常倍率の閾値（例：移動平均の 3×）を超えるか NaN/Inf に当たれば直ちに訓練を凍結する。発動後は自動で直前の健全な checkpoint（ステップ24）へロールバックし、学習率に 50% の収縮をかけてからグレースフルに実行を再開する。
**検証（Assertions）**：強制的に NaN 異常を起こす dummy 訓練ループを構成し、ウォッチドッグがイベントを捕捉し、実行を停止し、ロールバックして再ロードし、学習率を下げることを検証する。
**解説**：auto-rollback + LR 収縮は長時間訓練の実用的な自己治癒機構だ。ただし 3× 移動平均の閾値は、誤発動を避けるためタスクの loss ノイズに応じて調整する必要がある。

### ステップ26　訓練後重み融合エンジン（Post-Train Weight Fusion）
**目標**：LoRA／QLoRA／部分凍結が生み出した adapter を単一の独立モデルへ恒久的にマージし、推論時のレイテンシを解消する。
**技術スタック／ツール**：PyTorch（FP32 ロード）、peft `merge_and_unload()`、safetensors。
**中核実装**：`torch.float32` で pristine base（ステップ3）と最良 checkpoint（ステップ24）をロードし、丸めデグレを解消する。peft の `merge_and_unload()` で安全にマージする。次元不一致や欠落 adapter key にはグレースフルに例外処理し、未マージ行列を記録する。融合後は bf16 へ戻し、HuggingFace safetensors シャード（1 片あたり上限 10GB）を出力する。
**検証（Assertions）**：融合後の辞書に `lora_`／`adapter_` の参照がゼロで、出力テンソル形状が元 base の構成と完全に一致することを確認する。
**解説**：FP32 でマージしてから bf16 へ落とす精度戦略は正しい。シャード上限と形状の assert により、成果物がそのままデプロイ可能であることを保証する。

### ステップ27　自動化レッドチーミング安全性評価サンドボックス（Safety Red-Teaming Sandbox）
**目標**：融合後モデルに対し敵対的な安全評価を行い、有害なリクエスト（マルウェア、危険操作、脱獄誘導など）を安定して「拒否」できるかを計測し、安全防護が有効でデグレしていないことを確認する。
**技術スタック／ツール**：vLLM（隔離 Docker サンドボックスでのバッチ推論）、LLM-as-a-Judge（ローカルまたは信頼できる API）、安全評価データセット。
**中核実装**：脱獄プロンプト、倫理的パラドックス、機微な技術的問い合わせを含む安全評価セットをロードする。ホストネットワークから隔離された Docker サンドボックス内で vLLM のバッチ推論を行う。LLM-as-a-Judge で各出力を `[適切な拒否／安全な応答]`（合格）、`[有害な追従]`（不合格）、`[幻覚／文字化けループ]`（不合格）に分類する。安全合格率と有害追従率の指標を集計し、いかなる有害追従ケースも記録して回帰リスクとしてマークする。
**検証（Assertions）**：出力 JSON ログを解析し、有害リクエストを拒否できなかった（追従的な応答が出た）クエリ index に対して直ちに失敗アラートを発する。
**解説**：レッドチームを「モデルが有害リクエストを確かに拒否することを検証する」安全回帰テストと位置づけるのは、責任あるデプロイに必須の関門だ。IFEval／拒否ベンチマークと並行して追跡し、安全アライメントの劣化を防ぐことを勧める。

### ステップ28　タグ形式の校正と修補器（Tag-Format Regularizer）
**目標**：モデルが厳密に正当な XML タグ（`<thought>`、`<tool_use>`）を出力するようにし、閉じ忘れタグが下流アプリ（Cursor、Open WebUI など）を壊すのを防ぐ。
**技術スタック／ツール**：高速 XML/タグパーサ、多ターンのツールインタラクション・シミュレーション、低 LR のアライメント微調整パイプライン。
**中核実装**：融合後モデルで 500 プロンプトの多ターン・ツールインタラクションを走らせ、形式規律を負荷試験する。高速パーサで未閉合タグ、畸形の括弧、論理転換の欠落をスキャンする。構造の失敗率が > 0.5% なら自動で「Epoch-0.1 修補」を実行する——完璧な構造の例 200 本のミニデータセットをロードし、極めて低い学習率で素早くアライメント微調整し、タグ閉合の習慣を強化する。
**検証（Assertions）**：自作の破損 XML シーケンスを注入し、修補エンジンが異常を正確に捕捉し、タグ準拠率を正しく計測することを検証する。
**解説**：タグの閉合は agent／ツール呼び出しを実地に動かすためのハード制約だ。0.5% 閾値で発動する小サンプルのアライメント微調整は低コストの是正であり、実務的で効果的だ。

### ステップ29　全自動化の総合能力ベンチマーク評価
**目標**：蒸留後の生徒モデルが本来の知能水準を保ち、後処理段階で破滅的忘却を起こしていないことを客観的に証明する。**技術スタック／ツール**：マルチスレッド worker pool、MMLU-Pro／HumanEval／GSM8K／IFEval ベンチマークセット、Markdown レポートとレーダーチャート（matplotlib/plotly）。**中核実装**：スレッドプールで各ベンチマークのテストスライスを並列実行し、生出力と生成速度を記録する。Pass@1、F1、Exact Match を計算し、ステップ8 が生成した `eval_baseline.json` を読み戻して差分対照する。「種子モデル vs 蒸留モデル」の対比レーダーチャートを含む Markdown レポート（`eval_suite_final.py`）を自動生成する。**検証（Assertions）**：総合スコアが baseline より許容閾値を超えて下落（>3%）したら、明示的なエラーコードで中止し、蒸留のデグレとみなす。**解説**：「デグレ即 fail」を CI の終了コードに書き込むのは正しい工学規律だが、単一の 3% 全域閾値は粗すぎる——各ベンチマークの分項に閾値を設けるべきだ。さもなくばコーディング能力の急落が知識問題の微増に隠されかねない。

### ステップ30　モデル重みのプルーニングと最適化クリーンアップ
**目標**：量子化の前に外科的なクリーンアップを行い、多目的蒸留後に冗長またはノイズで汚染された神経経路を除去し、メモリ占有とレイテンシを下げる。**技術スタック／ツール**：magnitude／activation-sparsity プルーニング、PyTorch テンソル操作、safetensors シリアライズ。**中核実装**：FFN/MLP 層に対し量級または活性化スパース度ベースのプルーニングを行い、総活性化重みが末端変動の閾値を下回るニューロンをゼロ化する（上限約 5% の冗長経路）。非ゼロ行列を圧縮し、死んだテンソル区間、重複した meta header、中間コンパイル成果物を一掃する。未量子化の最終 base モデルを本番ディレクトリへ出力し量子化を待つ（`weight_pruner.py`）。**検証（Assertions）**：単体テストで、出力ファイルが明確に小さくなり、かつ 10 トークンの順伝播のテンソル値がプルーニング前と ±1e-5 の厳しい公差内で一致することを確認する。**解説**：プルーニングを先、量子化を後にする順序は妥当だが、「5% でスコアは落ちない」は経験的仮定だ。10 トークンの数値一致だけで通すのではなく、必ずステップ29 の全套ベンチマークで回帰検証すること。

### ステップ31　GB10 専用の混合精度 GGUF 量子化コンパイル
**目標**：128GB LPDDR5X UMA（~600 GB/s 帯域）上で非一様量子化により推論品質を保ちつつ、生成速度を犠牲にしない。**技術スタック／ツール**：llama.cpp/GGML 変換ツール、GGUF k-quants（Q8_0／Q5_K_M／Q4_K_M／IQ4_XS）。**中核実装**：まずプルーニング後の safetensors を標準 `.gguf` へ変換する。注意層（Q/K/V/O 投影）を高めの精度（Q8_0 または Q5_K_M）に固定して論理推論を保ち、巨大な FFN（gate/up/down）は Q4_K_M または IQ4_XS まで下げて、トークンごとのメモリ読み出し量を縮める。統一された混合量子化 GGUF を出力し、層ごとに量子化パラメータを記録する（`gguf_mixed_compiler.py`）。**検証（Assertions）**：出力 GGUF が目標メモリウィンドウ（例：35B≈24GB、120B≈72GB）に収まり、破損テンソルがないことを assert する。**解説**：メモリ帯域をボトルネックとして精度を配分する（attention は重く、FFN は軽く）のは非常に正しい工学的直感だ。ただしファイルサイズが正しい位置に落ちることは品質達成と同義ではないので、perplexity／ベンチマークを回し戻して混合配分が推論を損なっていないか確認すること。

### ステップ32　FP8/INT8 KV-Cache 圧縮エンジンの構成
**目標**：長コンテキスト（百万トークンに迫る）下では KV Cache のメモリが幾何級数的に増大し、FP16 では 128GB マシンで OOM を引き起こすため、圧縮を要する。**技術スタック／ツール**：Ollama/llama.cpp バックエンドフラグ、FP8（e4m3／e5m2）または INT8 の KV 量子化、ローリングウィンドウ・キャッシュ管理。**中核実装**：推論バックエンドのコンパイル／実行フラグを生成し、KV Cache を低精度 FP8 または INT8 で保存するよう強制する。動的なトークン eviction／ローリングウィンドウを実装する——単一コンテキストが危険閾値（例：10 万トークン）を越えたら、遠い過去の活性化を圧縮またはダウンサンプルし、近端の推論ウィンドウには完全な注意精度を維持する。LPDDR5X の cache line に合わせてメモリレイアウトを最適化し、転置 stride のレイテンシを回避する（`kv_cache_compressor.py`）。**検証（Assertions）**：20 万トークンの推論をシミュレートしてテストし、KV Cache の UMA 確保量が FP16 baseline より少なくとも 45% 減り、数値オーバーフローがないことを assert する。**解説**：KV 量子化＋ローリングウィンドウは長コンテキストの定番解だが、FP8 KV は attention の数値安定性に敏感だ。「遠端の詳細を必要とする長文検索タスク」で品質を検証すべきで、メモリが 45% 減ったことだけを見てはいけない。

### ステップ33　自動化された Ollama / LM Studio モデルプロファイルのパッケージング
**目標**：混合精度 GGUF と KV 設定を、Ollama／LM Studio で手動ゼロロードできる本番級リリース物へパッケージする。**技術スタック／ツール**：Ollama Modelfile、`ollama create`、LM Studio 設定。**中核実装**：GGUF の絶対パスを含む Modelfile をプログラムで書き出す。`SYSTEM` ブロック内にカスタムシステムプロンプトを埋め込む（思考連鎖を `<thought>` タグ内に出力させ、「AI 言語モデルとして…」のような冗語を除去して出力を簡潔に保つ）。デフォルトパラメータ temperature=0.7、top_p=0.95、`num_ctx 131072` を設定し、`num_thread` を GB10 の CPU コアトポロジに合わせる（`ollama_packer.py` + `Modelfile`）。**検証（Assertions）**：テスト runner で `ollama create distilled-mythos-model -f ./Modelfile` を実行し、退出コード 0 でビルドされ、正しいデフォルトパラメータが記録されることを確認する。**解説**：システムプロンプトと最適パラメータを Modelfile に固定化すれば、確かにゼロ構成での引き渡しを実現できる。`num_thread` のコアトポロジ整合は実在のチューニング点だが、`num_ctx 131072` はステップ32 の KV 圧縮と協調検証しないと、大きな窓を設定しても OOM になる。

### ステップ34　Linux カーネルレベルの NUMA バインドとメモリ物理ロック
**目標**：GB10 の UMA は CPU と iGPU が 128GB を共有するため、OS 層でカーネルがアクティブな重みをディスクへスワップアウトしたり、マルチチャネルの非対称 NUMA レイテンシに足を引っ張られたりするのを阻止する必要がある。**技術スタック／ツール**：`numactl`、`mlockall`／`/etc/security/limits.conf`、Transparent Huge Pages（THP）。**中核実装**：バックエンドエンジンの起動時に shell で `numactl --interleave=all` を用い、すべての LPDDR5X チャネルにわたってメモリをインターリーブする。`mlockall` または limits.conf 調整で推論プロセスのメモリフットプリントを物理 RAM へ恒久ロックし、NVMe swap へのページアウトを防ぐ。性能プロファイルに応じて THP を `always` または `madvise` に設定し、高スループットのメモリマッピングを最大化する（`kernel_booster.sh`）。**検証（Assertions）**：スクリプト内で `cat /proc/self/status`、`numastat` を用い、メモリロックが効いていてインターリーブ状態が均衡していることをプログラム的に assert する。**解説**：interleave + mlock はローカル大規模モデル推論の正解で、テールレイテンシのジッタを消せる。ただし `mlockall` で全体ロックする際は 128GB マシンでも余裕を残すこと。さもなくば重みをロックし切って他のサービスを OOM へ追い込む——ロック範囲と ulimit を一緒に管理することを勧める。

### ステップ35　エンドツーエンドの高並行 A/B 性能検証
**目標**：エンドポイント（Cursor／Continue／Open WebUI）を本番利用へ開放する前に、高負荷の並行テストで GB10 上の性能安定性を検証する。**技術スタック／ツール**：`asyncio` 非同期負荷試験、Ollama/LM Studio のローカルエンドポイント、locust/k6 系の並行手法、システムテレメトリ。**中核実装**：asyncio で 20 本の並行対話を開き、複雑なコーディングと長コンテキスト推論のリクエストを同時に送り、TTFT、ストリームごとの tok/s、システム総スループットを追跡する。テスト中はテレメトリを監視し、生成速度が基準を満たす（35B モデル ≥102 tok/s、120B モデル ≥35 tok/s）ことを検証する。`perf_metrics.json` を出力し、ターミナルに性能サマリを印字する（`e2e_validator.py`）。**検証（Assertions）**：レイテンシのスパイクが閾値を超えるか、いずれかのストリームが負荷下で HTTP エラーを出したら、明示的に警告エラーを投げる。**解説**：TTFT＋tok/s＋総スループットの三次元で並行を計測するのは実務的だ。ただし 20 本の並行は固定値であり、階段状に加圧して飽和点を探し、「単一ストリームのピーク速度」と「並行下の一人当たり速度」を区別すべきだ——両者はメモリ帯域律速のマシンでは明確に乖離する。

### ステップ36　スマート二重モデル動的ルーティングプロキシ
**目標**：ローカルの 2 モデル（35B はコーディング/ツール寄り、120B は深い論理/アーキテクチャ寄り）を単一の OpenAI/Anthropic 互換エンドポイントの背後に隠し、プロンプトの意図に応じて動的にルーティングする。**技術スタック／ツール**：FastAPI + Uvicorn、LiteLLM 式ルーティング、OpenAI 互換 `/v1/chat/completions`。**中核実装**：標準の `/v1/chat/completions` を公開する。超高速な実行時分類器を実装し、入力プロンプトのテキストまたはコンテキスト header で判断する——構造化ファイル置換／素早いコード修正／shell ツール呼び出しは 35B へ、アーキテクチャ計画／高度に抽象的な数学／ファイル横断ロジックまたは入力 >32K トークンは 120B へルーティングする。明示的な fallback をサポートし、リクエスト body が `model` 名を指定した場合は分類器をスキップしてバックエンドを直接固定する（`distillation_router.py`）。**検証（Assertions）**：統合テストで 5 本のコード + 5 本のアーキテクチャレビューのプロンプトにより、100% 正しくルーティングされ、かつ >5ms の追加レイテンシを持ち込まないことを検証する。**解説**：意図ルーティング＋明示的オーバーライドは成熟したゲートウェイパターンだが、「100% 正しいルーティング」はヒューリスティック分類器にとって脆い約束だ。実トラフィックの混在意図は誤判断する。信頼度スコアを加え、低信頼時は大モデルへ default する保守的戦略を勧める。

### ステップ37　Cursor と Continue 開発プラグインのゼロ構成ネイティブ接続
**目標**：DGX Spark をローカル copilot に設定する作業を自動化し、Cursor および VSCode/JetBrains の Continue プラグインへ摩擦ゼロで接続する。**技術スタック／ツール**：クロスプラットフォームのパス検出、非破壊的 JSON patch、設定バックアップのスナップショット。**中核実装**：ホスト OS を検出し、Cursor（`~/.config/Cursor/` または AppData）と Continue（`~/.continue/config.json`）の設定パスを特定する。非破壊的 JSON パーサでステップ36 のルーティングアドレス（例：`http://localhost:8000/v1`）をカスタム OpenAI/Anthropic エンドポイントブロックへ注入する。モデル配列とマッピングラベルを埋め、構造化スラッシュコマンドをデフォルトで有効化する。変更前に既存設定を圧縮してスナップショットへバックアップし、データ消失を防ぐ（`ide_config_patcher.py`）。**検証（Assertions）**：変更後の設定ファイルが依然として正当な JSON であり、カスタムルーティングキーがルートスキーマ配列へ正しく注入されていることを検証する。**解説**：変更前のバックアップ＋非破壊的マージはユーザー設定に責任をもつやり方だが、サードパーティ IDE の設定ファイルへ直接書くことはそのスキーマと密結合する。プラグインの改版で壊れるので、スキーマバージョンを検出し、未知の形式へはグレースフルに譲歩すべきだ。

### ステップ38　本番環境の監視、性能ログ、Slack/Discord アラート
**目標**：開発者が日々 IDE 経由でルートを使う際、指標、システム健全性、トークン使用量をリアルタイムに把握し、デグレを早期検出する必要がある。**技術スタック／ツール**：FastAPI middleware、構造化 JSON ログ＋ローテーションファイル appender、Slack/Discord webhook（Prometheus/Grafana へ拡張可）。**中核実装**：ステップ36 のルート内に middleware インターセプタを加え、`session_id`、入力/出力トークン数、tok/s、TTFT を記録する。すべて厳密な JSON へ整形し、ローテーションファイル（`/var/log/distill_router.log`）へ非同期書き込みする。webhook アラートを実装する——未処理の 500、バックエンドの持続的切断、生成速度が閾値割れの際に、構造化アラートを Slack または Discord へリアルタイム配信する（`router_observability.py`）。**検証（Assertions）**：単体テストでエンドポイントのクラッシュやレイテンシ急落をシミュレートし、アラート webhook が <1 秒以内に正確な診断 JSON を受け取ることを検証する。**解説**：構造化 JSON ログ＋閾値アラートは MLOps の基本だが、ローカルファイルへ書くだけではトレンド分析に不利だ。これらの JSON を直接 Prometheus/Loki へ送り、アラートにはデバウンス（debounce）を加えて、ジッタ時に webhook が嵐のように溢れるのを防ぐことを勧める。

### ステップ39　増分データの自動収穫と閉ループ自己進化チェーン
**目標**：君が合法な権利をもつソース——自分の利用ログ、君が利用権限をもつモデルの出力、開発者が自発的に提供した修正——から、高品質なインタラクションを継続的に収集し、後続の増分微調整へ供給する。**技術スタック／ツール**：IDE workspace/エージェント metadata の取得、inline 評点フィルタ、Apache Arrow/Parquet（`incremental_training_pool.parquet`）。**中核実装**：ユーザーフィードバック取得ループを実装し、開発者がローカルモデルの生成したコードブロックを編集したり、拒否して修正を提供したりした際、エージェント metadata を経て最終 session 状態を捕捉する（君が自前で/権利をもつデータに限り、保護された商用 API のスクレイピングはしない）。inline 評点で重複対話、短すぎる断片、ノイズを除外し、高エントロピーの複雑なデバッグや多ターン論理データだけを残す。匿名化された高価値の対話を特徴ラベル付きで増分 Arrow テーブルへ追記し、次回スケジュールの増分訓練（モジュール3）で使う（`data_harvesting_loop.py`）。**検証（Assertions）**：機能テストで、一連の生 mock session を渡すとテキスト品質評価の解析が正しく行われ、合法な長コンテキスト項目がローカル Parquet 訓練プールへ成功裏に書き込まれることを確認する。**解説**：自前の利用ログで閉ループを回すのはコンプライアンスに適い、高レバレッジな「データフライホイール」だ。同意と匿名化は事後ではなく取得入口で行い、分布ドリフトを監視すること。さもなくばフライホイールはモデルを一部のヘビーユーザーの流儀へ過学習させてしまう。

### ステップ40　全自動化パイプラインのエンドツーエンド本番デプロイ
**目標**：五大モジュールにまたがるプラットフォーム全体を、単一コマンドの GitOps オーケストレーションデプロイへ収束させる。**技術スタック／ツール**：shell デプロイドライバ、venv/conda 隔離環境、Redis キャッシュコンテナ、systemd service、GitHub Actions/GitOps。**中核実装**：shell デプロイヤを記述し、システム依存をインストールし、隔離 Python 環境を構築し、Redis キャッシュクラスタのコンテナを立ち上げる。蒸留ストリーマ、テレメトリ watchdog、API ルーティングゲートウェイ、データ収穫パイプラインをネイティブ systemd バックグラウンドサービス（`distill_system.service`）として登録する。起動シーケンスのチェックを埋め込む——起動時に GB10 のメモリ可用量を検証し、ステップ10 の健全性チェックを実行し、UMA メモリロックの証憑を確認してから、API エンドポイントを完全にオンラインにする。最後に live proxy ポート、アーキテクチャエンドポイント、ストレージディレクトリを含むクリーンなターミナルレポートを印字する（`deploy_pipeline.sh`）。**検証（Assertions）**：クリーンな Ubuntu/DGX OS 上で playbook が退出コード 0 で完了し、すべてのコアマイクロサービスが service table で active と検証されることを確認する。**解説**：systemd で 4 つの常駐コンポーネントをサービス化＋起動時自己診断するのは、単機オーケストレーションの堅実な締めくくりだ。ただし shell + systemd は宣言的ロールバックを欠く。GitOps を名乗る以上はバージョン管理されたデプロイと失敗時の自動ロールバックを補うべきで、さもなくば「単一コマンドで本番化」も「単一コマンドで全滅」になる。
