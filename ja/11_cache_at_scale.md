## 場面1　三段階異種ストレージのピラミッド：KV Cache を OS の仮想記憶として管理する

**背景**：動もすれば百万トークン級のコンテキストを支えるモデルにおいて、全ユーザーの KV Cache をすべて GPU/TPU の HBM（高帯域 VRAM）にデッドロックさせてしまえば、クラスタは並行数が少し上がっただけで OOM により瞬時に麻痺する。HBM はシステム全体で最速かつ最も高価で最も希少な資源だ——それを「三日帰ってこないユーザーの対話履歴」を保存するのに使うのは、犯罪的なまでに贅沢な浪費である。原素材が主張する解法の本質は、OS が五十年間管理してきた古い手——**仮想記憶と多段キャッシュ**——をそっくりそのまま KV Cache に持ち込むことだ。すなわち、三段階の異種ストレージピラミッド（Hierarchical KV Cache Management）を築く。

三段階のピラミッドは、速いものから遅いものへ、高価なものから安価なものへ、小さいものから大きいものへ:

```
            ┌───────────────────────────────────────────┐
            │  L1：TPU/GPU HBM  (High-Bandwidth Memory)  │  ← 最速·最高価·最小
            │  何を置く：いま Decode 中の Active Segments  │
            │  容量規模：数十 GB / カード  帯域：~TB/s       │
            └───────────────────────────────────────────┘
                          ▲   100 GB/s ~ TB/s
                          │   PCIe Gen5 / NVLink / ICI
                          ▼
            ┌───────────────────────────────────────────┐
            │  L2：Host Memory  (CPU DDR5)                │  ← 中速·中価·中量
            │  何を置く：当面不活発だがすぐ使うかもの多輪履歴│
            │  容量規模：数百 GB ~ TB / ノード              │
            └───────────────────────────────────────────┘
                          ▲   Direct I/O
                          │   NVMe-over-Fabrics / RDMA
                          ▼
            ┌───────────────────────────────────────────┐
            │  L3：Distributed NVMe SSD Pool             │  ← 遅い·安い·大容量
            │  何を置く：冷データ（長尺マルチモーダル/冷凍Session）│
            │  容量規模：数十 TB ~ PB / プール              │
            └───────────────────────────────────────────┘
```

対応関係は一目瞭然だ。**KV Cache における L1 は、主記憶における CPU L1 キャッシュにあたり、L1 における L3 は、RAM における HDD にあたる**。ユーザーがまさにタイプしていて、モデルがトークンを一つずつ生成している間、その KV は L1 になければならない。一言を打ち終えて「人間が次の一言を考えている」数秒の空白に入ったら、その KV を HBM から L2 の DDR5 へ退避(Offload)させ、HBM は即座に空いて別の人にサービスする。三十分帰ってこなければ、Session 全体を L3 の SSD プールへ沈めて冷凍する。

**鍵となる技術：非同期プリフェッチのパイプライン（Prefetching Pipeline）**。ピラミッド自体は難しくない。難しいのは「運搬」というこの一件だ——HBM と Host Memory の間には PCIe が横たわり、一度に数 GB の KV を運ぶには時間がかかる。運搬の間に計算が止まって待つなら、この階層化は負の最適化になる——VRAM を節約したが、レイテンシで損をするのだ。原素材の解法は、コンパイラ（XLA）と実行時のなかで、**計算スレッドと I/O スレッドを徹底的に分離**し、古くて強力な一語で運搬コストを覆い隠すことだ——**重ね合わせ(Overlapping)**:

```python
# 概念図示：「他人の decode 時間」を使って「自分の KV 運搬レイテンシ」を覆い隠す
# 実システムは XLA runtime / CUDA stream 層で非同期 DMA により実装。ここはスケジューリングの意味論を示すのみ

while serving:
    batch = scheduler.next_batch()          # decode 中のリクエストを一束取る

    # ── I/O スレッド（計算と並行、ブロックしない）──────────────────
    if detect_user_typing_next_turn(user_X):
        # ユーザー X が次の一言を入力中だと検知 → 彼の履歴 KV を前もって
        # L2 DDR5 から L1 HBM へ非同期プリフェッチ。この瞬間、計算スレッドは何も知らず、待つ必要もない
        io_thread.async_prefetch(user_X.kv, src="DDR5", dst="HBM")

    # ── 計算スレッド（現在の batch を満載で走らせ続ける）────────────────
    compute_thread.decode_step(batch)        # このステップの所要時間が、上の運搬をちょうど「覆い隠す」

    # ユーザー X が本当に送信を押すときには、彼の KV はとっくに HBM に横たわっている → 体感ゼロのレイテンシ
```

精髄はあのコメントにある——**他のリクエストがまさに実行中の decode 計算時間を使って、現在のリクエストの KV Cache の運搬レイテンシを「覆い隠す」**。大バッチ推論では、常に大量のリクエストが decode 中で、TPU の Tensor Core を満腹に食わせている。I/O エンジンはこの計算の陰に潜み、次に使う KV をこっそり DDR5 から HBM へ運び戻す。ユーザーが本当に Enter を押すころには、彼の履歴はとっくに HBM で待っている——レイテンシは、ハードウェアがもともと忙しくしていたその時間のなかに隠され、ハードウェア利用率（MFU）は常に高位で走り続ける。

> 💡 ひとこと助言
> 君に一言贈ろう。**「システムを速くする」あらゆる妙手を底まで分解すると、ほとんど同じ一句に行き着く——高価な資源に安価な資源を待たせるな。運搬が避けられないなら、それを計算の陰に隠せ。** 三段階ピラミッドは AI の新発明などではない。OS の教科書第三章の仮想記憶であり、CPU の L1/L2/L3 であり、データベースの buffer pool であり、「KV Cache」という衣装に着替えてもう一度歩いてきただけだ。本当に値打ちがあるのは「三段に分ける」という構造ではない——それは常識だ。値打ちがあるのは**非同期プリフェッチのあの一手「他人の計算時間を使って自分の運搬時間を覆い隠す」**である。新人が階層キャッシュを作ると、「退避も復帰も止まって待つ」同期版になり、結果として VRAM を節約してレイテンシで損をする、割に合わないものができあがる。熟練者が階層化を作るとき、最初に考えるのは「運搬のレイテンシを、どうせ費やすことになる時間の後ろに隠せないか」だ。**これが overlapping の哲学だ。システム工学の最高境地は、すべてを速くすることではなく、遅いものを、誰もそれを待っていないときに起こさせることにある。** 頭のなかで「計算スレッド」と「I/O スレッド」という二本の独立した時間軸を同時に回し、正しい点で交差させられるか——この一念の差が、「キャッシュを使える者」と「キャッシュを調達できる者」を分ける。

> 🔍 掘り下げ解説 — なぜ「KV Cache のページング」は「OS のページング」より凶険で、しかも値打ちがあるのか
> KV Cache を仮想記憶のページングとして扱う方向は絶対に正しい。だが OS のページングとの**三つの決定的な違い**を冷静に見ておかねば、丸写しすれば必ず転覆する。**その一、粒度と意味が違う**。OS のページングの単位は固定の 4KB ページで、中身に意味はない。KV Cache の単位は「あるトークン列の注意力状態」で、強烈な**時系列のプレフィックス依存**を持つ——履歴の途中だけを退避させることはできない。後ろのトークンを計算するとき、前のトークンが見えなければならないからだ。だから KV の退避単位は通常「プレフィックス丸ごと一段」か vLLM 式の固定 block であり、しかもプレフィックスが連続していなければならない。**その二、コスト構造が逆**だ。OS のページングが退避するのは「データ」で、戻せば元どおりだ。KV Cache は、退避せず捨てる選択をすれば、復元するには**prefill を再計算**せねばならない——そして prefill は compute-bound の重労働だ。これは「DDR5 へ退避する」と「いっそ捨てて再計算する」の間に精算問題があることを意味する。PCIe を一往復する I/O コストと、prefill をもう一度走らせる演算力コスト、どちらが安いか。列が長いほど再計算は高くつき、退避して運ぶべきだ。列が短いほど再計算は安く、捨てたほうが手間が省ける。この精算こそ、場面3「投機的キャッシュ移送」のミクロ版だ。**その三、これは vLLM がすでに製品化した方向**だ。vLLM の CPU offloading、`--swap-space`、そしてコミュニティが進める KV 階層化が、まさにこのピラミッドの L1↔L2 の実装である。さらに L3 SSD へ向かえば、Mooncake（Kimi の背後にある KVCache 中心の分離型アーキテクチャ）のような最前線システムの領域に入る。**だからこの節は空中楼閣ではない。すでに生産ラインで走っている方向を、その物理まで明快に説いたものだ。** 素材に対して割り引くべき唯一の点は、「体感ゼロのレイテンシ」は理想状態だということ——ユーザーがプリフェッチより速く返してきたり、プリフェッチが人を読み違えたりすれば、運搬の尻尾が露わになる。真にゼロのレイテンシなど存在せず、あるのは「隠し方が十分か否か」のレイテンシだけだ。

## 場面2　キャッシュに引き算をする：MHA から MLA へ、モデル構造層のダイエット術

**背景**：場面1は純粋な工学だ——モデルをいじらず、外側にピラミッドを建てて運ぶだけ。だが原素材の一言がその天井を見抜く——**「純粋な工学最適化には上限がある。必ずアルゴリズムチームと手を組み（Co-design）、モデル構造の側から Cache をダイエットさせねばならない。」** これは全章で最も重要な方法論の一句だ。KV Cache が大きいのは、根っこが Attention の構造にある——各層、各注意力ヘッドが、トークンごとに K を一つ、V を一つキャッシュせねばならない。源泉から削るには、Attention そのものに手を入れねばならない。この進化の道——MHA → MQA → GQA → MLA——は、ここ数年の大規模モデル推論効率において最も重要な一本の暗線だ。

まず KV Cache の体積の公式をはっきりさせよう。古典的な **Multi-Head Attention（MHA）** では、トークンごとにキャッシュすべき KV の大きさは次に比例する。`2（K と V）× L（層数）× H（ヘッド数）× D（ヘッドごとの次元）`。そのうちの **H（ヘッド数）** こそ、歴代の改良が繰り返しメスを入れてきた対象だ:

```
MHA  ：各 Q ヘッドが自分の K ヘッド・V ヘッドを持つ      KV ヘッド数 = H      （基準·最大）
        Q1 Q2 Q3 Q4 ... QH
        K1 K2 K3 K4 ... KH
        V1 V2 V3 V4 ... VH

MQA  ：すべての Q ヘッドが唯一の一組の K、V を共有       KV ヘッド数 = 1      （H 倍節約·能力を損なう）
        Q1 Q2 Q3 Q4 ... QH
        K（一つだけ、皆で共用）
        V（一つだけ、皆で共用）

GQA  ：Q ヘッドをグループ分けし、各組が一対の K、V を共有  KV ヘッド数 = G（組数、例 8）（業界主流）
        [Q1 Q2 Q3 Q4][Q5 Q6 Q7 Q8]...
         K_grp1        K_grp2
         V_grp1        V_grp2

MLA  ：高次元 K/V を保存せず、低ランク圧縮した潜在ベクトル c_t を保存  キャッシュ = 極小の latent ベクトル（復号時にオンライン展開）
        K, V ──低ランク射影──► c_t（次元は極小）   復号時 ──線形射影──► K, V を復元
```

このダイエットの道のトレードオフを一つずつ分解する:

| 機構 | KV Cache 相対サイズ | 中核のやり方 | 代償 / 現状 |
|---|---|---|---|
| **MHA** | 1×（基準） | 各 Q ヘッドが一対の KV ヘッドを専有 | 表現力は最強だが、KV 体積は最大、長コンテキストで最も VRAM を食う |
| **MQA** | ~1/H | すべての Q ヘッドが**唯一の一組の KV を共有** | VRAM が H 倍激減するが、表現力の低下が大きすぎ、大規模モデルの能力が深刻に損なわれる |
| **GQA** | ~G/H（例 8/H） | Q ヘッドを**グループ分け**し、各組が一対の KV を共有（Llama 系は 8 組をよく使う） | 速度と効果の折衷、**現在の業界主流**、Llama 3 などに広く採用 |
| **MLA** | 90% 以上の節約と称する | **低ランク圧縮**：K/V を低次元の潜在空間 c_t へ射影し、復号時にオンライン展開 | DeepSeek が提唱（V2/V3 が代表作）、表現力を保ちつつ KV を大幅縮小。アルゴリズム-基盤の協調設計の手本 |

MQA と GQA は「KV ヘッドの数を減らす」——MQA は一刀で一組だけに削り、激しすぎてモデルが馬鹿になる。GQA は賢く学び、数組（たとえば 8 組）まで削って、「VRAM の節約」と「能力の保持」の間にスイートスポットを見つけた。こうして Llama 3 のような主流オープンソースモデルの標準装備となった。

**MLA（Multi-head Latent Attention）** はまったく別の道を行く。「ヘッド数」で引き算をするのではなく、「次元」で圧縮をする。その洞察はこうだ——高次元の K と V には大量の冗長が潜んでおり、**低ランク行列分解（Low-rank Compression）** を使って、それらを次元が極小の潜在空間ベクトル `c_t` へ射影し圧縮できる。キャッシュするときは、この小さな `c_t` だけを保存し、元のあの大きな高次元 K、V は保存しない。復号段階に至って、線形射影行列で **`c_t` をオンライン(on-the-fly)で展開**し、本物の K と V に戻して注意力を計算する。本来トークンごとに `2 × L × H × D` を保存すべきところ、いまは小さなベクトル一つで済む。

> ⚠️ 真偽メモ
> 「MLA は KV Cache を 90% 以上削減し、並行数を数倍に引き上げる」というこの数字は DeepSeek の論文と技術報告に由来し、**特定モデル・特定構成下の測定値**であって、普遍定数ではない——圧縮比は潜在空間次元の選び方、モデル規模、列長に強く依存する。MLA は DeepSeek が提唱しそのオープンソースモデルで検証した、検証可能な実在の仕事だ。だが「DeepMind も類似方向を深く追っている」という一句は原素材の内部口吻の**主張**であり、本書は独立に検証できない。方向性の参考にとどめてほしい。平易に述べられる事実は——MQA / GQA / MLA はいずれも発表済みで、生産級モデルに採用された実在の技術であり、この「Attention の KV をダイエットさせる」進化線は本物だ、ということだ。

## 場面3　クラスタ分散：キャッシュに「位置」が生まれると、ルーティングは戦争になる

**背景**：単機で PagedAttention や Prefix Caching を極めても、まだ足りない。いったん管理対象が数千数万の TPU/GPU からなる超大規模推論クラスタになれば、ゲートウェイ（Gateway）とスケジューラは新しい能力へ進化せねばならない——**キャッシュ認識(Cache-Aware)** だ。クラスタにおいてキャッシュはもはや抽象的な「VRAM 空間」ではなく、**ある具体的な物理マシンの上に生えている**からだ。ユーザー A の履歴 KV を持っているのが、ノード 7 なのかノード 42 なのか——この一件が、次のリクエストをどこへ送るべきかを決める。これこそ本章冒頭のあの「電気代が灰になる」災害の根源だ。

**伝統的スケジューリング（Round-Robin / Least-Connection）の災害**:

```
ユーザー A が 5 句を連投、伝統的ロードバランサは「最も暇」か「ラウンドロビン」で分配する:

  msg#1 ──► ノード1   (A の全履歴 prefill を再計算)
  msg#2 ──► ノード2   (ノード2 に A のキャッシュなし → また全履歴を再計算)
  msg#3 ──► ノード3   (もう一度再計算)
  msg#4 ──► ノード1   (ノード1 のキャッシュは他人に押し出された可能性 → やはり再計算)
  msg#5 ──► ノード4   (再計算)

結果：同じ「五万字の履歴」の prefill が 5 回計算された。
      各マシンで作った Prefix Cache のヒット率は全部ゼロに帰す——
      次の一言が、前の一言のあったマシンに戻ってこないからだ。
```

ロードバランサの本能は「仕事を最も暇なマシンに振る」——これはステートレスな Web サービスでは金科玉条だ。だが LLM 推論は**ステートレスではない**——リクエスト N の安さは、「リクエスト N-1 の KV がまだこのマシンに載っている」ことの上に成り立つ。ステートを持つキャッシュを、ステートレスを前提とするスケジューラに渡すのは、全クラスタの演算力を「キャッシュを間違った位置に置いた」という死角の道連れにするに等しい。

**正解：プレフィックスハッシュ + アフィニティルーティング + 投機的キャッシュ移送**。ゲートウェイはグローバルな分散キャッシュインデックス（通常はインメモリDB か高効率 RPC で維持）を持たねばならない。三段構えだ:

| 手順 | 機構 | やり方 |
|---|---|---|
| ① 識別 | **Prefix Hashing（プレフィックスハッシュ）** | ゲートウェイがリクエストの `System Prompt + 履歴対話` についてプレフィックスハッシュを計算（**MurmurHash3** など、速くて分布が均一）。同じプレフィックス → 同じハッシュ |
| ② ルーティング | **Affinity Routing（アフィニティルーティング）** | 「最も暇な」ノードへ投げるのではなく、**すでにそのプレフィックスの KV をキャッシュしている物理ノード**を優先して投げ、クラスタ全体の Prompt Cache ヒット率を最大化する |
| ③ 比較衡量 | **Speculative Cache Transfer（投機的キャッシュ移送）** | キャッシュを持つノード A がすでに過負荷なら：「暇なノード B で prefill を再計算する」vs「クラスタの高速ネットワークで KV を A から B へ直接引き込む」を精算し、速いほうを選ぶ |

第三手はこの設計全体で最も精緻な一手だ。その精算問題を分解して見る価値がある:

```
状況：ユーザーのリクエストがノード A のキャッシュにヒットするが、ノード A はこの瞬間過負荷（キューが長い）。
スケジューラの前には二つの道:

  経路甲【再計算】：リクエストを暇なノード B に送り、B にゼロから全履歴の prefill を再計算させる
       コスト ≈ Prefill_FLOPs(履歴長) / B の演算力
       —— 列が長いほどこの道は高くつく（prefill は compute-bound）

  経路乙【移送】：A 上のすでに計算済みの KV を、高速ネットワーク RDMA で B へ直接運ぶ
       コスト ≈ KV_Bytes(履歴長) / クラスタネットワーク帯域
       —— 帯域が十分強ければ、出来合いを運ぶほうが、ゼロから計算するよりはるかに速い

  決定：if  移送コスト(乙) < 再計算コスト(甲):  RDMA 移送を選び、全履歴の prefill 演算力を節約
        else:                          損切りして B で再計算
```

この問題が場面1の「KV ページング vs 再計算」と**同じ問題を別のスケールで再演している**ことに注意。場面1は「隣の DDR5 へ運ぶ」対「捨てて再計算」、ここは「マシンをまたいでノード B へ運ぶ」対「B で再計算」だ。物理的直観は一致する——**prefill の演算力コストは列長に対して超線形に増えるが、出来合いの KV を運ぶコストはそのバイト数に対して線形にしか増えない**。だから列が長く、コンテキストが大きいほど、「出来合いを運ぶ」が割に合う——これこそ長コンテキストの時代に、クラスタ内のキャッシュ移送ネットワークが演算力と同じくらい重要になる理由だ。

> ⚠️ 真偽メモ
> 「Google Jupiter ネットワーク 1.6 Tbps」「キャッシュの直接転送が顕著な TTI 優位をもたらす」といった具体的な帯域数字と内部ネットワークの符号は、原素材が DeepMind 内部の口吻で与えた**主張**であり、本書は独立に検証できない。桁の目安として扱ってほしい。平易に述べられる実在技術は——**アフィニティルーティング + プレフィックスハッシュは、SGLang の RadixAttention、そして各社の推論ゲートウェイ（vLLM の prefix-aware スケジューリング、Mooncake のグローバル KV インデックスなど）が進めている本物の方向**だ。MurmurHash3 は実在の非暗号ハッシュであり、RDMA でノード間 KV を運ぶのも PD 分離（場面4）アーキテクチャの実在の機構だ。割り引くべきは「どの社のどのネットワークのどの帯域」という製品まで具体化した数字だけである。

> 🔍 掘り下げ解説 — アフィニティルーティングの暗礁：キャッシュアフィニティと負荷分散は、生まれながらに反目しあう双子だ
> アフィニティルーティングは完璧に聞こえるが、生産ラインでは君を一つの古典的な**ジレンマ**へ引きずり込む。これを見抜かねば、一つの穴から別の穴へ飛び込むことになる。**負荷分散が求めるのは「仕事を分散させる」こと、キャッシュアフィニティが求めるのは「仕事を、キャッシュを持つあのマシンに集中させる」こと——この二つの目標は物理的に真っ向から相殺する。** ユーザー A の全リクエストをノード 7 に厳格に釘付けにする（キャッシュヒット率を満タンに）ほど、**ホットスポット**を作りやすくなる——狂ったように呼び出される長い system prompt（たとえば爆発的にバズった Agent テンプレート）が、それにヒットする全トラフィックを最初にキャッシュしたあの二、三台へ集中投下し、それらを焦がし尽くす一方、クラスタの他のマシンは眠っている。これこそ成熟した cache-aware スケジューラが決して「脳死アフィニティ」ではなく、**負荷認識を伴うアフィニティ**である理由だ——SGLang のスケジューリング、そして各社ゲートウェイのやり方は、本質的に「キャッシュヒットの利得」と「ノード負荷のコスト」の間で動的にスコアリングしている。ヒットはもちろん良いが、保持者がすでに過負荷なら、場面3の第③手のあの「移送 vs 再計算」の精算を発動し、能動的にトラフィックを逸らすかキャッシュを二台目へ複製する。**さらに一層深く、これは一つのアーキテクチャ哲学を絞り出す——ステートを持つ推論クラスタには、「最適ルーティング」という静的な答えなど根本的に存在せず、あるのは「キャッシュアフィニティ、ノード負荷、移送コストの三者の間で持続的に再均衡をとる」動的なプロセスだけだ。** それを一度計算して終わるハッシュ関数として扱えば、君はホットスポットで死ぬ。それを持続的に調節する制御ループとして扱えば、君はようやく呼吸できるクラスタを本当に手にする。これは分散システム永遠の母題でもある——一貫性（ステートを固定位置に釘付け）と可用性／負荷（トラフィックを広げる）の間で、永遠に綱引きが続く——キャッシュ認識ルーティングは、この綱引きが LLM 時代に至った最新の戦場にすぎない。

## 場面4　PD 分離：Prefill と Decode のハードウェア要求が真逆になるとき

**背景**：キャッシュ認識ルーティングを極限まで推し進めると、より深い構造的矛盾に突き当たる。その解法こそ、現在の超大規模推論基盤の「究極の進化方向」だ——**PD 分離（Disaggregated Prefill & Decode）**。なぜそれが終局なのかを理解するには、まず多くの人が見落としている事実を見抜く必要がある——LLM 推論の二つの段階、**Prefill と Decode は、物理的性質がまったく逆の二種類の負荷**であり、それらを同じカード、同じ batch に押し込むのは、相性最悪の二人を同じ部屋に同居させ、互いに足を引っ張らせるようなものだ。

二段階の対照:

| 観点 | Prefill（プレフィル：prompt を消化） | Decode（デコード：トークンを逐次生成） |
|---|---|---|
| 計算特性 | **Compute-bound（演算力律速）** | **Memory-bound（メモリ律速）** |
| 何をしている | prompt 全体を一気に通し、KV Cache を産出 | 毎回 1 トークンだけ生成、巨大な KV を繰り返し読む |
| 何の資源を食う | **演算力**（Tensor Core 満載）、帯域 | **VRAM 容量**（全並行 KV を載せる）、メモリ帯域 |
| 理想ハードウェア | 高演算力・高帯域チップ（HBM3e など） | 大容量 VRAM |
| 痛点 | 長い prompt が来ると**演算力を独占**し、decode 中のリクエストを餓死させる | 演算力が大量に遊び、ひたすら KV を運ぶだけ |

矛盾はここにある——**Prefill に足りないのは演算力、Decode に足りないのは VRAM 容量。Prefill が演算力を絞り尽くすとき、Decode の演算力は遊んでいる。Decode が VRAM を埋め尽くすとき、Prefill の VRAM 需要が圧迫される。** この二つが同じ GPU batch に混ざると、最も見苦しいことが起こる——五万トークンの長い prompt が prefill に入り、瞬時に全演算力を独占すると、逐字回答中の数十の短いリクエストが一斉に固まり、それらの **TTFT（最初のトークンのレイテンシ）と TPOT（トークン間レイテンシ）が一緒に跳ね上がる**。ユーザーの目には「打っている途中で突然つっかえた」と映る。

業界はこの矛盾に二層の解法を持つ。一つは軽く、一つは重い:

```
─── 同機の緩和：Chunked Prefill（分割プレフィル）───────────────────
  長い prompt の prefill を固定サイズの chunk（毎回 512 トークンなど）に切り、
  一塊を計算 → 演算力を decode に譲って一歩走らせ → 次の塊を計算。
  「長い prefill を一口で飲み込む」を「小口でパイプライン化」に変え、長リクエストが短リクエストを餓死させるのを防ぐ。
  ↑ vLLM / DeepSpeed-FastGen の Dynamic SplitFuse がこれをやっている。「同じカード上の時間スライス」に属す。

─── 跨機の根治：PD Disaggregation（PD 分離）─────────────────
  いっそクラスタを二種類の専用ノードに割る:

   ┌────────────────────┐                      ┌─────────────────────┐
   │  Prefill ノード (P)  │   RDMA 直押し KV     │  Decode ノード (D)    │
   │  高演算力·高帯域      │ ───────────────────► │  大容量 VRAM          │
   │  (HBM3e など)       │   計算済みの KV Cache  │  逐トークン生成に専念  │
   │  prompt 消化に専念   │   を高速網で D へ送る  │  prefill に中断されない│
   └────────────────────┘                      └─────────────────────┘

  P ノードが KV を計算し終え、超高速 RDMA ネットワークで KV 全体を D ノードへ「押し」、
  D ノードは受け取って decode に専念する。二種類のハードウェアが各々の職に専念し、互いに干渉しない。
```

Chunked Prefill は**同じカード上の時間スライス**——緩和であり、対症療法だ。PD 分離は**マシンをまたぐ空間隔離**——根治であり、根本療法だ。二段階のハードウェア要求が逆である以上、最も徹底した解法は、それらを同じカード上で交互に我慢させることではなく、**各々に最も適したハードウェア**を当てがうことだ——prefill ノードは演算力を積み、decode ノードは VRAM 容量を積み、間を RDMA で KV を一押しして済ます。

さらにコンパイラ底層の一手があり、PD 分離と相補的だ——**XLA の演算子融合（Kernel Fusion）**:

```
未融合（汎用フレームワークの素朴なやり方）:
  PagedAttention アドレス計算 ──HBM へ書戻し──► RoPE 回転位置エンコーディング ──HBM へ書戻し──► Softmax
  └ 各ステップの中間結果を HBM へ書き戻してまた読み出す。HBM 往復はレイテンシと帯域の殺し屋

融合後（手作業最適化の単一 TPU Kernel）:
  ┌─────────────────────────────────────────────────┐
  │  PagedAttention + RoPE + Softmax を一つの演算子に融合 │
  │  中間変数はレジスタに留め、HBM へ書き戻さず、レジスタ級のデータ再利用 │
  └─────────────────────────────────────────────────┘

もう一手：固定 block size（16 / 32 トークン一塊など）→ 「動的な列長」を
        「静的な block 数列の組み合わせ」に変える → 動的形状が XLA の頻繁な再コンパイルを誘発するのを回避
```

二つの底層ディテールは同じ一件を指している——**不要な HBM 往復を消す、動的形状を消す**。演算子融合は attention のアドレス計算、RoPE、Softmax の三ステップを一つの kernel に揉み込み、中間結果をレジスタに留めて HBM を往復させない。固定 block size は「列長が動的に変わる」というコンパイラが最も嫌うものを、「固定サイズ block の配列」という静的で予測可能な形状に変え、コンパイル後の機械語を最高効率で走らせ、新しい長さが来るたびにコストの高い再コンパイルを誘発させない。

> ⚠️ 真偽メモ
> 「PD 分離は Google DeepMind / OpenAI の基盤層における究極の進化方向だ」は原素材の内部視点の主張だ。検証可能な事実は、PD 分離が実在しかつ盛んな研究・工学の方向だということ（DistServe、Splitwise、Mooncake などの公開された仕事が進めている）であり、vLLM もすでに disaggregated serving に対応している。「Prefill は compute-bound、Decode は memory-bound」は広く測定検証された実在の特性だ。割り引くべきは「某々トップ企業の内部ですでに全面採用」「某々具体的帯域」といった外部検証不能な断言だ——それを「業界公認の最前線の方向」として扱い、「某社の既成現実」としては扱わないこと。

> 💡 ひとこと助言
> 君に一言贈ろう。**あるシステムをどう調整しても調整しきれないとき、十中八九それはパラメータが合っていないのではなく、君が本性の逆な二つのものを無理やり同じ箱に押し込んでいるからだ——本当の解法はしばしば、より賢く同居させることではなく、相性が悪いと認めて、箱を二つに割ることにある。** Prefill と Decode の物語は、この知恵の最も純粋な実例だ。多くのチームが「同じカード上で prefill と decode を平和に同居させる」という一件に心血を注ぐ——batch を調整し、優先度を調整し、chunked prefill をますます細かく切る——本質はすべて、不幸な結婚のなかで夫婦調停をしているのだ。Chunked Prefill は巧みな調停だが、「一方は演算力を、一方は VRAM を求める」という根本的な不一致を変えることはできない。PD 分離の洞察は乱暴なまでに素朴だ——**お前たちが求めるものが逆なら、もう一緒に住むな。** これは分散システムでは「関心の分離」という荘厳な名を持つが、その最難の点は決して「分ける」という動作ではなく、**「一緒にすること自体が間違いだ」という判断を下す勇気**にある——なぜなら「一緒にする」のは通常、歴史的しがらみであり、「これまでずっとそうしてきた」であり、手間を省くことだからだ。**アーキテクトの成熟度は、どれだけ複雑なものを一つに揉み込めるかではなく、皆がある結合に慣れきった後で、立ち上がって『この二つは根本的に一緒にあるべきではない』と言い、移送コストを背負ってそれを割れるかどうかにある。** 不一致を見抜くことは、同居を最適化することより、はるかに難しく、はるかに値打ちがある。

## 場面5　二十大最前線の軍械庫：攻略小隊ごとにクラスタリングした一枚の作戦地図

**背景**：前の四つの場面は、素材のいくつかの主線を説き切ったものだ。だが原素材が最後に広げて見せるのは、より大きなもの——大規模モデルキャッシュ領域の**二十の最前線の方向**だ。トップ会議（NeurIPS / ICML / ICLR / MLSys）の最新論文、産業界の生産環境のオープンソースの神器、そしてトップクラスのアーキテクチャ方法論にまたがる。素材はこの二十の方向を、「どの攻略小隊が打つべきか」で三層に分けた——**1–6 アルゴリズム層**（アルゴリズム工学者がモデルを魔改造）、**7–12 インフラ層**（infra チームがエンジン選定と負荷試験）、**13–20 アーキテクチャ方法論層**（アーキテクトがシステムの HLD を書くとき全体配置する）。以下、この軍械庫を一枚の作戦地図に整理する——それはまさに本書付録G のより完全なキャッシュ技術早見表の種子だ。

**【アルゴリズム層 1–6】——アルゴリズム / 研究工学者の担当：モデル構造と KV そのものにメスを入れる**

| # | 方向 | 一言で技術の核心 | 代表 |
|---|---|---|---|
| 1 | **MLA** | 低ランク圧縮で K/V を低次元の潜在空間 c_t へ射影し、復号時にオンライン展開、KV 体積を大削減（場面2 参照） | DeepSeek-V2/V3 |
| 2 | **H2O（Heavy-Hitter Oracle）** | Attention 行列は**冪則分布**を呈し、少数の Token が圧倒的多数の重みを担う。これらの Heavy-Hitter の KV を動的に識別して保持し、残りを即時に切り捨てる → 定長キャッシュに無限長の文を載せる | 《H2O》論文 |
| 3 | **StreamingLLM** | **Attention Sink** 効果を発見（最初の 2〜4 個の Token が巨大な注意力を独占）。「最初の数トークン + 最新のスライディングウィンドウ」を保持すれば、prefill を再計算せずストリーミングで無限に復号できる | 《Attention Sinks》論文 |
| 4 | **Speculative Decoding + Tree-KV** | 投機的復号（小モデルが草稿を出し、大モデルが検証）は多分岐を生む。**木構造の KV Cache** で多分岐を並列検証し、多経路の重複計算を回避 | Medusa 系列 |
| 5 | **Infinite-LLM / Mamba-2 混合** | ウィンドウを超えた古い KV を**固定サイズの循環状態（Recurrent State）** へ圧縮し、空間計算量 O(N) → O(1) | Griffin / Hawk / Mamba-2 |
| 6 | **KV-Quant（低ビット量子化）** | Per-Channel/Per-Token 混合量子化 + 非一様量子化 + 外れ値保持で、KV を **2-bit/3-bit でほぼ精度を落とさず**圧縮 | 《KV-Quant》論文 |

この六つの方向には共通点がある——いずれも「**KV Cache そのものを、もっと小さく、もっと賢くできないか**」を問うている——MLA は次元を圧縮し、H2O と動的淘汰は無用なトークンを捨て、StreamingLLM は頭と尾だけ残し、Mamba 系は履歴を固定状態に揉み込み、KV-Quant はビットを下げる。1、2、3、5 の間には一本の暗線があることに注意——**いずれも「注意力のロングテール」と格闘している**——H2O は「大半のトークンは重要でない、捨てろ」と言い、StreamingLLM は「最初の数個と最近のものが最も重要、残せ」と言い、Mamba は「いっそ過去を一つの状態に圧縮しろ」と言う。三つの哲学が、同じ一つの問いに答える——**無限に長い履歴を前に、結局何を覚えておくべきで、何を忘れていいのか**。

**【インフラ層 7–12】——Infra チームの担当：推論エンジンの選定と負荷試験**

| # | ツール | 必殺技 | 適用シーン |
|---|---|---|---|
| 7 | **vLLM** | **PagedAttention** の元祖、VRAM 断片化を解決。Chunked Prefill / Prefix Caching / テンソル並列 | 業界のゴールドスタンダード、汎用の第一選択 |
| 8 | **SGLang** | **RadixAttention**——Prompt キャッシュを自動化された**基数木（Radix Tree）** にする | 複雑な制御フロー、Few-shot、Tool-use の多輪 Agent。ヒット率とスループットが強い |
| 9 | **LMDeploy** | KV Cache 量子化（W4A16、KV INT4/INT8）への対応が極めて完璧 + Persistent RPC | エッジ端 / オンプレの VRAM 制約シーンで限界まで絞る |
| 10 | **TensorRT-LLM** | In-flight Batching、FlashDecoding+、バイト級の VRAM 制御 | NVIDIA 一式、Triton と組んだ多ノードクラスタ |
| 11 | **LightLLM** | **Token Attention**——Token 級の細粒度 VRAM スケジューリング | 超大並行、リクエスト長が極端に不均一な過酷な環境、OOM 耐性 |
| 12 | **DeepSpeed-FastGen** | **Dynamic SplitFuse**——prefill と decode を同 batch 内で動的に織り交ぜて分割 | マイクロソフト系、二段階同居のキャッシュライフサイクルを最適化 |

この六つは「すぐ使える既製の兵器」だ——選定の鍵は「どれが最強か」ではなく、「**君の負荷がどんな形か**」だ——汎用なら vLLM、多輪 Agent や prompt の高度な再利用なら SGLang の RadixAttention（それはまさに場面3 のアフィニティルーティングの単機版だ）、VRAM 制約で極限量子化が要るなら LMDeploy、NVIDIA 一式に縛られているなら TensorRT-LLM。

**【アーキテクチャ方法論層 13–20】——アーキテクト / Tech Lead の担当：HLD を書くときの戦略配置**

| # | 方法論 | 実装の要点 |
|---|---|---|
| 13 | **Cache-Aware Routing** | ゲートウェイが MurmurHash3 でプレフィックスハッシュを計算し、同プレフィックスのリクエストをその KV を持つノードへルーティングし、クラスタのヒット率を最大化（場面3 参照） |
| 14 | **Tiered Cache Orchestration** | HBM(L1)/DDR5(L2)/NVMe(L3) の三段階、バックグラウンドスレッドで Offload/Prefetch（場面1 参照） |
| 15 | **Chunked Prefill** | 長い prefill を固定 chunk に切り、decode とパイプライン化、長リクエストが短リクエストを餓死させるのを防ぐ（場面4 参照） |
| 16 | **Dynamic KV Eviction** | もはや LRU で履歴を丸ごと乱暴に捨てない。**注意力の重要度**でスコアリングし、**句読点、機能語、つなぎ語を優先的に捨て**、名詞と中核的な意味実体を残す——「損失はあるが高知能」な回収 |
| 17 | **Deterministic Prompt Formatting** | 「**静的を前に、動的を後に**」。Tools 定義 / System Prompt / Few-shot を固定順序でシリアライズ。**prompt の冒頭に UUID、ランダムソルト、タイムスタンプを詰め込むのは絶対禁止**（さもなくばプレフィックスが永遠にヒットしない） |
| 18 | **Exact Prefix vs Semantic Cache** | Exact：トークンが完全一致なら RadixTree で KV を再利用。Semantic：**GPTCache + Milvus/Qdrant** でベクトル類似度を計算し、**>0.98 なら前回の答えを直接返してインターセプト、コストはゼロ** |
| 19 | **PD Disaggregation** | Prefill ノードが KV を計算し終え、RDMA で Decode ノードへ直接押す（場面4 参照） |
| 20 | **Context Cache Monetization** | **Anthropic Claude の `cache_control`** を参考に：コード内でキャッシュアンカーを明示的に宣言、大量の長文（法律契約、コードベース）タスクの入力費用が大幅減 |

この八つはアーキテクトの「全局の一手」だ。なかでも第 16、17 の二つは、最も安く、最もよく見落とされ、なのに収益が最も高いので、個別に名指しする価値がある。**第 17 の決定論的プロンプト整形は、ほぼゼロコストの純粋な規律だ**——モデルを一行も変える必要も、カードを一枚買う必要もなく、チームの開発規約に「prompt の冒頭は永遠に静的な内容、動的なものは最後に置く」という一条を書き付けるだけで、プレフィックスキャッシュのヒット率は「あってないも同然」から「常態的にヒット」へ変わる。逆に、不注意な工学者が一人、system prompt の冒頭に `request_id=<uuid>` や `現在時刻：...` を詰め込んだだけで、**チームが精緻に建設した prefix cache を全線無効化させる**に十分だ——プレフィックスの最初のトークンが変わってしまえば、後ろが同じでも無意味だからだ。**第 18 の意味キャッシュは、別次元のコスト削減の核兵器だ**——exact cache が節約するのは「同じプレフィックスの再計算」、semantic cache が節約するのは「意味の近い問いに、そもそもモデルを呼ばない」ことだ——GPTCache はベクトルデータベースで類似度を計算し、0.98 を超えれば前回の答えを直接返す。この一回の推論のコストはゼロだ。

> ⚠️ 真偽メモ
> 表中の数字は段階的に見るべきだ。**Claude `cache_control` は Anthropic が実際に提供している API 機能であり、prompt caching は確かに重複する長いプレフィックスの入力費用を大幅に下げられる**——だが「コスト 80% 削減」はキャッシュヒット率とプレフィックス比率に依存するシーン値であって、保証値ではない（実際には君がどれだけキャッシュを再利用したかによるし、キャッシュの読み取り自体にも課金がある）。「類似度 >0.98 で直接返す」の閾値は工学的経験値で、緩すぎると見当外れのキャッシュを返す——意味キャッシュは諸刃の剣で、ヒット率と正確率は自分で衡量せねばならない。SGLang の「ヒット率とスループットが vLLM を数倍超える」は特定ベンチマークの結果で、負荷を変えれば成り立つとは限らない。**この二十の方向自体はすべて実在の論文、ツール、方法論だ。割り引くべきは永遠に、一桁まで精密なパーセンテージのほうだ。**

> 🔍 掘り下げ解説 — この地図の最大の価値は、二十の技ではなく、あの「誰が打つべきか」というクラスタリングの線だ
> 二十の方向を暗記するのは助手工学者の腕前だ。それらが**なぜ 1–6 / 7–12 / 13–20 の三層に分けられたのか**を見抜くことこそ、アーキテクトの視野だ。このクラスタリングの線には、組織と技術についての深い判断が潜んでいる——**この三層が必要とするのは、三種類の根本的に異なる人間が、三種類の根本的に異なるものに手を入れ、三種類の根本的に異なるリスクを背負うことだ。** アルゴリズム層（1–6）が手を入れるのはモデルの数学的構造で、間違えればモデルは精度を落とし、馬鹿になる。リスクは「能力」にある。だから attention の数学を理解し、大規模なアブレーション実験を走らせられる研究工学者が背負わねばならない——infra 工学者がどれだけ強くても、MLA の潜在空間次元に手を出す勇気はない。インフラ層（7–12）が手を入れるのはエンジン選定と負荷試験で、間違えればシステムは OOM し、スループットが崩壊する。リスクは「安定性とコスト」にあり、必要なのは CUDA kernel を読めて、ストレステストができて、P99 のテールレイテンシを見られるシステム工学者だ。アーキテクチャ方法論層（13–20）が手を入れるのはサービスをまたぐ全体配置で、間違えればクラスタ全体が「キャッシュを間違った位置に置いた」「prompt 形式が不確定」といった**単一障害点はないのに、至るところで漏電している**問題で慢性的に失血する。必要なのは、ルーティング、階層化、形式規約、商業課金をホワイトボード上で同時に見渡せるアーキテクトだ。**最も致命的なミスマッチは、間違った層の人間に間違った戦いを打たせることだ**——アーキテクトに MLA の低ランク次元を調整させる（彼はアブレーションを知らない）、あるいはアルゴリズム工学者にクラスタルーティングを設計させる（彼はホットスポットと負荷を知らない）——どちらも事故る。**そして本当にトップクラスの技術 leader の価値は、まさにこのクラスタリングの線の上にある——彼は二十件を自ら完遂する必要はない。彼に必要なのは『この方向はモデルの問題か、システムの問題か、それともアーキテクチャの問題か』を正確に判断し、それを正しい小隊に渡し、三つの小隊の成果を co-design のホワイトボード上で揃えること（場面2 のあの塀）だ。** 一件を誰がやるべきか知ることは、どうやるか知ることより、より高次で、より希少な能力だ——この二十の方向の階層化は、技術リストというより、むしろ**組織がいかに認知を配分するかの地図**なのだ。

## 本章のまとめ

- **核心の転換**：単機では、キャッシュは「節約すべき VRAM」。クラスタでは、キャッシュは「調達すべき戦略資産」——位置があり、アフィニティがあり、移送コストがあり、演算力と同じく SRE を夜中に叩き起こすものだ。
- **場面1　三段階異種ピラミッド**：L1 HBM（active decode）/ L2 DDR5（当面不活発な多輪履歴）/ L3 NVMe（冷データ）、OS の仮想記憶を KV Cache に持ち込む。魂は**非同期プリフェッチ + Overlapping**——他人の decode 時間で自分の運搬レイテンシを覆い隠す。
- **場面2　構造層のダイエット**：MHA → MQA（KV を 1 組に削り、能力を損ないすぎ）→ GQA（グループ共有、業界主流、Llama 3）→ MLA（DeepSeek の低ランク圧縮で潜在空間 c_t へ、オンライン展開、KV 大削減）。鍵は**アルゴリズム-基盤の co-design**——二つのチームの間の塀を取り壊すこと。
- **場面3　キャッシュ認識ルーティング**：Round-Robin はステートを持つ推論では災害（同ユーザーが多ノードに散らばり、prefill を再計算）。正解 = プレフィックスハッシュ（MurmurHash3）+ アフィニティルーティング + 投機的キャッシュ移送（再計算 vs RDMA 引き込みの精算）。暗礁：**キャッシュアフィニティと負荷分散は生まれながらに反目する**、負荷認識を伴う動的な再均衡が要る。
- **場面4　PD 分離**：Prefill は compute-bound、Decode は memory-bound、ハードウェア要求が逆。Chunked Prefill は同機の時間スライス（対症療法）、**PD 分離は跨機の空間隔離（根本療法）**——両者の不一致を認め、二種類の専用ノードに割り、RDMA で KV を押す。XLA の演算子融合（HBM 往復を消す）+ 固定 block size（動的形状の再コンパイルを消す）を併用。
- **場面5　二十大軍械庫**：アルゴリズム層 1–6（MLA / H2O / StreamingLLM / Tree-KV / Mamba 混合 / KV-Quant）は研究工学者の担当。インフラ層 7–12（vLLM / SGLang / LMDeploy / TensorRT-LLM / LightLLM / DeepSpeed-FastGen）は infra の担当。アーキテクチャ層 13–20（ルーティング / 階層化 / Chunked Prefill / 動的淘汰 / 決定論的整形 / 意味キャッシュ / PD 分離 / Claude `cache_control` 商業化）はアーキテクトの担当。**クラスタリングの線そのものがリストより値打ちがある**——一件を誰がやるべきか知ることは、どうやるか知ることより希少だ。
- **真偽の規律**：MLA、H2O、StreamingLLM、KV-Quant、vLLM、SGLang、Claude `cache_control` などはすべて実在の仕事だ。割り引くべきは永遠に「90% 削減、コスト 80% 削減、数倍超え、Jupiter 1.6 Tbps」といった精密な数字と、「某トップ研究所が内部でどうやっているか」の主張だ。

この二十の方向は種子にすぎない。それらを、前の十一章の各所に散らばったキャッシュの技——第3章の UMA ゼロコピー、第5章の KV Cache FP8 量子化、本章の三段階ピラミッドと MLA まで——一枚の手元でいつでも引ける早見表にまとめあげること、それこそ**付録G「キャッシュ技術早見表」**がやることだ。中核概念、VRAM 圧縮、推論エンジン、アーキテクチャ方法論、底層演算子、意味キャッシュ、ヒット率の落とし穴回避、さらに「90% 削減、コスト 80% 削減、Jupiter 1.6 Tbps」といった数字を個別に釘付けにする真偽校正区を一枚加えて。

## 終章：三つの楽章、一本の背骨

軍械庫が扉を閉じ、本書もまた終わりに辿り着いた。この旅を振り返ると、それは実のところ一つの問いを、三度問うただけだった。

第一楽章、我々は「**それをどう造るか**」を問うた——あの反蒸留の盾の下で、合法的に、君だけのもので、君の机の上で走るローカルモデルを蒸し出す。第二楽章、我々は「**それをどう使うか**」を問うた——この小さく安価なモデルに眼と手を生やし、人間のために設計された一台のコンピュータを見て、操作させ、さらに顔を上げてこの技芸が立つ範式そのものを問い質す。第三楽章、我々は「**それをどう養うか**」を問うた——この眼と手が一つの demo から万級の並行へ拡大し、あの最も目に見えないメモリとキャッシュの戦いを打つとき、「廉価」というこの約束を規模の上で空手形にしないために。

造る、使う、養う。この三件が根本的に分けられないことに君は気づくだろう——君が第三楽章で死力を尽くして争う一 GB ごとの KV Cache を節約するのは、すべて第一楽章のあの DGX Spark に、第二楽章のあの決して閉じない眼を支えさせるためだ。**これは三つの主題を釘で打ち付けた合輯ではなく、同じ一つの工学的理想——「真に君のもので、君に従い、臓器にできるほど安く、養えるほど強い知能」——を三つの角度から徹底的に問い詰めたものだ。**

全書が繰り返し磨いたのは、実のところ二つのものだけだ。一本の**真偽の物差し**——あの一連の魅惑的な名前（Mythos、Qwythos、Aluminium OS、Jupiter 1.6 Tbps、78.4 点）を前に、本能的に「どれが自分で検証できるもので、どれが他人が信じさせたがっているものか」を見分ける——この一線が、考える工学者と、復唱する伝声管とを分ける。一条の**潔白な道**——技術は元来中性で、分水嶺は意図と許諾にある。使用権のあるモデルを蒸留し、合法のデータを使い、自分のマシンで養うのは工学だ。他人の盾を回避して影を一つ盗むのは攻撃だ。

技術は時代遅れになる。この本のなかの一つ一つの型番、一つ一つの点数、一つ一つの発表会の名前は、来年には誰も覚えていないかもしれない。だがあの物差しとあの道は、どんなモデルよりも長く生きるだろう。
