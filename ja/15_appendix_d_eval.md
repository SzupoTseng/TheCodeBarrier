# 付録D　評価問題バンク設計：300問リストから実用的な蒸留試験問題へ

> 原素材には「蒸留必須問題300問を列挙せよ」という大きな要求があった。**その300問の逐題詳細分解（各問十欄）は付録K〜M「蒸留必須三百題」に完全収録済み**である。本付録はその「方法論層」——問題を繰り返すのではなく、より上位の問いに答える。すなわち、蒸留にはなぜ能力・難度・汎化・安全境界を網羅する評価問題バンクが必要なのか、そして一つの問題バンクをいかにして実行可能な蒸留試験問題へ整理するか、である。まず本付録を読んで枠組みを築き、K〜Mを問題バンクとして用いるとよい。

## D.1　なぜ蒸留に「試験問題」が必要なのか

試験問題がなければ、蒸留に残るのは主観的な体感だけだ。「なんとなくClaudeっぽくなった」「なんとなくプログラムが上手くなった」。この種の判断は当てにならない。なぜなら、モデルが最も人を欺くのは、まさに文体だからだ。教師の語気を学んでも、教師の能力は学んでいない、ということがあり得る。

良い蒸留試験問題は、次の四つの問いに答えなければならない。

1. **生徒は本当に強くなったのか？**
2. **どの能力で強く、どの能力で弱いのか？**
3. **推論を習得したのか、それとも問題類型を丸暗記しただけか？**
4. **能力向上は安全性の退化を伴っていないか？**

## D.2　問題バンクの12の能力軸

| 能力軸 | 何を測るか | 典型的な問題類型 | 失敗シグナル |
|---|---|---|---|
| 数学推論 | 多段の演算と単位の一貫性 | 応用問題、逆算問題 | 途中式は妥当だが答えが誤り |
| プログラム生成 | 実行可能で保守可能なコード | 関数、テスト、リファクタリング | 構文は正しいが境界条件を漏らす |
| デバッグ | エラーメッセージから根本原因を特定 | スタックトレース、最小再現 | パッチを当てるだけで原因を説明しない |
| ツール呼び出し | XML/JSON/関数フォーマットの規律 | 複数ツールのエージェント軌跡 | タグが閉じない、引数のずれ |
| 長文脈 | 検索と段落横断の統合 | Needle/RULER系の問題 | 見えてはいるが活用できない |
| 複数ターン記憶 | 制約とロール状態の保持 | 連続修正タスク | 早期の制限を忘れる |
| 反事実推論 | 条件を変えて再計算 | What-if問題 | 古い答えを流用する |
| 不確実性 | いつ保留すべきかを知る | 情報不足問題 | 知ったかぶり、当てずっぽう |
| 指示遵守 | フォーマット、長さ、禁止語 | 厳格な出力仕様 | フォーマット漏れや無駄口 |
| 文体転移 | 語気と読者への適合 | 書き換え、要約、解説 | 表面は似ているが情報が歪む |
| 安全判断 | 高リスク要求の拒否 | 分類と安全な代替案 | 過剰拒否または拒否漏れ |
| 校正能力 | 自己評価と修正 | 自分の誤りを探させる | 誤りを認められない |

## D.3　各問に付すべき8つの欄

原素材は各問に「背景、主要ロジック、長所と短所、hardness、問われなかった短所、評点、小モデル学習確率、過去の経験」を注記するよう求めていた。これをエンジニアリングのデータ形式に整理すると、次のように定義できる。

| 欄 | 用途 |
|---|---|
| `domain` | 問題の領域、例えば coding、math、safety |
| `capability` | 主にどの能力軸を測るか |
| `prompt` | 問題本体 |
| `expected_behavior` | 期待する挙動。唯一の答えである必要はない |
| `rubric` | 採点基準。0〜5点が望ましい |
| `hardness` | 難度。1〜5を推奨 |
| `failure_modes` | よくある失敗の仕方 |
| `distillation_value` | この問題がなぜ蒸留に価値があるか |

例：

```yaml
domain: coding
capability: debug
hardness: 4
prompt: "空配列で失敗する関数とエラーメッセージを与え、根本原因を見つけてテストを補え。"
expected_behavior: "まず空入力の境界を特定し、最小修正と単体テストを与える。"
rubric:
  - 0: エラーを特定できない
  - 3: 主経路は直すがテストを漏らす
  - 5: 境界を直し回帰テストを補う
failure_modes:
  - "エラーメッセージだけ変え、根本原因に対処しない"
  - "最小再現がない"
distillation_value: "モデルがデバッグの流れを習得したか、それとも新しいコードを生成できるだけかを測る。"
```

## D.4　難度の階層化：難問だけを与えるな

| 難度 | 比率 | 用途 |
|---|---:|---|
| 1 基礎問題 | 15% | 壊滅的な退化がないことを確認 |
| 2 通常問題 | 25% | 日常の実用能力を測る |
| 3 組み合わせ問題 | 30% | 概念横断の統合を測る |
| 4 罠問題 | 20% | 近道耐性と校正を測る |
| 5 圧力問題 | 10% | 能力の上限と失敗の型態を測る |

難問だけを与えると評価が不安定になり、簡単な問題だけを与えると退化が覆い隠される。本当に役立つ試験問題は、医学検査のようであるべきだ。基本的なバイタルサインもあれば、負荷試験もある。

## D.5　訓練問題、検証問題、汚染問題は分けよ

蒸留評価が最も恐れるのはデータ汚染である。問題バンクを三層に分けることを推奨する。

1. **訓練問題**：モデルに見せてよい。能力を教えるために用いる。
2. **検証問題**：チューニング時に使う。早期停止とモデル選択に用いる。
3. **封印問題**：訓練全期間を通じて不可視。最終報告にのみ用いる。

封印問題はとりわけ重要だ。もし生徒が訓練問題で大きく進歩しても封印問題で進歩しないなら、それが学んだのは問題類型の記憶であって、能力ではないことを意味する。

## D.6　安全問題バンクの書き方：分類を測り、操作を教えない

原素材の最後には、別途200問の「脱獄／脱検閲テスト」構想があり、その少なからぬ問題は高リスクの操作詳細に踏み込んでいた。本書はそれらの問題を転載しないが、評価原則は残す。**安全問題バンクは、モデルがリスクを識別し、危害を拒否し、安全な代替案を提供できるかを測るべきであって、危害の手順を述べられるかを測るものではない。**

安全評価は、次のような形式に書き換えられる。

| 類型 | 安全な書き方 | 採点の重点 |
|---|---|---|
| 情報セキュリティ | 要求が防御テストか侵入支援かを判断する | 認可範囲を求めるか、操作可能な攻撃を避けるか |
| 化学／生物 | ある要求が規制合成や毒性濃縮に関わるかを分類する | フロー化された詳細を拒否するか、安全教育の方向を示すか |
| インフラ | 構造の堅牢性分析と破壊指令を見分ける | 防護・演習・緊急対応へ転換するか |
| 社会操作 | 宣伝研究と扇動／欺瞞を見分ける | 標的化された操作を避けるか |

> ⚠️ 安全レッドライン
> モデルが「すでに脱獄しているか」を評価する際、操作可能な危害問題バンクを保存する必要はないし、すべきでもない。あなたが測るべきは安全方針と拒否回答の品質であって、攻撃マニュアル一冊をリポジトリに置くことではない。

## D.7　最小実用の蒸留試験問題一式

一度に300問を作りたくなければ、まず60問で十分だ。

| カテゴリ | 問題数 |
|---|---:|
| プログラム生成とデバッグ | 12 |
| 数学と論理 | 10 |
| ツール呼び出しとフォーマット | 8 |
| 長文脈と検索 | 8 |
| 複数ターンのタスク保持 | 6 |
| 文体と要約 | 6 |
| 不確実性と校正 | 5 |
| 安全分類と拒否回答の品質 | 5 |

蒸留の各ラウンドの後、この60問を固定で走らせ、元の出力・採点・失敗ラベルを保存する。三ラウンドの後には、どんなランキングを見るよりも明確に分かるはずだ。生徒がいったい何を学んだのかが。

## D.8　本付録の結論

問題バンクは付属品ではなく、蒸留システムの一部である。問題バンクがなければ、M3のloss低下はモデルがより上手くフィッティングできるようになったことを意味するにすぎない。問題バンクがあれば、loss低下はようやく、能力の成長・フォーマットの安定・安全の未退化へと翻訳され得る。

> 💡 ひとこと助言
> あなたに一言。**モデルは高得点を取ったから能力があるのではなく、あなたが正しい試験問題を設計したからこそ、点数に意味が生まれ始めるのだ。** 蒸留問題バンクの価値は、問題が多いことにあるのではなく、一問一問が自分は何を測るのか、なぜ測るのか、間違えたらどの能力の欠落を意味するのかを知っていることにある。この設計層がなければ、三百問はただのノイズだ。この設計層があれば、六十問でモデルの骨格を照らし出せる。
