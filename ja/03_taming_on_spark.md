# 第三部　馴龍場

# 第3章　DGX Spark で龍を馴らす：ハードウェアの Roofline と選定

> シリコンバレー、麗臺（Leadtek）代理店の出荷伝票に並ぶ、その一台の **NVIDIA DGX Spark** には目を疑う価格が付いている。Mac mini ほどの小ささでありながら、「個人向け AI スーパーコンピュータ」を自称する。ファンが回り始めたその瞬間、本当に問うべきは「どれだけ大きなモデルを動かせるか」ではなく、もっと鋭いひと言だ——**「この 128GB に高い割増金を払った以上、いったい何を動かせば宝の持ち腐れにならないのか？」**

> 本章は技術的な含金量が最も高く、最も論拠が堅い。GB10、統合メモリ（UMA）、GGUF、Roofline はいずれも本物の工学だ。選定の結論は一種のよくある推論だが、その裏にある**帯域幅の論理は検証可能なハードウェアの常識**である——まさに前章の「神話的宣伝」を突き崩す物差しだ。

## 場面1　この怪物を知る：Roofline でその気性を見抜く

**背景**：ハードウェアは **NVIDIA DGX Spark（GB10 チップ、データセンター級の GB200/GB100 ではない）**。まず仕様を釘付けにしておく。なぜなら**選定の一歩一歩は、すべてこれらの数字から導かれる**からだ。

| 仕様 | 数値 | 確信度 |
|---|---|---|
| チップ | GB10 Grace-Blackwell スーパーチップ（20 コア Arm CPU + Blackwell iGPU）| 🟢 公式 |
| メモリ | **128GB LPDDR5X 統合メモリ（UMA）**、CPU と iGPU が物理的に同一プールを共有・PCIe を経由しない | 🟢 公式 |
| メモリ帯域幅 | **~273 GB/s**（LPDDR5X クラス）| ⚠️ よく見る公称値；バス幅から算出した理論ピークは ~301 GB/s とする説もあり、**精確な定説とみなすな** |
| 演算力 | ~1 PFLOP **FP4（NVFP4）** スパースピーク（第 5 世代 Tensor Core）| ⚠️ 公式宣伝の「スパース FP4」の数字、密／実測ではかなり低くなる |
| 量子化ネイティブ | Tensor Core が **FP4 / NVFP4 / FP8 / BF16** をネイティブ対応 | 🟢 アーキテクチャ特性 |

**多くの二次的記事が「600 GB/s」あるいはそれ以上と書いている帯域幅——あれは間違いだ。** DGX Spark が使うのは LPDDR5X であって、データセンターカードの HBM3e ではない。その帯域幅の桁は「2、300 GB/s」であって、HBM の「TB/s」ではない。**この数字を倍ほど間違えると、以下のあらゆる tok/s 推定が丸ごと倍ずれる**ので、まず ~273 GB/s クラスへ校正しておく。NVFP4 については——Blackwell が導入した 4-bit 浮動小数点フォーマット（E2M1 ＋ 二段のマイクロブロックスケーリング）で、ブロック単位の scale によって 4-bit の品質を FP8 に迫るところまで強引に引き上げる。これこそ後の選定で Q4 を使う度胸の根拠だ（詳細は第 4 章の蒸留の数学に譲る）。

**問題**：その気性を知らなければ、「4GB の VRAM で Claude を動かす」といった類の主張に振り回され、最も値打ちのある本領をみすみす無駄にする。その気性を見抜く道具こそ、エンジニアなら必修のモデル——**Roofline（ルーフライン）**だ。

**LLM 推論の二つの段階は、それぞれ別の壁にぶつかる**：

- **プレフィル（Prefill、入力の処理）**：プロンプト全体の数百から数千の token を一気に流し込んで大きな行列積を行う——同じ重みがこのバッチ全体の token に繰り返し使われるため、**演算力律速（compute-bound）**で、効くのは FLOPS だ。
- **デコード（Decode、token を逐次生成）**：自己回帰で、一度に一つの token しか生成しない。一つ生成するたびに、**演算に関わる重みをメモリから丸ごと一度読み出す**必要があり、その重みは「この一つ」の token のためにしか働かない——読み込んで、計算して、捨てる。**メモリ帯域幅律速（memory-bandwidth-bound）**で、効くのは GB/s だ。

**なぜ prefill は演算力の壁、decode は帯域幅の壁にぶつかるのか？** ひと言で言えば、**算術強度（Arithmetic Intensity、重み 1 byte を読むごとに何回の浮動小数点演算に換えられるか）が天と地ほど違う**からだ。Prefill は一度に N 個の token を処理し、重みを一度読めば N 個の token に共用される。算術強度は N に比例して拡大し、難なく「演算力律速」領域へ突入する。Decode は batch=1 で、重みを一度読んでも一つの token の積和しか計算しない——算術強度は理論的最低（パラメータあたり約 2 FLOP）に近づき、GPU の兆級の演算力は大半が遊び、ボトルネックは「重みをメモリから運び出す」この一段にがっちり固定される。**これこそ、DGX Spark で一対一の対話を動かすと、GPU 使用率がしばしば 2、3 割しかないのにすでに最高速度である理由だ——力不足ではなく、メモリを待っているのだ。**

**ここから、速度を直接推定できる鉄則が導かれる**：

```
デコード速度 (tok/s) ≈  メモリ帯域幅 (GB/s)  ÷  token あたりに読む必要のある重み (GB)
```

DGX Spark の ~273 GB/s を当てはめる（これは帯域幅の「天井」で、アテンションの KV 読み出しとフレームワークのオーバーヘッドを差し引くと、実測は通常さらに 6〜8 割になる）：

- **70B 密モデル @ Q4_K_M**（token ごとに**全部**の ~40GB の重みを一度読む）：273 ÷ 40 ≈ **理論 ~6.8 tok/s、実測 ~5 tok/s**——一文字ずつ絞り出すようで、使い物にならない。
- **120B-A4 MoE @ Q4**（token ごとに「活性化した ~4 個のエキスパート ~2〜3GB ＋ 各層常駐のアテンション/共有重み ~3〜4GB」≈ 6GB だけ読む）：273 ÷ 6 ≈ **理論 ~45 tok/s、実測 35〜50 tok/s**——快適だ。

からくりが見えただろうか？ **同じ一台のマシンで、70B 密を動かすと ~5 tok/s で詰まり、「総パラメータがより大きい」120B MoE を動かすと逆に約 9 倍速い**——違いは総パラメータ数ではなく、「token ごとに本当に何 byte を運ぶか」にある。**この一本の式さえあれば、ダウンロードボタンを押す前にどれだけ速いかを計算できる**。70GB のダウンロードを終えてから後悔する必要はない。

> 🔍 掘り下げ解説 — Roofline の「リッジポイント」：一つの数字でマシンの性格を見抜く
> Roofline モデルの精髄はすべて、一つの座標に凝縮できる——**リッジポイント（Ridge Point）＝ ピーク演算力 ÷ ピーク帯域幅**、単位は「FLOP/byte」だ。これはマシンの「性格の分水嶺」である。負荷の算術強度がこれより**高ければ**、ぶつかるのは**演算力の壁**（屋根の水平部分）；**低ければ**、ぶつかるのは**帯域幅の壁**（屋根の斜面部分）だ。DGX Spark を代入する：~1 PFLOP(FP4) ÷ ~273 GB/s ≈ **~3600 FLOP/byte**。これが意味するのは、**算術強度が 3600 を下回るあらゆる負荷は、帯域幅に詰まらされる**ということだ。では decode の算術強度はいくつか？ batch=1 のとき約 **2〜4 FLOP/byte**——リッジポイントより**まるまる三桁**低い。言い換えれば、**LLM の token 逐次生成というこの作業は、このマシン（そしてすべての UMA / Apple Silicon）の上では、帯域幅の壁に組み伏せられて床に擦りつけられる極端なメモリ律速の負荷**であり、あの 1 PFLOP の演算力はそれにとってほぼ飾りだ。これを理解すれば、三つの事柄の根がすべて分かる。①**なぜ高速化は「token あたりの読み出し量を減らす」（量子化、MoE、KV Cache 量子化）に頼るべきで、「より強い演算力を買う」ではないのか**；②**なぜ batch を上げる（複数のリクエストを同時に処理する）と総スループットがほぼ線形に向上するのか**——複数の token が同じ一回の重み読み出しを共用し、算術強度をリッジポイントへ押し上げるからだ（これが vLLM の連続バッチ処理の価値でもある）；③**なぜ投機的デコード（場面3）が有効なのか**——大モデルに「一度重みを読んで、複数のドラフト token を並列に検証させる」ことで、本質的には算術強度を引き上げているのだ。**リッジポイントは一本の物差しであり、君のボトルネックが演算力なのか帯域幅なのかを測り出す。そして個人の LLM 推論にとって、その答えはほとんど常に帯域幅だ。**

> 🧠 脳に焼き付けるべきハードウェアの鉄則
> **容量が「動かせるかどうか」を決め、帯域幅が「どれだけ速く動くか」を決める。そして MoE の「活性化パラメータ」こそ、帯域幅の請求書に載る本当の数字だ。** DGX Spark は「容量の巨人、帯域幅の凡人」である。128GB のおかげで、他者が呑み込めない大モデルを呑み込めるが、~273 GB/s の LPDDR5X は、ディスクリート GPU がしばしば持つ 1TB/s（GDDR7）どころか ~8TB/s（HBM3e）の帯域幅より一桁遅い。だからその最適戦略は永遠に「**大容量で、他者が動かせない大 MoE を動かす**」ことであって、「小モデルでどちらが速いかを競う」ことではない。

**この鉄則がいかに前章の神話を突き崩すか**：DGX Spark で 9B モデルを動かせば、**当然** 180+ tok/s まで吹き上げられる——だがそれは無駄だ。9B なら二万元台の M チップ Mac でも十分に動く。君が 128GB を買ったのは容量のためであって、小モデルをより速くするためではない。

> 💡 ひとこと助言
> 君と一席を交わそう：**ハードウェアを選ぶこととモデルを選ぶことは、本質的に同じ問題だ——君のボトルネック資源の上で最大の産出を求めること。** DGX Spark のボトルネックは容量ではなく（むしろ過剰だ）、帯域幅だ（これが稀少だ）。稀少な資源を「活性化パラメータが小さく、総パラメータが大きい」MoE に与えることは、過剰な容量で帯域幅の不足を補うに等しい——これこそ Roofline 思考の精髄だ。**仕様表の中で最も大きな数字に惑わされるな、システムの中で最も小さな一枚の壁を見つけ出し、君の負荷をその壁にぴたりと沿わせて走らせよ。** これが分かれば、君が選ぶのはもはやモデルだけではなく、システム全体の最適動作点だ。

## 場面2　六つの影の横並び比較：誰がこの怪物にふさわしいか

**背景**：DGX Spark GB10 に対して、前章のモデルを以下のように横並びで比較する（Q4_K_M 量子化下で評価）。

| モデル | メモリ占有 | DGX Spark 予想速度 | 強み | 弱み |
|---|---|---|---|---|
| **GPT-OSS 120B Fable-5**（MoE）| ~72GB | 35〜50 tok/s | 128 エキスパート MoE、複雑な多段コードと数学 | 体積が大きく、コンテキスト >32K で明らかに遅くなる |
| **Qwable-v1**（35B）| ~24GB | ~102 tok/s | Fable 5 の Agent 軌跡を完璧に継承、Coding エージェントに最適 | 知識の広さは 120B に劣る |
| **clzoro/Qwen3.5-27B** | ~19GB | 110+ tok/s | Opus の対話で育成、文学/ロールプレイが極めて強い | ツール呼び出しは Qwable-v1 に及ばない |
| **Qwen3.6-14B-FableVibes** | ~11GB | 140+ tok/s | 剪定 MoE、「Claude らしい思考感」を保持 | 複雑なロジックでときおり反芻（幻覚）|
| **Qwythos-9B**（abliterated 含む）| ~6.5GB | 180+ tok/s | 超高速、徹底した無検閲、ボロい GPU でも動く | パラメータが小さすぎ、深い推論の完全性に欠ける |
| **Dzluck/Qwen3.5-2B** | ~2.1GB | 250+ tok/s | 極限の軽量、エッジ機器向け設計 | 役不足、知能の低下が明らか |

> ⚠️ 真偽メモ
> 速度とメモリの数字は**いずれも実測を経ていない推定値/巷で流れている値**だ。相対関係は Roofline の直感に合致する（パラメータが小さいほど速い、MoE は同サイズの密より速い、量子化が激しいほど占有が小さい）ので、「合理的な推定」として扱えるが「実測基準」ではない。コミュニティには実際に流れている参照点が一つある：「Solved the DGX Spark, 102 stable tok/s on Qwen3.5-35B-A3B」（付録 A 参照）——表中の Qwable-v1（35B-A3B 同級）の 102 tok/s と一致する。なぜなら 3B 活性化 @ Q4 ≈ 1.7GB、273 ÷ 1.7 ≈ 160 が理論上限で、実測 102 は合理的な割引の範囲に収まるからだ。**だが同じ物差しが、表の破綻も照らし出す**：Qwythos-9B は 180+ tok/s と表示——9B 密 @ Q4 ≈ 5.4GB、273 ÷ 5.4 ≈ **50 tok/s こそが物理的上限**であり、180 はすでに 3 倍超、**単一リクエストの decode では到達不可能だ。高並列バッチの総スループットか、表自体の水増しでもない限り。** これはまさに本章の核心的規律を示している：**いかなる tok/s の数字も、まず「帯域幅 ÷ token あたり読み出し量」というこの物差しで一度測れ。物理上限を越えられないものは、一律に疑え。**

**表を眺めるだけで終わるな——「入るかどうか」を自分で計算する術を学べ**：表の「メモリ占有」は重みだけだ。本当に 128GB に収まるのは三つのものの積み重ねであり、一つでも計算し損ねると長文時に OOM する。まず公式を暗記せよ：

```
総占有 = 重み + KV Cache + フレームワーク/活性化オーバーヘッド(~2〜4GB)

重み (GB) ≈ パラメータ量(B) × パラメータあたりビット ÷ 8
   BF16/FP16 : 16 bit → ×2.00      FP8 / Q8_0 : 8 bit → ×1.00 (Q8_0 実 ~8.5 bit)
   Q4_K_M    : ~4.8 bit → ×0.60     NVFP4/Q4   : 4 bit → ×0.50

KV Cache (GB) ≈ 2 × 層数 × KVヘッド数 × ヘッド次元 × コンテキスト長 × 要素あたりバイト ÷ 1e9
   ★ ×2 = K と V それぞれ一份
   ★ GQA は「KVヘッド数」をアテンションヘッド数より遥かに小さくする（例：8 vs 64）、KV 節約の命脈
   ★ 経験値：GQA-8 + FP16 KV、1K token あたり約 0.3〜0.5GB（層数による）
```

四つの代表的構成を 128GB に代入して実際に計算する（⚠️ 数字は目安、実際の層数/GQA 比率で変動）：

| モデル @ 量子化 | 重み | KV @ 32K | KV @ 256K | 128GB の結論 |
|---|---|---|---|---|
| **70B dense Q4_K_M**（GQA）| ~40GB | ~10GB | ~80GB | 32K は余裕(~50GB)；**256K=120GB はほぼ破綻、KV 量子化なしには不可** |
| **120B-A4 MoE Q4** | ~72GB | ~5GB | ~40GB | 32K は楽々；256K≈112GB は逼迫、KV FP8 で救える |
| **35B-A3B Q8_0**（GQA）| ~37GB | ~6GB | ~48GB | 単体は余裕；大モデルと二枚同時起動には長さ制御が必要 |
| **8B dense Q4** | ~5GB | ~2GB | ~16GB | 好きに動かせるが、128GB でこれを動かすのは役不足 |

その二列の KV をじっと見よ——**重みは一度きりの固定オーバーヘッドだが、KV Cache はコンテキスト長に伴って「線形に膨張」する。** 32K ではほとんど感じないが、256K まで引くと、KV は重みに迫る、いや重み以上のメモリを食いつぶしかねない。**これこそ「70B が入る」と「70B が入った上で 256K の長文も動かせる」が全く別の問題である理由だ**——後者の本当のメモリ殺し屋は、決してモデルの重みではなく、KV Cache だ。（この伏線、第 10・11 章で丸々一章を使ってその借りを返す。）

**よくある選定の結論**：**まず選ぶべきは GPT-OSS 120B Fable-5 か Qwable-v1（35B）**——「128GB で大モデルを動かさないのはもったいなすぎる」。Roofline と上記の fit 算式で検算すると、この結論は成り立つ：

- **GPT-OSS 120B-A4 は「知能担当」**：120B 総パラがそれに知識の深さを与えるが、token ごとに ~4 エキスパート（約 20〜30B が実際に参与）しか活性化しないため、帯域幅の死穴を回避し、35〜50 tok/s は許容範囲だ。**これは個人機器で「70B 級の知能上限」を突破する唯一の現実的な道だ。**
- **Qwable-v1（35B）は「速度担当」**：24GB の占有は 128GB に対して何の負担もなく、102 tok/s はほぼ「瞬時噴射」、しかもネイティブで `str_replace_editor` XML を吐き、OpenHands/Aider との接続にゼロ変換——Coding Agent としての効率は 120B より数倍高い。

> 🔍 掘り下げ解説──MoE はなぜ帯域幅律速マシンの救世主なのか、そしてその隠れた代償
> これは本章で最も値打ちのある洞察だ：**いかなる帯域幅律速のハードウェアでも（DGX Spark、Apple Silicon、すべての UMA アーキテクチャ）、「活性化パラメータ量」は「総パラメータ量」より遥かに重要だ。** 一つの 120B-A4 は、走らせると 20〜30B 密モデルに近い速度なのに、知能は 120B に近い——これこそ 2026 年に「個人スパコン」向けのオープンウェイトモデルがほぼ全て MoE に賭けた理由だ。だが MoE はタダ飯ではなく、三つの隠れた代償があり、選定時には併せて天秤にかけねばならない。**その一、メモリ税**：MoE は**全 128 個のエキスパート**をメモリに常駐させる必要がある（次の token がどれにルーティングされるか分からないから）。だから「節約するのは帯域幅、費やすのは容量」——これはちょうど DGX Spark の「容量過剰、帯域幅稀少」という特性と相補的で、まさに天作の妙だが、小 VRAM の機器ではかえって災難となる。**その二、ロジックの反芻**：エキスパート間のルーティング切り替えが、長鎖推論に不連続を生じさせうる（表中 FableVibes の「複雑なロジックでときおり反芻」はまさにこれが原因）。これが、clzoro の**密 27B** が「思考の連続性」でかえって文学/ロールプレイの場面に好まれる理由でもある。**その三、バッチ効率**：MoE の高並列バッチ下でのエキスパート負荷の不均衡は、本番運用の古い問題だ。**だから選定の完全な心得はこうだ：「いくつ活性化するか」を問う（速度を決める）、「全部でいくつか」を問う（知能を決める）、「密か MoE か」を問う（連続性とメモリ形態を決める）——この三問を経て、初めて正しく選ぶ話ができる。**

## 場面3　最も賢い遊び方：二モデル並行と、一つの上級隠し玉

**背景**：128GB があるなら、より賢い遊び方は「一つを選ぶ」ではなく、**二モデル並行**だ——大小一つずつ同時にロードし、タスクに応じて切り替える。

**参考に値する二モデル構成**：

- **同時ロード**：GPT-OSS 120B（~72GB）+ Qwable-v1 35B（~24GB）= ~96GB、残り ~32GB を両者の **KV Cache** に。
- **バックエンド**：**Ollama** に動的マルチモデル管理が内蔵。端末二つか API を並行で：`ollama run gpt-oss:120b` / `ollama run qwable:35b`。
- **フロントエンド**：**Open WebUI / Cursor**、プルダウンでワンクリック切り替え、さらには「**二モデル対決（Arena）**」——両モデルが同じ問いに答え、君が最良を選ぶ。
- **ホットスワップ（hot-swap）**：大モデルを動かすとき、遊休モデルを LPDDR5X から自動的に解放し、超長コンテキスト用に空きを作る。~273 GB/s の UMA 帯域幅と「ゼロコピー、PCIe を経由しない」特性のおかげで、入れ替えはディスクリート GPU の PCIe 経由の運搬より遥かにスムーズだ（ただし HBM のような TB/s 速度があると誤解するな）。

**黄金のワークフロー**：

- **Qwable-v1（35B）が「日常の高頻度」を担当**：書きながら Debug、コード修正、短文作成、翻訳、コマンド検索——102 tok/s の噴射速度のおかげで、思考が AI を待って止まることがない。
- **GPT-OSS 120B が「重度の思考」を担当**：35B で解けない Bug、アーキテクチャ一式の計画、三篇の長論文を投げての領域横断分析——切り替えて、128 エキスパートに攻略させる。

> 💡 ひとこと助言
> 君と一席を交わそう：**本当に道具を使いこなす人は、一本のスイスナイフで全てを解決しようとはせず、正しい刃が正しい瞬間に無意識のうちに手元に現れるようにする——この「階層的認知」は実は人間の脳の鏡像だ。** 心理学の System 1（速い、直感）と System 2（遅い、熟考）が対応するのは、まさに「速いモデルが日常の九割を、大モデルが一割の硬い骨を攻める」ことだ。二モデル並行の精髄は「二つのモデルがある」ことではなく、「切り替えそのものが無意識になる」ことにある——深い思考を呼び出すコストがゼロに近づいて初めて、君は本当に熟考すべきときに熟考し、「切り替えが面倒だから」と速いモデルで間に合わせることがなくなる。**道具の最高の境地は、正しい選択を最も省力な選択にすることだ。**

> 🔍 掘り下げ解説──二モデルの上級隠し玉：投機的デコード、二つのモデルを合体させて加速する
> 前段では「タスクに応じた切り替え」しか語らなかったが、すでに大小二つのモデルを 128GB にロードしてある以上、彼が触れなかったものの、両者を**協調加速**させられる上級技術が一つ、知る価値がある——**投機的デコード（Speculative Decoding）**だ。原理はこうだ：**小モデル（Qwable-v1 やさらに小さい draft model）にまず素早く一連のドラフト token を生成させ**、次に**大モデル（GPT-OSS 120B）にこのドラフトを一括で並列検証**させ、正しいものを受理し、誤ったものを否決する。「K 個の token を検証する」ほうが「K 個の token を逐次生成する」よりメモリ帯域幅上はるかに割に合う（一度重みを読んで複数を検証する）ため、実測ではしばしば大モデルの実効出力速度を 1.5〜3 倍にでき、**しかも出力分布は大モデルと完全に一致する（無損加速）**。これはまさに DGX Spark の「大小二つのモデルを同時に持つ」という条件を極限まで活かす——小モデルは「日常を素早く答える」予備ではなく、大モデルの「予言加速器」なのだ。ツール面では、**llama.cpp（`--draft` / draft model）、vLLM、SGLang** がいずれも内蔵対応済みだ。**もう一つの隠れたコストの注意**：KV Cache こそ長コンテキストの本当のメモリ殺し屋であることを忘れるな——コンテキストが 20 万 token まで積み上がると、残りの ~32GB は**長さに伴って線形に急騰する** KV Cache に容赦なく食い尽くされる（場面2 の fit 算式を振り返れ：70B 級モデルの 256K の KV は ~80GB に達しうる）。このときは **FP8/INT8 の KV Cache 量子化**（vLLM、llama.cpp が対応済み）でそれを半減させるべきで、「モデルを一つ退避すれば足りる」と無邪気に思ってはいけない。**長コンテキストのボトルネックは決して重みではなく、KV Cache だ。**

## 場面4　デプロイ実務：モデルをあの 128GB に釘付けにする

**背景**：モデルを選び終えたら、最後の一マイルは「DGX Spark でいかに極限の速度を引き出すか」だ。以下、いくつかの具体的提案で、ツールの詳細を補う。

**デプロイのツールチェーン**：

- **フォーマットとエンジン**：GGUF をダウンロード（まず **Q4_K_M**、コア層は Q8_0 に上げて混合量子化が可能）；エンジンは **llama.cpp / Ollama**（最も手軽）、スループットを追うなら **vLLM / SGLang**（連続バッチで算術強度をリッジポイントへ押し上げ、KV PagedAttention）；DGX OS 内蔵の **NVIDIA AI ソフトウェアスタック / NIM / TensorRT-LLM** で直接立ち上げられる。
- **クロスプラットフォーム対照（生態系を取り違えるな）**：DGX Spark は **CUDA** 生態系（llama.cpp-CUDA / Ollama / vLLM / SGLang / TensorRT-LLM）；同じ UMA 陣営の **Apple Silicon** は **MLX / llama.cpp-Metal** だ。両者は**原理が共通**——どちらも「統合メモリ＋端側量子化モデル」というこの道に賭けている——が、kernel と binary は互換でなく、**Mac でうまく動く MLX モデルをそのまま GB10 に持ってこられない**し、逆もまた然りだ。これを見極めて初めて、デプロイスクリプトを写し間違えずに済む。
- **量子化キャリブレーション**：GGUF 量子化を行うとき、**imatrix（importance matrix）**を使う——少量のキャリブレーション用コーパスで各重みの重要度を統計し、低ビット量子化が重要な重みを優先的に守るようにする。**同じ Q4 の体積で、明らかに高品質**だ。これは llama.cpp コミュニティの標準的な上級手法である。
- **カーネル級の固定（UMA 帯域幅を絞り尽くす鍵）**：
  - `numactl --interleave=all`：モデルの重みを LPDDR5X の全メモリチャネルに**交互**配置し、全 channel を同時に全力稼働させ、CPU/iGPU が帯域幅を奪い合う際のキャッシュミスを避ける。
  - `mlock`：重みを**物理メモリに固定**し、OS がそれを低速ディスクへページング（swap）するのを防ぐ——これは高 tok/s を維持する硬派なディテールだ。
  - **汎用 Docker のデフォルトコンテナに頼るな**：そのメモリ戦略は UMA に必ずしも合致せず、手動で NUMA と mlock を調整して初めて GB10 を発揮できる。

**「104 万コンテキスト」神話を突き崩す最後の一刀**——ベンダー仕様だけを見るな、**自分で検証せよ**：

```
どれだけの長さが入る？  →  RoPE/YaRN 設定を見る（最も宣伝しやすい数字）
どれだけの長さを動かせる？ →  KV Cache と帯域幅に制限される実用可能長
どれだけの長さを読み解ける？ →  RULER / Needle-in-a-Haystack を走らせれば真偽即判明
```

**この三問のうち、第三問「どれだけ読み解けるか」が最も飛ばされやすく、最も致命的だ**——それこそ **NIAH** と **RULER** が測るものだ：

- **NIAH（Needle-in-a-Haystack、干し草の中の針）**：数十万字の無関係な長文（haystack）の中に、一つの鍵となる事実（needle、例えば「2026 年の合言葉は nol9981」）を隠し、コンテキストを指定の長さまで詰めてからそのひと言を問う——測るのは**最低水準の純粋検索**だ。これすら通らなければ、「百万コンテキスト」は純然たる広告だ。
- **RULER**：NIAH の工業強化版で、単一針検索に加えて、**複数針検索、変数の多段追跡（variable tracking）、集計統計、長文 QA** など十数種のタスクを重ね、コンテキストを段階的に伸ばし、モデルの正確率が**閾値（しばしば 85% に設定）を割り込むその長さ**を記録する——この長さこそが「**有効コンテキスト長（effective context length）**」だ。残酷な現実はこうだ：**公称 1M のモデルが、RULER 実測の有効長はしばしば 64K〜128K しか残らない**。その後の大部分は「入りはするが、読み解けない」水ぶくれだ。だから君が信じるべきは仕様表の RoPE base ではなく、RULER 曲線上の、正確率が崩壊するその変曲点だ——そしてこの曲線は、`lm-eval-harness` にオープンソースの RULER スイートを組み合わせれば、君自身が DGX Spark で一晩で出せる。

> 💡 ひとこと助言
> 君と一席を交わそう：**仕様表の数字は、ベンダーが君に見せたい真実だ。ベンチマークを一度走らせることこそ、マシンが君に語ってくれる真実だ。** 「1M コンテキスト」「180 tok/s」「無検閲」——これらの宣伝文句は、どれも三次元の問題のうち最も見栄えのする一次元しか明かしていない。成熟したエンジニアの本能は、いかなる単一の数字も、その裏にある「入る／動く／読み解ける」「総パラ／活性化/アーキテクチャ」という独立した次元へ分解し直し、既成のオープンソースベンチマーク（RULER、lm-eval-harness、HumanEval）で自らの手で一度測ることだ。**数字への畏敬とは不信ではなく、「信頼せよ、されど検証せよ」だ——そして検証の道具はすべて無料のオープンソースだ。検証しないのは、君自身の選択だ。**

> 🔍 掘り下げ解説──UMA はメモリアーキテクチャの大博打であり、DGX Spark は何に賭けて当てたのか
> DGX Spark の **統合メモリアーキテクチャ（UMA）** は NVIDIA の小手先の賢さではなく、「未来の個人向け AI 演算力がどんな姿になるか」をめぐるアーキテクチャの大博打であり、より高みからその賭けの中身を見抜く価値がある。従来の GPU は「速いが小さい」HBM/GDDR をカードに直付けし、CPU のメモリは「大きいが遅い」、両者の間には PCIe という狭い橋が横たわる——大モデルを動かすとき、重みは二つのメモリ間を行き来し、PCIe がボトルネックになる。UMA はこの壁を取り払った：CPU と iGPU が同一プールの 128GB LPDDR5X を共有し、**運搬なし、PCIe ボトルネックなし、「VRAM に入りきらない」もなし**だ。代償は、このプールのメモリの帯域幅（~273 GB/s）が、HBM3e（~8TB/s、熱くなるほど速い）と一般的なデュアルチャネル DDR5（~100 GB/s 以下、遅い）の間の中途半端な領域に留まること——**それが賭けているのは「個人/エッジの場面では、容量のほうが極限の帯域幅より稀少だ」ということだ**。個人/エッジの場面では、この賭けはおおむね正しい：君がより頻繁に詰まるのは「モデルが 24GB の VRAM に入りきらない」ことであって、「帯域幅が足りない」ことではない。Apple Silicon（M シリーズ）はとうにこの道を実証しており、DGX Spark はそれを 128GB + Blackwell 演算力まで推し進めた版だ。**これはまた選定哲学の長期的転換を予示している：未来、この種のマシンで勝つのは「最も速い小モデル」ではなく、「最大の、活性化パラメータを制御可能な MoE」だ——ハードウェアアーキテクチャの賭けは、最終的に逆方向から、どんなモデルが蒸留され、デプロイされる価値があるかを定義する。** 君が今日 DGX Spark のために選定するロジックは、そのまま今後数年の個人向け AI 演算力の縮図だ。

## 本章のまとめ

- **まず仕様を校正**：DGX Spark = GB10 + 128GB LPDDR5X UMA + **~273 GB/s**（⚠️ 二次的記事がよく書く 600 ではない；倍間違えれば tok/s も丸ごと間違う）+ ~1 PFLOP FP4/NVFP4。
- **Roofline で DGX Spark を見る**：リッジポイント = 演算力÷帯域幅 ≈ **~3600 FLOP/byte**、一方 decode の算術強度はわずか **2〜4**——帯域幅の壁に深く沈む；デコード速度 ≈ 帯域幅 ÷ token あたり読み出し重み。容量の巨人、帯域幅の凡人。
- **選定の三問**：いくつ活性化するか（速度）、全部でいくつか（知能）、密か MoE か（連続性/メモリ形態）。結論：まず大 MoE（GPT-OSS 120B）+ 速度型（Qwable-v1 35B）。
- **fit を自分で計算**：総占有 = 重み + KV Cache + フレームワークオーバーヘッド；重み = パラメータ × bpw ÷ 8、KV はコンテキストに伴い線形に膨張。32K は重みを見て、256K は KV を見る——**長文のメモリ殺し屋は KV Cache であって重みではない**（第 10・11 章への伏線）。
- **二モデル並行 = 階層的認知**、上級には**投機的デコード**で大小モデルを合体させ無損加速；KV Cache 量子化（FP8）こそ長コンテキストの真のボトルネック解だ。
- **デプロイは UMA を絞り尽くす**：GGUF + imatrix キャリブレーション、`numactl --interleave=all` + `mlock`、汎用 Docker を避ける。
- **長コンテキストは三問に分解**、RULER/Needle で自ら検証せよ、仕様表を信じるな。

ハードウェアと選定を語り尽くした。次の問題は本質へ戻る：**これらの影は、いったいどうやって蒸留されるのか？** ハードウェアを離れ、蒸留の科学へ踏み込む——第 4 章。
