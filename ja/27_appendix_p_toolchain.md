## P.1　知識蒸留・微調整ツール

教師（大規模モデル）の能力を生徒（小規模モデル）へ圧縮する実務は、三つのルートに分かれる：**SFT（ハードラベルを学ぶ）→ オンライン蒸留（logits／on-policy を学ぶ）→ 選好アライメント（DPO/ORPO/GRPO）**。下表のツールの多くは相互に接続できる（同一の `transformers` 重み + `datasets` コーパス）。

| ツール | 中核技術 | 接続モジュール | ハード負荷 | 利点 | リスク／注意 |
|---|---|---|---|---|---|
| **LLaMA-Factory** | 一体型微調整：SFT / DPO / ORPO / **GRPO**、LoRA/QLoRA/Full FT、Sequence Packing | `transformers`+`peft`+`trl`+`deepspeed`、WebUI/YAML | 中→極高（Full FT は ZeRO-3 シャーディングが必要） | 設定駆動、アルゴリズム網羅性が最も高い、習得が速い | 抽象層が厚く、エラー時に下層へ降りてデバッグしにくい。学習率の暴走で NaN になりやすい |
| **axolotl** | YAML 駆動の微調整フレームワーク。多様なデータ形式、packing、RoPE 拡張に対応 | `transformers`+`accelerate`+`deepspeed` | 中→高 | コミュニティのレシピが豊富、再現性が良い | バージョンと依存関係のドリフトが速く、バージョン固定が必要 |
| **Unsloth** | 手書き Triton kernel + 融合演算子、QLoRA で VRAM 節約と高速化 | `transformers`+`peft`+`bitsandbytes` | 低（シングルカードに優しい） | VRAM 半減、速度2×級、コンシューマ向け GPU でも動く | 対応するモデルアーキテクチャが限定的。マルチカード拡張は DeepSpeed に劣る |
| **HF `trl`** | `SFTTrainer` / `DPOTrainer` / **`GKDTrainer`**（一般化知識蒸留、on-policy reverse-KL） | `transformers`+`accelerate`+`peft` | 中 | 公式メンテナンス、エコシステムとシームレス、`GKDTrainer` は「真の蒸留」の入口 | 低レイヤー寄りで、自分で pipeline を組む必要がある |
| **DeepSpeed（ZeRO-3）/ FSDP** | 重み／勾配／オプティマイザ状態を多カードや CPU/NVMe へ**シャーディング** | 上記フレームワークのバックエンド | 大規模モデルの Full FT を解禁 | 限られた VRAM で超大規模モデルを動かせる。ZeRO-Offload が小 VRAM を救う | 通信オーバーヘッドが大きく、帯域不足だとネットワーク律速になる。設定がシビア |
| **distilabel（+ Argilla）** | 合成データ／CoT 生成／意味的書き換え／選好対の注釈 pipeline | LLM API またはローカル推論エンジン | 低（データ側） | 「蒸留コーパスの生成」を標準化・監査可能・重複除去可能にする | 合成データの品質＝下流の天井。人手抽出検査＋汚染除去が必要 |

**解説**

- **LLaMA-Factory**：オープンソース微調整エコシステムにおける、事実上の「オールインワン」。その価値は、DPO/ORPO/GRPO という数式的に厄介なアライメントアルゴリズムを YAML に封じ込め、勾配の数値安定性ではなく**データの選好そのもの**に集中させてくれる点にある。代償は抽象層の厚さ——いったん loss が飛んだら、フレームワークを迂回して `trl` の下層を見るはめになりがちだ。**初めての蒸留はこれを選び、深くデバッグしたくなったら `trl` まで降りる。**
- **`trl` の `GKDTrainer`**：本書全体で語る「生徒が教師の完全な確率分布（ソフトラベル／暗黙知）を学ぶ」を、コードに落とすとこれになる。ハードラベルだけを学ぶ純粋な SFT に比べ、GKD は **on-policy + reverse-KL** であり、生徒に自分自身が生成した軌跡上で教師へアライメントさせるため、過学習（丸暗記）を顕著に緩和できる。**「教師の出力を SFT コーパスとして使う」のではなく「真の蒸留」をやりたいなら、これが正門だ。**
- **Unsloth**：コンシューマ向け GPU の救世主。融合 kernel で QLoRA の VRAM を半減させる。だがこれは**分散ソリューションではない**——70B 級の Full FT をやるならやはり DeepSpeed/FSDP に戻る必要がある。位置づけは「シングル／デュアルカードで LoRA 微調整を極める」だ。
- **distilabel**：蒸留の成否を分ける隠れたカギはデータにある。これは合成コーパス生成を再現可能な pipeline にし、Argilla と組んで人手レビューを行う。**忘れるな：合成データの品質こそが生徒の知能の上限であり、訓練セットには必ずベンチマークリーク（benchmark contamination）チェックを施すこと。**

## P.2　推論・デプロイツール

蒸留／微調整が終わったら、モデルを「動かして」スループットを絞り出す。ボトルネックは通常、演算力ではなく**メモリ帯域と KV-Cache の断片化**だ（付録 B.4「容量 vs 帯域の鉄則」を参照）。

| ツール | 中核技術 | 適用シーン | ハード負荷 | 利点 | リスク／注意 |
|---|---|---|---|---|---|
| **vLLM** | **PagedAttention**（KV-Cache のページ化、断片ゼロ）+ continuous batching + **Speculative Decoding** | 高並行 GPU サービス | 高（重みの常駐が必須。投機的デコードはさらに draft モデルのロードも要る） | スループットは業界基準、padding ゼロ、OpenAI 互換 API | CUDA kernel に縛られ、ハード間の可搬性が低い。長 context は依然 VRAM を食う |
| **TGI（Text-Generation-Inference）** | 連続バッチ + テンソル並列 + 量子化推論、本番級のサービス化 | クラウド／企業エンドポイントのデプロイ | 高 | すぐ使える HTTP サービス、可観測性が良い、HF 公式 | 機能の進化とライセンス条項の変更に注意が必要。カスタム kernel は vLLM ほど多くない |
| **llama.cpp** | **GGUF** 形式 + K-quants + CPU/Metal/CUDA の全プラットフォーム、imatrix 校正 | ローカル／エッジ／Apple Silicon | 低（量子化後は純 CPU でも可） | クロスプラットフォーム、デプロイが容易、量子化エコシステムが最も成熟 | 単一リクエストのスループットは vLLM に劣る。高並行は得意分野ではない |
| **SGLang** | RadixAttention（プレフィックス KV 再利用）+ 構造化出力の高速デコード | マルチターン／共有プレフィックス／Agent ワークフロー | 高 | プレフィックス再利用が few-shot/Agent の高速化に明確に効く | 比較的新しく、エコシステムとドキュメントはまだ vLLM を追いかけている |
| **TensorRT-LLM** | NVIDIA 公式、AOT コンパイル engine + in-flight batching + FP8/INT4 | 単一 NVIDIA GPU の遅延／スループットを極限まで絞る | 高 | NVIDIA ハード上ではしばしば性能の上限 | engine の事前コンパイルが必須、NVIDIA に縛られる、可搬性が最低、習得が急峻 |
| **Ollama** | llama.cpp の上層ラッパー + Modelfile + ワンクリックでモデル取得 | ローカルでの素早い試用／デスクトップ統合 | 低 | インストールしてすぐ使える、API が親切、エコシステムが広い | 下層はそのまま llama.cpp。高並行／細かな制御は vLLM や llama.cpp 本体に劣る |

**解説**

- **vLLM**：本付録「デプロイ」節の中核。PagedAttention は OS の仮想ページングを借用し、KV-Cache を非連続な物理 block に切り分け、従来の事前確保が生むメモリの空隙を消し去る（大量の無駄を節約できる）。これが高並行の根本だ。**Speculative Decoding**（小さな draft モデルが予測し、大きな target モデルが一度に複数 token を検証）はさらに遅延を圧縮できる。代償は CUDA kernel への縛りで、クロスプラットフォーム移行が苦痛なこと。
- **llama.cpp**：vLLM と相補的だ。vLLM は**サーバーの高並行**を主戦場とし、llama.cpp は**ローカル単機の可搬性**を主戦場とする——GGUF + K-quants でノート PC、さらには純 CPU 上で量子化モデルを動かせ、`imatrix`（重要度行列の校正）を組めば同じサイズでより高い品質に化ける。**個人のローカルなら第一選択。トラフィックを捌くなら vLLM へ。**
- **SGLang**：ワークフローが「大量の共有プレフィックス」（同一の system prompt を繰り返し呼ぶ、Agent のマルチターン）のとき、RadixAttention のプレフィックス KV 再利用が最大の威力を発揮する。特定の負荷形態に向けた最適化に属する。
- **TensorRT-LLM vs Ollama（両極端）**：どちらも「汎用の第一選択」には入らないが、スペクトラムの両端を代表する。TensorRT-LLM は **AOT コンパイル engine** で NVIDIA カード上の最後の一片の遅延／スループットを引き出す——代償は事前コンパイルのコストと**NVIDIA への完全な縛り**で、「ハードが確定し、最後の10%を絞り出す」ときにだけ持ち出す重火器だ。Ollama はもう一端：本質は **llama.cpp の上層ラッパー**で、売りは「`ollama run` で即使える」デスクトップ体験。ローカル試用や統合には向くが、その手軽さを新しい推論エンジンと取り違えてはいけない——**batch／量子化／並行を調整したいなら、やはり llama.cpp や vLLM 本体へ戻る必要がある。**

> 注意：元素材は「ローカルワークステーションで 100+／102 tok/s を叩き出す」と謳うが、これは具体的な性能宣伝であり、**ハード・モデルサイズ・量子化レベル・batch に強く依存し、一概には言えない**。自分のマシンでの実測を基準とすること。

## P.3　モデルマージ・評価ツール

訓練せず、複数の重みを**算術的にマージ**するだけで、低コストに能力を「縫合」する手法は既に主流だ。そして蒸留／マージのいかなる成果も、**統一された評価**を通さなければ数えられない。

| ツール／手法 | 中核技術 | 用途 | 注意 |
|---|---|---|---|
| **mergekit** | 複数のマージアルゴリズムを統一したフレームワーク | 複数の微調整モデルを一つに融合 | マージ≠必ず強くなる、評価による検証が必要 |
| └ **SLERP** | 球面線形補間、二モデルの重みを滑らかに補間 | 二モデルの作風／能力の折衷 | 一度に二モデルのみ |
| └ **TIES-Merging** | 微小な差分を剪定 + 符号衝突を解消してからマージ | 多モデル融合、相互干渉の低減 | 疎度比率の調整が必要 |
| └ **DARE** | タスクベクトルをランダムに drop して再スケールしてからマージ | パラメータ冗長の低減、TIES と併用 | ランダム性はシード固定が必要 |
| **lm-evaluation-harness（EleutherAI）** | MMLU-Pro / GSM8K / HumanEval などのベンチマークを統一して実行 | 横並び評価、独りよがりの防止 | データリークの確認が必須。prompt テンプレートがスコアに影響 |
| **AlpacaEval 2 / MT-Bench / Arena-Hard** | LLM-as-judge で指示遵守と対話品質を評価 | 静的ベンチで測れない「使い勝手」を補う | judge に選好バイアス（長さ／作風）があり、judge とベンチ対戦相手の固定が必要 |
| **RULER / Needle-in-a-Haystack** | 長コンテキストの検索／推論のストレステスト | 「謳い文句の context を本当に読めるか」を検証 | YaRN/RoPE 拡張と合わせて検証 |

**解説**

- **mergekit**：ほぼゼロの演算力で、複数の専門特化モデルを一つに「縫合」する——例えば推論が強い体と中国語が強い体の微調整版をマージする。**SLERP** は二モデルの滑らかな折衷に向き、**TIES/DARE** は多モデル時のパラメータ符号衝突と冗長を処理する。だが肝に銘じよ：**マージは経験的な操作であり、しばしばより悪いモデルを生む。必ず P.3 の評価ツールで回帰検証し、感覚でリリースしてはならない。**
- **lm-evaluation-harness**：生徒モデルが蒸留後に知能を落としていないか、これ一本のものさしで測る。**最大の落とし穴はベンチマークリーク（test set が訓練データに混入）による見かけ上の高得点**——必ず汚染除去チェックを行い、prompt テンプレートの違いがスコアを大きく左右する点に注意せよ。
- **AlpacaEval / MT-Bench / Arena-Hard**：静的ベンチ（MMLU/GSM8K）は「できるか」を測り、judge 系評価は「使い勝手」を測る——アライメント／蒸留後の対話モデルは、しばしばベンチは落ちないのに体感が悪化し、それは LLM-as-judge でしか捕まえられない。だが**落とし穴は静的ベンチより隠れている**：judge には系統的な選好（より長い、より丁寧、自分と同源の出力を好む）があり、スコアは judge モデルのバージョンとともにドリフトする。**相対比較に使い、judge とベンチ対戦相手を固定し、絶対スコアを真理だと思うな。**
- **RULER**：付録 B.2 の YaRN+RoPE 拡張と組み合わせて使う。モデルが 1M context を「謳う」ことは「読める」ことを意味しない。RULER と Needle が測るのは公称長ではなく実効長だ。

## P.4　アライメントと「脱検閲」ツール（防御的説明）

> ⚠️ **本節はリスク開示であり、操作ガイドではない。** 以下のツールは**それ自体は公開・合法で、正当な研究用途を持つ**（機械的解釈可能性／安全研究）が、同じ技術群が**モデルの安全アライメント除去（脱検閲／abliteration）に使われると濫用となる**。本書は abliteration の手順・CLI・Python グルースクリプト・YAML 設定を**提供も転載もしない**——対応するレッドラインの判定基準は一律で**付録E**と**第6章の六つの鉄則**を参照せよ。

| ツール | それは何か | 正当な用途 | 二重用途リスク ⚠️ |
|---|---|---|---|
| **TransformerLens** | 機械的解釈可能性（Mechanistic Interpretability）ライブラリ。HookPoint を提供し、Transformer 内部の活性を読み取り／介入する | 学術的に注意ヘッド・ニューロンの機能を分析し、安全・アライメント研究を行う | 同じ活性介入能力が、安全関連の回路を特定し弱体化させるために使われうる ⚠️ |
| **llm-abliteration 系スクリプト** | 「拒否方向（refusal direction）」を残差ストリームから除去する脱検閲ツール | （研究の文脈で）安全アライメントの脆弱性を定量化し、防御を評価する | **設計目的がそもそも安全アライメントの除去**であり、拒否しない重みを生み出す高リスク濫用 ⚠️⚠️ |

**これらのツールが何をしているか（事実レベル）**

学界の2024年の *refusal-direction* 研究（**Arditi et al., 2024**、「Refusal in LLMs is mediated by a single direction」）は、次を発見した：多くのオープンソースモデルの「回答拒否」挙動は、高次元の活性空間において**単一の方向**が主導している。これは**実在し、検証可能**な解釈可能性の発見だ。その防御面での価値は、現在の安全アライメントの脆弱性を露呈させ、デプロイ側に**安全はモデル自身の重みだけに頼れない**と警告する点にある。

**なぜリスクがあるのか（防御意識）**

同じ発見をひとたび逆用すれば——その方向を重みから消し去れば——ほとんど再訓練なしに**モデルの安全ガードレールを除去**でき、有害リクエストに応じる重みを生み出せる。これこそ abliteration ツールの争点だ：技術的ハードルが低く、演算力の要求が小さく、しかも**開発者が投じたアライメント作業を迂回する**。本書の立場は明確だ：

- 元素材にある「三ステップの作戦接続／ワンクリック消融」レシピ、`pip install`、対照プロンプト、層数の範囲、`save_pretrained` の流れ、DPO 脱検閲 YAML を**転載しない**——これらは操作的内容であり、**一律で省略する**。
- 安全アライメントを除去したモデルはマルウェア、攻撃コード、違法コンテンツを生成しうる。**法的・倫理的責任は使用者が自ら負う**。この種の重みの配布は、多くの司法管轄区とプラットフォーム規約の下で制限される。
- 「能力のために安全を外す」というあらゆる訴えは、**まず付録Eのレッドライン一覧と第6章の六つの鉄則**に立ち戻って越境していないか照合し、それからやるか否かを決める——たいていの答えは「やらない」だ。

> **ひとことで言えば**：TransformerLens は命を救うことも傷つけることもできるメスだ。本書はそれを**見分け、濫用を防ぐ**ことを教えるが、**どう振るうか**は教えない。操作の境界 → 付録E／第6章。
