# 第六部　現実とのすり合わせ

# 第6章　あの一線はどこまで赤いのか：法律、倫理、そして最後まで弾き切る勘定

> シリコンバレー、弁護士事務所のガラス張りの会議室。「蒸留モデルでプロダクトを加速する」という提案書がテーブルの中央へ押し戻され、法務はただ一言だけ尋ねた——「あなたたちの訓練データのなかに、よそ様の有料 API 由来のものが一本でも混じっていないか？」会議室は三秒間しんと静まった——その三秒の沈黙は、この章のどの一文よりも価値がある。

> 本書でもっとも重要な章である。前の五章は「何ができるか」を解きほぐしてきたが、この章が論じるのは「やるべきか否か」と「割に合うか否か」だ。これは道徳の説教ではなく、一人のエンジニアが Enter を押す前に、自分と会社のために弾き切っておかねばならない、あの勘定である。

## 場面1　法律と規約：先に知っておくべき四つの事実

**背景**：巷に流れるあの断片的な警告——「ToS 違反だ」「刑事犯罪化が推進されている」「アリババが告発された」——を、使える判断材料へとまとめ上げる。

**四つの事実（公開資料と常識に基づく。法的助言ではない）**：

1. **ToS はほぼ確実に蒸留を禁じている**。主流の商用 LLM ベンダーの規約は、おしなべて「我が社のモデルの出力を使って競合モデルを訓練すること」を明文で禁じている。大規模に API 応答を採取して訓練に回す行為は、**契約違反がほぼ確定**であり、その帰結はアカウント停止、損害賠償請求から訴訟にまで及ぶ。
2. **「防護の回避」は、事を「違約」から「違法」へと格上げしうる**。単なる ToS 違反は通常は民事の問題だ。だが、**技術的な防護措置を自ら能動的に回避する**（第5章 M1 のあのプロキシ・ローテーション、敵対的プロンプト、ダウングレード検知）と、法域によっては「技術的保護措置の回避」「コンピュータシステムへの無権限アクセス」といった**刑事ないし準刑事**の条項に触れる可能性がある。巷で言われる「大規模蒸留攻撃を刑事犯罪と定める動き」とは、まさにこの方向のことだ。
3. **データセットそのものが「法的な放射能」を帯びている**。出所不明の `claude-traces` をダウンロードしただけで、たとえ採取したのが自分でなくとも、**それを使って商用モデルを訓練すれば、侵害の連鎖の一環になりうる**（第2章）。
4. **企業の場面ではリスクが N 倍に拡大される**。個人の遊び半分なら捕まってもアカウント停止で済むかもしれない。だが、会社が競合の蒸留に頼って商用モデルを構築したと認定されれば、待っているのは**商業訴訟、賠償、ブランド信頼の崩壊**——桁が完全に違う。

> ⚠️ 免責
> 以上は法的助言ではない。商業の場面でこの一線に本気で触れるなら、**専門の弁護士に相談せよ**。本書の立場は単純だ——**クローズドソースの商用モデルへの無権限蒸留は、技術的には可能でも、法的・倫理的に取るべきではない。**

> 🔍 掘り下げ解説──なぜ「防護の回避」が全体のなかで法的性質のもっとも重い一歩なのか
> 第2項は単独で深掘りする価値がある。多くのエンジニアがもっとも過小評価しがちな法的断層だからだ。**ToS 違反と技術防護の回避は、まったく桁の違う二つの事柄である。** 前者の本質は「違約」——あなたとベンダーは契約を結んでおり、あなたが利用規約に違反した、という話で、これは通常は民事の範疇（アカウント停止、賠償請求）に収まる。ところが、プロキシ IP プールを展開してレート制限をかいくぐり、敵対的プロンプトで防護分類器を欺き、ダウングレード検知でベンダーの汚染機構に対抗するとき、あなたがしているのはもはや「一つの契約に違反する」ことではなく、「**あなたを阻むために設けられた技術的統制システムを能動的に破る**」ことだ——これは多くの法域（米国 CFAA の「無権限アクセス」、DMCA の「技術的保護措置の回避」をめぐる議論、および各国の類似立法など）で、行為の法的性質を民事から刑事へと押しやる。**これこそ M1 が全パイプラインのなかでもっとも危険な一片である理由だ**：それは単にあなたを「違約」させるだけでなく、あなたの採取の一回一回を「統制回避」の証拠にしてしまう。一つ重要な判断の示唆がある——**相手があなたを阻むために専用に設けた機構を「欺く」または「回避する」必要があると気づいたとき、あなたはおそらくすでに「グレーゾーン」から「ブラックゾーン」へ踏み込んでいる——技術的に「回避できる」こと自体が、法的には「故意性（intent）」のもっとも有力な証明になる。** 防護シールドの存在それ自体が「立入禁止」と書かれた看板であり、それを回避した時点で、あなたはもう「許されていないとは知らなかった」とは主張できなくなる。

### 規約の構造：三大ベンダーが「何を禁じるか」は実は高度に一致している

「ToS が蒸留を禁じている」を曖昧なスローガンとして受け取ってはならない——それは主流ベンダーそれぞれの契約のなかで、**構造のほとんど同じ「出力による競合訓練の禁止」条項**に対応している。三社を並べて見れば、これが特定の一社の気性ではなく、業界全体の共通防衛線だと分かる：

| ベンダー（条項の所在）| 一般的に含まれる制限の類型 | 蒸留者にとっての意味 |
|---|---|---|
| OpenAI（Usage Policies / Business Terms）| 「出力」を用いて競合するモデルを開発してはならない | 出力で生徒を訓練 = 直撃 |
| Anthropic（Usage Policy / Commercial Terms）| サービスを用いて競合品を構築してはならない、**競合的 AI モデルの訓練を含む** | Claude 系列の蒸留 = 直撃 |
| Google（Gemini API 追加条項 / 禁止ポリシー）| サービスの出力で競合的 ML モデルを開発することを制限 | Gemini の蒸留 = 直撃 |

> ⚠️ 真偽メモ
> 上表は**構造的な要約であって逐語引用ではない**。また各社の条項は**版の更改が頻繁**だ（近年、三社とも関連文言を調整しており、一部の製品ラインは緩和、一部は強化されている）。本当に判断するなら、**あなたが契約した時点での、その製品ラインの最新条項の原文を基準とし**、法務に確認せよ。本書が記憶してほしいのは特定の一文ではなく——**あの防衛線はずっとそこにあり、三社ともに引いている**ということだ。

**多くのエンジニアが整理しきれていない法的断層：これは主に「契約問題」であって「著作権問題」ではない。** 米国著作権局はすでに繰り返し表明している——**人間の創作を欠いた純粋な AI 生成コンテンツには、登録可能な著作権が存在しない**。つまり、教師モデルが吐き出す生のトークンそれ自体は、**多くの場合、著作権の保護を受けない**。これは「では使ってしまっても問題ない」と誤解させやすいが、真相はまさに逆だ：**著作権というこの壁が存在しない可能性があるからこそ、ベンダーの保護は主にあなたが「同意」を押した瞬間に発効するあの一片の契約（ToS）の上に築かれており**、そこへさらに**営業秘密**や、前述の**無権限アクセス／防護回避**といったコンピュータ犯罪法理が積み重なる。言い換えれば、クローズドソースモデルを蒸留する法的リスクは、**出力に著作権があるか否かではなく、あなたが違約したか、統制を回避したか否かに左右される**——だからこそ「私はそれが公開で吐き出した文字を使っただけだ」という弁解は、契約や CFAA の類の法理の前ではほとんど意味をなさない。

> ⚠️ 真偽メモ
> 「AI 出力が著作権の保護を受けられるか」「モデル出力を別のモデルの訓練に使うことが侵害を構成するか」は、現時点で**世界的に結論が出ていない**——各国で立場が分かれ、訴訟が進行中であり、しかも「訓練データ側の争点」と「モデル出力側の争点」は**二つの異なる問題**なので混同してはならない。本段が述べるのは**現時点での未解決の問題と主流の方向性**であって、確定した法律ではない。

### 立ち上がりつつある二枚の新しい壁：規制と輸出管理

レッドラインはベンダーの契約だけから来るのではない。国家からも来る。モデルを作る誰もがレーダーに入れておくべき二つの方向がある：

- **EU AI Act（汎用型 AI／GPAI の義務）**：蒸留後に汎用型モデルを**対外公開**するなら、EU 法の下であなたはほぼ確実に「GPAI 提供者」となり、技術文書、**著作権コンプライアンス方針**、訓練データ要約の開示などの義務を負う。システミックリスクの閾値（訓練演算力で測る）に達した者の義務はさらに重い。蒸留はこれらの義務を「回避」させてはくれない——むしろ**何もないところから提供者一式の責任を継承させかねない**。
- **輸出管理（米国 EAR など）**：先進 AI の**モデル重みそのもの**が、徐々に輸出管理の議論と規則の枠組みに組み込まれつつある。あなたが蒸留して頒布する重みは、状況によっては管理対象そのものになりうる——これは「ソフトウェアはオープンソースにすれば自由」というのとは別の話だ。

> ⚠️ 真偽メモ
> EU AI Act の GPAI 細則と、米国のモデル重みに対する輸出管理の枠組みは、**いずれも高速で変化し、ときに反復している段階にある**（関連する輸出管理規則は 2025 年に数度の調整と再評価を経た）。具体的な閾値、発効時点、適用範囲は、**最新の公式法規の原文**を基準とせよ。本書が請け負うのは「**この二枚の壁が立ち上がりつつある**」という方向を指し示すことだけで、いかなる具体的数値も保証しない。

## 場面2　経済の勘定：盗む方が、買うより高くつくかもしれない

**背景**：多くの人は蒸留を「節約」だと思っている——高額な API 費を払わず、自分で一つ蒸留してローカルに置く、と。だが第1章のあの「汚染の盾」のコストを勘定に入れると、この計算は割に合うとは限らない。

**著しく過小評価されている一つの勘定**：

| コスト項目 | 内容 | なぜ過小評価されるか |
|---|---|---|
| **採取コスト** | 複数アカウント、プロキシ IP プール、API 費。アカウントが停止されたら買い直し | 「一度採れば十分」と思いがちだが、実際は防護の盾が繰り返しの採取を強いる |
| **除染コスト** | 教師がダウングレードして汚染したデータ（第1章）。十万本採っても使えるのは三万本かもしれない | さらにもう一巡分の演算力＋別の信頼できるモデルでどれが汚染されたかを判定する必要がある |
| **訓練コスト** | 35B のフルパラメータ微調整に要する GPU 時間 | 一度で調整が決まらなければやり直し。反復コストが積み上がる |
| **保守コスト** | 教師が更新／取り下げられると、あなたのデータは陳腐化し、再採取・再訓練が必要 | あなたのモデルの「鮮度の有効期限」が他人の公開ペースに縛られる |
| **法的期待コスト** | 追及される確率 × 帰結 | 企業にとってこれは**最大の項目**になりうるが、もっともスプレッドシートに勘定しづらい |

これらを合算すると、**「いっそ合法的に対価を払って公式 API を使う」のと比べて、蒸留の節約神話はしばしば成り立たない**。これこそ第1章のあの盾のもっとも巧妙なところだ——**それはあなたを阻みはしない、ただ「盗む」ことを割に合わなくするだけだ。**

**「合法に自前のものを蒸留する」と「他人のクローズドソースを盗む」を並べれば、勘定は一目瞭然だ。** 上の表が勘定したのは「盗む」ことの隠れたコストだった。下の表は、二つの道を同じ物差しで測る——節約神話が「成り立つとは限らない」どころか、**企業規模ではほぼ必然的に逆転する**ことが分かるだろう：

| 観点 | 合法の道：自前／オープン重みを蒸留 | 非合法の道：クローズド API を盗む |
|---|---|---|
| 採取の合法性 | 潔白、ライセンスは白紙黒字 | 違約、刑事に触れる可能性 |
| 供給の安定性 | 重みは手元にあり、永久に取り下げられない | 他人の公開ペースに縛られ、改版／取り下げ一つで供給断絶 |
| 追跡可能リスク | なし | 生徒が**教師の統計的指紋／透かしを継承**しうるため、事後に検出されうる |
| 商用可／監査可 | 商用可、監査を通過可、デューデリジェンスに書ける | いかなるコンプライアンス文書にも書けず、デューデリジェンスで一発露見 |
| 規制義務 | GPAI などの義務を通常どおり負うだけ | 義務はそのまま、加えて違法の事実が一つ乗る |
| 総コスト曲線 | 初期投資は高いが、その後は均され、予測可能 | 一見初期は安いが、後期に「継続採取＋継続除染＋法的期待コスト」で底なし沼に引きずり込まれる |

**「追跡可能リスク」のこの行は、単独で警報を鳴らす価値がある。** 学術界には早くから「放射性データ（radioactive data）」の研究がある：訓練データはモデル重みのなかに**統計的に検出可能な痕跡**を残す。近年のテキスト透かし（統計式 watermark の類）やメンバーシップ推論攻撃は、「この生徒は私の出力を蒸留したのか」を玄学から**立証可能な技術的問題**へと変えた。これは背筋が寒くなる事実を意味する：盗んだ能力は**採取スクリプトを止めたところで綺麗にはならない**——それはすでに重みに溶接され、相手が将来いつでも持ち出せる法廷証拠になっている。**盗み終えれば船が水を渡るように跡形なく消えると思っていたら、実際はあなた自身の手で指紋をモデルに焼き付けており、しかも一度焼き付けたら取り出せないのだ。**

> 💡 ひとこと助言
> あなたとの一席の話——**最良の盗難対策は、より高い壁ではなく、盗みの総コストを購入より高くすることだ。「盗む」が「買う」より高くつくとき、誠実さは最高潔な選択ではなく、最も理性的な選択になる。** これはセキュリティ経済学の最も深い一課だ：すべての攻撃者を阻むことはできないが、彼らの損益関数を作り替え、攻撃を「期待値」のうえで赤字の商売にすることはできる。Anthropic のあの汚染の盾の天才は、技術がどれほど破りにくいかにあるのではなく、蒸留者のコスト構造を「一回限りの採取」から「継続採取＋継続除染＋継続保守＋継続的な法的リスク負担」へと書き換えたところにある——四つの「継続」が積み重なり、それを公式 API のあの明快で合法で SLA のある価格と対比すれば、理性的な人間は自ら諦める。**真に成熟した防御とは、相手が計算機の前で弾いているうちに、自分でため息をついて端末を閉じてしまうようにすることだ。**

## 場面3　六つの鉄則：「蒸留したい」を安全に着地させる

**背景**：「ローカルに強いモデルが欲しい」という完全に正当な需要を、安全で合法な軌道へと導く。

1. **まず教師が誰かを問え**。**オープン重みモデル**（Qwen、Llama、Mistral、GPT-OSS、そのライセンスを遵守して）を蒸留する——青信号。**クローズドソースの商用 API** を蒸留する——赤信号。この一条が面倒の九割をふるい落とす。
2. **ToS だけでなくライセンスを読み解け**。オープン重み ≠ 自由に使ってよい。商用禁止のものもあれば、表示を要求するもの、派生モデルの再ライセンスを制限するものもある。**使う前に LICENSE を読み切れ**（Llama の許容利用ポリシー、Qwen のライセンス、Apache-2.0／MIT の違いをきちんと区別すること）。
3. **データセットは素性を調べよ**。あらゆる `claude-traces`、`gpt-dump` の類のデータセットは、使用前に出所が合法か、ライセンスが明確か、アップロード者が信頼できるかを確認せよ。**出所不明の原料は、どれほど香ばしくても鍋に入れるな。**
4. **技術防護を能動的に回避してはならない**。レート制限、防護分類器、アクセス制御——これはベンダーが引いた境界だ。それらを回避することは「違約」を「違法」に格上げする一歩であり、**決して踏み出してはならない**（場面1を参照）。
5. **自己申告の除去 ≠ 出所の抹消**。自前のモデルでブランド中立化を行うのは正常なエンジニアリングだ。だが他人のモデルのデータで出所の痕跡を抹消するのは、証拠隠滅だ。**同じ技術でも、自分がどちら側に立っているかを見極めよ。**
6. **企業の場面では、法務が先行する**。商用モデルの出力に関わるいかなる訓練計画も、**先に法務を通し、それからコードを書け**。エンジニアのロマンが、会社の訴訟になってはならない。

**第1・第2条は展開する価値がある：オープン重みは「自由に使ってよい」を意味せず、ライセンスの地勢は思うよりも険しい。** いくつかの主流ライセンスを並べれば、なぜ「LICENSE を読む」が官僚的な建前ではなく、保身なのかが分かる：

| ライセンス類型（代表）| 商用 | 主要な制限／義務 |
|---|---|---|
| Apache-2.0 / MIT（多くの Qwen、一部の Mistral）| ✅ | 最も緩い。著作権表示を保持すればよく、Apache は別途特許ライセンスを含む |
| Llama コミュニティライセンス（Meta Llama 系）| ✅ 条件付き | 独自ライセンス＋許容利用ポリシー。超大規模ユーザー数（MAU 閾値）は別途ライセンス交渉が必要。派生モデルに命名／表示の要求あり |
| Gemma ライセンス（Google）| ✅ 条件付き | Gemma 条項＋禁止ポリシーに拘束される |
| 一部ベンダーの独自ライセンス（一部の大サイズ重み）| 場合による | MAU 閾値、再ライセンス制限、特定領域の使用禁止を含む可能性 |
