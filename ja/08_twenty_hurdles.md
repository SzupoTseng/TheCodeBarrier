# 第八部　難関と人間的操作

# 第8章　二十の難関：「歩いては止まる」を「筋肉の記憶」へ変える

> シリコンバレー、とある Computer Use スタートアップのデモルーム。スクリーンには、君が自ら調整したエージェントが、ある投資家のために航空券を予約している――スクリーンショットを「見」て、三秒「考え」て、一度「クリック」し、また一枚スクリーンショットを「見る」。優雅だが、まるで手術を終えたばかりで、手を上げるたびに半秒その手を見つめなければならない人のように遅い。投資家は何も言わず、ただ椅子を後ろに引いて、君の背筋を凍らせる一言を放った。「**こいつは分かっているのか、さっきの決済確認ポップアップが、もう自分で閉じてからもう一度開いたことを？**」君はうつむいて log を見る――分かっていない。その目は「スクリーンショットを撮るべき時」だけ開き、それ以外の時間は閉じている。その瞬間、君は悟った。君が作ったのは電卓を操作できる AI ではない、**三秒おきに一度目を覚まし、目覚めるたびに記憶を失う夢遊病者**なのだと。この章は、その夢遊病者を叩き起こすためにある――無数のチームを転覆させてきた二十の工学的難関から語り起こし、さらに「歩いては止まる」を「筋肉の記憶」へ変えるその物理的な最下層をはっきりさせる。

> ⚠️ 本章の素材の出所と真偽の規律
> 本章が統合した原素材は、「Google L7 / Gemini 3.5 Flash」の口調で書かれた一段のアーキテクチャ論述である。そのうち**真に成熟した工学原理（Fitts's Law、Optical Flow、H.264 GOP、SIGSTOP、gRPC、TCC、Accessibility API、Apple Silicon UMA、ViT Patch）は本書も通常どおり記述する**。しかし**具体的な製品名、未公開の内部数値、未実証の脆弱性のタイムライン**（「遅延 40% 減」「TTFT 1.2 秒」「あるバージョンの macOS が仮想ストリーミングを内蔵」「Windows Recall がある時点であるチームに破られた」など）に関わるものは、すべて `> ⚠️ 真偽メモ` で標示し、「既成事実」ではなく「妥当な工学的仮説」として受け取ってほしい。製品成熟度の比較は観察と推論であり、ベンダーの保証ではない。

## 場面1　二十の難関の全景：混沌を五つに分類して、初めて勝てる

**背景**：大模型に「完璧にコンピュータを操作させる」とき、ぶつかるのは決して「API を少し書く」ほど単純なものではない。それはマルチモーダル知覚の限界、分散システムの遅延、非決定的環境、セキュリティ隔離、人間とエージェントの協働のエッジケースを横断する。二十の難関を手当たり次第にリストへ並べても意味はない――**工学の第一歩は常に分類である**。なぜなら、異なる種類の難関は、異なる層のアーキテクチャで防がねばならないからだ。

以下は二十の難関を「どの層を叩くか」で五大カテゴリに分けたものだ。各難関に一句の精錬された記述と、それが log に残す**失敗シグネチャ（failure signature）**を付す――これは定義よりも役に立つ。本番で君が目にするのは常にシグネチャであって、定義ではないからだ。

> 💡 ひとこと助言
> 君に一言――**この二十の難関のうち、本当に致命的なのは決して「モデルが賢くない」ことではなく、「モデルが自分の誤りを知らない」ことだ――失敗シグネチャのこの一欄は、難関そのものより百倍値打ちがある。** クリックを誤るエージェントは怖くない。怖いのは、クリックを誤ったのに自分は正しく押したと思い込み、その誤りの上にさらに三十階建ての楼閣を積み上げるエージェントだ。第 3、4、10、19 のこの四つを見よ――スケルトンスクリーン、無限スクロール、イベント返却なし、幻覚クリック――表面上は四つの異なる問題だが、**骨の髄では同じ一つのことだ：モデルが「さっきの一手は本当に効いたのか」という閉ループの確認を欠いている。** 人間のコンピュータ操作が滑らかなのは、手が正確だからではなく、**目が一刻も止まらず手の結果を検証しているからだ**。指を押し下げると同時に、目はすでにそのボタンが色を変えるのを、その入力欄に文字が出るのを待っている。**アマチュアのエージェントエンジニアは「どうすればもっと正確に押せるか」を最適化し、プロは「押した後にどうすれば押し当てたか分かるか」を最適化する**――前者は開ループの精度を追い、後者は閉ループの神経を作る。この境界線が、君のエージェントが夢遊病者か、それとも本当に目覚めた操作者かを決める。

> 🔍 掘り下げ解説 ―― なぜ SoM と Context Caching は止血であって根治ではないのか
> 業界が第 1、2 関（ピクセルずれ、DPI 自適応）に対して最も流行している解法は **Set-of-Mark（SoM）の視覚オーバーレイ**だ：スクリーンショット上にあらかじめプログラムで番号付きの枠を描き、モデルに「座標 (x,y)」ではなく「点番号 7」を出力させ、連続的な座標回帰問題を離散的な分類問題へ次元削減する――これは確かにピクセルずれを大幅に下げる。第 6、8 関（Token 爆発、KV 圧迫）に対する主流の解は **Context Caching / Prompt Caching** だ：変わらないシステムプロンプトと履歴のプレフィックスをキャッシュし、毎ステップの再計算を避ける。この二手はどちらも真に有効な工学だが、その**境界**を見極めねばならない：SoM は「クリック可能な要素をまず枠で囲める」前置の検出器に依存する――ところが第 5 関（カスタムコントロールの意味的断層）の、画像で継ぎ接ぎしたあのボタンを、前置の検出器はそもそも枠で囲めず、SoM は無効化する。Context Caching は「重複するプレフィックス」の計算を節約できるが、第 11 関（コンテキストの焦点喪失）は救えない――キャッシュは君に百ステップの履歴を**収められる**ようにするが、注意が最初の一歩の目標を**見えている**ことを意味しない。**すべての「圧縮 / キャッシュ / マーク重ね」系の最適化は、本質的に同一の開ループ管路の帯域を広げているにすぎない。それらは歩いては止まるのをより速く、止まりをより短くするが、「歩いては止まる」という範式そのものは変えていない。** 真に根治するには範式を換えねばならない――これこそが場面2以降で語る二環アーキテクチャだ。

## 場面2　二環デカップリング：「大脳皮質」と「小脳・脊髄」を切り離す

**背景**：もし従来の「スクリーンショットを送る → 考える → クリック」という RPC 式の歩いては止まるアーキテクチャだけを使うと、AI の操作は一動きごとに三秒静止する人のようになり、現代の OS で瞬く間に消える動的ウィンドウにはまるで対応できない。前述の丸ごと三類の難関（知覚、遅延、状態機械）を解くには、単一ループを速く調整するだけでは無理だ――光速はそこにあり、クラウドの往復はどうしても時間を食う。**唯一の出口は、物理的にループを二つに分け、それぞれ異なる周波数で別々に走らせることだ。**

この分け方は人間の脳の分業をそのまま写し取る。人の**大脳皮質（Cortex）**は深い思考と計画を担う――遅く、高スループットで、「何をするか」を司る。**小脳と脊髄（Cerebellum/Spine）**は筋肉反射と動的微調整を担う――極めて速く、低遅延で、「手をどう動かすか」を司る。君が飛んできたカップに手を伸ばして受け止めるとき、皮質は「受け止めろ」というマクロな意図を下しただけで、最後の 50 ミリ秒で指の閉じを微調整しているのは小脳だ。皮質はそもそも間に合わない。

Computer Use の二環アーキテクチャは、まさにこの分業の工学化だ：

```
        ┌──────────────────────────────────────────────┐
        │   クラウド遅延大脳環 (Slow Loop)  ~1.0s 遅延   │
        │   [ 大模型プランナー / Planner ]               │
        │   責務：「マクロな意図」+「予期した視覚アンカー」を出力 │
        │   例:「ログインフォームを探す→アカウント入力→送信後2FAへ遷移を予期」 │
        └──────────┬───────────────────────▲────────────┘
                   │ Macro Plan             │ Key-Frame イベントログ
                   │ (低頻度で下達)         │ (増分で返報, 全画像ではない)
                   ▼                        │
   ┌───────────────────────────────────────┴──────────────────┐
   │   ローカル高速反射環 (Fast Loop)   < 16ms 遅延 (60 FPS)    │
   │   [ Local Agent Core — 軽量 C++/Rust 制御コア ]           │
   │                                                            │
   │  ┌─────────────┐   ┌──────────────────┐   ┌────────────┐ │
   │  │1.ストリーム映像感知 │   │2.局所状態機械の評価 │   │3.OS級実行器 │ │
   │  │ OD/OCR/追跡  ├──►│ Delta-View 解析   ├──►│ マウス/キーボード │ │
   │  │ 60FPS 映像流 │   │ V-OS Event Loop  │   │ ドライバ注入 │ │
   │  └─────────────┘   └──────────────────┘   └────────────┘ │
   └────────────────────────────────────────────────────────────┘
```

二環の責務分担こそ、アーキテクチャ全体の魂だ：

| 次元 | 遅延大脳環（Slow Loop） | 高速反射環（Fast Loop） |
|---|---|---|
| 載体 | クラウド GPU/TPU クラスタの大模型 | ローカル / Edge MicroVM の軽量 C++/Rust コア |
| 遅延 | ~1.0 秒（一回の計画） | < 16 ミリ秒（60 FPS に対応） |
| 頻度 | 低頻度、「判断が要る」時のみトリガー | 持続的、毎フレーム走る |
| 出力 | マクロな意図 + 予期アンカー、**ピクセルは管理しない** | マイクロ秒級の精密アライメント、連続入力、軌跡修正 |
| 例え | 指揮官（戦略を司る） | 特殊部隊（引き金を引くその 16ms を司る） |
| 走らせるモデル | 大型マルチモーダル LLM | **端側小モデル（YOLO / OCR 級）** |

鍵となる設計原則を一言で：**我々はクラウドの大模型に「マウスが一ミリ秒ごとにどこへ動くか」を管理させない**。大脳は「ログインフォームを見つけて入力を済ませる」という意図だけを下し、ローカルの超高速 Event Loop が自前で 60 FPS の監視・アライメント・連続入力を完成させ、ほとんどの動作はクラウドへ返さない。画面に大脳が「予期していなかった」激変が起きた時にだけ、大脳を驚かせる。

> ⚠️ 真偽メモ
> 原素材は遅延大脳環を具体的な製品型番に結び付け、遅延を「~1.0 秒 / < 16 ミリ秒」と精密に記している。実際の数値はモデルサイズ、ネットワーク RTT、ローカル演算力に大きく依存するため、**本書はこれを「桁の目安」として扱う**：遅延環は「秒」級、高速環は「ミリ秒～十ミリ秒」級、この**桁の差（約二桁）こそがアーキテクチャ成立の前提**であり、具体的な数値を定論としてはならない。

> 🔍 掘り下げ解説 ―― 二環アーキテクチャは同時にどの難関を撃ち抜くのか
> 二環は美しい比喩ではなく、前述の三類の難関への**体系的な解**だ。第 7 関（P99 TTFT）に対して：滑らかさはもはやクラウドの往復で決まらない。「手に追従する」部分はそもそもクラウドへ行かず、人間の目が見る反応は 16ms のローカル環が担い、クラウドの 1 秒の遅延は「次の戦略を考える」バックグラウンドに隠れ、ユーザーには感知されない。第 6、8 関（Token / KV 爆発）に対して：ローカル環は毎秒 60 フレームを見るが、**毎フレームをクラウドへ送るのではない**――key-frame だけが Vision Token へ符号化され、平時に送るのは軽量なイベントログ（場面3を参照）で、token 消費は「毎フレーム一画像」から「激変ごとに一画像」へ下がる。これは桁レベルの節約だ。第 9、12 関（環境の副作用、複数ウィンドウ切り替え）に対して：ローカル環は画面を持続的に睨み、突発ポップアップは 16ms 以内に捕捉され、三秒に一度のスクリーンショットで「たまたま見える」のを待たない。**最も深い意味はこうだ：二環アーキテクチャは「知能」と「即時性」という生来矛盾する二つの要求を、空間で時間を買う方式で解いた――知能は遠隔に置き（遅延を許容し、深さを追う）、即時性はローカルに置き（深さを犠牲にし、速さを追う）、その間を狭帯域の意図チャネルでつなぐ。** これはまさに分散システムの最も古典的な知恵だ：二つの要求が同一ノードで同時に満たせないなら、遅延特性の異なる二つのノードへ分け、最小化した同期プロトコルを設計する。二環が Computer Use にとって持つ意味は、CPU の L1 キャッシュがメインメモリにとって持つ意味と同じだ――君はメインメモリを速くしたのではなく、その前に小さくて極めて速いものを置いたのだ。

## 場面3　人間的視覚：H.264 の頭で画面を見る（Delta-Frame 増分符号化）

**背景**：人がコンピュータを見るのはスライド（一枚一枚の静的 slide）を見ているのではなく、連続的な視覚の流れだ。だが本当に毎秒 60 枚のスクリーンショットをクラウドの大模型へ送れば、帯域と演算力は瞬時に崩壊する――これこそ第 6 関 Vision Token 爆発の極端版だ。どうすれば AI に「見続け」させつつ「燃やし続け」させないか？答えは映像圧縮の工学に二十年も眠っていた：**H.264 の I-frame / P-frame 機構**だ。

H.264 は映像の毎フレームを完全な画像として保存しない。フレームを二種に分ける：**I-frame（キーフレーム / Intra-frame）**は完全に自己完結した一枚の画像、**P-frame（予測フレーム / Predicted-frame）**は「前フレームに対し、どの領域が動いたか、どう動いたか」だけを記録する。画面が静止した映像なら、P-frame はほぼゼロまで小さくできる。一組の I-frame に後続の一連の P-frame を加えたものを、**GOP（Group of Pictures）**と呼ぶ。

これを Computer Use の視覚パイプラインへ持ち込む：

| フレーム種別 | トリガー時機 | どう処理するか | 誰に送るか |
|---|---|---|---|
| **キーフレーム（Key Frame）** | 画面の激しい遷移（ページのリダイレクト、新ウィンドウ、Modal のポップアップ） | スクリーンショット全体を Vision Tokens へ符号化 | クラウド遅延大脳へ送り、全体認知を再構築 |
| **増分フレーム（Delta Frame）** | 平時の操作（タイピング、メニューのドロップダウン、アニメーション、loading の回転） | ローカルで CV を使い「ピクセル変化が起きた局所の Bounding Box」をロック | **画像は送らず**、軽量なイベントストリームへ変換して大脳へ送る |

ローカル側はどうやって「どこが動いたか」を判断するのか？二本足で歩く：

1. **ピクセル層 — 光流（Optical Flow）/ 端側検出**。Optical Flow はコンピュータビジョンの古典的アルゴリズムで、隣接する二フレーム間の各ピクセルの運動ベクトルを計算し、「画面の中で変化している領域」を精密に枠で囲める。端側の小型検出モデル（YOLO 級）と組み合わせて物体レベルの変化定位を行えば、ローカル環はミリ秒以内に「画面のどの矩形が動いたか、どう動いたか」を答えられる。

2. **意味層 — OS イベントストリーム**。光流は「ピクセルが変わった」ことしか知らず、「なぜ変わったか」を知らない。だからローカル環は同時に OS の構造化イベントを購読する――ウェブページ内の DOM 変動（現代では `MutationObserver` がノードの挿入/属性変更を返報し、廃止された `DOMNodeInserted` 突然変異イベントに取って代わった）、デスクトップ側の **Accessibility API イベント**（macOS の AX / Windows の UI Automation がコントロールのフォーカス、値の変更、ウィンドウのアクティブ化を返報する）。これらのイベントはテキストで、極小で、しかも意味を帯びている。

両者を結合し、ローカル環は「画面で今何が起きたか」を一本の **Event Token Stream**（イベント token 流）へ圧縮して大脳に与える：

```
従来範式 (毎フレーム一画像):
  frame_0 [全画像 ~2000 tok] → frame_1 [全画像 ~2000 tok] → frame_2 [全画像 ~2000 tok] ...
  → 60 フレーム/秒 × 2000 tok = 災害

Delta-Frame 範式:
  [I-frame 全画像 ~2000 tok]   ← ページ遷移時のみ
     ├─ P: {evt: text_input, box:[210,240,330,260], val:"nol"}   ~15 tok
     ├─ P: {evt: dropdown_open, box:[400,300,520,460]}            ~12 tok
     ├─ P: {evt: a11y_focus, ctrl:"submit_btn"}                   ~10 tok
     └─ ... (大脳は「テキストログ」を読むだけで世界モデルを更新, もう画像は不要)
```

この設計の精髄は最後の一句にある：**大脳は多くの場合「画像を見る」必要はなく、「ログを読む」だけで環境に対する内部状態をリアルタイムに更新できる**。画像を見るのは高い（数千 token、秒級の推論）、ログを読むのは安い（数十 token、秒に何度でも可能）。これは第 6 関（Token 爆発）と第 7 関（TTFT ボトルネック）を一挙に押し下げる――大脳のほとんどの「更新」が、純テキストの安価な推論になる。

> ⚠️ 真偽メモ
> 「変化領域だけ送れば N% の帯域 / token を節約できる」といった具体的なパーセンテージは、原素材が検証可能な出典を示しておらず、UI の動的度合いに強く依存する（狂ったように跳ね回る広告ページなら、P-frame も大きくなる）。本書は**方向が正しい**ことだけを主張する：「毎フレーム全画像」を「キーフレーム全画像 + 増分イベントストリーム」へ換えれば桁レベルで必ず節約できるが、**どれだけ節約できるかは場面次第で、普遍的な数値は存在しない**。Optical Flow、H.264 GOP、Accessibility / DOM イベント自体はいずれも真に成熟した技術だ。

> 💡 ひとこと助言

## 場面4　弾道マウスと固有感覚：「動作」もまた人間らしく仕立てる

**背景**：人間らしく見えた後は、動きも人間らしくせねばならない。人がマウスを動かすとき、決して目標座標へ瞬間移動（teleportation）はしない。加速度を帯びた放物線を描き、まず加速し、後で減速し、最後に目標付近で微調整して締めくくる。人がタイピングするのも、文字列を一気に入力欄へ押し込むのではなく、連続して叩き、目はカーソルを睨んで随時誤りを正す。従来の Bot はポインタを「パッ」と `(x, y)` へ設定する――これは第 18 関（ドラッグ/ホットキーの物理シミュレーション）に対応できず、反クローラの行動軌跡分析に極めて見破られやすく、さらに第 1、3 関を突く：目標が動いていれば（スライドする広告バナー、ロード完了直後に位置が変わったボタン）、瞬間移動の座標は必ず空振りする。

**第一の手：弾道マウス軌跡（Ballistic Trajectory）**。ローカルの高速反射環はここで人間工学の古典的法則――**Fitts's Law（フィッツの法則）**を導入する。Fitts's Law は人間のポインティング動作の時間が、「距離」と「目標サイズ」と**対数関係**にあると記述する：目標が遠いほど、小さいほど、所要時間は長くなる（正確には、移動時間は log₂(2D/W) に比例する。D は距離、W は目標幅）。そして人間のポインティングがこれほど速く正確なのは、動作が自然に「高速突進 + 末端微調整」の二段に分かれるからだ――一段の**開ループ**の弾道突進でまず手をおおよそ目標付近へ振り、続いて一連の**閉ループ**の小修正動作で締めくくる。この二段式構造こそ、後の最適化サブ動作モデル（optimized-submovement）が Fitts's Law の成因を説明したものであり、まさにローカルコントローラが複製すべき対象だ。これに基づき、ローカルコントローラは座標へ直接ジャンプせず、一本の **最小ジャーク軌跡（Minimum-Jerk Trajectory）**を生成する：

> 「ジャーク（Jerk）」は加速度の変化率（位置の三階導関数）だ。人間の自然な肢体運動は**ジャークの最小化**へ傾く――つまり加速度が滑らかに変化し、「瞬間的な急加速 / 急ブレーキ」の硬さがない。最小ジャーク軌跡は数学的には**五次多項式の曲線**で、位置は滑らかな S 字を描き、速度は**対称な釣鐘曲線**（出だしは徐々に速く、中段が最速、末端はゼロへ近づく）となる。これこそ人の手の移動の物理的特徴だ。

```
従来 Bot:                       弾道軌跡 (Minimum-Jerk):
  (起点) ──[瞬移]──► (終点)      (起点)
  速度 ▔▔│▔▔▔▔▔▔▔│              速度  ╱▔▔▔╲          ← 滑らかに加速→巡航→減速
        瞬間満速 瞬間停                ╱       ╲
                                ─────╯         ╰──── 末端微調整でアライン
  → 行動分析に一目で見破られる   → 人間の生理に合致, かつ移動中も視覚追跡を継続
```

しかも――ここが鍵だ――**マウスが移動するその二百ミリ秒の間、視覚追跡（Visual Tracking）は持続的に開いている**。もし目標ボタンが移動の途中で左へ 5 ピクセル漂えば（広告のスクロール、レイアウトの再配置）、ローカル反射環は**リアルタイムに終端座標を修正**し、一度の「動的追跡クリック」を完成させる。これこそ二環の威力だ：このミリ秒級の途中補正は、クラウドのあの 1 秒遅延の大脳には到底間に合わず、ローカルの 16ms の環だけができる。

**第二の手：固有感覚フィードバック（Proprioceptive Feedback）――Type-and-Verify**。人はタイピングするとき、一文字打つたびに画面を見たりしない。目はカーソルを睨み、指は持続的に入力する。これに対応し、ローカル側は非同期の **Type-and-Verify バッファ**を実装する：

```
[大脳が下達] 入力指令: "nolacc00103"
        │
        ▼
[ローカル実行器] 擬人的なリズムで OS ドライバ層へキーボードイベントを連続送出
        │
        ├──► (並行) [ローカル OCR スレッド] 入力欄の文字変化を持続的に監視
        │              │
        │              ▼
        │         ウェブページのもたつきで 2 文字落ちたのを発見 → "nolac00103" (c0 が欠落)
        │              │
        ▼              ▼
[ローカル自動訂正] 自前で Backspace を実行し打ち直す ── クラウド大脳を驚かせる必要なし!
```

最後の一句に注目：欠字の訂正はローカル OCR スレッドがミリ秒以内に閉ループで完了し、**クラウド大脳が「削除して打ち直せ」と下達するのを待つ必要はない**。これが「固有感覚」の工学的含意だ――身体は自分がどこまでやったか、歪んでいないかを自分で知っており、いちいち中枢へ報告する必要はない。これは第 10 関（イベント返却の欠如）と第 19 関（幻覚クリックのデッドロック）の一部を直接撃ち抜く：ローカル環が「さっきの一手は効いたか」という閉ループの知覚を手にする。

> 🔍 掘り下げ解説 ―― なぜ「人間らしく動く」ことが同時に滑らかさであり、反検出であり、正確さでもあるのか
> 弾道軌跡という手が最も巧妙なのは、それが一石三鳥であり、しかも三羽の鳥がまったく異なる三類の難関に属することだ。**その一は滑らかさ（第 7 関）**：滑らかな最小ジャーク曲線はユーザーの目には「自然な手」に映り、瞬間移動は機械的で不快なものだ。**その二は反検出（第 15 関）**：Cloudflare 級の行動分析はまさに「軌跡が人間の生理に合致するか」で人と bot を区別しており、本物の Fitts's Law 曲線はどんな User-Agent 偽装よりも見破られにくい――ここには微妙な倫理的緊張があり、本書は明言せねばならない：**同じ擬人軌跡技術を、「自動化テストが誤って弾かれないようにする」のに使えば正当だが、「自動化を明確に禁じた反クローラの壁を回避する」のに使えば第 15 関の法的グレーゾーンを踏む――技術は中立で、境界は君が叩くのがどの壁かにある**。**その三は正確さ（第 1、3 関）**：移動中の持続的な視覚追跡による補正は、エージェントが「移動中の目標」を命中させることを可能にする。これは「先に座標を計算してから瞬間移動する」あらゆる開ループ方式にはできないことだ。一つの設計が同時に三類の難関を解けるとき、それは通常、それがある**より下層の共通原因**を突いたことを意味する――そしてここでの共通原因はこうだ：**現実世界の UI は動的で、連続的で、物理的慣性を持つ。それを「静的座標系上の離散的なクリック」としてモデリングするあらゆる方式は、世界の本質に逆らっている。弾道軌跡が全面的に勝るのは、ただ「UI は動き、手は追い、目は追う」というこの物理的事実を、正直に認めたからにすぎない。**

## 場面5　能動的プリエンプティブ割り込み：ポップアップが第 2 ステップで突然飛び出した時

**背景**：本当に「人間のように絶えずウィンドウのメッセージに反応する」には、システムは**割り込み（Interrupt）能力**を備えねばならない。エージェントが十ステップの帳票エクスポートタスクを実行中、第 2 ステップまで進んだところで、ウェブページが突然「接続がタイムアウトしました、再ログインしてください」または「システム異常が発生しました」という遮断的な Modal をポップアップしたとしよう。夢遊病者エージェントはどうなるか？それは元の計画どおり第 3 ステップで押すべき位置を押しに行く――が、その位置は今やポップアップに覆われており、ポップアップ上をクリックして、さらに予測不能な結果を引き起こす。これは第 9 関（環境の副作用 / Task Drifting）の最も典型的な爆発現場だ。

人間がこの種のポップアップに遭遇した時の反応は**反射的で、プリエンプティブだ**：君は「計画どおり打ち終えてからそのポップアップに対処する」ことはしない。手を止め、目を上げ、まずポップアップを処理する。この反射を複製するため、アーキテクチャは**能動的プリエンプティブなイベント駆動実行（Preemptive Event-Driven Execution）**を採用する：

```
[ローカル高速反射環] ── 持続的に 60FPS で画面を監視
        │
        ▼
  ┌─────────────────────────────────────────────┐
  │ 軽量視覚分類器 (特化版の端側小モデル)          │
  │ 専門に走査: 赤/黄の警告色ブロック、Modal の背景色、 │
  │           error/warning/notification アイコン │
  └───────────────┬─────────────────────────────┘
                  │ 深刻な遮断ポップアップを検出 (Severity: High)
                  ▼
  ① [スレッドプリエンプション] 即座に OS へ SIGSTOP を送出
       └─ 現在のすべてのマウス/キーボードのシミュレーション入力を凍結 (Freeze Action)
                  │
                  ▼
  ② [帯域外呼び出し] 通常の対話キューを迂回し, 高優先の gRPC Out-of-Band Call を通す
       └─ "大脳へ報告: タスク中断! 画面に予期せぬ遮断が出現, ポップアップのスクショ添付, 指令を請う"
                  │
                  ▼
  ③ [大脳が再計画] 瞬時に旧 Plan をクリアし, General Reasoning で干渉を評価
       ├─ ゴミ広告なら → X をクリックして閉じる指令 → 元タスクへ復帰
       └─ システムエラーなら → Exception Handling 分岐へ切り替え
```

この機構で使われる真の工学的プリミティブを一層ずつ分解する：

| 機構 | 技術プリミティブ | なぜそれを使うのか |
|---|---|---|
| 異常分級 | 端側軽量視覚分類器 | クラウドを煩わせず、ローカルでミリ秒級に「これは割り込みが要るポップアップか」を判断 |
| スレッドプリエンプション | **SIGSTOP** 信号 | POSIX の本物の信号、対象（入力実行器）プロセスを瞬時に凍結する。その鍵となる特性は**プロセス自身が catch / 無視 / ブロックできない**こと（SIGTERM と異なる）で、だから必ず凍り、「ブレーキ信号が忙しいスレッドに食われる」ことが起きない――「現在の動作が終わるのを待つ」より遥かに速く、信頼性も高く、ポップアップ上の誤タッチを確実に防ぐ |
| 帯域外呼び出し | **高優先 gRPC Out-of-Band Call** | 通常の対話から分離した高優先チャネル（HTTP/2 ストリーム優先度）を通し、緊急の割り込みが前で待つリクエストに**ヘッドオブラインブロッキング（head-of-line blocking）**されるのを避け、十ステップ計画の後ろに並ぶのではなく割り込んで大脳へ届く |
| 動的自適応 | 大模型 General Reasoning | 広告なら閉じ、エラーなら例外分岐へ――この「見知らぬ干渉を秒級で評価する」のはまさに大模型がやるべきで、小モデルにはできないことだ |

この設計は責務をきれいに切り分ける：**「異常検出 + 緊急凍結」は反射であり、ローカルに置く（速く、知能不要）。「異常評価 + 再計画」は判断であり、クラウドに置く（遅く、知能が要る）**。ローカルは「ブレーキを踏む」を担い、クラウドは「この後どこへ行くか決める」を担う――人間の「まず反射的に手を引っ込め、それから理性で対策を考える」二段階の反応と、寸分の隙もなく合致する。

> ⚠️ 真偽メモ
> 「分類器が 30 ミリ秒以内に検出」「ポップアップのスクショを即座に至急便で送る」といった具体的な遅延やフロー詳細は、アーキテクチャの図示であって実測指標ではない。SIGSTOP、gRPC、Out-of-Band 通信自体はいずれも真に成熟した技術だが、「ある製品が本番環境である遅延を出した」という類の主張を本書は保証しない。

> 💡 ひとこと助言
> 君に一言――**システムが成熟しているかどうかは、追い風で何速で走れるかではなく、予期せず中断された時に、即座に「手を止める」のか、それとも「強行する」のかで見る――優雅に中断できることは、滑らかに実行できることより十倍難しく、十倍値打ちがある。** このプリエンプション機構を見よ、最も直感に反するのは：**それは「停止する」という動作を、「続行する」よりも高い優先度で、より速い反応で設計していることだ。** ほとんどのエンジニアの本能は「メインフローがどれだけ滑らかに走るか」を最適化することだが、真の達人は知っている、システムの命はメインフローで死ぬのではなく、「計画にないポップアップ、タイムアウト、エラー」で死ぬのだと――そしてそれらに対し、最も致命的なのは反応の遅さではなく、**停まるべき時に停まらず、誤った状態の上に操作を積み重ね続けること**だ。SIGSTOP が凍結するその瞬間の知恵は、本質的に飛行機の失速保護と、データベースのトランザクションロールバックと同じ種類だ：**「自分は理解できない状態の中にいるかもしれない」と認め、だから第一反応は現場を凍結し、不可逆な副作用を守り、上へ助けを求めることで、時代遅れの計画を頼りに前へ突っ込み続けることではない。** アマチュアのシステムの勇敢さは「事に遭っても続行する」ことで、プロのシステムの勇敢さは「理解できない事に遭ったら、その場で手を止めて潔く負けを認める」ことだ――後者のあの急ブレーキには、「世界は私の計画より複雑だ」ということへのシステム全体の畏敬のすべてが宿っている。

## 場面6　なぜ Mac の Computer Use は Windows より成熟しているのか

**背景**：多くの人にある直感がある――Mac 上の Computer Use / 自動化制御は、Windows より安定し、成熟しているようだ、と。これは錯覚ではない。マルチモーダル大模型が OS と「知覚―動作の閉ループ」を組むとき、Mac の最下層アーキテクチャは生来五つの工学的優位を占める。だがこの比較は客観的でなければならない――Mac は「環境が綺麗」で勝ち、Windows は「歴史の重荷」で負ける。だが Windows こそ、企業の旧システムでは避けられない戦場なのだ。

五つの次元の比較：

| 次元 | macOS の優位 | Windows の痛点 | どの難関を突くか |
|---|---|---|---|
| **ハードウェアの単一性** | Apple は M 系列チップから macOS まで垂直統合し、Retina スケーリング、ディスプレイドライバ、メモリアーキテクチャが完全に一貫し、座標マッピングにハードウェア由来のずれがない | 何千何万種ものハードウェア組み合わせ上で走り、各社の DPI スケーリング（125%/150%）で最下層レンダリングに微細な差が生じ、系統的な「照準のずれ」を招く | 第 2 関 DPI 自適応 |
| **Accessibility の一貫性** | Cocoa/SwiftUI が高度に統一され、Accessibility API が成熟し強制的に規範化され、コントロールの位置・ラベル・階層が明確で綺麗 | 同一画面に Win32 / WinForms / WPF / UWP / WinUI 3 / 企業の旧フレームワークが混在し、UI Automation ツリーがしばしば断裂し、あるいはそもそも未実装 | 第 5、10 関 コントロールの意味 / イベント返却 |
| **ウィンドウストリーミングへの投資** | Continuity（デバイスミラーリングなど）のため、WindowServer グラフィックサーバに低遅延で解像度を動的調整できる仮想ウィンドウストリーミングを深く植え込んでいる | 従来は RDP / 仮想ディスプレイドライバに頼り、歴史の重荷が重く、高フレームレートのウィンドウキャプチャが CPU/GPU を食う | 第 7、16 関 ストリーミング遅延 / サンドボックスのオーバーヘッド |
| **TCC サンドボックス認可** | 早期に **TCC（Transparency, Consent, and Control）**を確立し、画面キャプチャ/マウス制御はすべてユーザーの明示的な認可を要し、開発者は強制的に安全防衛線の内に枠付けられる | AI の画面記録/制御を推進する際、最下層の隔離が不十分で、セキュリティ論争を指摘されたことがある | 第 13、14 関 インジェクション / PII 漏洩 |
| **UMA ゼロコピー** | M 系列の**統一メモリ（UMA）**で、CPU/GPU/Neural Engine が高帯域メモリプールを共有し、ローカル小モデルがスクリーンショットのキャッシュを**ゼロコピー**で読み、ミリ秒級の反応 | 新世代の NPU でさえ、システムメモリ/単体 GPU の VRAM との間でデータ移動のバス遅延が残る | 第 8 関 + 高速反射環の性能 |

最後の行 UMA は、ちょうどこの章を本書全体の主軸へ接続する。場面2の**高速反射環は、ローカルで軽量な視覚/OCR モデルを走らせ 60 FPS のリアルタイム反応を行う必要がある**――このローカル小モデルこそ、前の各部で語った**蒸留 / 量子化で生み出した端側小モデル（YOLO / OCR 級）**の完璧な活躍の場だ。そしてそれが速く走るかどうかは、「スクリーンショットのキャッシュを読むのに PCIe バスのコピー遅延を経るか否か」に極度に依存する。UMA アーキテクチャ上では、CPU が撮った画像と、GPU/NPU が推論する画像は、**同じ一塊のメモリの中の同じ一份のデータで、ゼロコピー**だ――これこそ第 3 章で DGX Spark UMA を語った際に繰り返し強調したキラー級の特性だ。**高速反射環の物理性能の上限は、本質的に「端側小モデル + ゼロコピーメモリ」というこの組み合わせで決まる**：モデルは十分小さくあらねばならず（前の各部の蒸留/量子化が担う）、メモリはゼロコピーでなければならない（UMA が担う）。両者のどちらを欠いても、60 FPS のローカル閉ループは走らない。

> ⚠️ 真偽メモ
> 表中の「あるバージョンの macOS がある仮想ストリーミング技術を内蔵」「Windows のある AI 記録機能がある時点であるチームにある手法でデータを抽出された」といった**具体的な製品、バージョン、脆弱性のタイムラインと事件**は、原素材の記述を本書が独立に検証できないため、**一律に未実証の主張とみなし**、「方向性のある比較」の参考としてのみ扱い、事実としては引用しない。通常どおり記述できるのは技術機構そのものだ：TCC は本物の macOS 権限フレームワーク、Apple Silicon UMA は本物のアーキテクチャ、Accessibility API / UI Automation はいずれも本物のアクセシビリティ層だ。

> 🔍 掘り下げ解説 ―― Mac の「成熟」は閉鎖がもたらす配当、Windows の「混沌」は君が避けられない現実世界
> この五つの次元をつなげて見ると、より深い判断が浮かび上がる：**Mac が Computer Use で先行しているのは、Apple が AI のために特別に設計した何かをしたからではなく、過去二十年の「閉鎖型垂直統合 + 高基準の UI 規範」の副産物として、図らずも AI エージェントが最も好む「高確定性環境」を作り出したからだ。** ハードウェアの単一性→座標が漂わない、API の統一→状態が綺麗、UMA→ローカルモデルがゼロコピー、TCC→セキュリティにガードレール――どれもが「環境のエントロピーが低い」ことだ。そして AI エージェントは本質的に環境の不確定性に抗うシステムであり、環境のエントロピーが低いほど、成功しやすい。**だがここには、ベテランエンジニアが醒めていなければならない罠が潜む：低エントロピー環境で demo が通っても、問題を解決したことにはならない――君はただ環境の綺麗さを借りただけかもしれない。** Windows のあの、三十年五世代の UI フレームワーク、万種の DPI 組み合わせ、無数のアクセシビリティツリーの断裂が混ざり合った旧システムの世界こそ、ほとんどの企業自動化の真の戦場だ――銀行の基幹システム、工場の MES、政府の帳票端末、ただの一つも macOS 上では走っていない。**だから真の工学的成熟度とは、「Mac でどれだけ綺麗に作ったか」ではなく、「自分のアーキテクチャが Windows のあの高エントロピーの荒れ放題の中でも立っていられるか」だ。** 二環アーキテクチャ、Delta-Frame、固有感覚フィードバックといった設計が重要なのは、まさにそれらが**環境の不確定性に抗う武器**だからだ――そして Windows のような不確定性に満ちた場所でこそ、これらの武器の価値が際立つ。Mac は君に問題が簡単だと思わせ、Windows こそ問題がどれほど難しいかを教える：**Mac でしか検証していない Computer Use 方式と、Windows の旧システムで揉まれてきた方式の差は、プラットフォームではなく、それが本当に「世界の混沌」を見たことがあるかどうかだ。**

## 本章のまとめ

- **二十の難関は五類に分かれる**：知覚と定位（ピクセルずれ/DPI/スケルトンスクリーン/無限スクロール/意味的断層）、システム遅延コスト（Token 爆発/TTFT/KV 圧迫）、実行時の状態機械（副作用/イベント返却なし/焦点喪失/複数ウィンドウ）、セキュリティ隔離（インジェクション/PII/CAPTCHA/サンドボックス）、人間とのエッジ（MFA/ドラッグ/幻覚クリック/監査可能性）。**失敗シグネチャを見る方が定義を暗記するより役に立つ**――本番で君が見えるのはシグネチャだけだ。
- **核心の病根**：従来の「スクリーンショット→考える→クリック」は開ループの夢遊病者範式であり、モデルは自分の誤りを知らない。SoM / Context Caching は同一の開ループ管路を広げるだけで、対症療法にすぎず根治ではない。
- **人間的操作の四本柱**：① **二環デカップリング**（クラウド遅延大脳~秒級が戦略を司る / ローカル高速反射~16ms が引き金を司り、空間で時間を買って「知能 vs 即時性」の矛盾を解く）；② **Delta-Frame 増分符号化**（H.264 I/P-frame + 光流 + OS イベントストリームを借り、大脳は多くの場合ログを読み画像を見ず、Token 爆発を押さえ込む）；③ **弾道マウス + Type-and-Verify**（Fitts's Law 最小ジャーク曲線 + 移動中の視覚追跡補正 + ローカル OCR の欠字自動補打、一手で滑らかさ/反検出/正確さを同時に解く）；④ **能動的プリエンプティブ割り込み**（端側分類器が警告ポップアップを検出 → SIGSTOP で凍結 → 高優先 gRPC 帯域外呼び出し → 大脳が再計画、「手を止める」を「続ける」より優先に設計する）。
- **Mac vs Windows**：Mac はハードウェアの単一性/Accessibility の一貫性/WindowServer ストリーミング/TCC サンドボックス/UMA ゼロコピーで勝つ――だがそれは閉鎖統合の低エントロピー配当だ。Windows の高エントロピーの混沌こそ、企業の旧システムでは避けられない真の戦場であり、アーキテクチャの成熟度を試す試金石でもある。
- **主軸への接続**：高速反射環のローカル小モデル（YOLO/OCR 級）こそ、前の各部で**蒸留/量子化**した端側モデルの主戦場であり、その性能の上限は「小モデル + UMA ゼロコピー」というこの組み合わせで決まる――第 3 章 DGX Spark UMA に呼応する。

「歩いては止まる」がついに「筋肉の記憶」へと磨き上げられたとき、次章では別の次元の問題へ移る：このローカルエージェントに「人間の目のために設計された画面」で苦労して腕を磨かせるくらいなら、いっそこう問おう――コンピュータはなぜ、いまだに人間に操作させるための姿をしていなければならないのか？
