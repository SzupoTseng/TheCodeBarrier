# 第五部　パイプライン

# 第5章　五大モジュールと四十ステップ：自動蒸留パイプラインの解剖

> シリコンバレー、あるAIスタートアップのホワイトボードの前。誰かが「自分でClaudeを蒸留する」という行為を、手なりのスクリプト一本から、五つのマイクロサービスと四十のステップにまたがるアーキテクチャ図へと描き起こした。図は美しく、実に「エンタープライズ級」だ。だがホワイトボードの隅には、赤ペンで丸く囲まれた小さな一行がある——**「M1：ベンダーの防御を回避する」**。この章は、この図をブロックごとに分解し、どのブロックは自分の生産ラインに写経する価値があり、どのブロックは触れると事故になるのかを伝える。

> ⚠️ 本章の立場とレッドライン
> このパイプラインは「エンタープライズ級自動蒸留システム」として包装されている。その**システムアーキテクチャ（疎結合マイクロサービス、非同期データフロー、分散学習、評価サンドボックス、量子化デプロイ）は正当かつ優れたエンジニアリング**であり、「自分が保有する/オープンウェイトのモデルを蒸留する」という合法的なシナリオに置き換えてもそのまま適用できる。本書はそれを惜しみなく分解する。**だがモジュール一（M1）の核心は、特定ベンダーの防御とレート制限を回避することにある**——本書は**それがアーキテクチャ上で果たす役割とリスクを記述するのみで、実行可能な回避スクリプト・敵対的プロンプト・検知回避パラメータは一切提供しない**。なぜこの線を引くのか、その理由は第6章で徹底的に語る。

## 場面1　俯瞰：五つの疎結合マイクロサービスと、あのアーキテクチャ図

**背景**：原素材は、エンタープライズ級の自動蒸留は「おもちゃのスクリプトを語らない」ものであり、五つの**疎結合（decoupled）マイクロサービス**で構成され、データがパイプラインのように一端から他端へと流れていくべきだと主張する。

**五大モジュール（データフロー順）**：

| モジュール | 名称 | 職責 | 信号 |
|---|---|---|---|
| **M1** | おとりと軌跡採取（Honeypot & Trace Ingestion）| 教師の防御分類器に対抗し、CoTとツール軌跡を非同期で取得 | ⚠️ ベンダー防御の回避、本書は実装化しない |
| **M2** | 二重アライメントとフィルタリング（Dual-Alignment & Quality Filter）| クレンジング、匿名化、自己宣言の除去、意味的リライト | 🟢 正当（自己宣言の除去は下記の注を参照）|
| **M3** | 分散非同期コントラスティブ学習（Distributed Contrastive Trainer）| 混合損失、ZeRO-3、長コンテキスト拡張 | 🟢 正当な核心 |
| **M4** | 評価とレッドチームサンドボックス（Eval & Red-Teaming Sandbox）| ベンチマーク、フォーマット校正、レッドチーム演習 | 🟢 正当 |
| **M5** | 極限量子化とエッジオーケストレーション（Quantize & Edge Orchestrator）| GB10混合量子化、KV Cache圧縮、本番投入 | 🟢 正当 |

**アーキテクチャ図（教育用、合法な蒸留の骨格データフローを示す）**：

```
              [ Teacher モデル：利用権限のある大規模モデル / オープンウェイトモデル ]
                              │  (文字ストリーム / 非同期)
        ┌─────────────────────▼──────────────────────┐
        │ M1 軌跡採取   ⚠️(クローズドAPIへの回避版は本書の範囲外) │
        │   合法版 = 自前モデルへの通常バッチ推論による採取       │
        └─────────────────────┬──────────────────────┘
                              │  (raw traces)
        ┌─────────────────────▼──────────────────────┐
        │ M2 二重アライメントとフィルタリング             │
        │   PII匿名化 → 文体/フォーマット洗浄 → 軌跡反転 → 意味的リライト │
        └─────────────────────┬──────────────────────┘
                              │  (Apache Arrow 高スループットバッチ / Redis キャッシュ)
        ┌─────────────────────▼──────────────────────┐
        │ M3 分散コントラスティブ学習                     │
        │   損失エンジン: CE + KL(ソフトラベル) + CoTアライメント + (脱検閲) │
        │   骨格: DeepSpeed ZeRO-3 / BF16 / FlashAttn / YaRN  │
        └─────────────────────┬──────────────────────┘
                              │  (checkpoint 重み)
        ┌─────────────────────▼──────────────────────┐
        │ M4 評価とレッドチームサンドボックス             │
        │   MMLU-Pro / HumanEval / GSM8K → フォーマット校正 → レッドチーム │
        └─────────────────────┬──────────────────────┘
                              │  (ベンチマークを通過した verified checkpoint)
        ┌─────────────────────▼──────────────────────┐
        │ M5 量子化とエッジオーケストレーション（DGX Spark GB10） │
        │   混合量子化(Attn Q8/MLP Q4)+imatrix → KV Cache FP8  │
        │   → numactl/mlock → Modelfile → Registry へ本番投入 │
        └─────────────────────┬──────────────────────┘
                              ▼
              [ デプロイ: Ollama / LM Studio / vLLM @ 128GB UMA ]
```

**なぜ疎結合にするのか**：これは分散システムの基本動作だ。採取・クレンジング・学習・評価・デプロイを独立したサービスに分解する——**どこか一箇所が落ちても全ラインを巻き込まない**。採取側の高並行I/Oと学習側の大計算量はそれぞれ独立してスケールできる。**Redisクラスタ + Apache Arrow**でメモリのゼロコピー（zero-copy）バッファを作れば、教師APIが突発的にレート制限をかけたり勾配更新が詰まったりしても、パイプラインは**データのデッドロック（deadlock）**を起こさない。アーキテクチャ図を魔法だと読み違えないよう、エンジニアリングの現実を一言添えておく。「五つの疎結合マイクロサービス」は**論理ビュー**であり、実装上はたいてい**Prefect / Airflow / Dagsterでオーケストレーションされた一枚のDAG**になる——各モジュールは単点リトライ可能・バックフィル（backfill）可能なtaskの集合であり、分散の採取と学習のfan-outはさらに**Ray**に支えてもらう。いわゆる「疎結合」とは、物理的にはこのオーケストレーション層があなたの代わりに依存・リトライ・ロールバックを管理してくれることであって、アーキテクチャ図を美しく描けば自動的に発生するものではない。

> 💡 ひとこと助言
> 君と交わすひとこと——**良いパイプラインは、どの一節も単独でストライキを起こせて、なお雪崩を誘発しないものだ。疎結合は見栄えのためではなく、午前三時にライン全体ではなくモジュール一つだけを再起動すれば済むようにするためにある。** このアーキテクチャの真の価値は、何か単一のテクニックにあるのではなく、「採取→クレンジング→学習→評価→デプロイ」を**ガードレールがあり、観測可能で、自動ロールバックできる**一本のパイプラインに仕立てた点にある。MLOpsの成熟度は「モデルを学習できるかどうか」では決して測れず、「壊れたとき五分以内に原因を特定し、ロールバックし、やり直せるかどうか」に現れる。**アマチュアはモデルが動いたかを気にし、プロはパイプラインが切れたときどう自己修復するかを気にする——この分界線こそ、どんなモデルスコアよりもエンジニアリングの段位を見抜く。**

## 場面2　第一段階：10個の前置ステップ（環境とデータの地盤）

**背景**：この手法は40ステップを三段階に分ける。第一段階はインフラとデータの地盤に焦点を当てる。以下に一条ずつ列挙する（これが「原資料の全覧」だ）。ツールと信号も併記する。

1. **動的APIローテーション集約プール**　⚠️ *原文ではベンダーのレート制限と検知を回避する設計——本書はその存在を挙げるのみで、実装は提供しない*。（合法的な類比：自前のモデルサービスに対する負荷分散のクライアント接続プール）
2. **敵対的おとりプロンプトライブラリ**　⚠️ *原文では「オニオン包み」で防御分類器を欺く設計——本書はいかなるプロンプトテンプレートも提供しない*。
3. **Studentシード重みの定義（Base Model Sourcing）**　🟢 GB10の帯域に合った基底（Qwen3.6-35B-A3B / Llama系）を選び、全パラメータFTかQLoRAかを決める。ツール：`transformers`、`peft`。
4. **1MコンテキストKVキャッシュ構造の初期化**　🟢 UMAに領域を事前確保し、**FlashAttention-3 + YaRN**を有効化して、百万コンテキスト外挿能力を付与する。
5. **安全な匿名化サンドボックス（PII Anonymizer）**　🟢 ローカルの軽量**NER（Presidio / spaCy）**で鍵・メールアドレス・人名をフィルタする。あらゆる学習データ処理の必修科目。
6. **分散ストレージキャッシュ（Cache Pipeline）**　🟢 **Redisクラスタ + Apache Arrow**で高スループット・低レイテンシのインメモリデータ層を構築する。
7. **Tokenizerの意味的アライメント（Vocabulary Alignment）**　🟢 教師/学生の語彙が一致しない場合、マッピング表または**語彙横断KL補償（ULD）**を構築する（第4章参照）。
8. **ベースラインのデプロイ（Baseline）**　🟢 蒸留前にまず**MMLU-Pro、HumanEval、GSM8K**を走らせ、元のスコアを記録して進捗比較に使う。ツール：**lm-evaluation-harness**。
9. **ハードウェアテレメトリ監視（Telemetry）**　🟢 **Prometheus + Grafana**でGB10のLPDDR5X帯域占有率・温度・UMAスワップ遅延を監視する。
10. **自動ヘルスチェック（Smoke Test）**　🟢 100本のエンドツーエンドテストを走らせ、五モジュールと下層ドライバ（CUDAスタック）が疎通しているか確認する。

> 🔍 掘り下げ解説──地盤段階が暴く真実：蒸留の難所は学習ではなく、データと観測可能性にある
> 注目すべきは、この10ステップのうち**実際に「学習」に触れるステップが一つもない**ことだ——すべてデータパイプライン、語彙アライメント、ベンチマークのベースライン、ハードウェアテレメトリである。これは初心者の最大の誤解をまさに突き崩す。蒸留の核心は「train.pyを一本走らせること」だと思い込んでいる誤解だ。**現実には、蒸留のエンジニアリング作業の80%は学習の前にある**。クリーンで、匿名化され、フォーマットの統一されたデータを手に入れられるか（ステップ5-6）、教師と学生の語彙をアライメントできるか（ステップ7）、蒸留が本当に強くなったと証明できる**蒸留前のベースライン**を持っているか（ステップ8）、そしてGB10をフル稼働させたときに帯域ボトルネックをリアルタイムで見られるか（ステップ9）。**とりわけステップ8のベースラインはよく飛ばされるが、最も致命的だ**——蒸留前のスコアがなければ、「今回のラウンドは進歩したのか後退したのか」にそもそも答えられず、パイプライン全体が感覚頼みのパラメータ調整というギャンブルに化ける。成熟したエンジニアリングの証は、**システムを変えに手を付ける前に、まず変化を測る物差しを建てておくこと**にある。地盤段階の真の成果物は「環境が整った」ことではなく、「**信頼できる物差し一本 + クリーンなデータパイプライン一本**」だ——この二つは、いかなる学習テクニックよりも先決である。

> ⚠️ 「40ステップ」の完全さは語りの完全さであって、エンジニアリングの完全さではない——丸ごと重複排除が抜けている
> 工程を「美しい40ステップ」に分解するのは教育の骨格として実に優れているが、それを「写経すればエンタープライズ級が保証される」リストだと思ってはいけない——**「Nステップでエンタープライズ級を保証する」と謳うパイプラインで、まず疑うべきは、それが何をこっそり抜き落としたかだ**。この図の最も明白な綻び：**40ステップのどこにもデータ重複排除（dedup）をするステップがない**。だが重複排除こそ、実際の蒸留データパイプラインで最終品質に最も影響する工程の一つだ。教師が同類のプロンプトで繰り返し採取されれば、必然的に大量の近似重複軌跡が生まれる。重複排除しなければ、学生は重複サンプルに過学習し、評価スコアが水増しされ、汎化が悪化する。実務の標準装備は二層だ。**MinHash / SimHashで近似的な字面の重複排除**（`datasketch`、Sparkレベルまでスケール可能）、**SemDeDupまたは埋め込みクラスタリングで意味的な重複排除**（「同義で異なる字面」の潜在的重複を捕まえる）。さらに**Lilac / Argilla**でデータセットの可視化巡検と人手によるサンプリング検証を加える。dedupとデータ巡検のない40ステップは、ただの40個の動作であって、成熟したパイプラインではない。

## 場面3　第二段階：20個の中核実行ステップ（データを重みに変える）

**背景**：第二段階は蒸留の核心であり、データはここで学生のシナプス重みへと変換される。一条ずつ列挙する。

**採取とデータアライメント（ステップ1-6）**：

1. **非同期並行軌跡採取（Asynchronous Tracing）**　文字ストリームを一時領域に書き込み、採取帯域を最大化する。（クローズドAPI向けの版はToSの制約を受ける、第6章参照）
2. **リアルタイム降格検知機構**　⚠️ *原文では、教師が「ダウングレード」された後に鍵を替えてリトライするのを検知するために使う——本書はその存在を挙げるのみ*。
3. **XMLタグとツール呼び出しの抽出**　🟢 `<thinking>`、`<tool_use>`と最終結果をパースし、ツール能力が学習可能であることを保証する。
4. **文体の脱タグ化と自己同一性の修正（Identity修正）**　⚠️ *自前モデルに対して「ブランド中立化」を行うのは正当だが、他者のモデルのデータで出所の痕跡を消すのは証拠隠滅だ——同じ技術、二つの道徳的重み、第6章参照*。
5. **思考連鎖の軌跡反転（Trace Inversion）**　🟢 推論ステップを逆順/再構成して丸暗記を防ぐ（第4章）。ただし反転後も論理が自己整合的であることを検証せねばならない。
6. **意味的多様化バリアント（Synthetic Paraphrasing）**　🟢 高温（T=0.85）で複数の言い回しにリライトし、過学習を防ぐ。ツール：**distilabel**（**Argilla**と組み合わせて人手巡検と選好アノテーションを行う）。

**損失と学習（ステップ7-20）**——この14ステップはM3の精髄であり、すべて正当かつ実用的な硬派の技術だ：

7. **ハードラベルの交差エントロピー損失**　🟢 学生の予測トークンと教師の最終トークンのCE。
8. **ソフトラベルのKLダイバージェンス損失**　🟢 （局所的にでも）logitsを取得できるなら、確率分布をアライメントする（第4章の温度付きKL）。
9. **思考連鎖強化損失（CoT Alignment Loss）**　🟢 `<thinking>`ブロックのトークンに重み付け（例：2.0）し、compute-over-thinkingを促す。
10. **脱検閲特徴アライメント（Abliterated Loss）**　⚠️ *原文では拒否率を下げるために使う——第2章の警告に注意：過度な脱検閲は判断力と安全直感を鈍らせる*。
11. **動的勾配クリッピング（Gradient Clipping=1.0）**　🟢 教師の異常データによる勾配爆発を防ぐ。
12. **DeepSpeed ZeRO-3の重みシャーディング**　🟢 Optimizer States/勾配/重みを動的に128GB UMAへシャーディングし、バッチスループットを最大化する（等価な選択肢：**PyTorch FSDP**）。
13. **混合精度BF16**　🟢 安定的で、GB10のTensor Coreをフルに活用する。
14. **超長コンテキストの漸進的拡張**　🟢 学習初期は8Kから始め、epochに応じて**RoPE baseを1万から100万へ拡張（YaRN）**し、シーケンスを128K→1Mまで伸ばす。
15. **マルチタスク並列スケジューリング（Sequence Packing）**　🟢 複数の短いシーケンスを一本の長いシーケンスにパッキングし、paddingの無駄を消して、GPU効率を+30%。
16. **動的学習率（Cosine Annealing + Warmup）**　🟢 最初の5% stepsをwarmupし、コサインアニーリングで安定的に収束させる。
17. **リアルタイム重みチェックポイント（Checkpointing）**　🟢 Nステップごとに評価し、validation lossが下がったときだけ非同期でNVMeに書き込む。
18. **動的データデッドロッククリーニング（GC & Memory Purge）**　🟢 epochごとに残留KV Cacheが占有するUMA領域を解放する。
19. **学習/検証損失の異常遮断（Early Stopping）**　🟢 lossが急騰するか、連続三ノードで過学習したら自動停止 + アラート。
20. **全パラメータ/LoRA重み融合（Weight Merger）**　🟢 LoRA学習後にFP32の精密融合を行い、最終的なdense/MoE重みを生成する。ツール：**mergekit**。

> 💡 ひとこと助言
> 君と交わすひとこと——**モデルを一つ学習させるのは、何か天才的なパラメータによってではなく、十数本の寡黙なガードレールの合力による。勾配クリッピングは爆発を防ぎ、early stoppingは過学習を防ぎ、checkpointは無駄を防ぎ、GCはOOMを防ぐ。どれも地味だが、一本欠けただけで三日分の計算力が一夜で消えうる。** ステップ11から19はすべて「悪いことが起きるのを防ぐ」ガードレールであり、「モデルを強くする」魔法は一つもない——これは直感に反する真実を明かす。**大規模学習の勝敗は、何を正しくやったかではなく、何種類の死に方を防いだかで決まる。** 初心者のパラメータ調整は「より高い学習率、より大きなbatch」を追い、ベテランのパラメータ調整は「何が起きてもこのラインは安全に収束するか、安全に止まれる」ことを追う。これは飛行機の操縦と同じ種類の知恵だ。パイロットの腕前は派手に飛ぶことではなく、無数の墜落の可能性を、一つずつチェックリストで塞ぎ切ることにある。

> 🔍 掘り下げ解説──この14ステップの損失と学習は、そのまま再利用できる「大規模モデル蒸留MLOpsテンプレート」だ
> 議論のある採取ステップ（1-2、4、10）を取り除けば、ステップ3、5-9、11-20は実のところ**極めて優秀で、合法的なシナリオにそのまま写経できる蒸留学習テンプレート**を構成している。その価値は、散らばったテクニックを**収束が安全で、メモリ効率が高く、長コンテキストに優しい**完全なレシピへとオーケストレーションした点にある。単独で挙げる価値のある精妙な設計が三箇所ある。**その一、損失関数の重ね合わせ戦略**（ステップ7-9）：CE（基本的な正しさを保証）+ KL（暗黙知を伝える）+ CoT重み付け（推論過程を強化）の三つを係数で配分し、「正解する・教師に似る・思考できる」の三つを同時に管理する——これは単一の損失よりはるかに精密だ。**その二、ZeRO-3 + Sequence Packingのメモリ複合技**（ステップ12、15）：前者がモデル状態を128GBにシャーディングし、後者がpaddingの無駄を消す。両者により、DGX Sparkのような「単機大容量メモリ」が、本来なら複数枚のGPUでしかファインチューニングできない大規模モデルを動かせるようになる——これはUMAアーキテクチャのキラーユースだ。**その三、漸進的なコンテキスト拡張**（ステップ14）：いきなり1Mにするのではなく、8K→128K→1Mと段階的に登り、モデルにまず短距離依存を学ばせてから長距離へ外挿させる。これは長コンテキスト学習が安定して収束するための鍵となるエンジニアリングの知恵だ。「14段を自分で手書きするプログラム」と読み違えないよう、実装の注意を一つ添える。現実のプロジェクトではこの14ステップはほぼ`train.py`を一本書くことはない——これらは**axolotl / Llama-FactoryのYAML recipe一枚**であり、LoRA/QLoRA、ZeRO-3かFSDP、sequence packing、cosine schedule、early stoppingはすべて設定項目だ。単一GPUや小資源のシナリオなら**Unsloth**に替えて2-4×のスループットとより低いVRAMを得る。この14ステップを「設定ファイルの項目表」と理解するほうが、「ゼロから実装する14個のアルゴリズム」と理解するより現実に近い。**もし社内の大規模モデルを合法的に軽量モデルへ蒸留してエッジにデプロイしたいなら、この14ステップをそのままレシピとして使い、採取を自前モデルへの通常推論に置き換えればいい——これで生産級のソリューションになる。技術それ自体は中立であり、分水嶺はただM1の教師が、あなたに搾取する権利のあるものかどうかにのみある。**

## 場面4　第三段階：10個の後処理ステップ（粗材から完成品へ）

**背景**：第三段階は、ファインチューニング後の「粗材モデル」を、GB10上で長文を高速に走らせられる生産形態へと仕立て上げる。一条ずつ：

1. **レッドチームの安全性と無検閲のバランステスト**　🟢/⚠️ 大量の極端なプロンプトで破壊的テストを行い、クラッシュや無限ループしないことを保証する（原文には「abliteratedの成功を保証する」も含む——第2章の警告参照）。
2. **文体とフォーマットの校正（Format Regularizer）**　🟢 500本のXMLツール呼び出しをテストし、タグの閉合を検査する。欠陥があれば微量の二次ファインチューニング（Epoch-0.1 Patching）を行う。
3. **能力ベンチマーク評価（Automated Eval）**　🟢 **lm-evaluation-harness**で統一してMMLU-Pro/HumanEval/GSM8Kを再実行し、ベースラインとレーダーチャートで比較して、本当に強くなったか確認する。⚠️ **ステップ14でコンテキストを128K–1Mまで一気に外挿した以上、ここには長コンテキスト評価（RULER / NIAHの干し草の山から針探し）を必ず一条追加せねばならない**——MMLU/HumanEvalしかテストせずに百万コンテキストを謳うのは、パイプライン全体で最もよくある、そして最も気まずい評価の盲点だ。
4. **重みのプルーニングと重複排除（Pruning & Dedup）**　🟢 低活性ニューロンを剃り落とし、体積を5-10%圧縮しても知能を損なわない（第2章FableVibesと同源）。
5. **GB10混合精度量子化**　🟢 アテンション層はQ8_0 / MLP層はQ4_K_M、+ imatrixキャリブレーション（第4章）。
6. **KV Cache量子化（FP8/INT8）**　🟢 長コンテキストのメモリを半減させ、GB10の大規模テキストのポテンシャルを解放する。
7. **Modelfileの自動生成**　🟢 **Ollama / LM Studio**が直接受け取れるフォーマットにパッケージし、最適化されたSystem Promptを内蔵する。
8. **NUMAバインディングとメモリロック**　🟢 `numactl --interleave=all` + `mlock`で、LPDDR5Xの全チャネルを搾り尽くす（第3章）。
9. **エンドツーエンドA/Bと性能検証**　🟢 10並行sessionをシミュレートし、tok/sと初トークン遅延（TTFT）が基準に達するか測る。
10. **自動本番投入とモデルレジストリへのアーカイブ（Registry）**　🟢 全テスト通過後にプライベートRegistry / Ollamaリポジトリへアップロードし、Cursor/ContinueのAPIルーティングを更新する。

> 🔍 掘り下げ解説──後処理こそ「使えるかどうか」の真の分水嶺であり、それが試すのはシステム思考だ
> 学習が終わればモデルは完成で、後処理は「ちょっとパッケージするだけ」だと多くの人は思う——大間違いだ。この10ステップには、「**動くだけのモデル**」と「**本番で使えるモデル**」の間の距離のすべてが潜んでおり、それを越えるのに必要なのはML知識ではなく、**システムエンジニアリングの全局観**だ。ステップ5-8の連鎖の噛み合いを見れば分かる。混合量子化（5）がモデルの大きさを決め、KV Cache量子化（6）が長文を走らせられるかを決め、Modelfile（7）がツールから呼び出せるかを決め、NUMAロック（8）がUMA上でどれだけ速く走るかを決める——**この四ステップのどれか一つでも正しくやらなければ、前段の30ステップの苦労が割り引かれる**。Q4まで量子化したのにimatrixキャリブレーションをしていないモデルは、正しく量子化したものより一段間抜けになる。NUMAをロックしていないモデルは、DGX Spark上で訳もなく半分の速度を叩き出す。XMLタグの閉合に欠陥のあるモデル（ステップ2）は、Aiderに繋ぐと頻繁にエラーを吐く。**ステップ3の「ベースラインとの比較」こそ、パイプライン全体のクローズドループ検収だ**——これは第一段階のステップ8と呼応しており、この前後二本の物差しがなければ、蒸留が結局成功したのか永遠に分からない。**後処理段階の哲学はこうだ。モデルの価値は重みそれ自体にあるのではなく、それが現実のツールチェーンとハードウェアにシームレスに組み込まれ、安定して呼び出されうるかにある——そしてこれこそ、大多数のオープンソースの「影モデル」が最も雑で、最も転びやすいところだ。**
