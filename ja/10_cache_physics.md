## 場面1　二大キャッシュ分類：推論層の KV Cache とサービス層の Prompt Cache

**背景**：大規模言語モデルの開発とデプロイにおいて、**キャッシュ（Cache）は推論レイテンシを下げ、計算コストを削るための中核の命脈**である。だが「大規模モデルのキャッシュ」は、実は一緒くたにされた漠然とした言い方だ——それは少なくとも、異なるフロアに住み、異なる問題を解き、異なる役割によって管理される二種類のものである。それらを明確に区別することが、後続のすべての最適化を理解する前提となる。**工学の第一歩は常に分類だ**：君はまず、あの百数十 GB がいったいどちらのキャッシュに食われたのかを知らなければ、節約の話など始まらない。

| 次元 | 推論層 KV Cache（キー・バリューキャッシュ） | サービス層 Prompt Cache（プロンプト / 文脈キャッシュ） |
|---|---|---|
| どこに住む | **GPU VRAM（HBM）内**、推論エンジンが動的に管理 | **サービス側**（クラウド API バックエンド / 自前ゲートウェイ）の永続層 |
| 何を解決 | 同一の生成内で、生成済み token の**再計算**を回避 | 複数の独立リクエストをまたぎ、**同一プレフィックス**の再 Prefill を回避 |
| 計算量の利得 | 自己回帰デコードを O(N²) から **O(N)** へ低減 | ヒット後、system prompt 全体の Prefill 演算力を節約 |
| 誰が管理 | vLLM / TensorRT-LLM / SGLang などのエンジン | Anthropic Claude / 阿里雲百煉 / 自前ゲートウェイ |
| ライフサイクル | **一リクエストが終わると即解放**（prefix caching を有効化しない限り） | **リクエストをまたいで生存**、複数のセッションにヒットされうる |
| 信号機 | 🟢 エンジンが自動でやるが、VRAM を食い尽くす（場面4の主役） | 🟢 明示的に標記すれば即節約（本場面の主役） |

**第一類、推論層の KV Cache**。これが解決するのは、きわめて具体的な無駄だ：Transformer は自己回帰生成のとき、各ステップで新しい token を一つしか吐けない。もしキャッシュしなければ、第 N 番目の token を生成するとき、前の N−1 個の token をもう一度すべて注意機構の行列演算にかけ直さねばならず、計算量は **O(N²)** の二乗オーダーで爆発する。KV Cache のやり方は、これまで各ステップで算出済みの Key と Value の行列を**VRAM に保存**し、後続では新たに加わったその一つの token だけを計算することで、時間計算量を O(N²) から **O(N)** に圧縮する。それは GPU VRAM の奥深くに住み、低レイヤーの推論エンジンがひそかに管理しているので、普段はまったく目に見えない——君のメモリを食い尽くすその瞬間までは。場面2・3・4で、これを数学から VRAM まで徹底的に解剖する。

**第二類、サービス層の Prompt Cache**、別名 Context Cache、文脈キャッシュ。これが解決するのは別の階層の無駄だ：君が大規模モデルに対して**同じ内容を繰り返し入力する**とき——冗長な製品マニュアル、固定された System Prompt の人格設定、一組の few-shot 例示——サービス側は毎回この文章を改めて token 化し、KV 状態を改めて計算し直さねばならない。Prompt Cache のやり方は、このプレフィックスの KV 状態を**サーバー側にキャッシュ**し、次に同じプレフィックスが来たら直接再利用して Prefill を飛ばす、というものだ。

ここに、君が必ず覚えておくべきキーワードがある：**明示的アンカー**。Anthropic の Claude API は `cache_control` という明示的標記を提供する——君はリクエスト内でそれを使って「この一段をキャッシュしてくれ」と囲い込み、ヒットすれば、**この入力 token の費用が大幅に割引される**。それと「暗黙的キャッシュ」（ゲートウェイが自動で傍受し、完全に同一のプレフィックスをマッチさせる）との違いは、明示的キャッシュは制御権を君の手に渡し、どの一段を安定したアンカーとみなすかを君自身が決める点にある。

```python
# Claude Prompt Caching：cache_control で不変プレフィックスを明示的にアンカリング
# （Anthropic Messages API、概念図示）
import anthropic
client = anthropic.Anthropic()

resp = client.messages.create(
    model="claude-opus-4-1",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": LONG_STABLE_SYSTEM_PROMPT,   # 数千 token の固定人格/ルール/ツール説明
            "cache_control": {"type": "ephemeral"}  # ★ 明示的アンカー：この KV 状態をキャッシュする
        }
    ],
    messages=[
        {"role": "user", "content": user_question}  # 動的内容は最後に置き、プレフィックスを壊さない
    ],
)
# 初回：キャッシュへ書き込み（cache_creation_input_tokens の課金はやや高め）
# 後続のヒット：cache_read_input_tokens を大幅割引で課金
print(resp.usage)  # cache_creation / cache_read の二欄を見ればヒットしたか一目でわかる
```

> ⚠️ 真偽メモ
> 原素材は「ヒット後の入力 token 費用は通常 1〜2 割にでき、最大 80%〜90% のコストを削減できる」と述べている。**キャッシュ読み取りが原価より顕著に安いという方向は本当で、各社 API のセールスポイントでもある。だが「何割になるか、何パーセント削減か」は、モデル・ベンダー・キャッシュ生存時間（TTL）・書き込み vs 読み取りによって変動する課金パラメータだ**。君が呼び出すまさにその時点の公式価格表の `cache_read` と `cache_write` の実数値を基準にしてほしく、いかなる固定パーセンテージも定論にしてはならない。本書が保証できるのはただ一点：**安定したプレフィックスの明示的キャッシュは、「同一の長い system prompt を繰り返し呼び出す」場面では、節約できるのは正真正銘の現金であり、桁数として顕著だ**ということである。

二類のキャッシュの分水嶺は、実は一言で言い切れる**黄金法則**だ：**安定した、静的な内容を最も前に置き、動的な、変化の激しい内容を最も後ろに置く**。なぜなら二種類のキャッシュは本質的にどちらも**プレフィックスマッチ（prefix matching）**だからだ——先頭の token が一つでも変われば、後ろのキャッシュ全体が丸ごと無効になる。この法則が金科玉条たるゆえんは、新人が最もよく犯す過ちが、まさに動的な変数を system prompt の先頭に押し込むことだからである：

| 🔴 よくある過ち（キャッシュを完全に無効化） | 🟢 正しい改善（ヒット率を最大化） |
|---|---|
| System Prompt に現在のタイムスタンプやユーザー ID を入れる | 動的なユーザー情報をメッセージ**末尾の user 欄**に移し、先頭を固定 |
| 毎ターンの対話で履歴サマリ（summary）を再生成する | 履歴は**追記のみで改変しない**；切り詰めるなら**最も古い**ものから削り、プレフィックスの一貫性を保つ |
| ツール定義（tools/functions）の順序がランダム | JSON ツールリストの内部の並びとフィールド構造を**厳格に固定** |
| テキストの空白・改行の細部が不統一 | クレンジングロジックを統一し、**余分な `\n` 一つ**が token 列を変えるのを防ぐ |

> 💡 ひとこと助言
> 君に一言贈ろう：**キャッシュの世界で最も高くつくのは「計算ミス」ではなく、「1 バイトを変えたこと」だ——一つの放浪するタイムスタンプ、一つの余分な改行が、君が精緻に設計したプレフィックスキャッシュ全体を煙のように消し去る。安定こそ、キャッシュ唯一の信仰だ。** あの過ちの対照表を見てほしい、一つとして「アルゴリズムが間違っている」ものはなく、すべて「構造が不安定」なのだ。これは反直感的な真実を暴く：**Prompt Cache のヒット率は、君のプロンプトエンジニアリングの「規律」によって決まるのであって、モデルによって決まるのではない。** system prompt を清潔に、固定的に書き、変わりやすい内容をすべて規律正しく末尾に追いやるエンジニアは、その請求書が、先頭に何気なくユーザー名を差し込むエンジニアの端数しかないかもしれない——二人が使っているのは同じモデル、同じ API なのに。**アマチュアは「自分が何を訊いたか」を最適化し、プロは「自分が訊くものが、何度も思い出されるほどプレフィックスとして安定しているか」を最適化する。** キャッシュは賢さを決して報いない、それが報いるのは一貫性だけだ——これはあらゆる高性能システムの底流の美学と通じている：予測可能性そのものが、一種の性能なのだ。

## 場面2　なぜ「メモリ律速」なのか：Prefill は演算力を食い、Decode はメモリを待つ

**背景**：「なぜキャッシュが大規模モデルにこれほど致命的に重要なのか」を根本から理解するには、まず一つの反直感的な事実を見抜かねばならない——**大規模モデル推論のボトルネックは、多くの場合まったく演算力ではなく、メモリ帯域幅なのだ**。これを見抜くには、推論を性格の正反対な二つの段階に分解する必要がある。

**第一段階、Prefill（プリフィル段階）**。これは君が入力したあの prompt を処理する段階だ。このとき、モデルは君が入力した**すべて**の token を同時に処理し、これらの token を高次元のテンソル（Tensor）にまとめて、一回の巨大な行列乗算に流し込める。行列乗算は GPU の大好物だ——それは Tensor Core を満腹まで養え、演算密度が極めて高い。この段階は典型的な**演算律速（Compute-Bound）**である：GPU の演算ユニットは満載で働き、メモリ帯域幅はむしろ暇を持て余している。

**第二段階、Decode（生成段階）**。これは自己回帰で字を吐く段階であり、真のボトルネックの在りかでもある。なぜなら自己回帰の本質は**一度に一つの token しか生成できない**ことだからだ。各ステップで君が手にするのは**一つ**の新しい token の Query ベクトルだけで、それを過去の**すべて**の token の Key/Value に対して注意計算にかける。行列乗算は「ベクトル × 行列」に退化する——演算量は極めて小さく、極めて速い。だが君はまず、あの巨大な歴史 KV（ともすれば数 GB）を**VRAM の HBM からチップ上の SRAM レジスタへ運んで**こなければ計算できない。こうして GPU の大半の時間は、計算しているのではなく、**運んでいる**のだ。これが典型的な**メモリ律速（Memory-Bound）**である：演算ユニットは大量に空転し、ボトルネックは HBM → SRAM の運搬帯域幅に詰まる。

```
        Prefill 段階                      Decode 段階
   ┌────────────────────┐          ┌────────────────────┐
   │ 全 prompt token     │          │ 1 個の新 token      │
   │ 大行列にまとめる     │          │ × 全歴史 KV 行列     │
   │                    │          │                    │
   │  GEMM 行列×行列      │          │  GEMV ベクトル×行列  │
   │  Tensor Core 満腹   │          │  計算は速く、運搬は遅い│
   └────────────────────┘          └────────────────────┘
   ▏Compute-Bound 演算律速▕        ▏Memory-Bound メモリ律速▕
   ▏GPU は「計算」中       ▕        ▏GPU は「メモリ運搬待ち」 ▕
```

この分裂した性格こそが、KV Cache が存在する**根本理由**であり、この章全体の物理的基盤でもある。Decode 段階は各ステップで全歴史 KV を読まねばならない——もしキャッシュせず毎ステップ再計算すれば、それは O(N²) の演算力の災厄だ。キャッシュすれば、演算は節約できるが、その代わり圧力を**演算力からメモリ帯域幅と容量へと転嫁する**。言い換えれば、**KV Cache はボトルネックを消滅させたのではなく、ボトルネックを「演算律速」から「メモリ律速」へ移したのだ**。場面1で君の 128GB が蒸発したのは、まさに Decode 段階で並行する各リクエストが、どんどん長くなる KV の尻尾を引きずって VRAM に住み着いているからである。

> 🔍 掘り下げ解説 — なぜ GQA / MQA / MLA はそろって同じものを削っているのか
> ひとたび「ボトルネックはメモリ、KV の体積だ」という前提を受け入れれば、現代の大規模モデルアーキテクチャに散らばる一群の無関係に見える設計が、突然一本の線につながる。標準的なマルチヘッド注意（MHA）では、各 Query ヘッドに独立した Key ヘッドと Value ヘッドが一つずつ付く——KV Cache の体積はヘッド数に比例する。**MQA（Multi-Query Attention）**は極端に走る：すべての Query ヘッドが**一組の** KV を共有し、KV Cache の体積は一瞬でヘッド数で割られるが、表現力が損なわれる。**GQA（Grouped-Query Attention）**は折衷だ：Query ヘッドをいくつかのグループに分け、各グループが一組の KV を共有する——Llama 3-8B の 32 個の Query ヘッドには 8 個の KV ヘッドしか付かず、KV Cache は一気に四分の一に削られる（場面4で、この 8 がどう公式に入るかを見る）。DeepSeek の **MLA（Multi-head Latent Attention）**はさらに容赦ない：KV を低ランクの潜在ベクトルに圧縮してからキャッシュし、ほんの少しの解凍計算と引き換えに極小のキャッシュ体積を得る。**この三者は表面上は注意機構の変種だが、骨の髄まで同じ工学的意図である：Decode のメモリ律速という物理的現実のもとでは、KV Cache の一バイト一バイトが VRAM 帯域幅で運ばねばならないコストなのだから、アーキテクトはモデル容量をほんの少し犠牲にしてでも、KV を小さく削りたいのだ。** 「公式中の H は KV ヘッド数であって、Query ヘッド数ではない」とだけ覚えておけば、過去三年の注意機構アーキテクチャ進化の本筋をつかんだことになる——それらはより賢くなろうとしているのではなく、**よりメモリを節約しよう**としているのだ。

## 場面3　KV Cache の数学的導出：O(T²) から O(T) へ、あの一回の連結

**背景**：これまで「なぜキャッシュするのか」を語ってきた。この節では、それを**計算して見せる**。多くの人の KV Cache 理解は「計算したものを保存しておく」という口語レベルで止まっているが、公式を広げて初めて、あの二乗から線形へ落ちる魔法が、いったいどの演算子の上で起きているのかを真に見抜ける——答えは、君が思いもよらないほど、徹底的に素朴な操作だ：**連結（Concat）**。

標準的な Transformer の自己回帰デコードにおいて、第 t ステップのスケーリング・ドット積注意（Scaled Dot-Product Attention）の公式はこうだ：

```
                       ┌                  ┐
                       │   Q_t · K_≤tᵀ    │
  Attention(Q_t,       │ ─────────────── │
            K_≤t,  = softmax│      √d_k       │ · V_≤t
            V_≤t)          └                  ┘

  ここで：
    Q_t  ∈ ℝ^(1 × d_k)   ← 第 t ステップ「新たに生成したその一つの token」の Query ベクトル
    K_≤t ∈ ℝ^(t × d_k)   ← 第 1 から第 t 番目の token 全部の Key 行列
    V_≤t ∈ ℝ^(t × d_k)   ← 第 1 から第 t 番目の token 全部の Value 行列
```

この三つの記号の次元の差をよく見てほしい：**Q は一行しかない**（現在の token）が、**K、V には t 行ある**（現在の token を含む全歴史）。KV Cache の物語まるごとが、「K_≤t と V_≤t のこの t 行を、各ステップで再計算するのか再利用するのか」というこの選択の中に潜んでいる。

**もし KV Cache がなければ。** 新しい token を一つ生成するたびに、君は歴史の t−1 個の token を改めて Embedding と全 Transformer 層の線形射影 W_k、W_v に通し、K と V を一から計算し直さねばならない。第 t ステップの計算量は t に比例する。t=1 から T まで積み上げると：

```
  総計算量  =  Σ(t=1→T) O(t)  =  O(1) + O(2) + ... + O(T)  =  O(T²)   ← 二乗爆発
```

これが、人を絶望させるあの O(T²) だ。二千字の文章を生成するとき、第 2000 字目のために、君は前の 1999 字の K/V を計算し直した——しかも一つ前のステップで、ほとんど寸分違わぬ同じものを計算したばかりなのに。

**もし KV Cache があれば。** 鍵となる洞察はこうだ：歴史 token の K と V は**いったん算出されたら永遠に変わらない**（なぜならそれらは自分自身とその前の token にしか依存せず、未来には依存しないからだ）。だから第 t−1 ステップの時点で、システムはすでに K_≤(t−1) と V_≤(t−1) を VRAM に保存し終えている。第 t ステップに至っては、君がやるべきは小さな三つの仕事だけだ：

```
  ① 現在のこの一つの token の q_t, k_t, v_t だけ計算       ← O(1)、t とは無関係
  ② 連結してキャッシュを更新（これが魔法の起きる場所だ）：
        K_≤t = [ K_≤(t-1) ; k_t ]      ← 古い行列の後ろに新しい一行 k_t を接ぐ
        V_≤t = [ V_≤(t-1) ; v_t ]      ← 古い行列の後ろに新しい一行 v_t を接ぐ
  ③ q_t と連結済みの K_≤t, V_≤t を上の Attention 公式に代入し直す

  単ステップ計算量  =  O(1)（新 token のためだけに計算）
  総計算量          =  Σ(t=1→T) O(1)  =  O(T)              ← 線形！
```

O(T²) から O(T) への次元削減まるごとが、**いかなる高度な数学も使っておらず、一つの `concat` を使っているだけだ**。君は一つ前のステップで算出済みの K/V 行列を取っておき、新 token の k/v を計算したら後ろに一つ接ぐ、それで終わりだ。これこそが KV Cache が「空間で時間を買う」と呼ばれるゆえんである——君が払う代償は、あの長くなるほど大きくなる K_≤t、V_≤t を**ずっと VRAM に住まわせる**こと（場面4の 128GB はここから来る）。引き換えに得るのは、各デコードステップの演算力が「全歴史を再計算」から「新しい一字だけを計算」に変わることだ。

> 💡 ひとこと助言
> 君に一言贈ろう：**最も深遠な工学的最適化は、しばしばより賢いアルゴリズムを発明することではなく、「このものは一度計算したら二度と変わらない」と認めること——そして、ただそれを取っておくことだ。KV Cache の秘密のすべては、『歴史は変わらない、だから再計算は不要だ』というこの当たり前の一言が、人にまじめに受け取られたというだけのことにすぎない。** あの concat を振り返ってほしい：それは滑稽なほど素朴だが、まさにそれが二乗を線形に変えた。この背後にあるのは計算機科学で最も古い知恵の一つだ——**記憶（memoization）とは、空間で時間を贖い戻すことだ。** 動的計画法のメモから、ブラウザの HTTP キャッシュ、CPU の L1、そしてここの KV Cache まで、すべて同じ思いの異なる化身である：**「入力が変わらなければ出力も変わらない」純関数の計算は、二度目を再計算するのが一種の罪なのだ。** 新人は Decode が遅いのを見ると「もっと速い GPU に換えよう」と考える。熟練者は Decode が遅いのを見ると、まず一言問う「この中に、さっき計算済みなのに今また再計算しているものはどれだけあるのか？」——後者のその一問は、まるごと一桁の価値がある。**一片の計算の中で、どの部分が『不変の歴史』なのかを認め抜ければ、君はすべてのキャッシュ技術の母題を手にしたことになる。**

## 場面4　VRAM 計算器：君の 128GB を食い尽くしたのは KV Cache であって重みではない

**背景**：いまや場面1のあの謎に戻る——重みは 16GB しかないのに、128GB がなぜ溢れるのか。この節では、どんなアーキテクトも常に携えるべき一本の物差しを君に渡す：**KV Cache VRAM 計算器**だ。一度計算してしまえば、君は一生「モデルサイズ」と「デプロイに必要な VRAM」を取り違えることはなくなる。

中核の公式、シングルユーザー、単 token：

```
  Size_per_token  =  2  ×  2  ×  L  ×  H  ×  D   (Bytes)
                     │     │     │     │     │
                     │     │     │     │     └─ D : 各ヘッドの次元 (Head Dim = Hidden / Query Heads)
                     │     │     │     └─────── H : KV ヘッド数 (GQA なら KV Heads、Query Heads ではない!)
                     │     │     └───────────── L : モデル層数 (Layers)
                     │     └─────────────────── 2 : FP16 = 2 バイト (FP8→1, BF16→2)
                     └───────────────────────── 2 : Key と Value の二つの行列
```

二つの 2 を必ず区別すること：**第一の 2 は K と V の二つの行列を保存するから**であり、**第二の 2 は精度のバイト数**（FP16 は 2 バイト、FP8 に換えれば 1 になる、これが「KV Cache 量子化」で VRAM が半分になる原理であり、第5章の後処理と呼応する）だ。H は必ず **KV ヘッド数**を使う——これこそが場面2で GQA を語ったときに埋めた伏線だ：GQA を使えば、この H は Query ヘッド数より数倍小さくなり、VRAM 節約の大頭となる。

**実戦：Llama 3-8B を例に。** その公式 config を調べる：

```python
# KV Cache VRAM 計算器（そのまま実行可能）
def kv_cache_bytes_per_token(layers, kv_heads, head_dim, dtype_bytes=2):
    # 2 = Key + Value の二つの行列；dtype_bytes = 精度バイト (FP16=2, FP8=1)
    return 2 * dtype_bytes * layers * kv_heads * head_dim

def total_kv_cache_gb(layers, kv_heads, head_dim, batch, ctx_len, dtype_bytes=2):
    per_tok = kv_cache_bytes_per_token(layers, kv_heads, head_dim, dtype_bytes)
    total_bytes = per_tok * batch * ctx_len
    return per_tok, total_bytes / (1024**3)

# Llama 3-8B 公式パラメータ
L, HIDDEN, Q_HEADS, KV_HEADS = 32, 4096, 32, 8   # ← GQA に注意：KV ヘッドは 8 だけ
D = HIDDEN // Q_HEADS                              # 4096 / 32 = 128

per_tok, _ = total_kv_cache_gb(L, KV_HEADS, D, batch=1, ctx_len=1)
print(f"每 token: {per_tok} bytes = {per_tok/1024:.0f} KB")
# 2 × 2 × 32 × 8 × 128 = 131,072 bytes ≈ 128 KB / token

_, gb = total_kv_cache_gb(L, KV_HEADS, D, batch=128, ctx_len=8192)
print(f"batch=128, ctx=8K → {gb:.0f} GB")
# 128 KB × 128 × 8192 ≈ 128 GB  ← 犯人はここに
```

数字を代入すると、各 token が **128 KB** の VRAM を食う。大したことなく見える？ 並行数を掛け、さらに文脈長を掛ければ、幾何級数が即座に姿を現す：

| 場面 | batch | 文脈 | KV Cache 総量 | 重み 16GB との対比 |
|---|---|---|---|---|
| 君が単体テストしたあの一本の prompt | 1 | 2K | ~256 MB | 🟢 取るに足らない、だから問題に気づかなかった |
| 小並行・短文脈 | 8 | 2K | ~2 GB | 🟢 まだ大丈夫 |
| 長文脈リクエスト数本 | 16 | 8K | ~16 GB | ⚠️ もう重みと同じ大きさ |
| 中並行・長文脈 | 128 | 8K | **~128 GB** | ⚠️ 重みの 8 倍、**OOM** |

**結論を一言で：大規模モデル推論のボトルネックは、根本的にモデルの重みそのものではなく、KV Cache だ。** 君のあの DGX Spark の 128GB 統一メモリは、16GB の重みを積むには十分すぎる——だが並行数と文脈長が上がりさえすれば、KV Cache はそれを丸ごと飲み込める。場面1の謎はここで解ける：**消えた百数十 GB は、128 個の並行リクエストがそれぞれ引きずるあの 8K の長さの KV の尻尾なのだ。** これは、なぜ第5章の後処理段階で「KV Cache 量子化（FP8/INT8）」をわざわざ厳かに行うのかも説明する——公式中の第二の 2 を 1 に変えれば、長文脈のメモリはその場で半減する。

> ⚠️ 真偽メモ
> 上表の数字は、上の公式で**理論的に推算**したクリーンな値で、君が直感を築くのに便利なものだ。**実際のデプロイでは、実占有量にさらに以下が積み重なる：ページングのブロック粒度の浪費（場面5）、フレームワークの予約バッファ、activation の一時保存、そして満載でない断片化**——だから実機で走らせると、通常は理論値よりさらに逼迫し、緩くはならない。「128GB」というこの数字がちょうどあるマシンのメモリと等しいのは、原素材が意図的に選んだ整数の偶然であり、**それを「同じ桁数」の図示として受け取り、ある機種の実測上限とはしないでほしい**。君が持ち帰るべきは**方法**だ：自分のモデルの L/H/D を公式に代入し、目標の並行数と文脈長を掛けて、まず KV Cache の天井を算出し、それからどれだけの VRAM を買い、どれだけの batch を開くかを決める——**手を動かしてデプロイする前に、まずこの物差しで一度測ることが、プロと丸腰の分かれ目だ**。

> 🔍 掘り下げ解説 — なぜ「最大並行数」は演算力ではなく KV Cache に決められるのか
> この計算器の最も反直感的な帰結はこうだ：**一台の推論サーバーが同時に何人のユーザーにサービスできるかの上限は、通常その演算力（FLOPS）ではなく、VRAM が何本の KV の尻尾を収められるかだ。** これは容量計画の発想を徹底的に書き換える。従来の CPU サービスなら君は「QPS × 1 リクエストあたりの CPU 時間」を計算する。だが LLM 推論サービスでは、君はまず「VRAM 総量 ÷ 1 本の sequence あたりの KV Cache ピーク」を計算せねばならない——この商こそが、君の並行数の上限だ（vLLM の起動パラメータ `--max-num-seqs` と `--gpu-memory-utilization` が調整しているのはまさにこのことだ）。こうして一連の工学的決定がすべてこの物差しによって再定義される：**なぜ長文脈サービスはこんなに高いのか？** ctx_len が各 KV を直接線形に拡大するからだ、128K 文脈の単リクエスト KV は 8K の十六倍にもなりうるし、収められる並行数は同じ比率で暴落する。**なぜ GQA/MQA/MLA が花形なのか？** それらが H を削るからで、それは君の並行数の上限を直接拡大することに等しい（場面2）。**なぜ PagedAttention は革命なのか？** それが「最大長のために予約する」浪費を消滅させ、実際に収められる KV の本数を物理的極限まで押し上げるからだ（場面5）。**この四つをつなげれば気づく、現代 LLM 推論エンジンのすべての努力は、同じ制約のもとで文章を書いている：VRAM が KV をどれだけ収められるか、それで君が何人にサービスできるかが決まる。** 演算力こそ、Decode 段階で最も不足しないものなのだ。

## 場面5　工学進化史：Naive な事前確保から PagedAttention の仮想メモリ革命へ

**背景**：KV Cache が VRAM を食い尽くすことを知ったうえで、次の問題はこの有限な VRAM を**いかに極限まで使い切るか**だ。この道をオープンソースコミュニティは三世代かけて歩んだ。浪費が度を越して呆れるほどの Naive 方式から、オペレーティングシステムをそのまま写し取った天才的設計へと進化した——そして写し取ったその答案は、君が計算機構成原理の授業で習ったのに、まさかここで使われるとは思わなかった**仮想メモリのページング**である。

**第一世代、Naive 方式（初期 PyTorch の素朴な実装）。** やり方は単純粗暴だ：各リクエストのために、**事前に連続した、大きさが `Max_Seq_Len` に等しい VRAM の塊を確保する**。モデルが 4K 文脈をサポートするなら、君が実際に何 token 生成しようと、一律にまず 4K の空間を占有してしまう。これは二種類の致命的な断片化をもたらす：

- **内部断片化（Internal Fragmentation）**：ユーザーは実は 100 token しか出力していないのに、君が彼のために予約した 4K の空間の中で、残り 3900 token 分の場所が**強制的に占有され、他人が使えない**、純粋な浪費。
- **外部断片化（External Fragmentation）**：VRAM の総量は明らかにまだ足りているのに、「連続した」大きな塊を要求するせいで、細切れの空きが連続した一塊にまとまらず、**空間はあるのに新しいリクエストを初期化できない**。

実測してみると、Naive 方式の VRAM 有効利用率はしばしば二、三割しかない——七割の VRAM が「万一ユーザーがとても長く書いたら」という仮定のために空回りしている。

**第二世代、vLLM の PagedAttention（仮想メモリ革命）。** vLLM チームの洞察は古典的と言える：**KV Cache の断片化問題は、オペレーティングシステムが四十年前に直面したメモリ断片化問題と、同じ問題だ。** そして OS はとうに解決していた——**ページング（Paging）**を使って。そこで PagedAttention は KV Cache を**離散的に、固定サイズの一群の物理ブロック（Physical Block）に保存する**。これらのブロックは VRAM の HBM の中で**あちこちに散らばってよく、連続を要求しない**：

```
  論理的視点 (各 sequence は自分の KV が連続だと思っている):
    Logical Blocks:  [ Block 0 ] -> [ Block 1 ] -> [ Block 2 ]
                         │              │              │
  Block Table (マッピング表):           │              │
                         ▼              ▼              ▼
    Physical Blocks: [ Frame 42 ]   [ Frame 7 ]   [ Frame 99 ]
                     (GPU HBM 内で物理的にあちこちに散らばるが、論理的には連続)
```

各物理ブロックは固定で 16 個または 32 個の token を収める。一つの `BlockManager` が全空きブロックのプールを管理し、一つの `BlockTable` が「論理ブロック → 物理ブロック」のマッピングを維持する——これはまさに OS のページテーブル（page table）そのものだ。新しいリクエストが来たらプールから空きブロックをつかみ、生成した分だけ、伸びた分だけつかむ、**使った分だけつかむので、内部断片化はほぼゼロになる**。ブロックサイズは固定で任意に散布できるので、**外部断片化も消える**。VRAM 利用率はほぼ 100% まで追い込まれる。

```python
# vLLM PagedAttention 中核アーキテクチャ（ソースコード論理の疑似コード）
class PhysicalTokenBlock:
    def __init__(self, block_number, block_size):
        self.block_number = block_number  # GPU VRAM ブロックの実際の物理インデックス
        self.block_size = block_size      # 通常 16 または 32 個の token
        self.ref_count = 0                # 参照カウント → Block 級 Copy-on-Write をサポート

class BlockAllocator:
    def __init__(self, num_gpu_blocks, block_size):
        # 起動時に全利用可能 VRAM を固定サイズのブロックに切り、空きプールにする
        self.free_blocks = [PhysicalTokenBlock(i, block_size)
                            for i in range(num_gpu_blocks)]

class BlockTable:
    def __init__(self):
        self.mapping = {}  # 論理ブロックインデックス → 物理ブロック (これが「ページテーブル」)
    def append_token(self, logical_block_idx, block):
        self.mapping[logical_block_idx] = block
```

だがページングは新たな厄介を一つもたらす：KV が VRAM の中でもう連続ではなくなったので、標準の `torch.cat` のような連続メモリ操作が使えない。vLLM の解法は**自前で CUDA / Triton kernel を手書きする**ことだ。注意を計算するとき、**`block_table` を通じて動的にアドレッシングし**、あの散らばった物理ブロックの中をブロックごとに K、V をすくい上げる：

```
  // vLLM CUDA Kernel アドレッシング中核の疑似ロジック
  __global__ void paged_attention_kernel(
      float* out, const float* q, const float* k_cache, const float* v_cache,
      const int* block_table, const int* context_lens, int block_size) {
      int seq_id = blockIdx.y;
      int token_idx = threadIdx.x;
      // ① 論理 token インデックスで block_table を引き、物理ブロック番号を見つける
      int logical_block_idx = token_idx / block_size;
      int physical_block_num =
          block_table[seq_id * max_blocks_per_seq + logical_block_idx];
      // ② その token の VRAM 内での絶対オフセットアドレスを算出
      int block_offset = token_idx % block_size;
      float* k_ptr = k_cache
          + physical_block_num * block_size * num_heads * head_dim + ...;
      // ③ 散らばった物理ブロック上で通常の Attention 計算を実行
  }
```

あの `ref_count`（参照カウント）は、ついでに一つの美しい特性も解錠する：**Copy-on-Write のブロック共有**だ。並列サンプリング（一つの prompt から同時に複数の候補を生成）や beam search のとき、複数の sequence が同じプレフィックスの物理ブロックを共有し、本当に分岐して書き込むときにだけ複製する——これもまた OS の `fork()` のコピーオンライトを、KV Cache に持ち込んだものだ。

> 🔍 掘り下げ解説 — なぜ「OS を写す」が LLM 推論エンジンの最も深い堀なのか
> PagedAttention は表面上は一つの VRAM 管理テクニックだが、それが真に示すのは、**システム工学における最高級の能力の一つ：まったく新しい領域の中で、徹底的に解決され尽くした古い問題を認め、その成熟した解法を丸ごと持ってくることだ。** vLLM が 2023 年に一発で当たり、事実上の標準になれたのは、新しい数学を発明したからではなく、その作者が KV Cache の断片化を見て、頭に浮かんだのが「これは 1960 年代の仮想メモリじゃないか」だったからだ——そしてページテーブル、ページング、コピーオンライト、さらには「CPU メモリへのページアウト」（VRAM が足りないとき冷たい KV ブロックをメインメモリに swap するのは、まさに OS の swap だ）まで、一式そっくり持ってきた。**今日 KV Cache を取り巻く生態系まるごとが、ほぼ OS 概念の LLM 化身だ：** PagedAttention はページング、prefix caching は重複排除、RadixTree（次節）はプレフィックスツリーのインデックス、KV 量子化は圧縮、KV offloading は swap、NVIDIA の **FlashInfer**、Hugging Face の **TGI**、SGLang のブロック管理まで、骨の髄では同じメモリ階層の手品を演じている。**これは蒸留者にきわめて実際的な啓示を与える：君が自分のあの DGX Spark の本地推論を最適化したいとき、急いで GitHub で新しいフレームワークを探すな——まず自分に問え「オペレーティングシステムは当時この問題をどう解いたのか」と。** 計算機科学六十年のメモリ管理の知恵が、まるごと君の既製の武器庫なのだ。他人が新しい kernel を発明するのを待つだけの者は、永遠に一歩遅れる。

## 場面6　サービス側の RadixTree と演算子級の FlashAttention：メモリを君の DGX Spark に返す

**背景**：PagedAttention は「単機の VRAM をどう使い切るか」を解決したが、まだ語っていない二つの階層の最適化がある——**サービス側がいかに異なるユーザーのリクエストどうしにプレフィックスを再利用させるか**（これは場面1の Prompt Cache に対応する）と、**最も底層の演算子がいかにメモリ帯域幅を浪費しないか**だ。この二つは一方が最上層、一方が最底層にありながら、ともに本章冒頭の問いに答える：あの百数十 GB をどう君に返すのか。

**サービス側：SGLang の RadixTree（基数木）プレフィックスキャッシュ。** 場面1で語ったとおり、サービス層 Prompt Cache は同一プレフィックスを再利用すべきだが、問題は——何千何万のユーザーがいて、プレフィックスはそれぞれ異なりながら部分的に重なる、どうやって効率的に「千人千様に公共プレフィックスを共有する」のか？ SGLang フレームワークの答えは **Radix Tree（基数木）**だ。それは各ユーザーの入力 prompt を token ID 配列に変え、木の**各ノードに一段の token 列 + その段に対応する KV Cache の物理ポインタを保存する**。公共プレフィックスは自動的に根に近い枝幹に集約され、それにヒットするすべてのリクエストに再利用される：

```
  三つのリクエストが同じ system prompt の一文を共有：
    リクエスト1: "You are a helpful assistant. Please tell me a joke."
    リクエスト2: "You are a helpful assistant. Please write Python code."
    リクエスト3: "You are a helpful assistant. Translate this to Chinese."

                        [ Root ]
                           │
        ( tokens of "You are a helpful assistant. " )  ← 公共プレフィックスにヒット！KV は一度だけ計算
                           │
            ┌──────────────┼──────────────┐
            │              │              │
      ["Please tell   ["Please write  ["Translate
       me a joke."]    Python code."]   to Chinese."]   ← それぞれ分岐した尻尾
```

あの繰り返し現れる system prompt は、その KV 状態が**木全体で一度だけ計算され、すべてのリクエストに共有される**——これこそが場面1の「安定したプレフィックスを最も前に置く」黄金法則の、サービス側における物理的な実現だ。VRAM が逼迫して空間を空ける必要が出ると、RadixTree は **LRU（最近最も使われていない）追い出し**を発動するが、その追い出し方が賢い：**まず葉ノードを斬り**（すでに終わった長い対話の末尾、最も再ヒットしそうにない枝）、**根ノードと公共枝幹を残す**（system prompt のような、最もヒット確率の高い部分）。まず葉を斬り、後に根を残す——この戦略そのものが、「プレフィックスは根に近いほど再利用価値が高い」ことの精密な活用だ。

**演算子級：FlashAttention と FlashDecoding。** ページングとプレフィックスツリーがあるだけではまだ足りない。最底層の注意演算子が間抜けに書かれていれば、やはりメモリ帯域幅を浪費する。場面2の中核の矛盾に戻ろう——Decode はメモリ律速であり、ボトルネックは HBM の運搬にある。

- **FlashAttention** が攻めるのは「中間行列を HBM へ書き戻す」というこの浪費だ。素朴な実装は、あの N×N の注意行列を**完全に算出し、遅い HBM に書き戻し、また読み戻して** softmax にかける——この一往復だけで膨大な帯域幅を燃やす。FlashAttention は **Tiling（分割）**でその計算をチップ上の**高速 SRAM** に収まる小ブロックに切り、**online softmax**（掃きながら softmax の正規化の分子分母を増分更新する）の数学的技巧を使って、**全行程で完全な N×N 行列を HBM に書き戻さない**。結果として VRAM の読み書き量は一桁下がる——それは GPU により速く計算させたのではなく、GPU に**一桁少ないデータを運ばせた**のであり、これがまさにメモリ律速の急所を突いている。

- **FlashDecoding** は場面2の Decode 段階を、しかも**超長文脈**を専門に治す。Decode 時には batch とヘッド数がしばしば GPU の全計算ユニットを満腹にするには足りず、歴史 KV がさらに極端に長い（たとえば 128K 文脈）と、あの一本の長い KV を sequence 次元に沿って直列スキャンするせいで大量の SM（ストリーミングマルチプロセッサ）が遊休する。FlashDecoding の鍵は**KV の sequence 次元（batch/ヘッド次元ではなく）に沿ってもう一刀入れる**ことだ：**あの超長の KV を複数の段に切り、GPU 上の複数の計算ユニット（SM）に並列に割り当てて**それぞれ局所注意と局所 softmax 統計量を計算させ、最後に一度 **Reduction（リダクション）**を行い、各段の softmax 分母で重み付けし直して正しい結果に合成する。それは「一本の長い尻尾の直列メモリアクセス」を「複数段の並列メモリアクセス」に変える。まさに「batch が小さく、文脈が長い」という従来の並列次元では GPU を満腹にできない死角で、利用率を再び引き上げるのだ。

**このすべてを君の DGX Spark に接ぎ戻す。** 場面1で君のあのマシンはメモリを溢れさせたが、いまや君の手には、メモリを返す一式の道具がそろっている：**vLLM / SGLang** で起動し、**PagedAttention** で断片化を消滅させて VRAM 利用率を 100% まで追い込み（場面5）、**RadixTree prefix caching** で複数のセッションにあの固定の system prompt を共有させて重複 Prefill を省き、**FlashAttention / FlashDecoding** でメモリ律速な Decode 段階のメモリ帯域幅を絞り尽くし、さらに第5章後処理の **KV Cache 量子化（FP8）**で公式中の精度バイトを 2 から 1 に削る。この四層を積み重ねれば、もともと溢れれば即 OOM だった 128GB が、支えられる並行数と文脈長を何倍にもできる——**これこそが「あの百数十 GB を返す」具体的な工学的経路だ**。

> 💡 ひとこと助言
> 君に一言贈ろう：**この章は最初から最後まで、「モデルをより賢くする」技術は一つもない——KV Cache、PagedAttention、RadixTree、FlashAttention、すべては「同じ賢さを、より少ないメモリで走らせる」ためのものだ。大規模モデル工学の真の戦場は、いまだかつて演算力の頂にあったことはなく、メモリの深淵にある。** この六つの場面の伏線を振り返ってほしい：場面2はボトルネックがメモリであって演算力でないことを暴き、場面3は一つの concat で演算力を線形に節約し代償としてメモリを常駐させ、場面4はそのメモリが恐ろしいほど溢れることを算出し、場面5はページングでメモリを使い切り、場面6はプレフィックスツリーと演算子でメモリ帯域幅を絞り尽くす——**章全体が「メモリを中心に文章を書く」という単一の本筋だ**。これは第3章の DGX Spark のあの 128GB 統一メモリのキラー級の価値と呼応し、また一つの残酷な現実を予告する：**君が強いモデルを本地に蒸留し、走れるまで量子化したとき、それが『本物のトラフィックにサービスできるか否か』を真に決めるのは、それがどれほど賢いかではなく、並行圧力下でのそのメモリ曲線が安定しているか否かだ。** アマチュアは「本地で 70B を走らせられる」と誇り、プロは「君は本地でどれだけの並行、どれだけの長さの文脈で、70B の KV を OOM させず安定させられるのか」と問い詰める——前者が競うのはピーク値、後者が競うのはメモリの深淵の縁で綱渡りする力量だ。**モデルの知能は天井、メモリの管理は床。そして一つのシステムが使い物になるか否かは、いつだって床が決める。**

> ⚠️ 真偽メモ
> 本場面で触れた FlashAttention（v2/v3 を含む）、FlashDecoding、SGLang RadixTree、vLLM PagedAttention、NVIDIA FlashInfer、Hugging Face TGI は、いずれも**公開論文 / オープンソース実装のある実在の技術**であり、自分で確認できる。だが「一桁下げる」「利用率を満たす」といった幅の描写は**方向性の結論**だ——実際の利得は sequence 長、batch、ハードウェア世代、kernel バージョンに強く依存する。上のあの vLLM 起動パラメータは**実在のパラメータ、概念図示の設定**であり、具体的な数値（0.90 / 64 / 8192）は自分のモデルで場面4の計算器を使って計算してから埋めてほしく、そのまま書き写してはならない。

## 本章のまとめ

- **二大キャッシュ分類**：① 推論層 **KV Cache**（VRAM に住み、vLLM/TensorRT-LLM が管理、自己回帰デコードを O(N²) から O(N) へ低減、一リクエストのライフサイクル）；② サービス層 **Prompt / Context Cache**（サービス側に住み、リクエストをまたいで同一プレフィックスの KV 状態を再利用、Claude は `cache_control` で明示的にアンカリング、ヒット後は入力 token が大幅割引）。黄金法則：**安定した静的内容を最も前に、動的で変化の激しい内容を最も後に**。両者の本質はどちらもプレフィックスマッチであり、一つの放浪するタイムスタンプや余分な `\n` 一つで、キャッシュ全体が無効になる。
- **メモリ律速が物理的な底層**：Prefill は Compute-Bound（テンソルをまとめ、Tensor Core を満腹に）、Decode は Memory-Bound（各ステップ 1 token のみ、GPU の大半の時間は HBM→SRAM の運搬）。**KV Cache はボトルネックを消滅させたのではなく、ボトルネックを演算力からメモリへ移した**——これが全章の物理的基盤であり、GQA/MQA/MLA が必死で KV ヘッド数を削る根本的動機でもある。
- **数学的導出**：キャッシュなしでは各ステップで歴史 K/V を再計算 → Σ O(t) = **O(T²)**；キャッシュありでは新 token だけ計算して古い行列に `concat` → 単ステップ O(1)、総 **O(T)**。次元削減の魔法は一つの素朴な連結しか使っておらず、背後には「歴史は変わらない、だから再計算は不要だ」という memoization の母題がある。
- **VRAM 計算器**：`Size_per_token = 2(K+V) × 2(FP16バイト) × L × H(KVヘッド) × D`。Llama 3-8B（L=32, KV Heads=8, D=128）= **128 KB/token**；batch=128 × 8K ctx ≈ **128 GB** だが、重みは 16GB しかない。**ボトルネックは KV Cache であって重みではない**。最大並行数は VRAM が何本の KV の尻尾を収められるかで決まり、演算力では決まらない。
- **工学進化**：Naive な連続事前確保 → 内部/外部断片化で利用率を二、三割に叩き落とす；vLLM **PagedAttention** が OS の仮想メモリのページングを写し、KV を離散的に固定サイズの物理ブロックに保存、`BlockManager`/`BlockTable` が論理→物理マッピング、自前 CUDA/Triton kernel が `block_table` 経由で動的アドレッシング、`ref_count` がコピーオンライトのブロック共有を解錠し、利用率は 100% に迫る。KV 生態系まるごとが OS 概念の LLM 化身だ（ページング/重複排除/プレフィックスツリー/圧縮/swap）。
- **サービス側 + 演算子級**：SGLang **RadixTree** は各ノードに一段の token + KV ポインタを保存、公共プレフィックス（system prompt）にヒットして再利用、LRU 追い出しはまず葉を斬り根の枝を残す；**FlashAttention** は SRAM tiling + online softmax で N×N 行列を HBM に書き戻さない；**FlashDecoding** は超長 KV を段に切って複数 stream で並行させ、その後 reduction する。四層の積み重ね（PagedAttention + prefix caching + Flash 演算子 + KV 量子化）こそが、あの消えた百数十 GB を DGX Spark に返す具体的な経路だ。

KV Cache は計算し尽くし、ページングもページ分けし、プレフィックスツリーも建てた——だがこのすべては、まだ「一台のマシン」の尺度にある。リクエスト量が数本から毎秒数十万本へ膨れ上がるとき、キャッシュが何百何千枚もの GPU をまたいで共有されねばならないとき、君が「ヒット率」と「一貫性」の間で分散的なトレードオフをしなければならないとき、単機のこの一式の物理学（physics）ではもう足りなくなる。次章では、キャッシュを一台のマシンからデータセンター全体へと拡大し、超大規模下のキャッシュアーキテクチャを、そしてそれがいかにして、攻防のために生まれたあの二十の方向の武器庫へと収束していくかを見る。
