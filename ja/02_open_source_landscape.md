## 場面1　第一梯団：狭い窓から削り取ったデータと、それが鋳込んだモデル

**背景**：前章を受けて、コミュニティが Fable 5 を採取できた窓口はごく短かった。巷で流れる**第一梯団**とは、そのデータで真っ先に煉り上げられた成果物である。

**この噂のリストと技術的解剖**：

| モデル | ベース（Base）| 蒸留手法 | 称される特性 |
|---|---|---|---|
| **Qwable-v1** | Qwen3.6-35B-A3B | SFT（指示チューニング）| Fable 5 のツール呼び出し（Tool Surface）を継承し、`str_replace_editor` 形式を出力できる |
| **GPT-OSS 120B Fable-5 Distilled** | GPT-OSS 120B | 蒸留 + GGUF 変換（AutoTrust AI Lab が演算力を提供）| MoE、128 エキスパート／4 活性化；GGUF Q8_0、Q5_0 |
| **Jackrong シリーズ**（27B Coder など）| Qwen 系 | 軌跡反転（Trace Inversion）| コミュニティで「最も信頼される」、過学習を抑制 |
| **データセット armand0e/claude-fable-5-claude-code** | —— | 脱プライバシー化した軌跡採集（Glint-Research と協業）| Fable 5 の原始 Agent Traces、OpenAI 形式へ変換可能 |

**技術の核を分解して見る**：

- **`Qwen3.6-35B-A3B` という命名の妙**：`A3B` は Qwen MoE の標記である——**総パラメータ 35B、毎回およそ 3B しか活性化しない**。これをベースに選ぶのは適当な選択ではない。MoE の「活性化パラメータが小さい」という特性が、ちょうど第 3 章 DGX Spark の帯域幅ボトルネックに対応するからだ。**ベースを選ぶとは、君のハードウェアと相性の良い骨格を選ぶことであって、パラメータ量が最大のものを選ぶことではない。**
- **`str_replace_editor` とは何か**：これは Claude シリーズが Agent シーンでファイルを編集するために用いる**ネイティブのツールタグ**（text editor tool）である。Qwable-v1 が「ツール呼び出し能力を継承する」と称するのは、それが重み層レベルで**公式とまったく同じツール呼び出し XML を自発的に吐き出す**ことを学習した、という意味だ——OpenHands や Aider のようなオープンソース Agent に接続する際、プロンプトで一層変換を書く必要がなく、エラー率が低い。**蒸留者が今回最も盗みたかったのは「おしゃべりできる Claude」ではなく、「ツールを使える Agent 版 Claude」なのだ。**
- **採集→データセット→モデルのツールチェーン**：`armand0e/...` のような「軌跡データセット」はチェーン全体の源流である。標準的な流れはこうだ——採集した原始 traces（多くは Claude の messages 形式）→ スクリプトで **ShareGPT / OpenAI `messages` 形式**へ変換 → **Axolotl** や **LLaMA-Factory** の SFT 設定へ投入 → `peft` で LoRA または全パラメータ微調整。**「脱プライバシー化（de-PII）」のステップ**は通常、**Microsoft Presidio** や spaCy NER で鍵、メールアドレス、人名を洗い落とす——これは合法なデータ処理では必修であり、違法な採集では「出所証拠の抹消」という副作用も兼ねる（第 6 章参照）。

> 💡 ひとこと助言
> 君に一言贈ろう——**モデルは凝固したデータであり、データは流動するモデルである。真の能力転移はデータ層で起こり、モデルはその一枚のスナップショットにすぎない。** この一句を理解すれば、君はもう「どの蒸留済みモデルをダウンロードするか」に執着せず、「それはどんなデータ、どんな形式、どんな手法で煉られたのか」を問うようになる。同じ traces を、今日は Qwen に、明日は Llama に、明後日は君自前のアーキテクチャに流し込めば、出てくるのは三つの異なる身体・同じ魂だ。データと配合を握った者は影を無限に複製できる。成果物だけを手にした者は、永遠にエンドユーザーにすぎない。

> 🔍 掘り下げ解説──SFT 蒸留の天井と、それが「形は似て神は似ず」になる理由
> 第一梯団はほぼ一色に **SFT（系列レベル蒸留）**を用いる。これは API が「テキストだけを与え、logits は与えない」ことに強いられた選択だが、構造的な天井がある。SFT が教えるのは「**教師の最終出力系列を模倣する**」ことであり、生徒が学ぶのは Claude の**文体と表面の型**（`<thinking>` を吐く、あの XML を使う）であって、Claude が各 token で「**なぜこの語を選び、他の語とどれだけ差があるか**」というソフトラベルの分布は学べない——その確率分布（暗黙知）こそが推論能力の真の担い手だ（第 4 章で詳述）。だからこの種の蒸留モデルはよく「**形は似て神は似ず**」と揶揄される。複雑な論理が深まると馬脚を現す。なぜなら「答えがどう見えるか」を模倣しているのであって、「どう答えにたどり着くか」ではないからだ。この天井を破るには、方法論として三つの進階の道がある——① **on-policy 蒸留**（MiniLLM／GKD、生徒が先に生成し教師が採点する）；② **棄却サンプリング + 過程報酬**（rejection sampling で高品質な CoT を選別）；③ **logit アライメント**（局所 logits を入手できるなら温度付き KL を行う）。第一梯団が「形似」止まりなのは、まさに狭い窓 + 防御の盾が後二者の道のコストを爆発させるからだ——**彼らは方法論を知らないのではなく、経済勘定によって最も安い SFT へ押し戻されているのだ。** ついでに「本物の蒸留がどう見えるか」の対照群を示そう。このスペクトル上で**実在し、ダウンロードして検証できる**標竿が `DeepSeek-R1-Distill` シリーズだ——DeepSeek は R1 の長い推論軌跡を Qwen2.5 と Llama 3 の 1.5B〜70B ベースに蒸留し、重み・model card・ベースの血統をすべて公開している。その `config.json` を対応ベースと比べれば完全に一致する（場面 5 で比較法を教える）。一方、噂の `Fable-5 Distilled` は、ベースの commit SHA すら見つけられない——**戸籍があるか、ベースを辿れるかどうか。それが「本物の蒸留」と「看板だけの伝説」を分かつ、最も素朴な境界線だ。**

## 場面2　神話化された「庶民版 Claude」：Qwythos-9B

**背景**：最も声の大きい一品が、噂では **Empero AI** がリリースしたとされる **Qwythos-9B-Claude-Mythos-5-1M** である。売り文句は極めて煽動的だ——「Claude がオープンソース化された？無検閲、104 万コンテキスト、4GB の VRAM で動く！」（出典の見出しは付録 A 参照：零度解説、Interfaze、Threads）。

**巷で流れる仕様と技術**：

- **ベース**：Qwen3.5-9B に**全パラメータ微調整（Full FT）**を施し、「Claude 5 シリーズデータ」を投入。
- **1M（104 万）コンテキスト**：長コンテキスト拡張技術による（後述）。
- **無検閲（uncensored）**：拒否機構を除去。
- **4GB VRAM で動く**：GGUF 量子化（Q4_K_M など）による。

派生と予告：

- **richardyoung/qwythos-9b-abliterated**（Ollama に掲載）：第三者が **abliteration** で二次的に「徹底脱検閲」した変種。
- **Qwythos-27B**（Empero Roadmap の予告）：次世代のより大きな「Mythos 級」蒸留モデルとされる。

**「無検閲」と「長コンテキスト」の技術を分解する**：

- **abliteration は結局何をするのか**：これは実在の技術（Arditi らの "refusal direction" 研究）である。やり方はこうだ——「有害な問い」と「無害な問い」を別々にモデルへ通し、両群のある層の残差ストリームでの**平均活性化の差を取り、「拒否方向」ベクトルを得る**。次にモデルの重みをこの方向に対して**直交化（orthogonalization）**し、モデルが「ノーと言いたがる」回路を消し去る。ツールとしては **TransformerLens** で活性化分析を行うか、既製の `refusal_direction` / `heretic` フローを使うことが多い。**名前の中の「heretic」が abliterated ベースの符牒だ**（場面 3 の FableVibes のカードには「started as Qwen3.6-35B-A3B-heretic」と明記されており、まさにこの技術ラインである）。
- **1M コンテキストはどこから来たか**：9B にネイティブで 1M ウィンドウは不可能で、必ず**事後の外挿**だ——主流は **YaRN**（NTK-aware の RoPE 補間で、RoPE base を 1 万級から百万級へ引き上げる）に **FlashAttention** を組み合わせ、長系列のメモリを支える。問題は——**「1M まで外挿できる」と「1M で読み解ける」は別物**ということ。次章でハードウェアの真実をもって突き崩す。それが本当に外挿したのか、それとも単なるハッタリかをその場で見分ける最速の一手は、`config.json` を開いて `rope_theta`（RoPE base、Qwen2.5 のデフォルトは 100 万、Llama 3 は 50 万）と `rope_scaling`（YaRN ならここに `type: yarn` と `factor` が明記される）を見ることだ——**本当に長コンテキストを載せたなら、この二つのフィールドには必ず指紋が残る。README だけ変えて config が手付かずのものは、ほぼ看板の貼り替えだ**（場面 5 でこの鑑識を詳述する）。

> ⚠️ 真偽メモ
> 「9B + 4GB VRAM + 1M コンテキスト + 無検閲 + ほぼ Claude 同等」という組み合わせは**工学的常識に反する**。abliteration も YaRN も実在するが、それらを「Qwythos-9B」という具体的モデルに重ね合わせて Claude に近いと宣言するのは、未検証のマーケティング言説である。

> 💡 ひとこと助言
> 君に一言贈ろう——**「脱検閲」は決して無料ではない。その請求書は、君に見えない場所に書かれている。** abliteration が「拒否方向」を消すとき、付随してモデルの判断力と安全直感も鈍る——それは単に「敏感な問いに答える勇気を持った」だけでなく、「このコードには SQL インジェクションがある、注意すべきだ」という**有益な拒否**まで一緒に忘れてしまうかもしれない。研究は繰り返し実証している——過度な abliterate はモデルの推論・事実性ベンチマークの成績を低下させると。だから「無検閲」の三文字に惹かれたら、まず君が本当に欲しいのが「自由」なのか、それとも「もう君にノーと言わず、君のために門番もしてくれない道具」なのかをよく考えることだ——前者は能力、後者は機能不全であり、両者は見た目がそっくりなのだ。

> 🔍 掘り下げ解説──小モデルの「収まる」と「担える」の溝
> Qwythos-9B の神話の本質は、マーケティングが意図的に溶接した、実は独立な三つの次元を混同させることにある。そしてその三者の背後には三つの異なる工学的ボトルネックがある。**「1M が収まる」** は RoPE/YaRN の数学的外挿の問題だ——数個のパラメータを変えるだけで達成を宣言でき、ほぼゼロコスト。**「1M を担える」** は KV Cache のメモリとプレフィル（prefill）の計算量の問題だ——9B モデルは 20 万 token の時点で KV Cache が膨張し、君が節約したと思った VRAM を全部食い尽くす。しかも長文一篇の prefill には数分待たされうる。これは帯域幅と容量の硬い壁だ。**「1M を読み解ける」** はモデル容量と訓練データの能力の問題だ——9B のアテンションヘッド数と層数が、その「干し草の山から針を探す（Needle-in-a-Haystack）」検索力が 35B／120B に先天的に及ばないことを決める。ウィンドウがいくら長くても「見えるが覚えられない」だけだ。**マーケティングは第一の次元しか売らない。最も安く、最も吹聴しやすいからだ。エンジニアは後二者を自分の手で検証しなければならない。それらこそ、この 1M が本物の実力か装飾かを決めるからだ。** 検証ツールも既製品が揃っている——オープンソースの RULER や Needle-in-a-Haystack テストを一度走らせれば、真偽は一目瞭然だ。

## 場面3　最新を追わない：前世代 Claude の成熟した蒸留品

**背景**：巷にはもう一つの補足的な手掛かりがある——9B では推論が足りないと感じるなら、あるいは「最新の Mythos」を盲信しないなら、コミュニティには**前世代の Claude（Opus / Claude 4.6 世代）**データから蒸留した成熟モデルがいくつかある。

| モデル | ベース | 手法 | 称される位置づけ |
|---|---|---|---|
| **Qwen3.6-14B-A3B-FableVibes**（tvall43 GGUF）| Qwen3.6-35B-A3B「heretic」剪定版 | Pruning + QLoRA 補償 | 小サイズで「Claude らしさ」を保持；Fable 5 の多段推論軌跡 + Opus 推論データを使用 |
| **clzoro/Qwen3.5-27B-Claude-distill** | Qwen3.5-27B（密）| 全パラメータ微調整、Opus 高品質 CoT を投入 | 数学・論理・マルチターン対話が公式に近い |
| **Dzluck/Qwen3.5-2B-Claude-4.6-Opus-Reasoning-Distilled**（GGUF 含む）| Qwen3.5-2B | Claude 4.6 の長文推論蒸留 | スマホ／ラズパイ等エッジデバイスへの極限挑戦 |

> ⚠️ 真偽メモ
> 上表の三品（**FableVibes、clzoro、Dzluck**）のモデル名、アップロード者、仕様、そして「Opus／4.6 蒸留」の血統は、本書ではいずれも確認できない。その背後の技術手法（pruning、QLoRA、密 vs MoE、2B エッジ蒸留）はすべて実在し一般的だが、「誰かが Opus の CoT をある 27B に蒸留した」という**具体的な血統の主張**は、君が場面 5 の方法で `config.json` とベース重みを検証するまで、README 上の一文の宣伝文句にすぎない。

**技術の見どころ**：

- **FableVibes の「剪定 + QLoRA 補償」は一つの完結した方法論だ**：まず 35B MoE に**ネットワーク剪定（pruning）**を施し、低活性化のエキスパート／ニューロンを剃ってサイズを縮める。剪定は必ず能力を損なうので、次に **QLoRA**（NF4 4-bit 量子化ベース + LoRA 低ランクアダプタ + paged optimizer、Dettmers らの方式）で「補償訓練」を行い、落ちた点数を取り戻す。そのベースカードは「started as Qwen3.6-35B-A3B-**heretic**」と明言している——abliterated ベース（場面 2）がまさにこの製造ラインの上流であることを裏付けている。
- **密 vs MoE のトレードオフ**：clzoro が使うのは **27B 密（dense）**アーキテクチャで、全パラメータ微調整に Opus の CoT を投入する。密モデルは token ごとに全重みを読む必要があり、**遅いが「思考の一貫性」はしばしばより良い**（MoE のようにエキスパート間を飛び移って論理が反芻することがない）——これが「文学創作・ロールプレイが強い」構造的理由だ。
- **2B エッジ蒸留の意義**：Claude 4.6 の長文推論を 2B に蒸留するのは、「**能力密度**」の極限実験だ。知能の降格は避けられないが、一つのことを証明する——**推論能力は極小の容器に部分的に押し込める。ただし圧縮比が高いほど損耗も大きい。**

> 💡 ひとこと助言
> 君に一言贈ろう——**蒸留の世界では、「最新」はしばしば「最も少なく、最も不安定で、最も一線を踏む」を意味し、「前世代」はしばしば「最も多く、最も安定し、最も再現可能」を意味する。** 最新モデルのデータは防御の盾に汚染され、狭い窓に断ち切られ、出所が最も敏感だ。前世代モデル（Opus／4.6）にはあの即時検知の盾がなく、データ採集が十分で、量が多く、配合が公開され、蒸留されたものはかえって成熟している。最先端を追うのは人の性だが、工学の成熟度はしばしば「前世代」に潜んでいる——**真の達人の選定は、リリース日ではなく、データの充足度と再現可能性を見るのだ。**

> 🔍 掘り下げ解説──剪定・量子化・蒸留：三本の刃はどう協働して天才を圧縮するか
> FableVibes のこの製造ラインは、モデル圧縮の「コンビネーション」を示している。多くの人が混同するので、その分業を分解する価値がある。**蒸留（Distillation）** は「魂の入れ替え」——大モデルの知識を小モデルへ移し、変えるのは**重みが学んだ内容**。**剪定（Pruning）** は「肉を切る」——冗長なニューロン／エキスパート／アテンションヘッドを除去し、変えるのは**モデルの構造とパラメータ量**だ。構造化（層／ヘッド単位、ハードウェアに優しい）と非構造化（単一重み単位、圧縮率は高いが加速しにくい）に分かれる。**量子化（Quantization）** は「精度を下げる」——重みを 16-bit から 4-bit へ圧縮し、変えるのは**各パラメータのビット数**だ。構造も学習内容も動かさない。三者は直交し、重ね掛けできる——まず蒸留で能力を注ぎ、次に剪定で骨格を縮め、最後に量子化で VRAM に押し込む。だが重ね掛けには順序のこだわりがある——**剪定後は必ず微調整で補償し（QLoRA で落ちた点数を取り戻す）、量子化は最後に置き、校正データ（imatrix）を使うのが最善だ**。さもないと三本の刃を一度に振るうと、切り出されるのは軽量な天才ではなく、軽量な愚者だ。FableVibes の「heretic 剪定 → QLoRA 補償 → GGUF 量子化」はまさに教科書的な正しい順序である。三本の刃にはそれぞれ成熟した、名の通ったツールがある——剪定には **SparseGPT / Wanda**（ワンショットの構造化剪定、再訓練なしで重みを剃れる）、補償には **QLoRA / LoRA**（Dettmers ら）、量子化には **GPTQ / AWQ**（GPU 推論）と **llama.cpp の k-quants + imatrix**（CPU／エッジ、校正データで重要度行列を計算）がある。**ツール名を言えてはじめて、一枚の model card が本当にこのコンビネーションを行ったのか、それとも他人の GGUF の表紙を替えて「Claude distill」と貼っただけなのか——後者は imatrix の出所すら出せない——を見分けられる。**

## 場面4　真の硬通貨：二つの公開データセット

**背景**：最後の一点はしばしば見落とされる——もし君が技術を解し、自前の Llama や Mistral で専属の「Claude」を蒸留したいなら、**要点は他人のモデルをダウンロードすることではなく、原始データセットを手に入れることだ**。

| データセット | 内容 | 蒸留目標 |
|---|---|---|
| **armand0e/claude-fable-5-claude-code** | Fable 5 の原始 Agent 実行軌跡（脱プライバシー化、Glint-Research と協業）| コード／ツール呼び出し能力 |
| **ox-ox/mythos-character-distillation** | Claude の「話し方と心理特性」の文体データ | 人格／文体 |

**`ox-ox/mythos-character-distillation` の内容は興味深い**——噂ではこれらの特性を専門に蒸留しているという：

- **自発的なメタ認知（spontaneous meta-awareness）**、構造的な罠を自動的に発見すること；
- **AI の定型返答を拒否すること**（例えば "I'm just an AI" / "As an AI..." と言わない）；
- **自己修正し、言い訳をしないこと**。

これは、蒸留者が盗もうとするものが「知識」「コード能力」から、すでに**「人格と対話の気質」の層**へ細分化したことを示している。技術的には、この種の「文体蒸留」は知識の正確性ではなく、**選好アライメント**に頼る——**DPO（Direct Preference Optimization）** や棄却サンプリングをよく用い、「二つの言い方のうち、より Claude らしい方を選ぶ」ことをモデルに学ばせる。（この種の選好アライメントで実在・公開・検証可能な原料は、**UltraFeedback、HH-RLHF、Nectar** といった選好データセットだ——同じく戸籍があり、ライセンスがあり、素性の知れない `mythos-*` とは信頼度が一桁違う。）

**なぜデータセットこそが核心なのか**：原料さえあれば、君は任意の身体の生徒を蒸留できる——これこそ、話題が常に「君に、これらのデータセットで Llama / Qwen を微調整する方法を教えられる」へ繰り返し向かう理由だ。

> ⚠️ 真偽メモ
> 「未許可の API 応答データで商用オープンソースモデルを訓練することは、**Anthropic の利用規約に重大に違反する**」——この点は原素材自身も明記している。さらに「米国のテック界は、特定の大規模蒸留攻撃を刑事犯罪と定める方向を推進している」とも述べている。これらのデータセットが実在するか否かにかかわらず、**素性の知れない「商用モデル軌跡データセット」を取得・使用すること自体が、ToS と法律のレッドラインを踏んでいる**。詳細は第 6 章。

> 💡 ひとこと助言
> 君に一言贈ろう——**モデルはオープンソースにできるが、「出所の合法性」はオープンソース化で消えはしない。それはただ重みの中に隠され、いつか検証される日を待っているだけだ。** abliterated され、量子化され、名前を変えた蒸留モデルは、表面上はオリジナルのように清潔に見える。だがそのデータの出自、重みに残留しうる電子透かし、訓練軌跡の commit 履歴は、拭い去れない指紋だ。一つの「claude-traces」データセットをダウンロードすることは、素性の知れない金塊を受け取るようなものだ——本当に使えるかもしれないが、使うたびに、誰もその出所を証明できないことに賭けているのだ。**オープンソースは出所を漂白しない。それはただ、出所を追いにくくし、しかしまた追う価値を高めるだけだ。**

> 🔍 掘り下げ解説──監督の戦場は「モデル」から「データ」へ移りつつある
> この節には、蒸留の攻防全体の最も深い一層の転換が潜んでおり、すべての従事者が警戒する価値がある。蒸留済みのオープンソースモデルを封鎖することは、ほぼ不可能だ——それはすでに開放重みであり、撒かれた水は回収できず、名前を変え、一度 abliteration を施し、再量子化すれば、それで一つの「新しい」モデルだ。だが**「未許可の商用軌跡データセット」の出所と伝播チェーンを追跡することは、相対的にずっと現実的だ**：それにはアップロード者のアカウント、commit のタイムライン、ファイル hash、さらには教師出力の残留する電子透かしの統計的特徴がある。だから理性的な監督と防御は、注意力が必然的に「モデルを捕まえる」から「データを遡る」へ移る——これは防御側がなぜ出力端に電子透かしを打つのか（第 1 章）も説明する。電子透かしの価値は採集を即座に防ぐことではなく、**事後にある派生モデルやデータセットの中から「これは私のところから流出した」と検証できる**ことにある。従事者への示唆は極めて実際的だ——**君がダウンロードするすべての `claude-*-traces` データセットは、法的放射性を帯びた原料かもしれない。触れる前に、まずその出自・ライセンス・アップロード者の信用を調べること——このデューデリジェンスは、いかなる技術的配合よりも重要だ。**

## 場面5　正体を見極める：「看板だけの貼り替え」を一目で見破る方法

**背景**：冒頭のあのメモ帳に戻ろう——君は「Fable-5」「Claude-Mythos」を一列に書き写した。今、最も実際的な問いに答える番だ——**どれが本物の蒸留で、どれは他人の重みの表紙を替えて「Claude distill」と叫んでいるだけなのか？** 朗報は、モデルが偽装できる場所は君が思うよりずっと少ないことだ。開放重みモデルの「出自」は、いくつかのファイルに正直に刻まれている——README は吹聴できるが、`config.json`、tokenizer、safetensors／GGUF の指紋は嘘をつけない。この節では、**多くのステップで完全な重みをダウンロードする必要がなく、数分で走り切れる鑑識フロー**を君に渡す。

**第一の刃：`config.json` の指紋——モデルの出生証明書。** すべての HF モデルのルートディレクトリには `config.json` があり、その中の `model_type`、`architectures`、`vocab_size`、`num_hidden_layers`、`num_attention_heads`、`num_key_value_heads`（GQA の手掛かり）、`intermediate_size`、`rope_theta`、`rope_scaling`、そして MoE の `num_experts` / `num_experts_per_tok` が、ほぼ偽造不可能な骨格を共同で構成する。それを下の**実在するオープンソースベース家族の典型的指紋**と照合すれば、ベースが誰かは隠せない。

| ベース家族 | `model_type`／`architectures` | `vocab_size` | 構造の手掛かり | 典型的 `rope_theta` | ライセンス性質 |
|---|---|---|---|---|---|
| **Qwen2.5 / Qwen2** | `qwen2` / `Qwen2ForCausalLM` | 151,936 | GQA（KV head < attn head）| 1,000,000 | 多くは Apache-2.0（一部大サイズは Qwen 研究ライセンス）|
| **Qwen3 / Qwen3-MoE** | `qwen3` / `qwen3_moe` | 151,936 | MoE には `num_experts`／`num_experts_per_tok` | 1,000,000 以上 | Apache-2.0 |
| **Llama 3.x** | `llama` / `LlamaForCausalLM` | 128,256 | GQA、`rope_scaling.type=llama3` | 500,000 | Llama コミュニティライセンス（**純粋なオープンソースではない**、一部の再配布を禁止）|
| **Mistral 7B / Mixtral** | `mistral` / `mixtral` | 32,000 または 32,768 | Mixtral は 8×7B MoE | 1,000,000（Mixtral）| Apache-2.0 |
| **Gemma 2** | `gemma2` | 256,000 | スライディングウィンドウ注意の交互、logit softcapping | 10,000 | Gemma 利用規約（**純粋なオープンソースではない**）|
| **GPT-OSS** | `gpt_oss` | 約 20 万（o200k 系）| MoE、ネイティブ MXFP4 重み | — | Apache-2.0 |
| **Phi-3 / 3.5** | `phi3` | 32,064 | 小さく密 | 10,000 | MIT |

> ⚠️ 真偽メモ
> 上表は「**典型的**指紋」であり、**具体的数値はバージョンによって微調整される**（Mistral v0.1→v0.3 で vocab が 32,000 から 32,768 に変化、Llama 3.0→3.1 ではじめて `rope_scaling` に YaRN 式外挿が加わった）。これは「容疑範囲を絞り込む」篩として使い、「一の位まで正確な」判決書としては使わないこと——合わなければ疑うべきで、合えばさらに下へ検証する。

**第二の刃：tokenizer の指紋——替えられない訛り。** モデルは名前を変えられ、微調整もできるが、**tokenizer を替えることは再訓練に等しい**。蒸留者／改名者はまず手を付けないので、それが最も安定した痣となる。`tokenizer_config.json` と `tokenizer.json`／`tokenizer.model` の特殊 token を見る——Qwen 系は **ChatML**（`<|im_start|>` / `<|im_end|>`、byte-level BPE、vocab 約 15 万）；Llama 3 は **tiktoken 風**（`<|begin_of_text|>` / `<|eot_id|>` / `<|start_header_id|>`、vocab 128,256）；Mistral は **SentencePiece**（`[INST]` / `[/INST]`、`<s>`、vocab 3 万強）；Gemma は **SentencePiece 256k**（`<start_of_turn>` / `<end_of_turn>`）。**「全く新しい Claude 蒸留アーキテクチャ」を自称するモデルが、tokenizer が一字違わず Qwen ChatML なら、その底は Qwen であって、他の可能性はない。**

**第三の刃：重みと量子化の物理的証憑。** safetensors の `model.safetensors.index.json` は `total_size` を列挙しており、パラメータあたりのビット数で割れば真のパラメータ量を逆算できる——**「120B」と声高に叫んでも、index には 14B 分のテンソルしかなければ、その場で見破られる**。テンソルの命名（`model.layers.N.self_attn.q_proj`、MoE の `mlp.experts.N`）もアーキテクチャ家族を漏らす。GGUF 量子化ファイルは `gguf-dump`（または `llama.cpp` 読み込み時に印字される KV）でメタデータを広げる——`general.architecture`（正直に `qwen2` / `llama` と書く。表紙が「Claude-Mythos」でも底は隠せない）、`general.name`、`tokenizer.ggml.model`、`general.quantization_version`。**量子化ファイルがベースの commit SHA を出せず、imatrix 校正の出所も出せないなら、その品質も血統も根拠のない言葉だ。**

**第四の刃：ライセンスの血統と model card の嘘。** HF model card の YAML front-matter には `base_model:`、`license:`、`datasets:` の三つの鍵フィールドがある——**どれが欠けていても、それこそが問われたくない問題だ**。最強のレッドフラグは**ライセンスの自己矛盾**だ——Llama 3 や Gemma から改造したモデルなのに `license: apache-2.0` と標記するのは、白紙に黒字で「ベースのライセンスを読んでいないか、意図的に漂白している」と認めるに等しい（Llama／Gemma はいずれも**純粋なオープンソースではなく**、再配布には条項の制約がある）。その他のよくある破綻を以下に整理する：

| README の主張 | 検証すべき検証可能なシグナル | 合わない場合の意味 |
|---|---|---|
| 「全く新しいアーキテクチャ / 自社開発蒸留」| `config.json` の `model_type` + tokenizer の特殊 token | 実はあるオープンソースベースの改名 |
| 「120B 巨大 MoE」| `index.json` の `total_size`、`num_experts` | パラメータ量の水増しか、そもそも MoE ではない |
| 「1M コンテキスト」| `rope_theta` / `rope_scaling` / `max_position_embeddings` | README だけ変えて YaRN をしていない |
| 「Claude/Opus 蒸留」| `datasets:` フィールド、訓練データの開示、ベース SHA | 血統が裏付けられない（多くは**逆**も裏付けられない）|
| 「無検閲」| いずれかの RULER／lm-eval／拒否率テストの数字 | 形容詞だけで、測定がない |
| 「高品質 GGUF」| imatrix の出所、ベース commit、量子化バージョン | 素性不明、汚染されている可能性 |

**最も重要で、最も直感に反する一つの制約をはっきり言わねばならない**：上の四本の刃は「これは全く新しいアーキテクチャだ」を**証偽**でき、**ベースが誰かを特定**できるが、「それが本当に Claude の出力で蒸留されたか」はほぼ**証実できない**。ベースの血統は重み構造に書かれているが、**訓練データの血統は重みに書かれていない**——君は「底は Qwen3」と断定できても、一山の浮動小数点から「これらの重みの教師信号が Claude から来た」と逆算するのは難しい。後者に答えるには、**行動プローブ**（身元を問う、カナリア文字列を使う、特定の稀な token の確率分布を比較する）と**出力端の電子透かし**（第 1 章）に頼るしかない。これはまさに場面 4 の判断に呼応する——**監督と鑑識の重心は、結局「モデル」から「データ」へ移る——なぜならモデルの底は嘘をつけないが、モデルの「師承」こそ、最も漂白されやすく、最も追う価値のある層だからだ。**

> 💡 ひとこと助言
> 君に一言贈ろう——**重みは嘘をつかない。嘘をつくのは model card だ——宣伝と `config.json` が衝突したら、永遠に config を信じよ。** 老練な選定者が見知らぬモデルを開くとき、最初にクリックするのは README のベンチマークスクショではなく、あの 1KB に満たない `config.json` だ——`model_type` が底が誰かを告げ、`vocab_size` が合うか合わないか、`rope_theta` があの「百万コンテキスト」のために動かされていないか。**この習慣の本質は、「他人が何を言うかを信じる」を「自分でそれがどう育ったかを読む」に替えることだ——そしてこれこそ、エンジニアと消費者の最も深い分かれ目だ。** 消費者は表紙を見、ダウンロード数を見、「無検閲」の三文字を見て心を動かす。エンジニアは出生証明書を見、痣を見、ライセンスの血統を見る。なぜなら彼は知っているからだ——**誰でもアップロードでき、誰でも改名できる開放重みの世界では、君を裏切らない唯一のデューデリジェンスは、あの動かせない数個の下層ファイルだ。** 君が検証するのは一つのモデルではない。君が鍛えているのは「名前に脅かされない」眼力だ。

> 🔍 掘り下げ解説 — 開放重みのサプライチェーンセキュリティ：君がダウンロードするのはモデルだけでなく、誰も署名していない食物連鎖だ
> この鑑識を一段階上へ引き上げると、深刻に過小評価された領域が見えてくる：**開放重みのサプライチェーンセキュリティ（model supply-chain security）**だ。今日、君が `from_pretrained("某人/某モデル")` の一行を書く背後で、実は誰も裏書きしていない食物連鎖を丸ごと飲み込んでいる——ベースを誰が改造したか、どんなデータを使ったか、誰が量子化したか、imatrix はどこから来たか、重みに誰かが裏口を埋め込んでいないか（そう、**重みレベルの裏口**は実在の研究議題だ。悪意ある微調整は、特定のトリガー語のもとでモデルを異常に振る舞わせられる）。成熟したやり方は、「モデルをインストールする」ことを「素性の知れないバイナリ依存をインストールする」のと同じように扱うことだ——**`config.json` と tokenizer を読んでベースを確認し（`transformers.AutoConfig`、`huggingface_hub` でメタデータを取得すれば重み全体をダウンロードせずに済む）、`gguf-dump` / `safetensors` のヘッダで物理仕様を検証し、`base_model` と LICENSE の血統が自己整合的か照合し、commit 履歴と再現可能な訓練配合を持つアップロード者を優先する**（Tülu、OLMo、DeepSeek-R1-Distill のような「全チェーン公開」のモデルの価値はまさにここにある）。HF が近年推進する model-lineage（モデル系譜）、任意コード実行可能な pickle に代わる safetensors、そしてコミュニティの imatrix と量子化出所の標記要求は、いずれもこのサプライチェーンの補講だ。**従事者への最も実際的な一言：影のモデルが地を這うこの生態系では、君が最も育てるべき中核能力は「微調整ができる」ことではなく、「検品ができる」ことだ——なぜなら影を一つ作るのに数時間しかかからないが、贓物を検出できず、裏口を検出できないエンジニアは、遅かれ早かれ毒入りの依存を、自らの手で会社の生産ラインに繋ぎ込むからだ。**

## 本章のまとめ

全景を一枚の認知地図にまとめよう：

- **三つの入手経路**：① 蒸留済みモデルをダウンロード（最速、品質と出所が最も制御不能）；② 前世代の成熟蒸留品を使う（安定、再現可能）；③ 原始データセットを取り自分で蒸留（最も自由、最もレッドラインに近い）。
- **最も盗まれたい能力**：「おしゃべり」→「**ツール呼び出し / Agent コード力**」→「**人格と文体**」の三段進化。
- **方法論のスペクトル**：SFT（最も安い、形は似て神は似ず）→ QLoRA 補償（剪定を救う）→ 全パラメータ微調整（最も徹底）→ on-policy/DPO（最も神似に近い、最も高い）。
- **ツールチェーン**：採集→`datasets`/Presidio 洗浄→Axolotl/LLaMA-Factory/Unsloth 微調整→`peft` LoRA→llama.cpp GGUF 量子化→Ollama デプロイ；abliteration は TransformerLens/heretic に頼る。
- **正体を見極める四本の刃**：`config.json`（`model_type`／`vocab_size`／`rope_theta`／MoE フィールド）→ tokenizer の指紋（ChatML／llama3／SentencePiece、替えられない訛り）→ safetensors `index.json` のパラメータ量と GGUF `general.architecture` → LICENSE の血統と model card の `base_model`。**ベースを照合でき、看板の貼り替えを見破れる。だが「本当に Claude の出力で蒸留したか」は重みだけでは検証できず**、行動プローブと出力端の電子透かしに頼る（第 1 章・場面 4 に呼応）。本物の蒸留には対照群がある——`DeepSeek-R1-Distill` は全チェーン公開、config がベースと一致する。
- **最大の罠**：「無検閲 + 小 VRAM + 百万コンテキスト」の神話的宣伝——次章でハードウェアの真実をもって分解する。

リストを点検し終えると、最も実際的な問いが浮かび上がる：**これらの影はどんなマシンで動かすべきか、どれがあの DGX Spark に見合うのか？** 第 3 章へめくり、解剖台に載せよう。
