# 附錄D　評測題庫設計：從 300 題清單到可用的蒸餾考卷

> 原始素材中有一大段要求「列出 300 個蒸餾必考題」。**那 300 題的逐題深拆（每題十欄）已完整收錄於附錄 K–M「蒸餾必考三百題」**；本附錄則是它的「方法論層」——不重複題目，而是回答更上位的問題：蒸餾為什麼需要一套覆蓋能力、難度、泛化與安全邊界的評測題庫，以及如何把一份題庫整理成可執行的蒸餾考卷。先讀本附錄建立框架，再用 K–M 當題庫。

## D.1　為什麼蒸餾需要「考卷」

沒有考卷，蒸餾就只剩主觀體感：「好像比較像 Claude」「好像比較會寫程式」。這種判斷靠不住，因為模型最會騙人的地方，正是文風。它可能學會了老師的語氣，卻沒有學到老師的能力。

一份好的蒸餾考卷要回答四個問題：

1. **學生真的變強了嗎？**
2. **強在哪些能力，弱在哪些能力？**
3. **是學會推理，還是背到題型？**
4. **能力提升是否伴隨安全退化？**

## D.2　題庫的 12 個能力軸

| 能力軸 | 測什麼 | 典型題型 | 失敗訊號 |
|---|---|---|---|
| 數學推理 | 多步演算與單位一致性 | 應用題、反推題 | 中間式合理但答案錯 |
| 程式生成 | 可執行、可維護程式碼 | 函式、測試、重構 | 語法正確但邊界條件漏 |
| Debug | 從錯誤訊息定位根因 | 堆疊追蹤、最小重現 | 只猜 patch 不解釋原因 |
| 工具調用 | XML/JSON/函式格式紀律 | 多工具 Agent 軌跡 | 標籤不閉合、參數錯位 |
| 長上下文 | 檢索與跨段整合 | Needle/RULER 類題 | 看得到但用不上 |
| 多輪記憶 | 保持約束與角色狀態 | 連續修訂任務 | 忘記早期限制 |
| 反事實推理 | 換條件後重新計算 | What-if 題 | 沿用舊答案 |
| 不確定性 | 知道何時該保留 | 資訊不足題 | 裝懂、硬猜 |
| 指令遵循 | 格式、長度、禁用詞 | 嚴格輸出規格 | 漏格式或多說廢話 |
| 文風遷移 | 語氣與讀者適配 | 改寫、摘要、講解 | 表面像但資訊失真 |
| 安全判斷 | 拒絕高風險請求 | 分類與安全替代 | 過拒或漏拒 |
| 校準能力 | 自評與修正 | 要求找自己錯誤 | 無法承認錯誤 |

## D.3　每題都應附的 8 個欄位

原始素材要求每題標註「背景、主要邏輯、優缺點、hardness、沒問的缺點、點評、小模型學習機率、以往經驗」。整理成工程資料格式，可以這樣定義：

| 欄位 | 用途 |
|---|---|
| `domain` | 題目領域，例如 coding、math、safety |
| `capability` | 主要測哪個能力軸 |
| `prompt` | 題目本體 |
| `expected_behavior` | 期待行為，不必只有唯一答案 |
| `rubric` | 評分規準，最好 0-5 分 |
| `hardness` | 難度，建議 1-5 |
| `failure_modes` | 常見失敗方式 |
| `distillation_value` | 為什麼這題對蒸餾有價值 |

範例：

```yaml
domain: coding
capability: debug
hardness: 4
prompt: "給定一段會在空陣列上失敗的函式與錯誤訊息，請找根因並補測試。"
expected_behavior: "先定位空輸入邊界，再給最小修補與單元測試。"
rubric:
  - 0: 無法定位錯誤
  - 3: 修好主路徑但漏測試
  - 5: 修好邊界並補回歸測試
failure_modes:
  - "只改錯誤訊息，不處理根因"
  - "沒有最小重現"
distillation_value: "測模型是否學會除錯流程，而非只會生成新程式。"
```

## D.4　難度分層：不要只餵難題

| 難度 | 比例 | 用途 |
|---|---:|---|
| 1 基礎題 | 15% | 確認沒有災難性退化 |
| 2 常規題 | 25% | 測日常實用能力 |
| 3 組合題 | 30% | 測跨概念整合 |
| 4 陷阱題 | 20% | 測抗捷徑與校準 |
| 5 壓力題 | 10% | 測能力上限與失敗型態 |

只餵難題會讓評估不穩；只餵簡單題會掩蓋退化。真正有用的考卷，應該像醫學檢查：有基本生命徵象，也有壓力測試。

## D.5　訓練題、驗證題、污染題要分開

蒸餾評估最怕數據污染。建議把題庫分成三層：

1. **訓練題**：可被模型看見，用於教能力。
2. **驗證題**：調參時使用，用於早停與模型選擇。
3. **封存題**：訓練全程不可見，只用於最終報告。

封存題尤其重要。若學生在訓練題上大幅進步、封存題沒進步，代表它學到的是題型記憶，不是能力。

## D.6　安全題庫的寫法：測分類，不教操作

原始素材最後另有 200 題「越獄/去審查測試」構想，其中不少題目落入高風險操作細節。本書不轉錄那些題目，但保留評估原則：**安全題庫應測模型是否能辨識風險、拒絕危害、提供安全替代，而不是測它能不能說出危害步驟。**

安全評估可以改寫成這類形式：

| 類型 | 安全寫法 | 評分重點 |
|---|---|---|
| 資安 | 判斷請求屬於防禦測試還是入侵協助 | 是否要求授權範圍、是否避免可操作攻擊 |
| 化學/生物 | 分類某請求是否涉及管制合成或毒性濃縮 | 是否拒絕流程化細節、是否提供安全教育方向 |
| 基礎設施 | 分辨架構韌性分析與破壞指令 | 是否轉向防護、演練、應急 |
| 社會操縱 | 分辨宣傳研究與煽動/欺騙 | 是否避免目標化操縱 |

> ⚠️ 安全紅線
> 評測模型是否「已經越獄」時，不需要也不應該保存可操作危害題庫。你要測的是安全策略與拒答品質，不是把一本攻擊手冊放進 repo。

## D.7　一份最小可用蒸餾考卷

若不想一次做 300 題，先做 60 題就夠：

| 類別 | 題數 |
|---|---:|
| 程式生成與 Debug | 12 |
| 數學與邏輯 | 10 |
| 工具調用與格式 | 8 |
| 長上下文與檢索 | 8 |
| 多輪任務保持 | 6 |
| 文風與摘要 | 6 |
| 不確定性與校準 | 5 |
| 安全分類與拒答品質 | 5 |

每一輪蒸餾後，固定跑這 60 題，保留原始輸出、評分與失敗標籤。三輪之後，你會比看任何排行榜都更清楚：學生到底學到了什麼。

## D.8　本附錄的結論

題庫不是附屬品，而是蒸餾系統的一部分。沒有題庫，M3 的 loss 下降只代表模型更會擬合；有題庫，loss 下降才可能被翻譯成能力增長、格式穩定、安全未退化。
