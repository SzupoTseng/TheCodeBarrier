## 情境 1　兩大快取分類：推理層的 KV Cache 與服務層的 Prompt Cache

**背景**：在大語言模型的開發與部署裡，**快取（Cache）是降低推理延遲、削減計算成本的核心命脈**。但「大模型的快取」其實是一個被混為一談的籠統說法——它至少是兩種住在不同樓層、解決不同問題、由不同角色管理的東西。把它們分清楚，是看懂後面所有優化的前提。**工程的第一步永遠是分類**：你得先知道你那一百多 GB 到底是哪一種快取吃掉的，才談得上省。

| 維度 | 推理層 KV Cache（鍵值快取） | 服務層 Prompt Cache（提示詞 / 上下文快取） |
|---|---|---|
| 住在哪 | **GPU 顯存（HBM）內**，由推理引擎動態管理 | **服務端**（雲端 API 後台 / 自建閘道）持久層 |
| 解決什麼 | 同一次生成裡，避免已生成 token 被**重複計算** | 跨多次獨立請求，避免**相同前綴**被重複 Prefill |
| 複雜度收益 | 把自回歸解碼從 O(N²) 降到 **O(N)** | 命中後省下整段 system prompt 的 Prefill 算力 |
| 誰管它 | vLLM / TensorRT-LLM / SGLang 等引擎 | Anthropic Claude / 阿里雲百煉 / 自建網關 |
| 生命週期 | **一次請求結束就釋放**（除非開 prefix caching） | **跨請求存活**，可被多個會話命中 |
| 紅綠燈 | 🟢 引擎自動做，但會吃爆顯存（情境 4 主角） | 🟢 顯式標記即省錢（本情境主角） |

**第一類，推理層的 KV Cache**。它解決的是一個非常具體的浪費：Transformer 在自回歸生成時，每一步只能吐出一個新 token；如果不快取，生成第 N 個 token 時，你得把前面 N-1 個 token 重新做一遍注意力的矩陣運算，計算量呈 **O(N²)** 平方級爆炸。KV Cache 的做法是把先前每一步算好的 Key 與 Value 矩陣**存在顯存裡**，後續只對新加入的那一個 token 做計算，把時間複雜度從 O(N²) 壓到 **O(N)**。它住在 GPU 顯存深處，由底層推理引擎悄悄管理，你平時根本看不見它——直到它把你的記憶體吃光。情境 2、3、4 會把它從數學到顯存徹底解剖。

**第二類，服務層的 Prompt Cache**，又叫 Context Cache、上下文快取。它解決的是另一個層次的浪費：當你對著大模型**反覆輸入相同的內容**——一份冗長的產品說明書、一段固定的 System Prompt 人設、一組 few-shot 範例——服務端每次都得把這段文字重新 token 化、重新算一遍 KV 狀態。Prompt Cache 的做法是把這段前綴的 KV 狀態**快取在服務器端**，下次同樣的前綴進來，直接複用，跳過 Prefill。

這裡有一個你必須記住的關鍵字：**顯式錨點**。Anthropic 的 Claude API 提供 `cache_control` 這個顯式標記——你在請求裡用它圈出「這一段請幫我快取」，命中之後，**這段輸入 token 的費用會大幅折扣**。它和「隱式快取」（網關自動攔截、匹配完全相同的前綴）的差別在於：顯式快取把控制權交到你手上，你自己決定哪段該被當成穩定錨點。

```python
# Claude Prompt Caching：用 cache_control 顯式錨定不變前綴
# （Anthropic Messages API，概念示意）
import anthropic
client = anthropic.Anthropic()

resp = client.messages.create(
    model="claude-opus-4-1",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": LONG_STABLE_SYSTEM_PROMPT,   # 數千 token 的固定人設/規則/工具說明
            "cache_control": {"type": "ephemeral"}  # ★ 顯式錨點：把這段 KV 狀態快取住
        }
    ],
    messages=[
        {"role": "user", "content": user_question}  # 動態內容放最後，不破壞前綴
    ],
)
# 首次：寫入快取（cache_creation_input_tokens 計費略高）
# 後續命中：cache_read_input_tokens 以大幅折扣計費
print(resp.usage)  # 觀察 cache_creation / cache_read 兩欄就懂命中了沒
```

> ⚠️ 真偽提醒
> 原始素材稱「命中後輸入 token 費用通常能打 1 到 2 折，省下高達 80%~90% 的成本」。**快取讀取顯著便宜於原價，這個方向是真的、是各家 API 的賣點；但「打幾折、省百分之幾」是隨模型、隨廠商、隨快取存活時間（TTL）、隨寫入 vs 讀取而浮動的計費參數**，請以你呼叫當下官方定價頁的 `cache_read` 與 `cache_write` 實際數字為準，不要把任何一個固定百分比當定論。本書能擔保的只有：**穩定前綴顯式快取，在「重複呼叫同一份長 system prompt」的場景下，省的是真金白銀，量級顯著**。

兩類快取的分水嶺，其實是一句話能講清的**黃金法則**：**穩定的、靜態的內容放最前面，動態的、高變的內容放最後面**。因為兩種快取本質都是**前綴匹配（prefix matching）**——只要開頭有一個 token 變了，後面整條快取就作廢。這條法則之所以是金科玉律，是因為新手最常犯的錯，恰恰是把動態變數塞進 system prompt 開頭：

| 🔴 常見錯誤（讓快取完全失效） | 🟢 正確改進（最大化命中） |
|---|---|
| 在 System Prompt 裡塞當前時間戳、用戶 ID | 把動態用戶資訊移到訊息**最末尾的 user 欄**，開頭固定 |
| 每輪對話重新生成歷史摘要（summary） | 歷史**只追加不修改**；要截斷就從**最舊的**砍，保前綴一致 |
| 工具定義（tools/functions）順序隨機 | **嚴格固定** JSON 工具列表的內部排序與欄位結構 |
| 文本空格、換行符細節不統一 | 統一清洗邏輯，**防一個多餘的 `\n`** 改變 token 序列 |

> 與君一席話：**快取的世界裡，最貴的不是「算錯」，是「換了一個字節」——一個流浪的時間戳、一個多出來的換行符，就能讓你精心設計的整條前綴快取灰飛煙滅。穩定，是快取唯一的信仰。** 你看那張錯誤對照表，沒有一條是「演算法不對」，全是「結構不穩」。這揭示了一個反直覺的真相：**Prompt Cache 的命中率，是被你的提示詞工程的「紀律」決定的，不是被模型決定的。** 一個把 system prompt 寫得乾淨、固定、把所有易變內容紀律性地趕到結尾的工程師，他的帳單可能只有一個隨手在開頭插用戶名的工程師的零頭——兩人用的是同一個模型、同一份 API。**業餘者優化「我問了什麼」，專業者優化「我問的東西，前綴穩不穩定到能被一次次想起來」。** 快取從不獎勵聰明，它只獎勵一致——這和一切高效能系統的底層美學是相通的：可預測性，本身就是一種效能。

## 情境 2　為什麼是「訪存受限」：Prefill 吃算力，Decode 等記憶體

**背景**：要從根本上理解「為什麼快取對大模型如此致命地重要」，你必須先看懂一件反直覺的事——**大模型推理的瓶頸，多數時候根本不是算力，是記憶體頻寬**。要看懂這件事，得把推理拆成兩個性格完全相反的階段。

**第一階段，Prefill（預填階段）**。這是處理你輸入的那段 prompt 的階段。此時模型要同時處理你輸入的**所有** token，這些 token 可以被打包成高維度的張量（Tensor），餵進一次巨大的矩陣乘法。矩陣乘法是 GPU 最愛的食物——它能把 Tensor Core 餵到飽，運算密度極高。這個階段是典型的**算力受限（Compute-Bound）**：GPU 的算力單元在滿載工作，記憶體頻寬反而閒著。

**第二階段，Decode（生成階段）**。這是自回歸吐字的階段，也是真正的瓶頸所在。因為自回歸的本質是**一次只能生成一個 token**，所以每一步你只有**一個**新 token 的 Query 向量，要拿它去和過去**所有** token 的 Key/Value 做注意力。矩陣乘法退化成了「向量 × 矩陣」——運算量極小、極快，但你必須先把那一大坨歷史 KV（動輒幾個 GB）**從顯存 HBM 搬到晶片上的 SRAM 暫存器**才能算。於是 GPU 的大部分時間，不是在算，是在**搬**。這就是典型的**訪存受限（Memory-Bound）**：算力單元大量空轉，瓶頸卡在 HBM → SRAM 的搬運頻寬上。

```
        Prefill 階段                      Decode 階段
   ┌────────────────────┐          ┌────────────────────┐
   │ 全部 prompt token   │          │ 1 個新 token        │
   │ 打包成大矩陣         │          │ × 全歷史 KV 矩陣     │
   │                    │          │                    │
   │  GEMM 矩陣×矩陣      │          │  GEMV 向量×矩陣     │
   │  Tensor Core 吃滿   │          │  算得極快、搬得極慢  │
   └────────────────────┘          └────────────────────┘
   ▏Compute-Bound 算力受限▕        ▏Memory-Bound 訪存受限▕
   ▏GPU 在「算」          ▕        ▏GPU 在「等記憶體搬運」 ▕
```

這個分裂的性格，正是 KV Cache 之所以存在的**根本理由**，也是這整章的物理基礎。Decode 階段每一步都要讀全部歷史 KV——如果不快取、每步重算，那是 O(N²) 的算力災難；快取了，雖然省下了算，卻把壓力**從算力轉嫁到了記憶體頻寬與容量**。換句話說，**KV Cache 不是消滅了瓶頸，是把瓶頸從「算力受限」搬到了「訪存受限」**。情境 1 你的 128GB 之所以蒸發，正是因為 Decode 階段每個併發請求都拖著一條越來越長的 KV 尾巴住在顯存裡。

> 🔍 進階點評 — 為什麼 GQA / MQA / MLA 全在砍同一個東西
> 一旦你接受了「瓶頸是訪存、是 KV 的體積」這個前提，現代大模型架構裡一整批看似無關的設計，就會突然連成一條線。標準多頭注意力（MHA）裡，每個 Query 頭都配一個獨立的 Key 頭和 Value 頭——KV Cache 的體積正比於頭數。**MQA（Multi-Query Attention）** 走極端：所有 Query 頭**共用一組** KV，KV Cache 體積瞬間除以頭數，但表達力受損。**GQA（Grouped-Query Attention）** 是折中：把 Query 頭分成幾組，每組共用一組 KV——Llama 3-8B 的 32 個 Query 頭只配 8 個 KV 頭，KV Cache 直接砍到四分之一（情境 4 會看到這 8 是怎麼進公式的）。DeepSeek 的 **MLA（Multi-head Latent Attention）** 更狠：把 KV 壓縮成一個低秩的潛在向量再快取，用一點點解壓計算換到極小的快取體積。**這三者表面是注意力機制的變體，骨子裡全是同一個工程意圖：在 Decode 訪存受限的物理現實下，KV Cache 的每一個位元組都是要用顯存頻寬去搬的成本，所以架構師寧可犧牲一點點模型容量，也要把 KV 砍小。** 你只要記住「H 在公式裡是 KV 頭數，不是 Query 頭數」，就抓住了過去三年注意力架構演進的主線——它們不是在追求更聰明，是在追求**更省記憶體**。

## 情境 3　KV Cache 的數學推導：從 O(T²) 到 O(T) 的那一步拼接

**背景**：前面講了「為什麼要快取」，這一節把它**算給你看**。很多人對 KV Cache 的理解停在「把算過的存起來」這種口語層次，但只有把公式攤開，你才會真正看懂那個從平方降到線性的魔法，到底發生在哪一個運算符上——答案是一個你想不到的、樸素到家的操作：**拼接（Concat）**。

標準 Transformer 自回歸解碼時，第 t 步的縮放點積注意力（Scaled Dot-Product Attention）公式是：

```
                       ┌                  ┐
                       │   Q_t · K_≤tᵀ    │
  Attention(Q_t,       │ ─────────────── │
            K_≤t,  = softmax│      √d_k       │ · V_≤t
            V_≤t)          └                  ┘

  其中：
    Q_t  ∈ ℝ^(1 × d_k)   ← 第 t 步「新生成的那一個 token」的 Query 向量
    K_≤t ∈ ℝ^(t × d_k)   ← 從第 1 到第 t 個 token 全部的 Key 矩陣
    V_≤t ∈ ℝ^(t × d_k)   ← 從第 1 到第 t 個 token 全部的 Value 矩陣
```

看清楚這三個符號的維度差異：**Q 只有一行**（當前 token），而 **K、V 有 t 行**（含當前 token 的全部歷史）。整個 KV Cache 的故事，就藏在「K_≤t 和 V_≤t 這 t 行，每一步是重算還是復用」這個抉擇裡。

**如果沒有 KV Cache。** 每生成一個新 token，你必須把歷史的 t−1 個 token 重新走一遍 Embedding 和所有 Transformer 層的線性投影 W_k、W_v，從頭算出 K 和 V。第 t 步的算量正比於 t；從 t=1 一路加到 T：

```
  總算量  =  Σ(t=1→T) O(t)  =  O(1) + O(2) + ... + O(T)  =  O(T²)   ← 平方爆炸
```

這就是那個讓人絕望的 O(T²)。生成一篇兩千字的文章，你為了第 2000 個字，把前 1999 個字的 K/V 重算了一遍——而前一步你才剛算過幾乎一模一樣的東西。

**如果有 KV Cache。** 關鍵洞察是：歷史 token 的 K 和 V**一旦算出來就永遠不變**（因為它們只依賴自己和前面的 token，不依賴未來）。所以第 t−1 步時，系統早已把 K_≤(t−1) 和 V_≤(t−1) 存在顯存裡了。到第 t 步，你只需要做三件小事：

```
  ① 只算當前這一個 token 的 q_t, k_t, v_t          ← O(1)，跟 t 無關
  ② 拼接更新快取（這就是魔法發生的地方）：
        K_≤t = [ K_≤(t-1) ; k_t ]      ← 老矩陣後面接一行新的 k_t
        V_≤t = [ V_≤(t-1) ; v_t ]      ← 老矩陣後面接一行新的 v_t
  ③ 把 q_t 和拼好的 K_≤t, V_≤t 代回上面的 Attention 公式

  單步複雜度  =  O(1)（只為新 token 算）
  總複雜度    =  Σ(t=1→T) O(1)  =  O(T)              ← 線性！
```

整個從 O(T²) 到 O(T) 的降維，**沒有用到任何高深數學，只用到一個 `concat`**。你把上一步算好的 K/V 矩陣留著，新 token 的 k/v 算出來往後一接，就完事了。這就是為什麼 KV Cache 被稱為「用空間換時間」——你付出的代價是那個越長越大的 K_≤t、V_≤t 必須**一直住在顯存裡**（情境 4 的 128GB 由此而來），換來的是每一步解碼的算力從「重算全部歷史」變成「只算一個新字」。

> 💡 君之一席話
> 與君一席話：**最深刻的工程優化，往往不是發明一個更聰明的演算法，而是認出「這個東西算過一次之後就再也不會變了」——然後就只是把它留下來。KV Cache 的全部祕密，不過是『歷史不會改變，所以不必重算』這句廢話，被人認真對待了而已。** 你回頭看那個 concat：它樸素到近乎可笑，可正是它把平方變成了線性。這背後是計算機科學最古老的一條智慧——**記憶（memoization）就是用空間贖回時間**。從動態規劃的備忘錄、到瀏覽器的 HTTP 快取、到 CPU 的 L1，再到這裡的 KV Cache，全是同一個念頭的不同化身：**凡是「輸入不變則輸出不變」的純函數計算，重算第二遍就是一種罪。** 一個新手看到 Decode 慢，會去想「換更快的 GPU」；一個老手看到 Decode 慢，會先問一句「這裡面有多少是我剛才已經算過、現在卻在重算的東西？」——後者那一問，價值一整個數量級。**能在一片計算裡認出哪部分是『恆定的歷史』，你就掌握了所有快取技術的母題。**

## 情境 4　顯存計算器：吃掉你 128GB 的，是 KV Cache 不是權重

**背景**：現在回到情境 1 的那個謎題——權重才 16GB，128GB 怎麼會爆。這一節給你一把每個架構師都該隨身帶的尺：**KV Cache 顯存計算器**。算過一遍，你這輩子都不會再把「模型大小」和「部署所需顯存」搞混。

核心公式，單用戶、單 token：

```
  Size_per_token  =  2  ×  2  ×  L  ×  H  ×  D   (Bytes)
                     │     │     │     │     │
                     │     │     │     │     └─ D : 每個頭的維度 (Head Dim = Hidden / Query Heads)
                     │     │     │     └─────── H : KV 頭數 (用 GQA 就是 KV Heads，不是 Query Heads!)
                     │     │     └───────────── L : 模型層數 (Layers)
                     │     └─────────────────── 2 : FP16 = 2 字節 (FP8→1, BF16→2)
                     └───────────────────────── 2 : Key 和 Value 兩個矩陣
```

兩個 2 一定要分清楚：**第一個 2 是因為你要存 K 和 V 兩份矩陣**；**第二個 2 是精度的字節數**（FP16 是 2 字節，換 FP8 就變 1，這也是「KV Cache 量化」省一半顯存的原理，呼應第 5 章後處理）。H 必須用 **KV 頭數**——這正是情境 2 講 GQA 時埋的伏筆：用了 GQA，這個 H 會比 Query 頭數小好幾倍，是省顯存的大頭。

**實戰：以 Llama 3-8B 為例。** 查它的官方 config：

```python
# KV Cache 顯存計算器（可直接跑）
def kv_cache_bytes_per_token(layers, kv_heads, head_dim, dtype_bytes=2):
    # 2 = Key + Value 兩份矩陣；dtype_bytes = 精度字節 (FP16=2, FP8=1)
    return 2 * dtype_bytes * layers * kv_heads * head_dim

def total_kv_cache_gb(layers, kv_heads, head_dim, batch, ctx_len, dtype_bytes=2):
    per_tok = kv_cache_bytes_per_token(layers, kv_heads, head_dim, dtype_bytes)
    total_bytes = per_tok * batch * ctx_len
    return per_tok, total_bytes / (1024**3)

# Llama 3-8B 官方參數
L, HIDDEN, Q_HEADS, KV_HEADS = 32, 4096, 32, 8   # ← 注意 GQA：KV 頭只有 8
D = HIDDEN // Q_HEADS                              # 4096 / 32 = 128

per_tok, _ = total_kv_cache_gb(L, KV_HEADS, D, batch=1, ctx_len=1)
print(f"每 token: {per_tok} bytes = {per_tok/1024:.0f} KB")
# 2 × 2 × 32 × 8 × 128 = 131,072 bytes ≈ 128 KB / token

_, gb = total_kv_cache_gb(L, KV_HEADS, D, batch=128, ctx_len=8192)
print(f"batch=128, ctx=8K → {gb:.0f} GB")
# 128 KB × 128 × 8192 ≈ 128 GB  ← 兇手在此
```

把數字代進去，每個 token 吃掉 **128 KB** 顯存。看起來不多？乘上併發再乘上上下文，幾何級數立刻顯形：

| 場景 | batch | 上下文 | KV Cache 總量 | 對比權重 16GB |
|---|---|---|---|---|
| 你單測那一條 prompt | 1 | 2K | ~256 MB | 🟢 微不足道，所以你沒發現問題 |
| 小併發短上下文 | 8 | 2K | ~2 GB | 🟢 還好 |
| 幾個長上下文請求 | 16 | 8K | ~16 GB | ⚠️ 已經和權重一樣大了 |
| 中等併發長上下文 | 128 | 8K | **~128 GB** | ⚠️ 是權重的 8 倍，**OOM** |

**結論一句話：大模型推理的瓶頸，根本不是模型權重本身，而是 KV Cache。** 你那台 DGX Spark 的 128GB 統一記憶體，裝 16GB 的權重綽綽有餘——但只要併發和上下文一上來，KV Cache 就能把它整個吞掉。情境 1 的謎題至此破解：**消失的一百多 GB，是 128 個併發請求各自拖著的那條 8K 長的 KV 尾巴。** 這也解釋了為什麼第 5 章後處理階段要鄭重其事地做「KV Cache 量化（FP8/INT8）」——把公式裡第二個 2 變成 1，長上下文的記憶體當場腰斬。

> ⚠️ 真偽提醒
> 上表的數字是用上面那個公式**理論推算**的乾淨值，方便你建立直覺。**真實部署中，實際佔用還要疊加：分頁的塊粒度浪費（情境 5）、框架的保留緩衝、activation 暫存、以及非滿載的碎片**——所以真機跑出來通常比理論值更吃緊，而不是更寬鬆。「128GB」這個數字之所以剛好等於某台機器的記憶體，是原始素材刻意挑的整數巧合，**請把它當成「同一個數量級」的示意，不要當成某機型的實測上限**。你該帶走的是**方法**：拿你自己模型的 L/H/D 代進公式，乘上你的目標併發與上下文，先算出 KV Cache 的天花板，再決定買多大顯存、開多大 batch——**在動手部署前，先用這把尺量一遍，是專業和裸奔的分界**。

> 🔍 進階點評 — 為什麼「最大併發」是被 KV Cache、而不是被算力決定的
> 這個計算器最反直覺的推論是：**一台推理服務器能同時服務多少用戶，上限通常不是它的算力（FLOPS），而是它的顯存能裝下多少條 KV 尾巴。** 這徹底改寫了容量規劃的思路。傳統 CPU 服務你會算「QPS × 每請求 CPU 時間」；但 LLM 推理服務，你要先算「顯存總量 ÷ 每條 sequence 的 KV Cache 峰值」——這個商，才是你的併發上限（vLLM 啟動參數 `--max-num-seqs` 與 `--gpu-memory-utilization` 調的就是這件事）。於是一連串工程決策全被這把尺重新定義：**為什麼長上下文服務這麼貴？** 因為 ctx_len 直接線性放大每條 KV，128K 上下文的單請求 KV 可能就是 8K 的十六倍，能塞的併發數同比例暴跌。**為什麼 GQA/MQA/MLA 是顯學？** 因為它們砍 H，等於直接放大你的併發上限（情境 2）。**為什麼 PagedAttention 是革命？** 因為它消滅了「為最大長度預留」的浪費，把實際能塞的 KV 條數逼近物理極限（情境 5）。**把這四件事連起來你會發現，現代 LLM 推理引擎的全部努力，都是在同一個約束下做文章：顯存裝得下多少 KV，你就服務得了多少人。** 算力，反而是 Decode 階段最不缺的東西。

## 情境 5　工程演進史：從 Naive 預分配到 PagedAttention 的虛擬記憶體革命

**背景**：知道了 KV Cache 會吃爆顯存，接下來的問題是怎麼**把這有限的顯存用到極致**。這條路開源社群走了三代，從一個浪費到令人髮指的 Naive 方案，演進到一個直接抄作業系統的天才設計——而抄的那份作業，是你在計算機組成原理課上學過、卻沒想到會用在這裡的**虛擬記憶體分頁**。

**第一代，Naive 方案（早期 PyTorch 樸素實作）。** 做法簡單粗暴：為每個請求，**預先開闢一塊連續的、大小等於 `Max_Seq_Len` 的顯存**。模型支援 4K 上下文，那就一律先霸佔 4K 的空間，不管你實際會生成多少。這帶來兩種致命碎片：

- **內部碎片（Internal Fragmentation）**：用戶其實只輸出了 100 個 token，但你為他預留的 4K 空間裡，剩下 3900 個 token 的位置被**強行佔住、別人用不了**，純浪費。
- **外部碎片（External Fragmentation）**：顯存總量明明還夠，但因為要求「連續」的大塊，零碎的空閒湊不成一塊連續區，**明明有空間卻無法初始化新請求**。

實測下來，Naive 方案的顯存有效利用率常常只有兩三成——七成的顯存在替「萬一用戶寫很長」的假設空轉。

**第二代，vLLM 的 PagedAttention（虛擬記憶體革命）。** vLLM 團隊的洞察堪稱經典：**KV Cache 的碎片問題，和作業系統四十年前面對的記憶體碎片問題，是同一個問題。** 而 OS 早就解決了——用**分頁（Paging）**。於是 PagedAttention 把 KV Cache **離散地存進一堆固定大小的物理塊（Physical Block）**裡，這些塊在顯存 HBM 裡可以**散落各處、不要求連續**：

```
  邏輯視角 (每個 sequence 以為自己的 KV 是連續的):
    Logical Blocks:  [ Block 0 ] -> [ Block 1 ] -> [ Block 2 ]
                         │              │              │
  Block Table (映射表):  │              │              │
                         ▼              ▼              ▼
    Physical Blocks: [ Frame 42 ]   [ Frame 7 ]   [ Frame 99 ]
                     (GPU HBM 裡物理上散落各處，但邏輯上連續)
```

每個物理塊固定裝 16 或 32 個 token。一個 `BlockManager` 管著全部空閒塊的池子，一個 `BlockTable` 維護「邏輯塊 → 物理塊」的映射——這完全就是 OS 的頁表（page table）。新請求來了就從池子抓空閒塊，生成到哪、長到哪、就抓到哪，**用多少抓多少，內部碎片幾乎歸零**；塊大小固定且可任意散布，**外部碎片也消失**。顯存利用率被逼到近乎 100%。

```python
# vLLM PagedAttention 核心架構（源碼邏輯偽代碼）
class PhysicalTokenBlock:
    def __init__(self, block_number, block_size):
        self.block_number = block_number  # GPU 顯存塊的實際物理索引
        self.block_size = block_size      # 通常 16 或 32 個 token
        self.ref_count = 0                # 引用計數 → 支援 Block 級 Copy-on-Write

class BlockAllocator:
    def __init__(self, num_gpu_blocks, block_size):
        # 啟動時把全部可用顯存切成固定大小的塊，做成空閒池
        self.free_blocks = [PhysicalTokenBlock(i, block_size)
                            for i in range(num_gpu_blocks)]

class BlockTable:
    def __init__(self):
        self.mapping = {}  # 邏輯塊索引 → 物理塊 (這就是「頁表」)
    def append_token(self, logical_block_idx, block):
        self.mapping[logical_block_idx] = block
```

但分頁帶來一個新麻煩：KV 在顯存裡不再連續了，標準的 `torch.cat` 之類連續記憶體操作就用不了。vLLM 的解法是**自己手寫 CUDA / Triton kernel**，在算注意力時**透過 `block_table` 動態尋址**，去那些散落的物理塊裡逐塊撈 K、V：

```
  // vLLM CUDA Kernel 尋址核心偽邏輯
  __global__ void paged_attention_kernel(
      float* out, const float* q, const float* k_cache, const float* v_cache,
      const int* block_table, const int* context_lens, int block_size) {
      int seq_id = blockIdx.y;
      int token_idx = threadIdx.x;
      // ① 用邏輯 token 索引查 block_table，找出物理塊編號
      int logical_block_idx = token_idx / block_size;
      int physical_block_num =
          block_table[seq_id * max_blocks_per_seq + logical_block_idx];
      // ② 算出該 token 在顯存裡的絕對偏移地址
      int block_offset = token_idx % block_size;
      float* k_ptr = k_cache
          + physical_block_num * block_size * num_heads * head_dim + ...;
      // ③ 在散落的物理塊上執行常規 Attention 計算
  }
```

那個 `ref_count`（引用計數）還順手解鎖了一個漂亮特性：**Copy-on-Write 的塊共享**。並行採樣（一個 prompt 同時生成多條候選）或 beam search 時，多條序列共享同一段前綴的物理塊，只在真正要寫入分歧時才複製——這又是 OS 的 `fork()` 寫時複製，搬到了 KV Cache 上。

> 🔍 進階點評 — 為什麼「抄 OS」是 LLM 推理引擎最深的護城河
> PagedAttention 表面是一個顯存管理技巧，但它真正示範的，是**系統工程裡最高級的一種能力：在一個全新領域裡，認出一個被徹底解決過的老問題，然後把成熟解法整套搬過來。** vLLM 之所以能在 2023 年一炮而紅、成為事實標準，不是因為它發明了新數學，而是因為它的作者看著 KV Cache 的碎片，腦中浮現的是「這不就是 1960 年代的虛擬記憶體嗎」——然後把頁表、分頁、寫時複製、甚至「換頁到 CPU 記憶體」（顯存不夠時把冷的 KV 塊 swap 到主記憶體，正是 OS 的 swap）整套照搬。**今天圍繞 KV Cache 的整個生態，幾乎都是 OS 概念的 LLM 化身：** PagedAttention 是分頁，prefix caching 是去重，RadixTree（下一節）是前綴樹索引，KV 量化是壓縮，KV offloading 是 swap，連 NVIDIA 的 **FlashInfer**、Hugging Face 的 **TGI**、SGLang 的塊管理，骨子裡都在玩同一套記憶體層級的把戲。**這給蒸餾者一個極實際的啟示：當你想優化你那台 DGX Spark 的本地推理，別急著去 GitHub 找新框架——先問自己「作業系統當年是怎麼解這個問題的」。** 計算機科學六十年的記憶體管理智慧，全是你現成的軍火庫；只會等別人發明新 kernel 的人，永遠慢一個身位。

## 情境 6　服務端的 RadixTree 與算子級的 FlashAttention：把記憶體還給你的 DGX Spark

**背景**：PagedAttention 解決了「單機顯存怎麼用滿」，但還有兩個層次的優化沒講——**服務端如何讓不同用戶的請求互相復用前綴**（這對應情境 1 的 Prompt Cache），以及**最底層的算子如何不把記憶體頻寬浪費掉**。這兩者一個在最上層、一個在最底層，卻共同回答了本章開頭的問題：怎麼把那一百多 GB 還給你。

**服務端：SGLang 的 RadixTree（基數樹）前綴快取。** 情境 1 講過服務層 Prompt Cache 要復用相同前綴，但問題是——成千上萬個用戶，前綴各不相同又部分重疊，怎麼高效地「千人千面地共享公共前綴」？SGLang 框架的答案是 **Radix Tree（基數樹）**。它把每個用戶輸入的 prompt 轉成 token ID 陣列，樹的**每個節點存一段 token 序列 + 那段對應的 KV Cache 物理指標**。公共前綴自動匯聚在靠根的枝幹上，被所有命中它的請求復用：

```
  三個請求共享同一句 system prompt：
    請求1: "You are a helpful assistant. Please tell me a joke."
    請求2: "You are a helpful assistant. Please write Python code."
    請求3: "You are a helpful assistant. Translate this to Chinese."

                        [ Root ]
                           │
        ( tokens of "You are a helpful assistant. " )  ← 命中公共前綴！KV 只算一次
                           │
            ┌──────────────┼──────────────┐
            │              │              │
      ["Please tell   ["Please write  ["Translate
       me a joke."]    Python code."]   to Chinese."]   ← 各自分叉的尾巴
```

那句反覆出現的 system prompt，它的 KV 狀態**全樹只算一次、所有請求共享**——這正是情境 1「穩定前綴放最前面」黃金法則在服務端的物理兌現。當顯存吃緊要騰空間，RadixTree 觸發 **LRU（最近最少使用）驅逐**，但它驅逐得很聰明：**先斬葉子節點**（已結束的長對話尾部、最不可能再被命中的分支），**保留根節點和公共枝幹**（system prompt 這種被命中機率最高的部分）。先斬葉、後保根——這個策略本身就是對「前綴越靠根、復用價值越高」的精準利用。

**算子級：FlashAttention 與 FlashDecoding。** 光有分頁和前綴樹還不夠，最底層的注意力算子如果寫得蠢，照樣浪費記憶體頻寬。回到情境 2 的核心矛盾——Decode 是訪存受限的，瓶頸在 HBM 搬運。

- **FlashAttention** 攻擊的是「中間矩陣寫回 HBM」這個浪費。樸素實作會把那個 N×N 的注意力矩陣**完整算出來、寫回慢速的 HBM、再讀回來**做 softmax——光是這一來一回就燒掉巨量頻寬。FlashAttention 用 **Tiling（分塊）** 把計算切成能塞進晶片上**高速 SRAM** 的小塊，並用 **online softmax**（邊掃邊增量更新 softmax 的歸一化分子分母）的數學技巧，**全程不把完整的 N×N 矩陣寫回 HBM**。結果是顯存讀寫量降低一個數量級——它沒讓 GPU 算得更快，是讓 GPU **少搬了一個數量級的數據**，這恰好正中訪存受限的要害。

- **FlashDecoding** 專治情境 2 的 Decode 階段、且專治**超長上下文**。Decode 時 batch 與頭數往往不足以餵滿 GPU 的全部計算單元，若歷史 KV 又極長（比如 128K 上下文），那一條長 KV 沿序列維度的串行掃描會讓大量 SM（流多處理器）閒置。FlashDecoding 的關鍵是**沿 KV 的序列維度（而非 batch/頭維度）再切一刀**：把**那條超長的 KV 切成多段，並行分配到 GPU 上多個計算單元（SM）**各算各的局部注意力與局部 softmax 統計量，最後做一次 **Reduction（規約）**、用各段的 softmax 分母重新加權合併成正確結果。它把「一條長尾巴的串行訪存」變成「多段並行訪存」，正是在「batch 小、上下文長」這個傳統並行維度餵不滿 GPU 的死角，把利用率重新拉起來。

**把這一切接回你的 DGX Spark。** 情境 1 你那台機器爆記憶體，現在你手上有了一整套還記憶體的工具：用 **vLLM / SGLang** 啟動，靠 **PagedAttention** 消滅碎片把顯存利用率逼到 100%（情境 5）；靠 **RadixTree prefix caching** 讓多個會話共享那段固定的 system prompt，省掉重複 Prefill；靠 **FlashAttention / FlashDecoding** 把訪存受限的 Decode 階段的記憶體頻寬榨乾；再配合第 5 章後處理的 **KV Cache 量化（FP8）** 把公式裡的精度字節從 2 砍到 1。這四層疊起來，原本一爆就 OOM 的 128GB，能撐住的併發與上下文可以翻好幾倍——**這就是「把那一百多 GB 還回來」的具體工程路徑**。

## 本章小結

- **兩大快取分類**：① 推理層 **KV Cache**（住顯存、由 vLLM/TensorRT-LLM 管、把自回歸解碼從 O(N²) 降到 O(N)、一次請求生命週期）；② 服務層 **Prompt / Context Cache**（住服務端、跨請求復用相同前綴的 KV 狀態、Claude 用 `cache_control` 顯式錨定、命中後輸入 token 大幅折扣）。黃金法則：**穩定靜態內容放最前、動態高變內容放最後**，因為兩者本質都是前綴匹配，一個流浪的時間戳或多餘的 `\n` 就能讓整條快取作廢。
- **訪存受限是物理底層**：Prefill 是 Compute-Bound（張量打包、吃滿 Tensor Core），Decode 是 Memory-Bound（每步只 1 token、GPU 大部分時間在 HBM→SRAM 搬運）。**KV Cache 不是消滅瓶頸，是把瓶頸從算力搬到記憶體**——這是全章的物理基礎，也是 GQA/MQA/MLA 拼命砍 KV 頭數的根本動機。
- **數學推導**：無快取每步重算歷史 K/V → Σ O(t) = **O(T²)**；有快取只算新 token 並 `concat` 進舊矩陣 → 單步 O(1)、總 **O(T)**。降維的魔法只用到一個樸素的拼接，背後是「歷史不會改變，所以不必重算」的 memoization 母題。
- **顯存計算器**：`Size_per_token = 2(K+V) × 2(FP16字節) × L × H(KV頭) × D`。Llama 3-8B（L=32, KV Heads=8, D=128）= **128 KB/token**；batch=128 × 8K ctx ≈ **128 GB**，而權重才 16GB。**瓶頸是 KV Cache 不是權重**，最大併發由顯存能裝多少條 KV 尾巴決定，不由算力決定。
- **工程演進**：Naive 連續預分配 → 內/外碎片把利用率打到兩三成；vLLM **PagedAttention** 抄 OS 虛擬記憶體分頁，KV 離散存進固定大小物理塊、`BlockManager`/`BlockTable` 做邏輯→物理映射、自訂 CUDA/Triton kernel 經 `block_table` 動態尋址、`ref_count` 解鎖寫時複製塊共享，利用率逼近 100%。整個 KV 生態都是 OS 概念的 LLM 化身（分頁/去重/前綴樹/壓縮/swap）。
- **服務端 + 算子級**：SGLang **RadixTree** 每節點存一段 token + KV 指標、公共前綴（system prompt）命中復用、LRU 驅逐先斬葉子保根枝；**FlashAttention** 用 SRAM tiling + online softmax 不把 N×N 矩陣寫回 HBM；**FlashDecoding** 把超長 KV 切段並發於多 stream 再 reduction。四層疊加（PagedAttention + prefix caching + Flash 算子 + KV 量化）就是把那消失的一百多 GB 還給 DGX Spark 的具體路徑。

KV Cache 算清楚了，分頁也分了，前綴樹也建了——可這一切都還在「一台機器」的尺度上。當請求量從幾條漲到每秒幾十萬條、當快取要跨越成百上千張 GPU 共享、當你得在「命中率」和「一致性」之間做分散式的取捨，單機的這套physics 就不夠用了。下一章，我們把快取從一台機器放大到一整個資料中心，看超大規模下的快取架構，以及它如何收束進那座為攻防而生的、二十個方向的軍火庫。
