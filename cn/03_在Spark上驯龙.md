# 第三篇　驯龙场

# 第3章　在 DGX Spark 上驯龙：硬件 Roofline 与选型

> 硅谷，丽台代理商的出货单上，那台 **NVIDIA DGX Spark** 标价不菲。它小得像台 Mac mini，却号称「个人 AI 超级电脑」。风扇转起来的那一刻，真正的问题不是「它能跑多大的模型」，而是更尖锐的一句——**「我为这 128GB 付了高昂溢价，到底该拿它跑什么，才不算暴殄天物？」**

> 本章技术含金量最高、也最站得住脚。GB10、统一内存（UMA）、GGUF、Roofline 都是真实工程。选型结论是一种常见推论，但背后的**带宽推理是可验证的硬件常识**——正是戳破上一章「神话宣传」的尺。

## 情境 1　认识这头怪兽：用 Roofline 看穿它的脾气

**背景**：硬件是 **NVIDIA DGX Spark（GB10 芯片，非数据中心级 GB200/GB100）**。先把规格钉死，因为**选型的每一步都是从这几个数字推出来的**：

| 规格 | 数值 | 确信度 |
|---|---|---|
| 芯片 | GB10 Grace-Blackwell 超级芯片（20 核 Arm CPU + Blackwell iGPU）| 🟢 官方 |
| 内存 | **128GB LPDDR5X 统一内存（UMA）**，CPU 与 iGPU 实体共享同一池、不经 PCIe | 🟢 官方 |
| 内存带宽 | **~273 GB/s**（LPDDR5X class）| ⚠️ 常见标称值；由总线宽度推算的理论峰值另有 ~301 GB/s 之说，**勿当精确定论** |
| 算力 | ~1 PFLOP **FP4（NVFP4）** 稀疏峰值（5 代 Tensor Core）| ⚠️ 官方宣传的「稀疏 FP4」数字，密集/实测会低不少 |
| 量化原生 | Tensor Core 原生支持 **FP4 / NVFP4 / FP8 / BF16** | 🟢 架构特性 |

**注意那个被很多二手文章写成「600 GB/s」甚至更高的带宽——那是错的。** DGX Spark 用的是 LPDDR5X，不是数据中心卡的 HBM3e；它的带宽量级在「两、三百 GB/s」，而非 HBM 的「TB/s」。**这个数字错一倍，下面所有 tok/s 估算就全错一倍**，所以先校准到 ~273 GB/s class。至于 NVFP4——它是 Blackwell 引入的 4-bit 浮点格式（E2M1 + 两级 micro-block 缩放），靠分块 scale 把 4-bit 的品质硬拉到接近 FP8，这正是后面选型敢用 Q4 的底气（细节留到第 4 章蒸馏数学）。

**问题**：不懂它的脾气，就会被「4GB 显存跑 Claude」这类说法牵着走，白白浪费它最值钱的本事。而看穿它脾气的工具，是一个工程师必懂的模型——**Roofline（屋顶线）**。

**LLM 推理的两个阶段，撞两道不同的墙**：

- **预填充（Prefill，处理你的输入）**：把整段 prompt 的几百上千个 token 一次性灌进去做大矩阵乘——同一份权重被这一整批 token 重复利用，**算力受限（compute-bound）**，吃的是 FLOPS。
- **解码（Decode，逐 token 生成）**：自回归，一次只生一个 token；每生一个，就得**把参与运算的权重从内存完整读一遍**，而这份权重只服务了「这一个」token——读进来、算一下、丢掉。**内存带宽受限（memory-bandwidth-bound）**，吃的是 GB/s。

**为什么 prefill 撞算力墙、decode 撞带宽墙？** 一句话：**算术强度（Arithmetic Intensity，每读 1 byte 权重能换来几次浮点运算）天差地别。** Prefill 一次处理 N 个 token，权重读一次被 N 个 token 共用，算术强度随 N 线性放大，轻松冲进「算力受限」区；Decode 的 batch=1，权重读一次只算一个 token 的乘加——算术强度趋近理论最低（每个参数约 2 FLOP），于是 GPU 的兆级算力大半闲置，瓶颈死死卡在「把权重从内存搬出来」这一步。**这就是为什么你在 DGX Spark 上跑单人对话，GPU 使用率常常只有两三成却已经是极速——它不是不够力，是在等内存。**

**这带出一条能直接估算速度的铁律**：

```
解码速度 (tok/s) ≈  内存带宽 (GB/s)  ÷  每 token 需读取的权重 (GB)
```

拿 DGX Spark 的 ~273 GB/s 套（这是带宽「天花板」；扣掉注意力 KV 读取与框架开销，实测通常再打 6–8 折）：

- **70B 密集模型 @ Q4_K_M**（每 token 把**全部** ~40GB 权重读一遍）：273 ÷ 40 ≈ **理论 ~6.8 tok/s，实测 ~5 tok/s**——一个字一个字往外挤，不堪用。
- **120B-A4 MoE @ Q4**（每 token 只读「激活的 ~4 个专家 ~2–3GB ＋ 每层常驻的注意力/共享权重 ~3–4GB」≈ 6GB）：273 ÷ 6 ≈ **理论 ~45 tok/s，实测 35–50 tok/s**——顺。

看出名堂没有？**同一台机器，跑 70B 密集卡成 ~5 tok/s，跑「总参更大」的 120B MoE 反而快近 9 倍**——差别不在总参数，在「每 token 真正要搬多少 byte」。**这一条算式，就能让你在按下载键之前先算出它跑多快**，不必等 70GB 下载完才后悔。

> 🔍 高端点评 — Roofline 的「屋脊点」：一个数字看穿机器的性格
> Roofline 模型的全部精华，可以浓缩成一个座标——**屋脊点（Ridge Point）= 峰值算力 ÷ 峰值带宽**，单位「FLOP/byte」。它是机器的「性格分水岭」：负载的算术强度若**高于**它，你撞的是**算力墙**（屋顶的水平段）；**低于**它，你撞的是**带宽墙**（屋顶的斜坡段）。把 DGX Spark 代进去：~1 PFLOP(FP4) ÷ ~273 GB/s ≈ **~3600 FLOP/byte**。这意味着：**任何算术强度低于 3600 的负载，都会被带宽卡死。** 而 decode 的算术强度是多少？batch=1 时约 **2–4 FLOP/byte**——比屋脊点低了**整整三个数量级**。换句话说，**LLM 逐 token 生成这件事，在这台机器（以及所有 UMA / Apple Silicon）上，是被带宽墙按在地上摩擦的极端内存受限负载**，那 1 PFLOP 算力对它几乎是摆设。理解这一点，三件事的根你就全懂了：①**为什么提速要靠「减少每 token 读取量」**（量化、MoE、KV Cache 量化）而不是「买更强算力」；②**为什么提高 batch（同时服务多个请求）能近乎线性提升总吞吐**——多个 token 共用同一次权重读取，把算术强度往屋脊点推（这也是 vLLM 连续批量的价值所在）；③**为什么投机解码（情境 3）有效**——它让大模型「一次读权重、平行验证多个草稿 token」，本质也是在拉高算术强度。**屋脊点是一把尺，量出你的瓶颈到底在算力还是带宽；而对个人 LLM 推理，答案几乎永远是带宽。**

> 🧠 必须焊进脑子的硬件铁律
> **容量决定你「能不能跑」，带宽决定你「跑多快」，而 MoE 的「激活参数」才是带宽账单上真正的数字。** DGX Spark 是「容量巨人、带宽凡人」：128GB 让它能吞下别人吞不下的大模型，但 ~273 GB/s 的 LPDDR5X 比独显动辄 1TB/s（GDDR7）甚至 ~8TB/s（HBM3e）的带宽慢了一个数量级。所以它的最优策略永远是「**用大容量去跑别人跑不动的大 MoE**」，而不是「用它去比小模型谁更快」。

**这条铁律如何戳破上一章的神话**：在 DGX Spark 上跑 9B 模型，你**当然**能飙到 180+ tok/s——但那是浪费。9B 一台两万多的 M 芯片 Mac 就跑得很好；你买 128GB，是为了容量，不是为了让小模型更快。

> 💡 君之一席话
> 与君一席话：**选硬件与选模型，本质是同一道题——在你的瓶颈资源上求最大产出。** DGX Spark 的瓶颈不是容量（它过剩），是带宽（它稀缺）。把稀缺资源喂给「激活参数小、总参数大」的 MoE，等于用过剩的容量去补带宽的不足——这就是 Roofline 思维的精髓：**不要被规格表上最大的数字迷惑，要找出系统里最小的那道墙，然后让你的负载刚好贴着那道墙跑。** 看懂这个，你选的就不只是模型，是整个系统的最优工作点。

## 情境 2　六款影子的横向对比：谁配得上这头怪兽

**背景**：针对 DGX Spark GB10，把上一章的模型做横向对比如下（Q4_K_M 量化下评估）。

| 模型 | 内存占用 | DGX Spark 预期速度 | 强项 | 弱项 |
|---|---|---|---|---|
| **GPT-OSS 120B Fable-5**（MoE）| ~72GB | 35–50 tok/s | 128 专家 MoE，复杂多步代码与数学 | 体积大，上下文 >32K 明显变慢 |
| **Qwable-v1**（35B）| ~24GB | ~102 tok/s | 完美继承 Fable 5 Agent 轨迹，最适合 Coding 代理 | 知识广度逊于 120B |
| **clzoro/Qwen3.5-27B** | ~19GB | 110+ tok/s | 喂 Opus 对话，文学/角色扮演极强 | 工具调用不如 Qwable-v1 |
| **Qwen3.6-14B-FableVibes** | ~11GB | 140+ tok/s | 剪枝 MoE，保留「Claude 思考感」 | 复杂逻辑偶有反刍（幻觉）|
| **Qwythos-9B**（含 abliterated）| ~6.5GB | 180+ tok/s | 极速、彻底无审查，烂显卡也能跑 | 参数太小，深度推理完整度差 |
| **Dzluck/Qwen3.5-2B** | ~2.1GB | 250+ tok/s | 极致轻量，为边缘设备设计 | 大材小用，智商降级明显 |

> ⚠️ 真伪提醒
> 速度与内存数字**均为未经实测的估算/坊间流传值**。相对关系符合 Roofline 直觉（参数越小越快、MoE 比同尺寸密集快、量化越激进占用越小），可当「合理估算」而非「实测基准」。社群有一条真实流传的参考点：「Solved the DGX Spark, 102 stable tok/s on Qwen3.5-35B-A3B」（见附录 A）——与表中 Qwable-v1（35B-A3B 同级）的 102 tok/s 吻合，因为 3B 激活 @ Q4 ≈ 1.7GB，273 ÷ 1.7 ≈ 160 理论上限，实测 102 落在合理折扣内。**但同一把尺照出表中的破绽**：Qwythos-9B 标 180+ tok/s——9B 密集 @ Q4 ≈ 5.4GB，273 ÷ 5.4 ≈ **50 tok/s 才是物理上限**，180 已超出 3 倍多，**单请求 decode 不可能达到，除非是高并发批量的总吞吐或表本身灌水**。这恰恰示范了本章的内核纪律：**任何 tok/s 数字，先拿「带宽 ÷ 每 token 读取量」这把尺量一遍，过不了物理上限的，一律存疑。**

**别只会看表——学会自己算「装不装得下」**：表上的「内存占用」只是权重，真正塞进 128GB 的是三样东西叠加，少算一样就会在长文时 OOM。先把公式背下来：

```
总占用 = 权重 + KV Cache + 框架/激活开销(~2–4GB)

权重 (GB) ≈ 参数量(B) × 每参数字元 ÷ 8
   BF16/FP16 : 16 bit → ×2.00      FP8 / Q8_0 : 8 bit → ×1.00 (Q8_0 实 ~8.5 bit)
   Q4_K_M    : ~4.8 bit → ×0.60     NVFP4/Q4   : 4 bit → ×0.50

KV Cache (GB) ≈ 2 × 层数 × KV头数 × 头维度 × 上下文长度 × 每元素字节 ÷ 1e9
   ★ ×2 = K 和 V 各一份
   ★ GQA 让「KV头数」远小于注意力头数（如 8 vs 64），是省 KV 的命脉
   ★ 经验值：GQA-8 + FP16 KV，每 1K token 约吃 0.3–0.5GB（依层数）
```

把四款代表性配置代进 128GB 实算（⚠️ 数字为示意，依实际层数/GQA 比例浮动）：

| 模型 @ 量化 | 权重 | KV @ 32K | KV @ 256K | 128GB 结论 |
|---|---|---|---|---|
| **70B dense Q4_K_M**（GQA）| ~40GB | ~10GB | ~80GB | 32K 从容(~50GB)；**256K=120GB 几乎撑爆，非 KV 量化不可** |
| **120B-A4 MoE Q4** | ~72GB | ~5GB | ~40GB | 32K 轻松；256K≈112GB 吃紧，KV FP8 可救 |
| **35B-A3B Q8_0**（GQA）| ~37GB | ~6GB | ~48GB | 单模从容；要与大模型双开需控长度 |
| **8B dense Q4** | ~5GB | ~2GB | ~16GB | 随便跑，但用 128GB 跑它是大材小用 |

盯着那两栏 KV 看——**权重是一次性的固定开销，KV Cache 却随上下文长度「线性膨胀」。** 在 32K 你几乎感觉不到它，但拉到 256K，KV 能逼近、甚至吃掉比权重更多的内存。**这就是为什么「装得下 70B」和「装得下 70B 还能跑 256K 长文」是两个完全不同的问题**——后者真正的内存杀手从来不是模型权重，是 KV Cache。（这条伏笔，第 10、11 章会用整章来还它的债。）

**一个常见的选型结论**：**首选 GPT-OSS 120B Fable-5 或 Qwable-v1（35B）**——「128GB 不跑大模型太浪费」。用 Roofline 与上面的 fit 算式验算，这结论成立：

- **GPT-OSS 120B-A4 是「智商担当」**：120B 总参给它知识深度，但每 token 只激活 ~4 专家（约 20–30B 实际参与），让它规避了带宽死穴，35–50 tok/s 可接受。**这是个人设备上突破「70B 级智商上限」的唯一务实途径。**
- **Qwable-v1（35B）是「速度担当」**：24GB 占用对 128GB 毫无压力，102 tok/s 近乎「瞬喷」，且原生吐 `str_replace_editor` XML，对接 OpenHands/Aider 零转译——当 Coding Agent 效率比 120B 高数倍。

> 🔍 高端点评──MoE 为何是带宽受限机器的救星，以及它的隐藏代价
> 这是全章最值钱的洞见：**在任何带宽受限的硬件上（DGX Spark、Apple Silicon、所有 UMA 架构），「激活参数量」比「总参数量」重要得多。** 一个 120B-A4 跑起来的速度接近 20–30B 密集模型，智商却接近 120B——这就是 2026 年面向「个人超算」的开放权重模型几乎全押 MoE 的原因。但 MoE 不是免费的午餐，它有三个隐藏代价，选型时要一并权衡。**其一，内存税**：MoE 要把**全部 128 个专家**都常驻内存（你不知道下个 token 会路由到哪个），所以它「省的是带宽、费的是容量」——这恰好和 DGX Spark「容量过剩、带宽稀缺」的特性互补，是天作之合，但在小显存设备上反而是灾难。**其二，逻辑反刍**：专家间路由切换可能让长链推理出现不连贯（表中 FableVibes 的「复杂逻辑偶有反刍」正是此因），这也是为什么 clzoro 的**密集 27B** 在「思考连贯性」上反而更受文学/角色扮演场景青睐。**其三，批量效率**：MoE 在高并发批量下的专家负载不均衡，是生产部署的老问题。**所以选型的完整心法是：问「激活多少」（定速度）、问「总共多少」（定智商）、问「密集还是 MoE」（定连贯性与内存形态）——三问之后，才谈得上选对。**

## 情境 3　最聪明的玩法：双模并行，与一个高端彩蛋

**背景**：有 128GB，更聪明的玩法不是「挑一个」，而是**双模并行**——同时加载一大一小，按任务切换。

**一个值得参考的双模方案**：

- **同时加载**：GPT-OSS 120B（~72GB）+ Qwable-v1 35B（~24GB）= ~96GB，剩 ~32GB 给两边 **KV Cache**。
- **后端**：**Ollama** 内置动态多模型管理；开两个终端或 API 并行：`ollama run gpt-oss:120b` / `ollama run qwable:35b`。
- **前端**：**Open WebUI / Cursor**，下拉一键切换，甚至「**双模对决（Arena）**」——两模型同答一题，你挑最好的。
- **热切换（hot-swap）**：跑大模型时自动把闲置模型从 LPDDR5X 释放，腾空间给超长上下文；靠 ~273 GB/s 的 UMA 带宽与「零拷贝、不过 PCIe」的特性，换入换出比独显经 PCIe 搬运顺手得多（但别误会它有 HBM 那种 TB/s 速度）。

**黄金工作流**：

- **Qwable-v1（35B）管「日常高频」**：边写边 Debug、改码、写短信、翻译、查指令——靠 102 tok/s 的喷射速度，思绪不停等 AI。
- **GPT-OSS 120B 管「重度思考」**：35B 解不掉的 Bug、规划整套架构、丢三篇长论文做跨域分析——切过去，让 128 专家攻坚。

> 💡 君之一席话
> 与君一席话：**真正会用工具的人，不追求一把瑞士刀解决所有事，而是让对的刀在对的时刻无感地出现在手边——这套「分层认知」其实是人脑的镜像。** 心理学里的 System 1（快、直觉）与 System 2（慢、深思），对应的正是「快模型管九成日常、大模型攻一成硬骨头」。双模并行的精髓，不在「有两个模型」，而在「切换本身变得无感」——当调用深度思考的成本趋近于零，你才会真的在该深思时深思，而不是因为「切换很麻烦」就凑合用快模型硬扛。**工具的最高境界，是让正确的选择变成最省力的选择。**

> 🔍 高端点评──双模的高端彩蛋：投机解码，让两个模型合体加速
> 前面只讲了「按任务切换」，但既然你已经把一大一小两个模型都载进了 128GB，有一个它没提、却能让两者**协同加速**的高端技术值得知道——**投机解码（Speculative Decoding）**。原理是：让**小模型（Qwable-v1 或更小的 draft model）先快速生成一串草稿 token**，再让**大模型（GPT-OSS 120B）一次性平行验证这串草稿**，接受对的、否决错的。因为「验证 K 个 token」比「逐个生成 K 个 token」在内存带宽上划算得多（一次读权重验证多个），实测常能让大模型的有效输出速度提升 1.5–3 倍，**且输出分布与大模型完全一致（无损加速）**。这恰恰把 DGX Spark「同时有大小两模型」的条件用到了极致——小模型不只是「快速回答日常」的备胎，更是大模型的「预言加速器」。工具上，**llama.cpp（`--draft` / draft model）、vLLM、SGLang** 都已内置支持。**另一个隐藏成本提醒**：别忘了 KV Cache 才是长上下文真正的内存杀手——当上下文堆到 20 万 token，那剩下的 ~32GB 会被**随长度线性飙升**的 KV Cache 无情吃光（回看情境 2 的 fit 算式：70B 级模型在 256K 的 KV 可达 ~80GB），这时就要靠 **FP8/INT8 的 KV Cache 量化**（vLLM、llama.cpp 已支持）把它腰斩，而不是天真地以为「卸载一个模型」就够了。**长上下文的瓶颈从来不是权重，是 KV Cache。**

## 情境 4　部署实务：把模型钉进那 128GB

**背景**：选好了模型，最后一哩是「怎么在 DGX Spark 上把它跑出极致速度」。以下是几个具体建议，补上工具细节如下。

**部署工具链**：

- **格式与引擎**：下载 GGUF（首选 **Q4_K_M**，内核层可上 Q8_0 做混合量化）；引擎用 **llama.cpp / Ollama**（最易上手）、追求吞吐用 **vLLM / SGLang**（连续批量把算术强度往屋脊点推、KV PagedAttention）；DGX OS 内置 **NVIDIA AI 软件堆栈 / NIM / TensorRT-LLM** 可直接拉起。
- **跨平台对照（别搬错生态）**：DGX Spark 走 **CUDA** 生态（llama.cpp-CUDA / Ollama / vLLM / SGLang / TensorRT-LLM）；同属 UMA 阵营的 **Apple Silicon** 则走 **MLX / llama.cpp-Metal**。两边**原理共通**——都在赌「统一内存 + 端侧量化模型」这条路——但 kernel 与 binary 不互通，**Mac 上跑得好的 MLX 模型不能直接搬上 GB10**，反之亦然。认清这点，才不会抄错部署脚本。
- **量化校准**：做 GGUF 量化时，用 **imatrix（importance matrix）**——拿一小批校准语料统计各权重重要度，让低比特量化优先保住关键权重，**同样 Q4 体积、明显更高品质**。这是 llama.cpp 社群的标配高端手法。
- **内核级锁定（榨干 UMA 带宽的关键）**：
  - `numactl --interleave=all`：把模型权重**交错**配置到 LPDDR5X 所有内存信道，逼所有 channel 同时全载运作，避免 CPU/iGPU 抢带宽时的缓存未命中。
  - `mlock`：把权重**锁在物理内存**，防止操作系统把它换页（swap）到慢速硬盘——这是维持高 tok/s 的硬核细节。
  - **不要依赖通用 Docker 缺省容器**：它的内存策略未必贴合 UMA，手动调 NUMA 与 mlock 才能发挥 GB10。

**戳破「104 万上下文」神话的最后一刀**——别只看厂商规格，**自己验**：

```
装得下多长？   →  看 RoPE/YaRN 设置（最易宣传的数字）
跑得动多长？   →  受 KV Cache 与带宽限制的实际可用长度
读得懂多长？   →  跑 RULER / Needle-in-a-Haystack，真假立判
```

**这三问里，第三问「读得懂多长」最常被跳过、也最致命**——它正是 **NIAH** 与 **RULER** 要量的东西：

- **NIAH（Needle-in-a-Haystack，海底捞针）**：在一篇几十万字的无关长文（haystack）里，藏一句关键事实（needle，例如「2026 年的通关密语是 nol9981」），把上下文塞到指定长度后问模型那句话——测的是**最低门槛的纯检索**。连这都过不了，「百万上下文」就是纯广告。
- **RULER**：NIAH 的工业强化版，除了单针检索，再叠上**多针检索、变量多跳追踪（variable tracking）、聚合统计、长文 QA** 等十几类任务，逐段加长上下文，记录模型准确率**跌破门槛（常设 85%）的那个长度**——这个长度才叫「**有效上下文长度（effective context length）**」。残酷现实是：**标称 1M 的模型，RULER 实测有效长度常常只剩 64K–128K**，后面那一大段是「装得下、却读不懂」的虚胖。所以你该信的不是规格表上的 RoPE base，是 RULER 曲在线那个准确率崩塌的拐点——而这条曲线，`lm-eval-harness` 配上开源 RULER 套件，你自己在 DGX Spark 上一晚就能跑出来。

> 💡 君之一席话
> 与君一席话：**规格表上的数字，是厂商想让你看见的真相；跑一遍基准测试，才是机器愿意对你说的真相。** 「1M 上下文」「180 tok/s」「无审查」这些宣传词，每一个都只揭露了三维问题里最好看的那一维。成熟工程师的本能，是把任何单一数字拆回它背后的「能装下／跑得动／读得懂」「总参／激活／架构」这些独立维度，再用现成的开源基准（RULER、lm-eval-harness、HumanEval）亲手量一遍。**对数字的敬畏不是不信，是「信任，但验证」——而验证的工具，全是免费开源的；不验，是你自己的选择。**

> 🔍 高端点评──UMA 是一场内存架构的豪赌，而 DGX Spark 押对了什么
> DGX Spark 的 **统一内存架构（UMA）** 不是 NVIDIA 的小聪明，是一场关于「未来个人 AI 算力长什么样」的架构豪赌，值得从更高处看懂它赌的是什么。传统 GPU 把「快但小」的 HBM/GDDR 焊在卡上，CPU 内存「大但慢」，两者间隔着 PCIe 这条窄桥——跑大模型时，权重在两块内存间搬来搬去，PCIe 成为瓶颈。UMA 把这堵墙拆了：CPU 与 iGPU 共享同一池 128GB LPDDR5X，**没有搬运、没有 PCIe 瓶颈、没有「显存装不下」**。代价是这池内存的带宽（~273 GB/s）卡在 HBM3e（~8TB/s，快得发烫）与一般双信道 DDR5（~100 GB/s 以下，慢）之间的尴尬地带——**它赌的是「对个人/边缘场景，容量比极致带宽更稀缺」**。对个人/边缘场景，这个赌注大概率正确：你更常被「模型装不进 24GB 显存」卡住，而非「带宽不够快」。Apple Silicon（M 系列）早已验证这条路，DGX Spark 是把它推到 128GB + Blackwell 算力的版本。**这也预示了选型哲学的长期转向：未来在这类机器上，胜出的不是「最快的小模型」，而是「最大的、激活参数可控的 MoE」——硬件架构的赌注，最终会反过来定义什么模型值得被蒸馏、被部署。** 你今天为 DGX Spark 选型的逻辑，就是未来几年个人 AI 算力的缩影。

## 本章小结

- **规格先校准**：DGX Spark = GB10 + 128GB LPDDR5X UMA + **~273 GB/s**（⚠️ 非二手文章常写的 600；错一倍 tok/s 全错）+ ~1 PFLOP FP4/NVFP4。
- **用 Roofline 看 DGX Spark**：屋脊点 = 算力÷带宽 ≈ **~3600 FLOP/byte**，而 decode 算术强度仅 **2–4**——深陷带宽墙；解码速度 ≈ 带宽 ÷ 每 token 读取权重。容量巨人、带宽凡人。
- **选型三问**：激活多少（速度）、总共多少（智商）、密集还 MoE（连贯性/内存形态）。结论：首选大 MoE（GPT-OSS 120B）+ 速度型（Qwable-v1 35B）。
- **自己算 fit**：总占用 = 权重 + KV Cache + 框架开销；权重 = 参数 × bpw ÷ 8，KV 随上下文线性膨胀。32K 看权重、256K 看 KV——**长文的内存杀手是 KV Cache，不是权重**（埋给第 10、11 章）。
- **双模并行 = 分层认知**，高端可上**投机解码**让大小模型合体无损加速；KV Cache 量化（FP8）才是长上下文的真瓶颈解。
- **部署榨干 UMA**：GGUF + imatrix 校准、`numactl --interleave=all` + `mlock`、避开通用 Docker。
- **长上下文拆三问**，用 RULER/Needle 亲自验，别信规格表。

硬件与选型讲透，下一个问题回到本质：**这些影子到底是怎么蒸出来的？** 离开硬件，走进蒸馏的科学——第 4 章。
