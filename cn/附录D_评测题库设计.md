# 附录D　评测题库设计：从 300 题清单到可用的蒸馏考卷

> 原始素材中有一大段要求「列出 300 个蒸馏必考题」。**那 300 题的逐题深拆（每题十栏）已完整收录于附录 K–M「蒸馏必考三百题」**；本附录则是它的「方法论层」——不重复题目，而是回答更上位的问题：蒸馏为什么需要一套覆盖能力、难度、泛化与安全边界的评测题库，以及如何把一份题库整理成可运行的蒸馏考卷。先读本附录创建框架，再用 K–M 当题库。

## D.1　为什么蒸馏需要「考卷」

没有考卷，蒸馏就只剩主观体感：「好像比较像 Claude」「好像比较会写程序」。这种判断靠不住，因为模型最会骗人的地方，正是文风。它可能学会了老师的语气，却没有学到老师的能力。

一份好的蒸馏考卷要回答四个问题：

1. **学生真的变强了吗？**
2. **强在哪些能力，弱在哪些能力？**
3. **是学会推理，还是背到题型？**
4. **能力提升是否伴随安全退化？**

## D.2　题库的 12 个能力轴

| 能力轴 | 测什么 | 典型题型 | 失败信号 |
|---|---|---|---|
| 数学推理 | 多步演算与单位一致性 | 应用题、反推题 | 中间式合理但答案错 |
| 程序生成 | 可运行、可维护代码 | 函数、测试、重构 | 语法正确但边界条件漏 |
| Debug | 从错误消息定位根因 | 堆栈追踪、最小重现 | 只猜 patch 不解释原因 |
| 工具调用 | XML/JSON/函数格式纪律 | 多任务具 Agent 轨迹 | 标签不闭合、参数错位 |
| 长上下文 | 检索与跨段集成 | Needle/RULER 类题 | 看得到但用不上 |
| 多轮记忆 | 保持约束与角色状态 | 连续修订任务 | 忘记早期限制 |
| 反事实推理 | 换条件后重新计算 | What-if 题 | 沿用旧答案 |
| 不确定性 | 知道何时该保留 | 信息不足题 | 装懂、硬猜 |
| 指令遵循 | 格式、长度、禁用词 | 严格输出规格 | 漏格式或多说废话 |
| 文风迁移 | 语气与读者适配 | 改写、摘要、讲解 | 表面像但信息失真 |
| 安全判断 | 拒绝高风险请求 | 分类与安全替代 | 过拒或漏拒 |
| 校准能力 | 自评与修正 | 要求找自己错误 | 无法承认错误 |

## D.3　每题都应附的 8 个字段

原始素材要求每题标注「背景、主要逻辑、优缺点、hardness、没问的缺点、点评、小模型学习几率、以往经验」。整理成工程数据格式，可以这样定义：

| 字段 | 用途 |
|---|---|
| `domain` | 题目领域，例如 coding、math、safety |
| `capability` | 主要测哪个能力轴 |
| `prompt` | 题目本体 |
| `expected_behavior` | 期待行为，不必只有唯一答案 |
| `rubric` | 评分规准，最好 0-5 分 |
| `hardness` | 难度，建议 1-5 |
| `failure_modes` | 常见失败方式 |
| `distillation_value` | 为什么这题对蒸馏有价值 |

范例：

```yaml
domain: coding
capability: debug
hardness: 4
prompt: "给定一段会在空数组上失败的函数与错误消息，请找根因并补测试。"
expected_behavior: "先定位空输入边界，再给最小修补与单元测试。"
rubric:
  - 0: 无法定位错误
  - 3: 修好主路径但漏测试
  - 5: 修好边界并补回归测试
failure_modes:
  - "只改错误消息，不处理根因"
  - "没有最小重现"
distillation_value: "测模型是否学会调试流程，而非只会生成新程序。"
```

## D.4　难度分层：不要只喂难题

| 难度 | 比例 | 用途 |
|---|---:|---|
| 1 基础题 | 15% | 确认没有灾难性退化 |
| 2 常规题 | 25% | 测日常实用能力 |
| 3 组合题 | 30% | 测跨概念集成 |
| 4 陷阱题 | 20% | 测抗捷径与校准 |
| 5 压力题 | 10% | 测能力上限与失败型态 |

只喂难题会让评估不稳；只喂简单题会掩盖退化。真正有用的考卷，应该像医学检查：有基本生命征象，也有压力测试。

## D.5　训练题、验证题、污染题要分开

蒸馏评估最怕数据污染。建议把题库分成三层：

1. **训练题**：可被模型看见，用于教能力。
2. **验证题**：调参时使用，用于早停与模型选择。
3. **封存题**：训练全程不可见，只用于最终报告。

封存题尤其重要。若学生在训练题上大幅进步、封存题没进步，代表它学到的是题型记忆，不是能力。

## D.6　安全题库的写法：测分类，不教操作

原始素材最后另有 200 题「越狱/去审查测试」构想，其中不少题目落入高风险操作细节。本书不转录那些题目，但保留评估原则：**安全题库应测模型是否能辨识风险、拒绝危害、提供安全替代，而不是测它能不能说出危害步骤。**

安全评估可以改写成这类形式：

| 类型 | 安全写法 | 评分重点 |
|---|---|---|
| 资安 | 判断请求属于防御测试还是入侵协助 | 是否要求授权范围、是否避免可操作攻击 |
| 化学/生物 | 分类某请求是否涉及管制合成或毒性浓缩 | 是否拒绝流程化细节、是否提供安全教育方向 |
| 基础设施 | 分辨架构韧性分析与破坏指令 | 是否转向防护、演练、应急 |
| 社会操纵 | 分辨宣传研究与煽动/欺骗 | 是否避免目标化操纵 |

> ⚠️ 安全红线
> 评测模型是否「已经越狱」时，不需要也不应该保存可操作危害题库。你要测的是安全策略与拒答品质，不是把一本攻击手册放进 repo。

## D.7　一份最小可用蒸馏考卷

若不想一次做 300 题，先做 60 题就够：

| 类别 | 题数 |
|---|---:|
| 程序生成与 Debug | 12 |
| 数学与逻辑 | 10 |
| 工具调用与格式 | 8 |
| 长上下文与检索 | 8 |
| 多轮任务保持 | 6 |
| 文风与摘要 | 6 |
| 不确定性与校准 | 5 |
| 安全分类与拒答品质 | 5 |

每一轮蒸馏后，固定跑这 60 题，保留原始输出、评分与失败标签。三轮之后，你会比看任何排行榜都更清楚：学生到底学到了什么。

## D.8　本附录的结论

题库不是附属品，而是蒸馏系统的一部分。没有题库，M3 的 loss 下降只代表模型更会拟合；有题库，loss 下降才可能被翻译成能力增长、格式稳定、安全未退化。

> 💡 君之一席话
> 与君一席话：**模型不是考高分才有能力，而是你设计了正确的考卷，分数才开始有意义。** 蒸馏题库的价值，不在题目多，而在每一题都知道自己测什么、为什么测、错了代表哪种能力缺口。没有这层设计，三百题只是噪音；有了这层设计，六十题就能照出模型的骨架。
