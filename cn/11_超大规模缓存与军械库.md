## 情境 1　三级异构保存金字塔：把 KV Cache 当操作系统的虚拟内存来管

**背景**：在动辄支持百万级 token 上下文的模型里，如果把所有用户的 KV Cache 全都死锁在 GPU/TPU 的 HBM（高带宽显存）里，集群会在并发稍微一涨时就因 OOM 瞬间瘫痪。HBM 是整个系统里最快、也最贵、最稀缺的资源——拿它去存「一个三天没回来的用户的对话历史」，是奢侈到犯罪的浪费。原始素材主张的解法，本质是把操作系统管了五十年的老把戏——**虚拟内存与多级缓存**——原封不动搬到 KV Cache 上：建一座三级异构保存金字塔（Hierarchical KV Cache Management）。

三级金字塔，由快到慢、由贵到便宜、由小到大：

```
            ┌───────────────────────────────────────────┐
            │  L1：TPU/GPU HBM  (High-Bandwidth Memory)  │  ← 最快·最贵·最小
            │  存什么：当前正在 Decode 的 Active Segments  │
            │  容量量级：数十 GB / 卡   带宽：~TB/s         │
            └───────────────────────────────────────────┘
                          ▲   100 GB/s ~ TB/s
                          │   PCIe Gen5 / NVLink / ICI
                          ▼
            ┌───────────────────────────────────────────┐
            │  L2：Host Memory  (CPU DDR5)                │  ← 中速·中价·中量
            │  存什么：暂不活跃、但可能马上用到的多轮历史    │
            │  容量量级：数百 GB ~ TB / 节点                │
            └───────────────────────────────────────────┘
                          ▲   Direct I/O
                          │   NVMe-over-Fabrics / RDMA
                          ▼
            ┌───────────────────────────────────────────┐
            │  L3：Distributed NVMe SSD Pool             │  ← 慢·便宜·海量
            │  存什么：冷数据（长视频多模态、冷冻 Session）  │
            │  容量量级：数十 TB ~ PB / 池                  │
            └───────────────────────────────────────────┘
```

对应关系一目了然：**L1 之于 KV Cache，正如 CPU L1 缓存之于主内存；L3 之于 L1，正如硬盘之于 RAM**。当一个用户正在打字、模型正在逐 token 生成，他的 KV 必须在 L1；当他发完一句、进入「人类在思考下一句」的几秒空档，把他的 KV 从 HBM 换出（Offload）到 L2 的 DDR5，HBM 立刻腾出来服务别人；当他半小时没回来，再把整段 Session 沉到 L3 的 SSD 池里冷冻。

**关键技术：异步预取流水线（Prefetching Pipeline）**。金字塔本身不难，难的是「搬运」这件事——HBM 和 Host Memory 之间隔着 PCIe，一次搬几 GB 的 KV 是要时间的。如果搬运期间计算停下来等，那这套分层就成了负优化：你省了显存，却赔了延迟。原始素材的解法是在编译器（XLA）与运行期里，把**计算线程与 I/O 线程彻底解耦**，并用一个古老而强大的词掩盖住搬运成本——**重叠（Overlapping）**：

```python
# 概念示意：用「别人的 decode 时间」掩盖「我的 KV 搬运延迟」
# 真实系统在 XLA runtime / CUDA stream 层用异步 DMA 实现，此处仅示意调度语义

while serving:
    batch = scheduler.next_batch()          # 取一批正在 decode 的请求

    # ── I/O 线程（与计算并行，不阻塞）──────────────────
    if detect_user_typing_next_turn(user_X):
        # 侦测到用户 X 正在输入下一句 → 提前把他的历史 KV
        # 从 L2 DDR5 异步预取回 L1 HBM。此刻计算线程根本不知道、也不用等
        io_thread.async_prefetch(user_X.kv, src="DDR5", dst="HBM")

    # ── 计算线程（持续满载跑当前 batch）────────────────
    compute_thread.decode_step(batch)        # 这一步的耗时，正好「盖住」上面的搬运

    # 等用户 X 真正按下送出时，他的 KV 早已躺在 HBM 里 → 零感延迟
```

精髓在那句注解：**用其他请求正在运行的 decode 计算时间，去「掩盖」当前请求 KV Cache 的搬运延迟**。在大批量推理里，永远有一堆请求正在 decode、把 TPU 的 Tensor Core 喂得饱饱的；I/O 引擎就躲在这片计算的阴影下，悄悄把下一个要用的 KV 从 DDR5 搬回 HBM。等用户真的按下 Enter，他的历史早就在 HBM 里等着了——延迟被藏进了硬件本来就忙着的那段时间里，硬件利用率（MFU）始终高位运行。

> 💡 君之一席话
> 与君一席话：**所有「让系统变快」的高招，拆到底层，几乎都是同一句话——别让昂贵的资源等便宜的资源；当搬运不可避免，就把它藏进计算的阴影里。** 三级金字塔不是什么 AI 新发明，它是操作系统教科书第三章的虚拟内存、是 CPU 的 L1/L2/L3、是数据库的 buffer pool，换了一身「KV Cache」的衣服又走了一遍。真正值钱的不是「分三层」这个结构——那是常识；值钱的是**异步预取那一手「用别人的计算时间掩盖我的搬运时间」**。新手做分层缓存，做出来的是「换出换入都要停下来等」的同步版本，结果省了显存、赔了延迟，得不偿失；老手做分层，第一件想的事就是「搬运的延迟，能不能躲到某段反正也要花的时间后面去」。**这就是 overlapping 的哲学：系统工程的最高境界，不是让每件事都变快，而是让慢的那件事，发生在没人等它的时候。** 你能不能在脑子里同时转动「计算线程」和「I/O 线程」两条独立的时间轴，并让它们在正确的点交错——这一念之差，分开了「会用缓存的人」和「会调度缓存的人」。

> 🔍 高端点评 — 为什么「KV Cache 换页」比「OS 换页」更凶险，也更值得
> 把 KV Cache 当虚拟内存换页，方向绝对正确，但要清醒地看到它和 OS 换页的**三个关键差异**，否则照搬必翻车。**其一，粒度与语义不同**：OS 换页的单位是固定的 4KB page，内容无语义；KV Cache 的单位是「一个 token 串行的注意力状态」，它有强烈的**时序前缀依赖**——你不能只换出一段历史的中间，因为后面的 token 算的时候要看见前面的。所以 KV 的换出单位通常是「整段 prefix」或 vLLM 那种固定 block，且必须前缀连续。**其二，代价结构是反的**：OS 换页换出的是「数据」，换回来就是原样；KV Cache 如果不换、选择丢掉，要还原它得**重算 prefill**——而 prefill 是 compute-bound 的重活。这意味着「换出到 DDR5」和「直接丢弃再重算」之间有一道精算题：搬一趟 PCIe 的 I/O 成本，对比重跑一遍 prefill 的算力成本，谁便宜？串行越长，重算越贵，越该留着搬；串行越短，重算越便宜，丢了反而省事。这道精算，正是情境 3「投机式缓存迁移」的微观版本。**其三，这是 vLLM 已经产品化的方向**：vLLM 的 CPU offloading、`--swap-space`、以及社群在做的 KV 分层，正是这套金字塔的 L1↔L2 落地；再往 L3 SSD 走，则进入 Mooncake（Kimi 背后那套以 KVCache 为中心的分离式架构）这类前沿系统的领域。**所以这一节不是空中楼阁，它是把一个已在生产线跑着的方向，讲清楚它的物理。** 唯一要对素材打的折是：「零感延迟」是理想态——只要用户回得比预取快，或者预取猜错了人，就会露出搬运的尾巴。没有真正的零延迟，只有「藏得够不够好」的延迟。

## 情境 2　给缓存做减法：从 MHA 到 MLA，模型结构层的瘦身术

**背景**：情境 1 是纯工程——不改模型，只在外面建金字塔搬来搬去。但原始素材一句话点破了它的天花板：**「纯粹靠工程优化是有上限的，必须和算法团队联手（Co-design），从模型结构上给 Cache 瘦身。」** 这是全章最重要的一句方法论。KV Cache 之所以大，根子在 Attention 的结构：每一层、每一个注意力头，都要为每个 token 缓存一份 K 和一份 V。要从源头砍，就得动 Attention 本身。这条演进路线——MHA → MQA → GQA → MLA——是过去几年大模型推理效率最重要的一条暗线。

先讲清楚 KV Cache 的体积公式。在经典 **Multi-Head Attention（MHA）**里，每个 token 要缓存的 KV 大小正比于：`2（K和V）× L（层数）× H（头数）× D（每头维度）`。其中那个 **H（头数）**，就是被历代改进反复开刀的对象：

```
MHA  ：每个 Q 头配自己的 K 头、V 头          KV 头数 = H      （基准·最大）
        Q1 Q2 Q3 Q4 ... QH
        K1 K2 K3 K4 ... KH
        V1 V2 V3 V4 ... VH

MQA  ：所有 Q 头共享 唯一 一组 K、V          KV 头数 = 1      （省 H 倍·伤能力）
        Q1 Q2 Q3 Q4 ... QH
        K（只有一个，大家共用）
        V（只有一个，大家共用）

GQA  ：Q 头 分组，每组共享一对 K、V          KV 头数 = G（组数，如 8）  （业界主流）
        [Q1 Q2 Q3 Q4][Q5 Q6 Q7 Q8]...
         K_grp1        K_grp2
         V_grp1        V_grp2

MLA  ：不存高维 K/V，存低秩压缩隐矢量 c_t     缓存 = 一个极小的 latent 矢量（解码时在线解压）
        K, V ──低秩投影──► c_t（维度极小）   解码时 ──线性投影──► 还原出 K, V
```

逐个拆解这条瘦身路线的取舍：

| 机制 | KV Cache 相对大小 | 内核做法 | 代价 / 现状 |
|---|---|---|---|
| **MHA** | 1×（基准） | 每个 Q 头独享一对 KV 头 | 表达力最强，但 KV 体积最大、长上下文最吃显存 |
| **MQA** | ~1/H | 所有 Q 头**共享唯一一组 KV** | 显存骤降 H 倍，但表达力下降太多，大模型能力受损严重 |
| **GQA** | ~G/H（如 8/H） | Q 头**分组**，每组共享一对 KV（Llama 系常用 8 组） | 速度与效果的折中，**当前业界主流**，被 Llama 3 等广泛采用 |
| **MLA** | 宣称省 90%+ | **低秩压缩**：把 K/V 投影进低维隐空间 c_t，解码时在线解压 | DeepSeek 提出（V2/V3 代表作），在维持表征力下大幅缩 KV；算法-基建协同设计的典范 |

MQA 和 GQA 是「减少 KV 头的数量」——MQA 一刀砍到只剩一组，太狠，模型变笨；GQA 学乖了，砍到剩几组（比如 8 组），在「省显存」和「保能力」之间找了个甜蜜点，于是成了 Llama 3 一类主流开源模型的标配。

**MLA（Multi-head Latent Attention）** 走的是完全不同的一条路。它不在「头数」上做减法，而是在「维度」上做压缩。它的洞察是：高维的 K 和 V 里藏着大量冗余，可以用**低秩矩阵分解（Low-rank Compression）**把它们投影压进一个维度极小的隐空间矢量 `c_t`——缓存的时候，只存这个小小的 `c_t`，而不是原来那一大坨高维 K、V；到了解码阶段，再用线性投影矩阵**在线（on-the-fly）把 `c_t` 解压**回真正的 K 和 V 去算注意力。本来每个 token 要存 `2 × L × H × D`，现在只要存一个小矢量。

> ⚠️ 真伪提醒
> 「MLA 把 KV Cache 砍掉 90% 以上、并发量提升数倍」这个数字来自 DeepSeek 的论文与技术报告，是**特定模型、特定配置下的测量值**，不是普适常数——压缩比强烈依赖隐空间维度的选取、模型规模与串行长度。MLA 由 DeepSeek 提出并在其开源模型上验证，是可查证的真实工作；但「DeepMind 也在深度跟进类似方向」这句是原始素材的内部口吻**宣称**，本书无法独立查证，当方向性参考即可。可平实陈述的事实是：MQA / GQA / MLA 都是已发表、已被生产级模型采用的真实技术，这条「给 Attention 的 KV 瘦身」的演进线是真的。

> 💡 君之一席话
> 与君一席话：**最深的性能优化，从来不发生在「工程」和「算法」各自的围墙里，而发生在那道墙被拆掉、两边的人坐到同一张桌子上的时候——MLA 就是这道墙倒下时长出来的东西。** 情境 1 的金字塔，是 infra 团队一个人能做完的事：模型不动，我在外面搬。但它有天花板——KV 本身有多大，金字塔就得搬多大，你只是把「贵的搬运」变成「便宜的搬运」，没让需要搬的东西变少。MLA 不一样，它要改的是注意力的数学形式，这是算法团队的领地；可决定「压到多少维才不掉点、解压的算子在 TPU 上跑不跑得动」的，又是 infra 的领地。**任何一边单独都做不出 MLA：算法团队不懂解压算子在硬件上的代价，会压出一个理论漂亮、实际慢的东西；infra 团队不敢动注意力公式，只会在外面搬箱子。** 这就是 co-design 的真义——不是「算法先设计好，丢给工程去实现」这种接力赛，而是两边在白板前同时被对方的约束逼着让步、长出一个谁都想不到的折中。**一个团队能不能做出结构性的突破，不看它招了多强的算法科学家或多硬的系统工程师，看它有没有勇气让这两种人，为了同一个 KV 数字吵到同一块白板上去。**

## 情境 3　集群分布式：当缓存有了「位置」，路由就成了战争

**背景**：单机把 PagedAttention、Prefix Caching 做到极致还不够。一旦你管的是由成千上万颗 TPU/GPU 组成的超大推理集群，网关（Gateway）与调度器就必须升级出一种新能力——**缓存感知（Cache-Aware）**。因为在集群里，缓存不再是一块抽象的「显存空间」，它**长在某一台具体的物理机器上**。谁持有用户 A 的历史 KV，是节点 7 还是节点 42，这件事决定了下一个请求该去哪——这正是本章开头那场「电费烧成灰」灾难的根源。

**传统调度（Round-Robin / Least-Connection）的灾难**：

```
用户 A 连发 5 句，传统负载均衡按「最闲」或「轮询」分发：

  msg#1 ──► 节点1   (重算 A 的全部历史 prefill)
  msg#2 ──► 节点2   (节点2 没有 A 的缓存 → 又重算一遍全部历史)
  msg#3 ──► 节点3   (再重算一遍)
  msg#4 ──► 节点1   (节点1 的缓存可能已被别人挤掉 → 还是重算)
  msg#5 ──► 节点4   (重算)

结果：同一段「五万字历史」的 prefill，被算了 5 遍。
      你在每台机器上做的 Prefix Cache，命中率全部归零——
      因为下一句话根本不回到上一句所在的那台机器。
```

负载均衡器的本能是「把活分给最闲的机器」——这在无状态的 Web 服务里是金科玉律。但 LLM 推理**不是无状态的**：请求 N 的便宜，创建在「请求 N-1 的 KV 还在这台机器上」之上。把有状态的缓存，交给一个假设无状态的调度器，就是在拿全集群的算力，为「缓存放错了位置」这个盲区陪葬。

**正解：前缀哈希 + 亲和性路由 + 投机式缓存迁移**。网关必须持有一张全局的分布式缓存索引（通常用内存数据库或高效 RPC 维护），三步走：

| 步骤 | 机制 | 做法 |
|---|---|---|
| ① 识别 | **Prefix Hashing（前缀哈希）** | 网关对请求的 `System Prompt + 历史对话` 算一个前缀哈希（如 **MurmurHash3**，快、分布均匀）。相同前缀 → 相同哈希 |
| ② 路由 | **Affinity Routing（亲和性路由）** | 不再投给「最闲」的节点，而是优先投给**已经缓存了该前缀 KV 的那台物理节点**，最大化集群整体的 Prompt Cache 命中率 |
| ③ 权衡 | **Speculative Cache Transfer（投机式缓存迁移）** | 若持有缓存的节点 A 已重载：精算「在闲置的节点 B 重算 prefill」vs「用集群高速网络把 KV 从 A 直接拉到 B」，谁快选谁 |

第三步是整套设计里最精密的一手，值得拆开看那道精算题：

```
情况：用户请求命中节点 A 的缓存，但节点 A 此刻重载（队列很长）。
调度器面前两条路：

  路径甲【重算】：把请求发给闲置的节点 B，让 B 从零重算整段历史 prefill
       成本 ≈ Prefill_FLOPs(历史长度) / B 的算力
       —— 串行越长，这条路越贵（prefill 是 compute-bound）

  路径乙【迁移】：把 A 上那段已算好的 KV，经高速网络 RDMA 直接搬到 B
       成本 ≈ KV_Bytes(历史长度) / 集群网络带宽
       —— 带宽够猛时，搬一份现成的，比从头算一份快得多

  决策：if  搬运成本(乙) < 重算成本(甲):  走 RDMA 迁移，省下整段 prefill 算力
        else:                          认赔，在 B 重算
```

注意这道题和情境 1「KV 换页 vs 重算」是**同一道题在不同尺度上的重演**：情境 1 是「搬到隔壁的 DDR5」对比「丢了重算」，这里是「跨机搬到节点 B」对比「在 B 重算」。物理直觉一致：**prefill 的算力成本随串行长度超线性增长，而搬一份现成 KV 的成本只随它的字节数线性增长**。所以串行越长、上下文越大，「搬现成的」就越划算——这正是为什么长上下文时代，集群内的缓存迁移网络会变得和算力一样关键。

> ⚠️ 真伪提醒
> 「Google Jupiter 网络 1.6 Tbps」「直接传输缓存带来显著 TTI 优势」这类具体带宽数字与内部网络代号，是原始素材以 DeepMind 内部口吻给出的**宣称**，本书无法独立查证，当量级示意看待。可平实陈述的真实技术是：**亲和性路由 + 前缀哈希是 SGLang 的 RadixAttention、以及各家推理网关（如 vLLM 的 prefix-aware 调度、Mooncake 的全局 KV 索引）正在做的真方向**；MurmurHash3 是真实的非加密哈希；RDMA 跨节点搬 KV 也是 PD 分离（情境 4）架构里的真实机制。要打折的只是「某厂某网络某带宽」这类具体到产品的数字。

> 🔍 高端点评 — 亲和性路由的暗礁：缓存亲和与负载均衡，是一对天生反目的孪生子
> 亲和性路由听起来完美，但生产在线它会把你拖进一个经典的**两难**，没看清这点就会从一个坑跳进另一个坑。**负载均衡要的是「把活分散开」，缓存亲和要的是「把活集中到持有缓存的那台」——这两个目标在物理上直接对冲。** 你越是严格地把用户 A 的所有请求钉在节点 7 上（缓存命中率拉满），就越容易制造**热点**：某个被疯狂调用的长 system prompt（比如一个爆红的 Agent 模板），会把所有命中它的流量全灌到最早缓存它的那两三台机器上，把它们烤糊，而集群里别的机器在睡觉。这就是为什么成熟的 cache-aware 调度器，从来不是「无脑亲和」，而是一套**带负载感知的亲和**——SGLang 的调度、以及各家网关的做法，本质都是在「缓存命中收益」和「节点负载代价」之间动态打分：命中当然好，但若持有者已经重载，就触发情境 3 第③步那道「迁移 vs 重算」的精算，主动把流量导开或把缓存拷贝到第二台。**更深一层，这逼出一个架构哲学：在有状态的推理集群里，根本不存在「最优路由」这种静态答案，只存在「在缓存亲和、节点负载、迁移成本三者之间持续再平衡」的动态过程。** 把它当成一个一次算完的哈希函数，你会死于热点；把它当成一个持续调节的控制回路，你才真正拥有了一座能呼吸的集群。这也是分布式系统永恒的母题：一致性（把状态钉在固定位置）与可用性／负载（把流量摊开）之间，永远在拔河——缓存感知路由，只是这场拔河在 LLM 时代的最新战场。

## 情境 4　PD 分离：当 Prefill 和 Decode 的硬件需求彻底相反

**背景**：把缓存感知路由推到极致，会撞上一个更深的结构性矛盾，它的解法是当前超大规模推理基建的「终极演进方向」——**PD 分离（Disaggregated Prefill & Decode）**。要理解它为什么是终局，得先看清一个被很多人忽略的事实：LLM 推理的两个阶段，**Prefill 和 Decode，是两种物理性质完全相反的负载**，把它们塞在同一张卡、同一个 batch 里，是在让两个八字不合的人合住一间房，互相拖累。

两个阶段的对照：

| 维度 | Prefill（预填：消化 prompt） | Decode（解码：逐 token 生成） |
|---|---|---|
| 计算特性 | **Compute-bound（算力受限）** | **Memory-bound（访存受限）** |
| 在干嘛 | 把整段 prompt 一次性过一遍，产出 KV Cache | 每次只生成 1 个 token，反复读取庞大的 KV |
| 吃什么资源 | 吃**算力**（Tensor Core 满载）、吃带宽 | 吃**显存容量**（装下所有并发的 KV）、吃访存带宽 |
| 理想硬件 | 高算力、高带宽芯片（如 HBM3e） | 大容量显存 |
| 痛点 | 一个长 prompt 进来会**霸占算力**，饿死正在 decode 的请求 | 算力大量闲置，只是在不停搬运 KV |

矛盾就在这：**Prefill 缺的是算力，Decode 缺的是显存容量；Prefill 把算力榨干的时候，Decode 的算力在闲置；Decode 把显存占满的时候，Prefill 的显存需求被挤压。** 它俩混在同一个 GPU batch 里，会发生最难看的事——一个五万 token 的长 prompt 进入 prefill，瞬间霸占全部算力，正在逐字回答的几十个短请求集体卡住，它们的 **TTFT（首 token 延迟）和 TPOT（token 间延迟）一起飙升**，用户眼里就是「打字打到一半突然顿住」。

业界对这个矛盾有两层解法，一轻一重：

```
─── 同机缓解：Chunked Prefill（分块预填）───────────────────
  把长 prompt 的 prefill 切成固定大小的 chunk（如每次 512 token），
  算完一块 → 让出算力给 decode 跑一步 → 再算下一块。
  把「一口吞下长 prefill」变成「小口流水线化」，避免长请求饿死短请求。
  ↑ vLLM / DeepSpeed-FastGen 的 Dynamic SplitFuse 都在做。属「同一张卡上的时间切片」。

─── 跨机根治：PD Disaggregation（PD 分离）─────────────────
  干脆把集群劈成两种专用节点：

   ┌────────────────────┐                      ┌─────────────────────┐
   │  Prefill 节点 (P)   │   RDMA 直推 KV       │  Decode 节点 (D)     │
   │  高算力·高带宽       │ ───────────────────► │  大容量显存          │
   │  (如 HBM3e)         │   把算好的 KV Cache   │  专心逐 token 生成   │
   │  专心消化 prompt    │   经高速网络推给 D    │  不被 prefill 打断   │
   └────────────────────┘                      └─────────────────────┘

  P 节点算完 KV，通过超高速 RDMA 网络把整份 KV「推」给 D 节点，
  D 节点接过来专心 decode。两种硬件各司其职，互不干扰。
```

Chunked Prefill 是**同一张卡上的时间切片**——缓解，治标；PD 分离是**跨机器的空间隔离**——根治，治本。因为两个阶段的硬件需求是相反的，最彻底的解法不是让它们在同一张卡上轮流忍让，而是给它们配**各自最适合的硬件**：prefill 节点堆算力，decode 节点堆显存容量，中间用 RDMA 把 KV 一推了事。

还有一层编译器底层的功夫，与 PD 分离互补——**XLA 的算子融合（Kernel Fusion）**：

```
未融合（通用框架的天真做法）：
  PagedAttention 寻址 ──写回HBM──► RoPE 旋转位置编码 ──写回HBM──► Softmax
  └ 每一步的中间结果都写回 HBM 再读出来，HBM 往返是延迟与带宽的杀手

融合后（手工优化的单一 TPU Kernel）：
  ┌─────────────────────────────────────────────────┐
  │  PagedAttention + RoPE + Softmax  融合成一个算子   │
  │  中间变量留在寄存器，不写回 HBM，寄存器级数据复用    │
  └─────────────────────────────────────────────────┘

另一手：固定 block size（如 16 / 32 token 一块）→ 把「动态串行长度」
        转成「静态 block 数组组合」→ 避免动态形状触发 XLA 频繁重编译
```

两个底层细节都指向同一件事：**消灭不必要的 HBM 往返、消灭动态形状**。算子融合把 attention 寻址、RoPE、Softmax 三步揉成一个 kernel，中间结果留在寄存器里，不来回写 HBM；固定 block size 则把「串行长度动态变化」这个编译器最讨厌的东西，转成「固定大小 block 的数组」这种静态、可预测的形状，让编译出来的机器码跑在最高效率上，不会因为每来一个新长度就触发一次代价高昂的重新编译。

> ⚠️ 真伪提醒
> 「PD 分离是 Google DeepMind / OpenAI 在基建层的终极演进方向」是原始素材的内部视角宣称；可查证的事实是 PD 分离是真实且火热的研究与工程方向（DistServe、Splitwise、Mooncake 等公开工作都在做），vLLM 也已支持 disaggregated serving。「Prefill compute-bound、Decode memory-bound」是被广泛测量验证的真实特性。要打折的是「某某顶尖公司内部已全面采用、某某具体带宽」这类无法外部核实的断言——把它当成「业界公认的前沿方向」，而非「某厂既成现实」。

> 💡 君之一席话
> 与君一席话：**当你发现一个系统怎么调都调不好，十有八九不是参数没调对，而是你硬把两个本性相反的东西，塞进了同一个盒子里——真正的解法往往不是更聪明地共处，而是承认它们合不来，然后把盒子拆成两个。** Prefill 和 Decode 的故事，是这条智能最干净的范例。多少团队在「同一张卡上让 prefill 和 decode 和平共处」这件事上耗尽心力——调 batch、调优先级、做 chunked prefill 切得越来越细——本质都是在一段不幸福的婚姻里做夫妻调解。Chunked Prefill 是高明的调解，但它改变不了「一方要算力、一方要显存」这个根本不合。PD 分离的洞察朴素到近乎粗暴：**既然你俩要的东西相反，那就别住一起了。** 这在分布式系统里有个庄严的名字叫「关注点分离」，可它最难的从来不是「分」这个动作，而是**敢于承认「合在一起本身就是错的」这个判断**——因为「合在一起」通常是历史包袱、是「一直以来都这么做」、是省事。**一个架构师的成熟度，不在他能把多复杂的东西揉到一起，而在他能不能在所有人都习惯了某种耦合之后，站出来说『这两件事根本不该在一起』，然后扛着迁移成本把它们劈开。** 看穿不合，比优化共处，难得多，也值钱得多。

## 情境 5　二十大前沿军械库：一张按攻坚小组分群的作战地图

**背景**：前四个情境，是把素材里几条主线讲透。但原始素材最后摊开的，是一份更大的东西——大模型缓存领域的**二十个前沿方向**，横跨顶级会议（NeurIPS / ICML / ICLR / MLSys）的最新论文、工业界生产环境的开源神兵、以及顶尖架构方法论。它把这二十个方向，按「该由哪个攻坚小组去打」分成了三层：**1–6 算法层**（给算法工程师魔改模型）、**7–12 基础设施层**（给 infra 团队做引擎选型压测）、**13–20 架构方法论层**（给架构师写系统 HLD 时通盘布局）。下面把这份军械库整理成一张作战地图——它正是本书附录 G 那份更完整的缓存技术速查的种子。

**【算法层 1–6】——归算法 / 研究工程师：在模型结构与 KV 本身上动刀**

| # | 方向 | 一句话技术内核 | 代表 |
|---|---|---|---|
| 1 | **MLA** | 低秩压缩把 K/V 投影进低维隐空间 c_t，解码时在线解压，KV 体积大砍（见情境 2） | DeepSeek-V2/V3 |
| 2 | **H2O（Heavy-Hitter Oracle）** | Attention 矩阵呈**幂律分布**，少数 Token 贡献绝大部分权重；动态识别并只保留这些 Heavy-Hitter 的 KV，即时剔除其余 → 定长缓存装无限长文 | 《H2O》论文 |
| 3 | **StreamingLLM** | 发现 **Attention Sink** 效应（最初 2–4 个 Token 锁住巨大注意力）；保留「初始几个 token + 最新滑动窗口」，不重算 prefill 即可流式无限解码 | 《Attention Sinks》论文 |
| 4 | **Speculative Decoding + Tree-KV** | 投机解码（小模型出草稿、大模型校验）会产生多分支；用**树状 KV Cache** 并行校验多分支，避免多路重复计算 | Medusa 系列 |
| 5 | **Infinite-LLM / Mamba-2 混合** | 把超出窗口的旧 KV 压进一个**固定大小的循环状态（Recurrent State）**，空间复杂度 O(N) → O(1) | Griffin / Hawk / Mamba-2 |
| 6 | **KV-Quant（低比特量化）** | Per-Channel/Per-Token 混合量化 + 非均匀量化 + 离群值保留，把 KV 压到 **2-bit/3-bit 几乎不掉点** | 《KV-Quant》论文 |

这六个方向有个共同点：它们都在问「**KV Cache 本身，能不能更小或更聪明**」——MLA 压维度、H2O 和动态淘汰丢无用 token、StreamingLLM 只留头和尾、Mamba 系把历史揉进固定状态、KV-Quant 降比特。注意 1、2、3、5 之间有一条暗线：**它们都在跟「注意力的长尾」较劲**——H2O 说「大部分 token 不重要，丢」，StreamingLLM 说「头几个和最近的最重要，留」，Mamba 说「干脆把过去压成一个状态」。三种哲学，回答同一个问题：**面对无限长的历史，到底该记住什么、可以忘掉什么**。

**【基础设施层 7–12】——归 Infra 团队：推理引擎的选型与压测**

| # | 工具 | 杀手锏 | 适用场景 |
|---|---|---|---|
| 7 | **vLLM** | **PagedAttention** 鼻祖，解决显存碎片化；Chunked Prefill / Prefix Caching / 张量并行 | 工业界黄金标准、通用首选 |
| 8 | **SGLang** | **RadixAttention**——把 Prompt 缓存做成自动化**基数树（Radix Tree）** | 复杂控制流、Few-shot、Tool-use 的多轮 Agent，命中率与吞吐强 |
| 9 | **LMDeploy** | KV Cache 量化（W4A16、KV INT4/INT8）支持极完美 + Persistent RPC | 边缘端 / 私有化显存受限场景榨极限 |
| 10 | **TensorRT-LLM** | In-flight Batching、FlashDecoding+，字节级显存控制 | NVIDIA 全家桶、配 Triton 多节点集群 |
| 11 | **LightLLM** | **Token Attention**——Token 级细粒度显存调度 | 超大并发、请求长度极不均匀的恶劣环境，抗 OOM |
| 12 | **DeepSpeed-FastGen** | **Dynamic SplitFuse**——把 prefill 与 decode 在同 batch 内动态交织切分 | 微软系，优化两阶段共处的缓存生命周期 |

这六个是「现成能用的兵器」——选型的关键不是「哪个最强」，而是「**你的负载长什么样**」：通用就 vLLM；多轮 Agent、prompt 高度复用就上 SGLang 的 RadixAttention（它正是情境 3 亲和路由的单机版）；显存受限要极致量化就 LMDeploy；绑死 NVIDIA 全家桶就 TensorRT-LLM。

**【架构方法论层 13–20】——归架构师 / Tech Lead：写 HLD 时的战略布局**

| # | 方法论 | 落地要点 |
|---|---|---|
| 13 | **Cache-Aware Routing** | 网关用 MurmurHash3 算前缀哈希，把同前缀请求路由到持有该 KV 的节点，最大化集群命中率（见情境 3） |
| 14 | **Tiered Cache Orchestration** | HBM(L1)/DDR5(L2)/NVMe(L3) 三级，背景线程 Offload/Prefetch（见情境 1） |
| 15 | **Chunked Prefill** | 长 prefill 切成固定 chunk，与 decode 流水线化，防长请求饿死短请求（见情境 4） |
| 16 | **Dynamic KV Eviction** | 不再 LRU 粗暴丢整段历史；按**注意力重要度**打分，**优先丢标点、虚词、过渡词**，保留名词与内核语意实体——「有损但高智能」的回收 |
| 17 | **Deterministic Prompt Formatting** | 「**静态在前、动态在后**」；Tools 定义 / System Prompt / Few-shot 按固定顺序串行化；**绝对禁止在 prompt 开头塞 UUID、随机盐、时间戳**（否则前缀永远不命中） |
| 18 | **Exact Prefix vs Semantic Cache** | Exact：token 完全一致走 RadixTree 复用 KV；Semantic：用 **GPTCache + Milvus/Qdrant** 算矢量相似度，**>0.98 直接拦截返回上次答案，成本归零** |
| 19 | **PD Disaggregation** | Prefill 节点算完 KV，RDMA 直推 Decode 节点（见情境 4） |
| 20 | **Context Cache Monetization** | 参考 **Anthropic Claude 的 `cache_control`**：代码里显式声明缓存锚点，海量长文（法律合约、代码库）任务输入费用大降 |

这八个是架构师的「全局棋」。其中第 16、17 两个值得单独点名，因为它们最便宜、最常被忽略、收益却最高。**第 17 确定性提示词格式化，几乎是零成本的纯纪律**——你不需要改一行模型、不需要买一张卡，只要在团队开发规范里写死一条「prompt 开头永远是静态内容，动态的东西放最后」，前缀缓存的命中率就能从「形同虚设」变成「常态命中」。反过来，只要有一个粗心的工程师在 system prompt 开头塞了个 `request_id=<uuid>` 或 `当前时间：...`，就足以让**整个团队精心建设的 prefix cache 全线失效**——因为前缀第一个 token 就变了，后面再相同也没用。**第 18 的语意缓存则是另一个维度的降本核武**：exact cache 省的是「相同前缀的重算」，semantic cache 省的是「意思相近的问题，根本不调模型」——GPTCache 用矢量数据库算相似度，超过 0.98 直接把上次的答案返回去，这一次推理的成本是零。

> ⚠️ 真伪提醒
> 表中数字需分级看待：**Claude `cache_control` 是 Anthropic 真实提供的 API 功能，prompt caching 确实能大幅降低重复长前缀的输入费用**——但「降本 80%」是依赖缓存命中率与前缀占比的场景值，不是保证值（实际取决于你重复用了多少缓存、缓存读取本身也有计费）。「相似度 >0.98 直接返回」的阈值是工程经验值，设太松会返回答非所问的缓存——语意缓存是把双刃剑，命中率与正确率要自己权衡。SGLang「命中率与吞吐超越 vLLM 数倍」是特定 benchmark 的结果，换负载未必成立。**这二十个方向本身都是真实存在的论文、工具或方法论；要打折的永远是那些精确到个位数的百分比。**

> 🔍 高端点评 — 这张地图最大的价值，不是二十个技巧，而是那条「该谁去打」的分群线
> 把二十个方向背下来，是助理工程师的本事；看懂它们**为什么被分成 1–6 / 7–12 / 13–20 三层**，才是架构师的视野。这条分群线藏着一个关于组织与技术的深刻判断：**这三层需要的，是三种根本不同的人，动三种根本不同的东西，承担三种根本不同的风险。** 算法层（1–6）动的是模型的数学结构，做错了模型会掉点、会变笨，风险在「能力」，所以必须由懂 attention 数学、会跑大规模消融实验的研究工程师来扛——一个 infra 工程师再强，也不敢去改 MLA 的隐空间维度。基础设施层（7–12）动的是引擎选型与压测，做错了系统会 OOM、会吞吐崩盘，风险在「稳定性与成本」，需要的是能读懂 CUDA kernel、会做压力测试、会看 P99 尾延迟的系统工程师。架构方法论层（13–20）动的是跨服务的整体布局，做错了整个集群会在「缓存放错位置」「prompt 格式不确定」这类**没有单点故障、却处处在漏电**的问题上慢性失血，需要的是能在白板上同时看见路由、分层、格式规范、商业计费的架构师。**最致命的错配，是让错的层的人去打错的仗**——让架构师去调 MLA 的低秩维度（他不懂消融），或让算法工程师去设计集群路由（他不懂热点与负载），都会出事。**而真正顶尖的技术 leader，价值恰恰在这条分群在线：他不需要亲手做完二十件事，他需要的是准确地判断『这个方向是模型问题、系统问题、还是架构问题』，然后把它交给对的小组，再让三个小组的成果在 co-design 的白板上对齐（情境 2 那道墙）。** 知道一件事该谁做，比知道怎么做，是更高端、也更稀缺的能力——这二十个方向的分层，与其说是技术清单，不如说是一张**组织如何分配认知的地图**。

## 本章小结

- **内核转变**：单机，缓存是「要省的显存」；集群，缓存是「要调度的战略资产」——它有位置、有亲和性、有迁移成本，跟算力一样是会让 SRE 半夜被叫醒的东西。
- **情境 1　三级异构金字塔**：L1 HBM（active decode）/ L2 DDR5（暂不活跃多轮历史）/ L3 NVMe（冷数据），把 OS 虚拟内存搬到 KV Cache；灵魂是**异步预取 + Overlapping**——用别人的 decode 时间掩盖我的搬运延迟。
- **情境 2　结构层瘦身**：MHA → MQA（砍到 1 组 KV，太伤能力）→ GQA（分组共享，业界主流，Llama 3）→ MLA（DeepSeek 低秩压缩进隐空间 c_t，在线解压，KV 大砍）。关键是**算法-基建 co-design**——拆掉两个团队之间的墙。
- **情境 3　缓存感知路由**：Round-Robin 在有状态推理里是灾难（同用户散落多节点、重算 prefill）；正解 = 前缀哈希（MurmurHash3）+ 亲和性路由 + 投机式缓存迁移（重算 vs RDMA 拉取的精算）。暗礁：**缓存亲和与负载均衡天生反目**，需带负载感知的动态再平衡。
- **情境 4　PD 分离**：Prefill 是 compute-bound、Decode 是 memory-bound，硬件需求相反；Chunked Prefill 是同机时间切片（治标），**PD 分离是跨机空间隔离（治本）**——承认两者不合，劈成两种专用节点，RDMA 推 KV。配 XLA 算子融合（消灭 HBM 往返）+ 固定 block size（消灭动态形状重编译）。
- **情境 5　二十大军械库**：算法层 1–6（MLA / H2O / StreamingLLM / Tree-KV / Mamba 混合 / KV-Quant）归研究工程师；基建层 7–12（vLLM / SGLang / LMDeploy / TensorRT-LLM / LightLLM / DeepSpeed-FastGen）归 infra；架构层 13–20（路由 / 分层 / Chunked Prefill / 动态淘汰 / 确定性格式 / 语意缓存 / PD 分离 / Claude `cache_control` 商业化）归架构师。**分群线比清单本身更值钱**——知道一件事该谁做，比知道怎么做更稀缺。
- **真伪纪律**：MLA、H2O、StreamingLLM、KV-Quant、vLLM、SGLang、Claude `cache_control` 等全是真实工作；要打折的永远是「砍 90%、降本 80%、超越数倍、Jupiter 1.6 Tbps」这类精确数字与「某顶尖实验室内部如何做」的宣称。

这二十个方向只是种子。把它们连同前十一章散落各处的缓存技巧——从第 3 章 UMA 零拷贝、第 5 章 KV Cache FP8 量化、到本章的三级金字塔与 MLA——收拢成一张可随手翻查的速查表，正是**附录 G「缓存技术速查」**要做的事：内核概念、显存压缩、推理引擎、架构方法论、底层算子、语意缓存、命中率避坑，外加一块把「砍 90%、降本 80%、Jupiter 1.6 Tbps」这类数字单独钉住的真伪校准区。

## 终章：三个乐章，一根脊椎

军械库关上门，整本书也走到了尽头。回望这趟旅程，它其实只问了一个问题，问了三遍。

第一乐章，我们问「**怎么造它**」——在那道反蒸馏的盾下，合法地蒸出一个只属于你、跑在你桌上的本地模型。第二乐章，我们问「**怎么用它**」——让这个又小又便宜的模型长出眼和手，去看懂、去操作一台为人类设计的电脑，再擡头质疑这门技艺所站的整个范式。第三乐章，我们问「**怎么养它**」——当这双眼手要从一个 demo 放大到万级并发，去打那场最隐形的内存与缓存之战，好让「廉价」这个承诺在规模上不跳票。

造它、用它、养它。你会发现这三件事根本分不开：你在第三乐章为之厮杀的每一 GB KV Cache，省下来都是为了让第一乐章那台 DGX Spark，能撑住第二乐章那双永不阖眼的眼睛。**这不是三个主题钉在一起的合辑，是同一个工程理想——「一个真正属于你、听你的、便宜到能当器官、强到能养得起的智能」——被从三个角度逼问到底。**

全书反复打磨的，其实也只有两样东西。一把**真伪的尺**：在那一连串诱人的名字（Mythos、Qwythos、Aluminium OS、Jupiter 1.6 Tbps、78.4 分）面前，本能地分清「哪些是我能验证的、哪些是别人要我相信的」——这条线，划开了会思考的工程师和会复述的传声筒。一条**清白的路**：技术从来中性，分水岭在意图与授权；蒸馏你有权使用的模型、用合法的数据、把它养在自己的机器上，是工程；绕过别人的盾去偷一个影子，是攻击。

技术会过时，这本书里的每一个型号、每一个分数、每一个发表会的名字，明年都可能无人记得。但那把尺和那条路，会比任何模型活得更久。
