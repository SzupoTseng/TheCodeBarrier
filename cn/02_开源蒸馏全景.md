## 情境 1　第一梯队：在窄窗里刮下的数据，与它铸成的模型

**背景**：承上一章，社群只有极短窗口采集 Fable 5。坊间流传的**第一梯队**，是用那批数据最先炼出的成品。

**这份传闻清单与技术解剖**：

| 模型 | 基底（Base）| 蒸馏方法 | 据称特性 |
|---|---|---|---|
| **Qwable-v1** | Qwen3.6-35B-A3B | SFT（指令微调）| 继承 Fable 5 工具调用（Tool Surface），能输出 `str_replace_editor` 格式 |
| **GPT-OSS 120B Fable-5 Distilled** | GPT-OSS 120B | 蒸馏 + GGUF 转换（AutoTrust AI Lab 提供算力）| MoE，128 专家／激活 4；GGUF Q8_0、Q5_0 |
| **Jackrong 系列**（如 27B Coder）| Qwen 系 | 轨迹反转（Trace Inversion）| 社群「最受信任」，减少过拟合 |
| **数据集 armand0e/claude-fable-5-claude-code** | —— | 去隐私化轨迹采集（与 Glint-Research 合作）| Fable 5 原始 Agent Traces，可转 OpenAI 格式 |

**把技术内核拆开看**：

- **`Qwen3.6-35B-A3B` 的命名玄机**：`A3B` 是 Qwen MoE 的标记——**总参 35B、每次只激活约 3B**。选它当基底不是随便挑的：MoE 的「激活参数小」特性，刚好对应第 3 章 DGX Spark 的带宽瓶颈。**选基底是在选一具与你硬件兼容的骨架，不是选参数量最大的那个。**
- **`str_replace_editor` 是什么**：这是 Claude 系列在 Agent 场景用来改文件的**原生工具标签**（text editor tool）。Qwable-v1 号称「继承工具调用能力」，意思是它在权重层面学会了**主动吐出与官方一模一样的工具调用 XML**——这对接 OpenHands、Aider 这类开源 Agent 时，不必再写一层 prompt 转译，出错率低。**蒸馏者这次最想偷的，不是「会聊天的 Claude」，是「会调工具的 Agent 版 Claude」。**
- **采集→数据集→模型的工具链**：`armand0e/...` 这种「轨迹数据集」是整条链的源头。标准流程是——采集的原始 traces（常是 Claude 的 messages 格式）→ 用脚本转成 **ShareGPT / OpenAI `messages` 格式** → 喂进 **Axolotl** 或 **LLaMA-Factory** 的 SFT 配置 → 用 `peft` 上 LoRA 或全参数微调。**「去隐私化（de-PII）」这一步**通常靠 **Microsoft Presidio** 或 spaCy NER 把密钥、信箱、人名洗掉——这在合法数据处理里是必修，在非法采集里则兼具「抹除来源证据」的副作用（见第 6 章）。

> 💡 君之一席话
> 与君一席话：**模型是凝固的数据，数据是流动的模型——真正的能力转移发生在数据层，模型只是它的一种快照。** 看懂这句，你就不会再执著于「下载哪个蒸好的模型」，而会去问「它是用什么数据、什么格式、什么方法炼的」。同一批 traces，今天灌进 Qwen、明天灌进 Llama、后天灌进你自家架构，出来的是三个不同身体、同一个灵魂。谁掌握了数据与配方，谁就能无限拷贝影子；只拿到成品的人，永远只是终端消费者。

> 🔍 高端点评──SFT 蒸馏的天花板，与它为何「形似神不似」
> 第一梯队几乎清一色用 **SFT（串行级蒸馏）**，这是被 API「只给文本、不给 logits」逼出来的选择，但它有一道结构性天花板。SFT 教的是「**模仿教师的最终输出串行**」，学生学到的是 Claude 的**文体与表面套路**（会吐 `<thinking>`、会用那套 XML），却学不到 Claude 在每个 token 上「**为什么选这个词、其他词差多少**」的软标签分布——而那层几率分布（暗知识）才是推理能力的真正载体（详见第 4 章）。这就是为什么这类蒸馏模型常被吐槽「**形似神不似**」：复杂逻辑一深就露馅，因为它在模仿「答案长什么样」，而非「怎么想出答案」。要突破这层天花板，方法论上有三条高端路：① **on-policy 蒸馏**（MiniLLM／GKD，让学生先生成、教师再评分）；② **拒绝采样 + 过程奖励**（rejection sampling 筛出高品质 CoT）；③ **logit 对齐**（若能拿到局部 logits，做温度 KL）。第一梯队之所以只到「形似」，正因为窄窗 + 防护盾让后两条路成本爆炸——**它们不是不懂方法论，是被经济账逼回了最便宜的 SFT。** 顺带给你一个「真蒸馏长什么样」的对照组：这条光谱上**真实存在、可下载验证**的标竿是 `DeepSeek-R1-Distill` 系列——DeepSeek 把 R1 的长推理轨迹蒸进 Qwen2.5 与 Llama 3 的 1.5B～70B 基底，权重、model card、基底血缘全部公开，拿它的 `config.json` 跟对应基底一比就完全对得上（情境 5 教你怎么比）；反观传闻中的 `Fable-5 Distilled`，你连基底的 commit SHA 都查不到——**有没有户口、查不查得到基底，就是「真蒸馏」与「贴牌传说」最素朴的分界。**

## 情境 2　被神话的「平民版 Claude」：Qwythos-9B

**背景**：声量最大的一款，是传闻中据称由 **Empero AI** 发布的 **Qwythos-9B-Claude-Mythos-5-1M**。行销话术极具煽动性：「Claude 被开源了？无审查、104 万上下文、4GB 显存可跑！」（出处标题见附录 A：零度解说、Interfaze、Threads）。

**坊间流传的规格与技术**：

- **基底**：Qwen3.5-9B，做**全参数微调（Full FT）**，喂「Claude 5 系列数据」。
- **1M（104 万）上下文**：靠长上下文扩展技术（见下文）。
- **无审查（uncensored）**：移除拒绝机制。
- **4GB 显存可跑**：靠 GGUF 量化（如 Q4_K_M）。

衍生与预告：

- **richardyoung/qwythos-9b-abliterated**（Ollama 上架）：第三方用 **abliteration** 做二次「彻底去审查」的变体。
- **Qwythos-27B**（Empero Roadmap 预告）：据称下一代更大的「Mythos 级」蒸馏模型。

**把「无审查」与「长上下文」的技术拆开**：

- **abliteration 到底做了什么**：这是真实技术（Arditi 等人的 "refusal direction" 研究）。做法是——拿一批「有害提问」与「无害提问」分别跑模型，取两组在某层残差流的**平均激活差，得到一个「拒绝方向」矢量**；再把模型权重对这个方向做**正交化（orthogonalization）**，抹掉模型「想说不」的那条回路。工具上常用 **TransformerLens** 做激活分析、或现成的 `refusal_direction` / `heretic` 流程。**名字里的「heretic」就是 abliterated 基底的代号**（情境 3 的 FableVibes 卡片写明「started as Qwen3.6-35B-A3B-heretic」，正是这条技术线）。
- **1M 上下文怎么来**：9B 原生不可能有 1M 窗口，必然是**事后外推**——主流是 **YaRN**（NTK-aware 的 RoPE 内插，把 RoPE base 从 1 万级拉到百万级）配 **FlashAttention** 撑住长串行的内存。问题在于：**「能外推到 1M」与「在 1M 上读得懂」是两回事**——下一章用硬件真相戳破。想当场分辨它是真做了外推还是纯吹，最快的一招是打开 `config.json` 看 `rope_theta`（RoPE base，Qwen2.5 缺省 100 万、Llama 3 为 50 万）与 `rope_scaling`（YaRN 会在这里写明 `type: yarn` 与 `factor`）：**真上了长上下文，这两个字段一定留指纹；只改 README、config 原封不动的，几乎都是贴牌**（情境 5 详解这套鉴识）。

> ⚠️ 真伪提醒
> 「9B + 4GB 显存 + 1M 上下文 + 无审查 + 约等于 Claude」这组合**违反工程常识**。abliteration、YaRN 都真实，但把它们叠在「Qwythos-9B」这个具体模型上并宣称接近 Claude，属未核实的行销叙事。

> 💡 君之一席话
> 与君一席话：**「去审查」从来不是免费的，它的帐单写在你看不到的地方。** abliteration 抹掉「拒绝方向」时，连带会钝化模型的判断力与安全直觉——它不只「敢回答敏感问题」了，它可能连「这段程序有 SQL 注入、我该提醒你」这种**有益的拒绝**也一并忘了。研究已反复证实：过度 abliterate 会拉低模型在推理与事实性基准上的表现。所以当你被「无审查」三个字吸引时，先想清楚你要的究竟是「自由」，还是「一个不再对你说不、也不再替你把关的工具」——前者是能力，后者是失能，而它们长得一模一样。

> 🔍 高端点评──小模型「装得下」与「扛得动」的鸿沟
> Qwythos-9B 的神话，本质是混淆了三个被行销刻意焊在一起、其实独立的维度，而这三者背后是三套不同的工程瓶颈。**「装得下 1M」** 是 RoPE/YaRN 的数学外推问题——改几个参数就能宣称达成，几乎零成本。**「扛得动 1M」** 是 KV Cache 的内存与预填充（prefill）算力问题——9B 模型在 20 万 token 时，KV Cache 会膨胀到吃掉你以为省下的所有显存，且 prefill 一篇长文可能要等好几分钟，这是带宽与容量的硬墙。**「读得懂 1M」** 是模型容量与训练数据的能力问题——9B 的注意力头数与层数，决定了它的「大海捞针（Needle-in-a-Haystack）」检索力天生不如 35B／120B，窗口再长也只是「看得到、记不住」。**行销只卖第一个维度，因为它最便宜、最好吹；工程师必须亲手验证后两个，因为它们才决定这 1M 是真本事还是装饰。** 验证工具也很现成：跑一遍开源的 RULER 或 Needle-in-a-Haystack 测试，真假立判。

## 情境 3　不追最新：前代 Claude 的成熟蒸馏品

**背景**：坊间还有一条补充的线索——若觉得 9B 推理不够、或不迷信「最新的 Mythos」，社群里还有几款基于**前代 Claude（Opus / Claude 4.6 世代）**数据蒸馏的成熟模型。

| 模型 | 基底 | 方法 | 据称定位 |
|---|---|---|---|
| **Qwen3.6-14B-A3B-FableVibes**（tvall43 GGUF）| Qwen3.6-35B-A3B「heretic」剪枝版 | Pruning + QLoRA 补偿 | 小体积保留「Claude 感」；用 Fable 5 多步推理轨迹 + Opus 推理数据 |
| **clzoro/Qwen3.5-27B-Claude-distill** | Qwen3.5-27B（密集）| 全参数微调，喂 Opus 高品质 CoT | 数学、逻辑、多轮对话接近官方 |
| **Dzluck/Qwen3.5-2B-Claude-4.6-Opus-Reasoning-Distilled**（含 GGUF）| Qwen3.5-2B | Claude 4.6 长文推理蒸馏 | 手机／树莓派等边缘设备的极限尝试 |

> ⚠️ 真伪提醒
> 上表三款（**FableVibes、clzoro、Dzluck**）的模型名、上传者、规格与「Opus／4.6 蒸馏」血缘，本书一律无法核实。其背后的技术手法（pruning、QLoRA、密集 vs MoE、2B 边缘蒸馏）全部真实且常见；但「某人把 Opus 的 CoT 蒸进某个 27B」这种**具体血缘声称**，在你按情境 5 的方法验过 `config.json` 与基底权重之前，只能当成 README 上的一句广告词。

**技术看点**：

- **FableVibes 的「剪枝 + QLoRA 补偿」是一套完整方法论**：先对 35B MoE 做**网络剪枝（pruning）**剃掉低活化专家/神经元缩小体积，剪枝必然损伤能力，于是再用 **QLoRA**（NF4 4-bit 量化基座 + LoRA 低秩适配 + paged optimizer，Dettmers 等人的方案）做「补偿训练」把掉的分数补回来。它的基底卡片直言「started as Qwen3.6-35B-A3B-**heretic**」——印证了 abliterated 基底（情境 2）正是这条产线的上游。
- **密集 vs MoE 的取舍**：clzoro 用的是 **27B 密集（dense）**架构，全参数微调喂 Opus 的 CoT。密集模型每 token 要读全部权重，**慢但「思考连贯性」往往更好**（不会像 MoE 那样在专家间跳变导致逻辑反刍）——这是它「文学创作、角色扮演强」的结构原因。
- **2B 边缘蒸馏的意义**：把 Claude 4.6 的长文推理蒸到 2B，是「**能力密度**」的极限实验。它注定智商降级，但证明了一件事：**推理能力可以被部分压进极小的容器，只是压缩比越高、损耗越大。**

> 💡 君之一席话
> 与君一席话：**在蒸馏的世界里，「最新」往往等于「最少、最不稳、最踩线」，而「前代」常常等于「最多、最稳、最可复现」。** 最新模型的数据被防护盾污染、被窄窗掐断、来源最敏感；前代模型（Opus／4.6）没有那道即时侦测之盾，数据采集充分、量大、配方公开，蒸出来反而更成熟。追逐尖端是人性，但工程的成熟度往往藏在「上一代」里——**真正的高手选型，看的不是发布日期，是数据的充分度与可复现性。**

> 🔍 高端点评──剪枝、量化、蒸馏：三把刀如何协同压缩一个天才
> FableVibes 这条产线，示范了模型压缩的「组合拳」，值得拆解它们的分工，因为很多人会混为一谈。**蒸馏（Distillation）** 是「换灵魂」——把大模型的知识迁移到小模型，改变的是**权重学到的内容**。**剪枝（Pruning）** 是「切肉」——移除冗余的神经元/专家/注意力头，改变的是**模型的结构与参数量**，分结构化（整层/整头剪，对硬件友善）与非结构化（单权重剪，压缩率高但难加速）。**量化（Quantization）** 是「降精度」——把权重从 16-bit 压到 4-bit，改变的是**每个参数的比特数**，不动结构也不动学到的内容。三者正交、可叠加：先蒸馏灌入能力、再剪枝缩小骨架、最后量化压进显存。但叠加有顺序讲究——**剪枝后必须微调补偿（QLoRA 补回掉的分数），量化最好放最后且要用校准数据（imatrix）**，否则三刀齐下，切出来的不是轻量天才，是轻量智障。FableVibes 的「heretic 剪枝 → QLoRA 补偿 → GGUF 量化」正是教科书式的正确顺序。三把刀各有成熟、叫得出名字的工具：剪枝有 **SparseGPT / Wanda**（一次性结构化剪枝，不需重训即可剃权重）、补偿有 **QLoRA / LoRA**（Dettmers 等人）、量化有 **GPTQ / AWQ**（GPU 推理）与 **llama.cpp 的 k-quants + imatrix**（CPU/边缘，靠校准数据算重要性矩阵）。**能叫得出工具名，你才分得清一张 model card 是真的做了这套组合拳，还是只把别人的 GGUF 换了个封面、粘贴「Claude distill」——后者连 imatrix 来源都拿不出。**

## 情境 4　真正的硬通货：两个公开数据集

**背景**：最后一点往往被忽略——若你懂技术、想用自己的 Llama 或 Mistral 蒸出专属「Claude」，**重点不是下载别人的模型，是拿原始数据集**。

| 数据集 | 内容 | 蒸馏目标 |
|---|---|---|
| **armand0e/claude-fable-5-claude-code** | Fable 5 原始 Agent 运行轨迹（去隐私化，与 Glint-Research 合作）| 程序/工具调用能力 |
| **ox-ox/mythos-character-distillation** | Claude 的「说话风格与心理特质」文风数据 | 人格/文风 |

**`ox-ox/mythos-character-distillation` 的内容耐人寻味**——传闻中它专门蒸馏这些特质：

- **自发的元认知（spontaneous meta-awareness）**、自动发现结构陷阱；
- **拒绝使用 AI 罐头回复**（例如不说 "I'm just an AI" / "As an AI..."）；
- **自我修正、不找借口**。

这说明蒸馏者想偷的东西，已经从「知识」「程序能力」细分到了**「人格与对话气质」的层次**。技术上，这类「文风蒸馏」不靠知识正确性，而靠**偏好对齐**——常用 **DPO（Direct Preference Optimization）** 或拒绝采样，让模型学会「在两种说法里，挑更像 Claude 的那种」。（这类偏好对齐真实、公开、可查的原料，是 **UltraFeedback、HH-RLHF、Nectar** 这类偏好数据集——同样有户口、有授权，与来路不明的 `mythos-*` 不是一个量级的可信度。）

**为什么数据集才是内核**：有了原料，你能蒸任意身体的学生——这正是话题总会反复导向「我可以教你怎么用这些数据集微调 Llama / Qwen」的原因。

> ⚠️ 真伪提醒
> 「用未授权 API 响应数据训练商业开源模型，**严重违反 Anthropic 服务条款**」——这一点原始素材自己也写明。它还提到「美国科技界正推动将特定大规模蒸馏攻击定为刑事犯罪」。无论这些数据集是否真实存在，**取得与使用来路不明的「商业模型轨迹数据集」，本身就踩在 ToS 与法律红在线**。详见第 6 章。

> 💡 君之一席话
> 与君一席话：**模型可以开源，但「来源的合法性」不会因为开源而消失——它只是被藏进了权重里，等着有一天被验出来。** 一个 abliterated、量化、改了名的蒸馏模型，表面上干净得像原创，但它的数据出身、它权重里可能残留的水印、它训练轨迹的 commit 历史，都是泼不掉的指纹。下载一份「claude-traces」数据集，就像收下一块来路不明的金条——它或许真能用，但每次动用，你都在赌没人能证明它的出处。**开源洗不白来源，它只是让来源更难追、却也更值得追。**

> 🔍 高端点评──监管的战场正从「模型」转移到「数据」
> 这一节藏着整个蒸馏攻防最深的一层转向，值得每个从业者警觉。封锁一个蒸馏好的开源模型，几乎不可能——它已是开放权重，泼出去的水收不回，改个名、做次 abliteration、重新量化，就是一个「新」模型。但**追查一个「未授权商业轨迹数据集」的来源与传播链，相对可行得多**：它有上传者帐号、有 commit 时间轴、有文件 hash、甚至有教师输出残留的水印统计特征。所以理性的监管与防守，注意力必然从「抓模型」转向「溯数据」——这也解释了防守方为何要在输出端打水印（第 1 章）：水印的价值不在于即时挡住采集，而在于**事后能在某个衍生模型或数据集里，验出「这是从我这里流出去的」**。对从业者的启示极其实际：**你下载的每一个 `claude-*-traces` 数据集，都可能是一份带法律放射性的原料；碰它之前，先查它的出身、授权与上传者信誉——这道尽职调查，比任何技术配方都重要。**

## 情境 5　验明正身：怎么一眼识破「套壳改名」

**背景**：回到开篇那张记事本——你抄下一排「Fable-5」「Claude-Mythos」，现在要回答最实际的问题：**哪些是真蒸馏、哪些只是把别人的权重换个封面再喊「Claude distill」？** 好消息是，模型能造假的地方比你想的少得多。一个开放权重模型的「出身」会诚实地刻在几个文件里——README 可以吹，但 `config.json`、tokenizer 与 safetensors／GGUF 的指纹骗不了人。这一节给你一套**多数步骤不必下载完整权重、几分钟就能跑完的鉴识流程**。

**第一刀：`config.json` 指纹——模型的出生证明。** 每个 HF 模型根目录都有一份 `config.json`，里面的 `model_type`、`architectures`、`vocab_size`、`num_hidden_layers`、`num_attention_heads`、`num_key_value_heads`（GQA 线索）、`intermediate_size`、`rope_theta`、`rope_scaling`、以及 MoE 的 `num_experts` / `num_experts_per_tok`，共同构成一组几乎无法伪造的骨架。把它跟下面这张**真实开源基底家族的典型指纹**对一遍，基底是谁，藏不住：

| 基底家族 | `model_type`／`architectures` | `vocab_size` | 结构线索 | 典型 `rope_theta` | 授权性质 |
|---|---|---|---|---|---|
| **Qwen2.5 / Qwen2** | `qwen2` / `Qwen2ForCausalLM` | 151,936 | GQA（KV head < attn head）| 1,000,000 | 多数 Apache-2.0（部分大尺寸走 Qwen 研究授权）|
| **Qwen3 / Qwen3-MoE** | `qwen3` / `qwen3_moe` | 151,936 | MoE 有 `num_experts`／`num_experts_per_tok` | 1,000,000 起 | Apache-2.0 |
| **Llama 3.x** | `llama` / `LlamaForCausalLM` | 128,256 | GQA、`rope_scaling.type=llama3` | 500,000 | Llama 社群授权（**非纯开源**，禁某些再分发）|
| **Mistral 7B / Mixtral** | `mistral` / `mixtral` | 32,000 或 32,768 | Mixtral 为 8×7B MoE | 1,000,000（Mixtral）| Apache-2.0 |
| **Gemma 2** | `gemma2` | 256,000 | 滑动窗口注意力交替、logit softcapping | 10,000 | Gemma 使用条款（**非纯开源**）|
| **GPT-OSS** | `gpt_oss` | ~20 万（o200k 系）| MoE、原生 MXFP4 权重 | — | Apache-2.0 |
| **Phi-3 / 3.5** | `phi3` | 32,064 | 小而密集 | 10,000 | MIT |

> ⚠️ 真伪提醒
> 上表是「**典型**指纹」，**具体数值会随版本微调**（Mistral v0.1→v0.3 的 vocab 从 32,000 变 32,768、Llama 3.0→3.1 的 `rope_scaling` 才加上 YaRN 式外推）。用它作「缩小嫌疑范围」的筛子，而非「精确到个位数」的判决书——对不上才该起疑，对上了再往下验。

**第二刀：tokenizer 指纹——换不掉的口音。** 模型可以改名、可以微调，但**换 tokenizer 等于重训**，蒸馏／改名者几乎不会动它，于是它成了最稳的胎记。看 `tokenizer_config.json` 与 `tokenizer.json`／`tokenizer.model` 的特殊 token：Qwen 系是 **ChatML**（`<|im_start|>` / `<|im_end|>`，byte-level BPE，vocab 约 15 万）；Llama 3 是 **tiktoken 风格**（`<|begin_of_text|>` / `<|eot_id|>` / `<|start_header_id|>`，vocab 128,256）；Mistral 是 **SentencePiece**（`[INST]` / `[/INST]`，`<s>`，vocab 3 万出头）；Gemma 是 **SentencePiece 256k**（`<start_of_turn>` / `<end_of_turn>`）。**一个自称「全新 Claude 蒸馏架构」的模型，若 tokenizer 是一字不差的 Qwen ChatML，它的底就是 Qwen，没有别的可能。**

**第三刀：权重与量化的物理凭证。** safetensors 的 `model.safetensors.index.json` 会列出 `total_size`，除以每参数字元数即可反推真实参数量——**「120B」喊得响，index 却只有 14B 的张量，当场拆穿**；张量命名（`model.layers.N.self_attn.q_proj`、MoE 的 `mlp.experts.N`）也透露架构家族。GGUF 量化档则用 `gguf-dump`（或 `llama.cpp` 加载时印出的 KV）摊开元数据：`general.architecture`（会老实写 `qwen2` / `llama`，封面叫「Claude-Mythos」也藏不住底层）、`general.name`、`tokenizer.ggml.model`、`general.quantization_version`。**一个量化档若拿不出基底 commit SHA、拿不出 imatrix 校准来源，它的品质与血缘就都是无据之词。**

**第四刀：授权血缘与 model card 的谎言。** HF model card 的 YAML front-matter 有 `base_model:`、`license:`、`datasets:` 三个关键字段——**缺哪个，哪个就是它不想让你问的问题**。最强的红旗是**授权自相矛盾**：拿 Llama 3 或 Gemma 改的模型却标 `license: apache-2.0`，等于白纸黑字承认它要嘛没读基底授权、要嘛蓄意洗白（Llama／Gemma 都**不是**纯开源，再分发有条款约束）。其余常见破绽整理如下：

| README 怎么说 | 该去验的可验证信号 | 对不上代表什么 |
|---|---|---|
| 「全新架构 / 自研蒸馏」| `config.json` 的 `model_type` + tokenizer 特殊 token | 其实是某开源基底改名 |
| 「120B 巨型 MoE」| `index.json` 的 `total_size`、`num_experts` | 参数量灌水或根本不是 MoE |
| 「1M 上下文」| `rope_theta` / `rope_scaling` / `max_position_embeddings` | 只改 README，没做 YaRN |
| 「Claude/Opus 蒸馏」| `datasets:` 字段、训练数据揭露、基底 SHA | 血缘无法佐证（多半也无法佐证**反**之）|
| 「无审查」| 任一 RULER／lm-eval／拒绝率测试数字 | 只有形容词、没有量测 |
| 「高品质 GGUF」| imatrix 来源、基底 commit、量化版本 | 来路不明、可能被污染 |

**最关键、也最反直觉的一条限制必须讲白**：上面四刀能**证伪**「这是全新架构」、能**锁定基底是谁**，却几乎**无法证实**「它真的是用 Claude 的输出蒸的」。基底血缘写在权重结构里，**训练数据的血缘却不写在权重里**——你能断定「底是 Qwen3」，却很难从一堆浮点数反推「这些权重的监督信号来自 Claude」。要回答后者，得靠**行为探针**（问它身份、用金丝雀字符串、比对特定罕见 token 的几率分布）与**输出端水印**（第 1 章）。这正好呼应情境 4 的判断：**监管与鉴识的重心，终究会从「模型」移到「数据」——因为模型的底骗不了人，但模型的「师承」，恰恰是最容易被洗白、也最值得追的那一层。**

> 💡 君之一席话
> 与君一席话：**权重不会说谎，会说谎的是 model card——当宣传与 `config.json` 起冲突，永远信 config。** 一个老练的选型者打开一个陌生模型，第一个点开的不是 README 的跑分截屏，是那份不到 1KB 的 `config.json`：`model_type` 告诉他底是谁、`vocab_size` 对不对得上、`rope_theta` 有没有为了那句「百万上下文」而动过。**这套习惯的本质，是把「相信别人怎么说」换成「自己去读它是怎么长的」——而这恰恰是工程师与消费者最深的分野。** 消费者看封面、看下载数、看「无审查」三个字就心动；工程师看出生证明、看胎记、看授权血缘，因为他知道：**在一个谁都能上传、谁都能改名的开放权重世界里，唯一不会背叛你的尽职调查，是那几个改不动的底层文件。** 你验证的不是某一个模型，你练的是一种「不被名字唬住」的眼力。

> 🔍 高端点评 — 开放权重的供应链安全：你下载的不只是模型，是一条没人签名的食物链
> 把这套鉴识往上拔一层，会看见一个被严重低估的领域：**开放权重的供应链安全（model supply-chain security）**。今天你 `from_pretrained("某人/某模型")` 一行，背后其实吞下了一整条无人背书的食物链——基底是谁改的、用什么数据、谁量化的、imatrix 哪来的、有没有人在权重里埋了后门（是的，**权重级后门**是真实研究议题，恶意微调可以让模型在特定触发词下行为异常）。成熟的做法，是把「装模型」当成「装一个来路不明的二进位依赖」一样对待：**读 `config.json` 与 tokenizer 确认基底（用 `transformers.AutoConfig`、`huggingface_hub` 拉 metadata 而不必下整包权重）、用 `gguf-dump` / `safetensors` header 验物理规格、核对 `base_model` 与 LICENSE 的血缘是否自洽、优先选有 commit 历史与可复现训练配方的上传者**（Tülu、OLMo、DeepSeek-R1-Distill 这类「全链公开」的模型，价值正在于此）。HF 近年推的 model-lineage（模型族谱）、safetensors 取代可任意运行代码的 pickle、以及社群对 imatrix 与量化来源的标注要求，都是这条供应链在补课。**对从业者最实际的一句话：在这个影子模型满地跑的生态里，你最该培养的内核能力，不是「会微调」，而是「会验货」——因为造一个影子只要几小时，而一个验不出赃物、验不出后门的工程师，迟早会把一条带毒的依赖，亲手接进公司的生产线。**

## 本章小结

把全景收拢成一张认知地图：

- **三条获取路径**：① 下载蒸好的模型（最快、品质与来源最不可控）；② 用前代成熟蒸馏品（稳定、可复现）；③ 拿原始数据集自蒸（最自由、最接近红线）。
- **最想被偷的能力**：从「聊天」→「**工具调用 / Agent 程序力**」→「**人格与文风**」三级进化。
- **方法论光谱**：SFT（最便宜、形似神不似）→ QLoRA 补偿（救剪枝）→ 全参数微调（最彻底）→ on-policy/DPO（最接近神似、最贵）。
- **工具链**：采集→`datasets`/Presidio 清洗→Axolotl/LLaMA-Factory/Unsloth 微调→`peft` LoRA→llama.cpp GGUF 量化→Ollama 部署；abliteration 靠 TransformerLens/heretic。
- **验明正身四刀**：`config.json`（`model_type`／`vocab_size`／`rope_theta`／MoE 字段）→ tokenizer 指纹（ChatML／llama3／SentencePiece，换不掉的口音）→ safetensors `index.json` 参数量与 GGUF `general.architecture` → LICENSE 血缘与 model card `base_model`。**能对上基底，戳破套壳改名；但「是否真用 Claude 输出蒸的」单靠权重验不出**，得靠行为探针与输出端水印（呼应第 1 章、情境 4）。真蒸馏有对照组：`DeepSeek-R1-Distill` 全链公开、config 对得上基底。
- **最大陷阱**：「无审查 + 小显存 + 百万上下文」的神话宣传——下一章用硬件真相拆解。

清单盘完，最实际的问题浮现：**这些影子该用什么机器跑、哪一只配得上那台 DGX Spark？** 翻到第 3 章，上解剖台。
