## 情境 1　两大缓存分类：推理层的 KV Cache 与服务层的 Prompt Cache

**背景**：在大语言模型的开发与部署里，**缓存（Cache）是降低推理延迟、削减计算成本的内核命脉**。但「大模型的缓存」其实是一个被混为一谈的笼统说法——它至少是两种住在不同楼层、解决不同问题、由不同角色管理的东西。把它们分清楚，是看懂后面所有优化的前提。**工程的第一步永远是分类**：你得先知道你那一百多 GB 到底是哪一种缓存吃掉的，才谈得上省。

| 维度 | 推理层 KV Cache（键值缓存） | 服务层 Prompt Cache（提示词 / 上下文缓存） |
|---|---|---|
| 住在哪 | **GPU 显存（HBM）内**，由推理引擎动态管理 | **服务端**（云端 API 后台 / 自建闸道）持久层 |
| 解决什么 | 同一次生成里，避免已生成 token 被**重复计算** | 跨多次独立请求，避免**相同前缀**被重复 Prefill |
| 复杂度收益 | 把自回归解码从 O(N²) 降到 **O(N)** | 命中后省下整段 system prompt 的 Prefill 算力 |
| 谁管它 | vLLM / TensorRT-LLM / SGLang 等引擎 | Anthropic Claude / 阿里云百炼 / 自建网关 |
| 生命周期 | **一次请求结束就释放**（除非开 prefix caching） | **跨请求存活**，可被多个会话命中 |
| 红绿灯 | 🟢 引擎自动做，但会吃爆显存（情境 4 主角） | 🟢 显式标记即省钱（本情境主角） |

**第一类，推理层的 KV Cache**。它解决的是一个非常具体的浪费：Transformer 在自回归生成时，每一步只能吐出一个新 token；如果不缓存，生成第 N 个 token 时，你得把前面 N-1 个 token 重新做一遍注意力的矩阵运算，计算量呈 **O(N²)** 平方级爆炸。KV Cache 的做法是把先前每一步算好的 Key 与 Value 矩阵**存在显存里**，后续只对新加入的那一个 token 做计算，把时间复杂度从 O(N²) 压到 **O(N)**。它住在 GPU 显存深处，由底层推理引擎悄悄管理，你平时根本看不见它——直到它把你的内存吃光。情境 2、3、4 会把它从数学到显存彻底解剖。

**第二类，服务层的 Prompt Cache**，又叫 Context Cache、上下文缓存。它解决的是另一个层次的浪费：当你对着大模型**反复输入相同的内容**——一份冗长的产品说明书、一段固定的 System Prompt 人设、一组 few-shot 范例——服务端每次都得把这段文本重新 token 化、重新算一遍 KV 状态。Prompt Cache 的做法是把这段前缀的 KV 状态**缓存在服务器端**，下次同样的前缀进来，直接复用，跳过 Prefill。

这里有一个你必须记住的关键字：**显式锚点**。Anthropic 的 Claude API 提供 `cache_control` 这个显式标记——你在请求里用它圈出「这一段请帮我缓存」，命中之后，**这段输入 token 的费用会大幅折扣**。它和「隐式缓存」（网关自动拦截、匹配完全相同的前缀）的差别在于：显式缓存把控制权交到你手上，你自己决定哪段该被当成稳定锚点。

```python
# Claude Prompt Caching：用 cache_control 显式锚定不变前缀
# （Anthropic Messages API，概念示意）
import anthropic
client = anthropic.Anthropic()

resp = client.messages.create(
    model="claude-opus-4-1",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": LONG_STABLE_SYSTEM_PROMPT,   # 数千 token 的固定人设/规则/工具说明
            "cache_control": {"type": "ephemeral"}  # ★ 显式锚点：把这段 KV 状态缓存住
        }
    ],
    messages=[
        {"role": "user", "content": user_question}  # 动态内容放最后，不破坏前缀
    ],
)
# 首次：写入缓存（cache_creation_input_tokens 计费略高）
# 后续命中：cache_read_input_tokens 以大幅折扣计费
print(resp.usage)  # 观察 cache_creation / cache_read 两栏就懂命中了没
```

> ⚠️ 真伪提醒
> 原始素材称「命中后输入 token 费用通常能打 1 到 2 折，省下高达 80%~90% 的成本」。**缓存读取显著便宜于原价，这个方向是真的、是各家 API 的卖点；但「打几折、省百分之几」是随模型、随厂商、随缓存存活时间（TTL）、随写入 vs 读取而浮动的计费参数**，请以你调用当下官方定价页的 `cache_read` 与 `cache_write` 实际数字为准，不要把任何一个固定百分比当定论。本书能担保的只有：**稳定前缀显式缓存，在「重复调用同一份长 system prompt」的场景下，省的是真金白银，量级显著**。

两类缓存的分水岭，其实是一句话能讲清的**黄金法则**：**稳定的、静态的内容放最前面，动态的、高变的内容放最后面**。因为两种缓存本质都是**前缀匹配（prefix matching）**——只要开头有一个 token 变了，后面整条缓存就作废。这条法则之所以是金科玉律，是因为新手最常犯的错，恰恰是把动态变量塞进 system prompt 开头：

| 🔴 常见错误（让缓存完全失效） | 🟢 正确改进（最大化命中） |
|---|---|
| 在 System Prompt 里塞当前时间戳、用户 ID | 把动态用户信息移到消息**最末尾的 user 栏**，开头固定 |
| 每轮对话重新生成历史摘要（summary） | 历史**只追加不修改**；要截断就从**最旧的**砍，保前缀一致 |
| 工具定义（tools/functions）顺序随机 | **严格固定** JSON 工具列表的内部排序与字段结构 |
| 文本空格、换行符细节不统一 | 统一清洗逻辑，**防一个多余的 `\n`** 改变 token 串行 |

> 💡 君之一席话
> 与君一席话：**缓存的世界里，最贵的不是「算错」，是「换了一个字节」——一个流浪的时间戳、一个多出来的换行符，就能让你精心设计的整条前缀缓存灰飞烟灭。稳定，是缓存唯一的信仰。** 你看那张错误对照表，没有一条是「算法不对」，全是「结构不稳」。这揭示了一个反直觉的真相：**Prompt Cache 的命中率，是被你的提示词工程的「纪律」决定的，不是被模型决定的。** 一个把 system prompt 写得干净、固定、把所有易变内容纪律性地赶到结尾的工程师，他的帐单可能只有一个随手在开头插用户名的工程师的零头——两人用的是同一个模型、同一份 API。**业余者优化「我问了什么」，专业者优化「我问的东西，前缀稳不稳定到能被一次次想起来」。** 缓存从不奖励聪明，它只奖励一致——这和一切高性能系统的底层美学是相通的：可预测性，本身就是一种性能。

## 情境 2　为什么是「访存受限」：Prefill 吃算力，Decode 等内存

**背景**：要从根本上理解「为什么缓存对大模型如此致命地重要」，你必须先看懂一件反直觉的事——**大模型推理的瓶颈，多数时候根本不是算力，是内存带宽**。要看懂这件事，得把推理拆成两个性格完全相反的阶段。

**第一阶段，Prefill（预填阶段）**。这是处理你输入的那段 prompt 的阶段。此时模型要同时处理你输入的**所有** token，这些 token 可以被打包成高维度的张量（Tensor），喂进一次巨大的矩阵乘法。矩阵乘法是 GPU 最爱的食物——它能把 Tensor Core 喂到饱，运算密度极高。这个阶段是典型的**算力受限（Compute-Bound）**：GPU 的算力单元在满载工作，内存带宽反而闲着。

**第二阶段，Decode（生成阶段）**。这是自回归吐字的阶段，也是真正的瓶颈所在。因为自回归的本质是**一次只能生成一个 token**，所以每一步你只有**一个**新 token 的 Query 矢量，要拿它去和过去**所有** token 的 Key/Value 做注意力。矩阵乘法退化成了「矢量 × 矩阵」——运算量极小、极快，但你必须先把那一大坨历史 KV（动辄几个 GB）**从显存 HBM 搬到芯片上的 SRAM 寄存器**才能算。于是 GPU 的大部分时间，不是在算，是在**搬**。这就是典型的**访存受限（Memory-Bound）**：算力单元大量空转，瓶颈卡在 HBM → SRAM 的搬运带宽上。

```
        Prefill 阶段                      Decode 阶段
   ┌────────────────────┐          ┌────────────────────┐
   │ 全部 prompt token   │          │ 1 个新 token        │
   │ 打包成大矩阵         │          │ × 全历史 KV 矩阵     │
   │                    │          │                    │
   │  GEMM 矩阵×矩阵      │          │  GEMV 矢量×矩阵     │
   │  Tensor Core 吃满   │          │  算得极快、搬得极慢  │
   └────────────────────┘          └────────────────────┘
   ▏Compute-Bound 算力受限▕        ▏Memory-Bound 访存受限▕
   ▏GPU 在「算」          ▕        ▏GPU 在「等内存搬运」 ▕
```

这个分裂的性格，正是 KV Cache 之所以存在的**根本理由**，也是这整章的物理基础。Decode 阶段每一步都要读全部历史 KV——如果不缓存、每步重算，那是 O(N²) 的算力灾难；缓存了，虽然省下了算，却把压力**从算力转嫁到了内存带宽与容量**。换句话说，**KV Cache 不是消灭了瓶颈，是把瓶颈从「算力受限」搬到了「访存受限」**。情境 1 你的 128GB 之所以蒸发，正是因为 Decode 阶段每个并发请求都拖着一条越来越长的 KV 尾巴住在显存里。

> 🔍 高端点评 — 为什么 GQA / MQA / MLA 全在砍同一个东西
> 一旦你接受了「瓶颈是访存、是 KV 的体积」这个前提，现代大模型架构里一整批看似无关的设计，就会突然连成一条线。标准多头注意力（MHA）里，每个 Query 头都配一个独立的 Key 头和 Value 头——KV Cache 的体积正比于头数。**MQA（Multi-Query Attention）** 走极端：所有 Query 头**共用一组** KV，KV Cache 体积瞬间除以头数，但表达力受损。**GQA（Grouped-Query Attention）** 是折中：把 Query 头分成几组，每组共用一组 KV——Llama 3-8B 的 32 个 Query 头只配 8 个 KV 头，KV Cache 直接砍到四分之一（情境 4 会看到这 8 是怎么进公式的）。DeepSeek 的 **MLA（Multi-head Latent Attention）** 更狠：把 KV 压缩成一个低秩的潜在矢量再缓存，用一点点解压计算换到极小的缓存体积。**这三者表面是注意力机制的变体，骨子里全是同一个工程意图：在 Decode 访存受限的物理现实下，KV Cache 的每一个字节都是要用显存带宽去搬的成本，所以架构师宁可牺牲一点点模型容量，也要把 KV 砍小。** 你只要记住「H 在公式里是 KV 头数，不是 Query 头数」，就抓住了过去三年注意力架构演进的主线——它们不是在追求更聪明，是在追求**更省内存**。

## 情境 3　KV Cache 的数学推导：从 O(T²) 到 O(T) 的那一步拼接

**背景**：前面讲了「为什么要缓存」，这一节把它**算给你看**。很多人对 KV Cache 的理解停在「把算过的存起来」这种口语层次，但只有把公式摊开，你才会真正看懂那个从平方降到线性的魔法，到底发生在哪一个运算符上——答案是一个你想不到的、朴素到家的操作：**拼接（Concat）**。

标准 Transformer 自回归解码时，第 t 步的缩放点积注意力（Scaled Dot-Product Attention）公式是：

```
                       ┌                  ┐
                       │   Q_t · K_≤tᵀ    │
  Attention(Q_t,       │ ─────────────── │
            K_≤t,  = softmax│      √d_k       │ · V_≤t
            V_≤t)          └                  ┘

  其中：
    Q_t  ∈ ℝ^(1 × d_k)   ← 第 t 步「新生成的那一个 token」的 Query 矢量
    K_≤t ∈ ℝ^(t × d_k)   ← 从第 1 到第 t 个 token 全部的 Key 矩阵
    V_≤t ∈ ℝ^(t × d_k)   ← 从第 1 到第 t 个 token 全部的 Value 矩阵
```

看清楚这三个符号的维度差异：**Q 只有一行**（当前 token），而 **K、V 有 t 行**（含当前 token 的全部历史）。整个 KV Cache 的故事，就藏在「K_≤t 和 V_≤t 这 t 行，每一步是重算还是复用」这个抉择里。

**如果没有 KV Cache。** 每生成一个新 token，你必须把历史的 t−1 个 token 重新走一遍 Embedding 和所有 Transformer 层的线性投影 W_k、W_v，从头算出 K 和 V。第 t 步的算量正比于 t；从 t=1 一路加到 T：

```
  总算量  =  Σ(t=1→T) O(t)  =  O(1) + O(2) + ... + O(T)  =  O(T²)   ← 平方爆炸
```

这就是那个让人绝望的 O(T²)。生成一篇两千字的文章，你为了第 2000 个字，把前 1999 个字的 K/V 重算了一遍——而前一步你才刚算过几乎一模一样的东西。

**如果有 KV Cache。** 关键洞察是：历史 token 的 K 和 V**一旦算出来就永远不变**（因为它们只依赖自己和前面的 token，不依赖未来）。所以第 t−1 步时，系统早已把 K_≤(t−1) 和 V_≤(t−1) 存在显存里了。到第 t 步，你只需要做三件小事：

```
  ① 只算当前这一个 token 的 q_t, k_t, v_t          ← O(1)，跟 t 无关
  ② 拼接更新缓存（这就是魔法发生的地方）：
        K_≤t = [ K_≤(t-1) ; k_t ]      ← 老矩阵后面接一行新的 k_t
        V_≤t = [ V_≤(t-1) ; v_t ]      ← 老矩阵后面接一行新的 v_t
  ③ 把 q_t 和拼好的 K_≤t, V_≤t 代回上面的 Attention 公式

  单步复杂度  =  O(1)（只为新 token 算）
  总复杂度    =  Σ(t=1→T) O(1)  =  O(T)              ← 线性！
```

整个从 O(T²) 到 O(T) 的降维，**没有用到任何高深数学，只用到一个 `concat`**。你把上一步算好的 K/V 矩阵留着，新 token 的 k/v 算出来往后一接，就完事了。这就是为什么 KV Cache 被称为「用空间换时间」——你付出的代价是那个越长越大的 K_≤t、V_≤t 必须**一直住在显存里**（情境 4 的 128GB 由此而来），换来的是每一步解码的算力从「重算全部历史」变成「只算一个新字」。

> 💡 君之一席话
> 与君一席话：**最深刻的工程优化，往往不是发明一个更聪明的算法，而是认出「这个东西算过一次之后就再也不会变了」——然后就只是把它留下来。KV Cache 的全部秘密，不过是『历史不会改变，所以不必重算』这句废话，被人认真对待了而已。** 你回头看那个 concat：它朴素到近乎可笑，可正是它把平方变成了线性。这背后是计算机科学最古老的一条智能——**记忆（memoization）就是用空间赎回时间**。从动态规划的备忘录、到浏览器的 HTTP 缓存、到 CPU 的 L1，再到这里的 KV Cache，全是同一个念头的不同化身：**凡是「输入不变则输出不变」的纯函数计算，重算第二遍就是一种罪。** 一个新手看到 Decode 慢，会去想「换更快的 GPU」；一个老手看到 Decode 慢，会先问一句「这里面有多少是我刚才已经算过、现在却在重算的东西？」——后者那一问，价值一整个数量级。**能在一片计算里认出哪部分是『恒定的历史』，你就掌握了所有缓存技术的母题。**

## 情境 4　显存计算器：吃掉你 128GB 的，是 KV Cache 不是权重

**背景**：现在回到情境 1 的那个谜题——权重才 16GB，128GB 怎么会爆。这一节给你一把每个架构师都该随身带的尺：**KV Cache 显存计算器**。算过一遍，你这辈子都不会再把「模型大小」和「部署所需显存」搞混。

内核公式，单用户、单 token：

```
  Size_per_token  =  2  ×  2  ×  L  ×  H  ×  D   (Bytes)
                     │     │     │     │     │
                     │     │     │     │     └─ D : 每个头的维度 (Head Dim = Hidden / Query Heads)
                     │     │     │     └─────── H : KV 头数 (用 GQA 就是 KV Heads，不是 Query Heads!)
                     │     │     └───────────── L : 模型层数 (Layers)
                     │     └─────────────────── 2 : FP16 = 2 字节 (FP8→1, BF16→2)
                     └───────────────────────── 2 : Key 和 Value 两个矩阵
```

两个 2 一定要分清楚：**第一个 2 是因为你要存 K 和 V 两份矩阵**；**第二个 2 是精度的字节数**（FP16 是 2 字节，换 FP8 就变 1，这也是「KV Cache 量化」省一半显存的原理，呼应第 5 章后处理）。H 必须用 **KV 头数**——这正是情境 2 讲 GQA 时埋的伏笔：用了 GQA，这个 H 会比 Query 头数小好几倍，是省显存的大头。

**实战：以 Llama 3-8B 为例。** 查它的官方 config：

```python
# KV Cache 显存计算器（可直接跑）
def kv_cache_bytes_per_token(layers, kv_heads, head_dim, dtype_bytes=2):
    # 2 = Key + Value 两份矩阵；dtype_bytes = 精度字节 (FP16=2, FP8=1)
    return 2 * dtype_bytes * layers * kv_heads * head_dim

def total_kv_cache_gb(layers, kv_heads, head_dim, batch, ctx_len, dtype_bytes=2):
    per_tok = kv_cache_bytes_per_token(layers, kv_heads, head_dim, dtype_bytes)
    total_bytes = per_tok * batch * ctx_len
    return per_tok, total_bytes / (1024**3)

# Llama 3-8B 官方参数
L, HIDDEN, Q_HEADS, KV_HEADS = 32, 4096, 32, 8   # ← 注意 GQA：KV 头只有 8
D = HIDDEN // Q_HEADS                              # 4096 / 32 = 128

per_tok, _ = total_kv_cache_gb(L, KV_HEADS, D, batch=1, ctx_len=1)
print(f"每 token: {per_tok} bytes = {per_tok/1024:.0f} KB")
# 2 × 2 × 32 × 8 × 128 = 131,072 bytes ≈ 128 KB / token

_, gb = total_kv_cache_gb(L, KV_HEADS, D, batch=128, ctx_len=8192)
print(f"batch=128, ctx=8K → {gb:.0f} GB")
# 128 KB × 128 × 8192 ≈ 128 GB  ← 凶手在此
```

把数字代进去，每个 token 吃掉 **128 KB** 显存。看起来不多？乘上并发再乘上上下文，几何级数立刻显形：

| 场景 | batch | 上下文 | KV Cache 总量 | 对比权重 16GB |
|---|---|---|---|---|
| 你单测那一条 prompt | 1 | 2K | ~256 MB | 🟢 微不足道，所以你没发现问题 |
| 小并发短上下文 | 8 | 2K | ~2 GB | 🟢 还好 |
| 几个长上下文请求 | 16 | 8K | ~16 GB | ⚠️ 已经和权重一样大了 |
| 中等并发长上下文 | 128 | 8K | **~128 GB** | ⚠️ 是权重的 8 倍，**OOM** |

**结论一句话：大模型推理的瓶颈，根本不是模型权重本身，而是 KV Cache。** 你那台 DGX Spark 的 128GB 统一内存，装 16GB 的权重绰绰有余——但只要并发和上下文一上来，KV Cache 就能把它整个吞掉。情境 1 的谜题至此破解：**消失的一百多 GB，是 128 个并发请求各自拖着的那条 8K 长的 KV 尾巴。** 这也解释了为什么第 5 章后处理阶段要郑重其事地做「KV Cache 量化（FP8/INT8）」——把公式里第二个 2 变成 1，长上下文的内存当场腰斩。

> ⚠️ 真伪提醒
> 上表的数字是用上面那个公式**理论推算**的干净值，方便你创建直觉。**真实部署中，实际占用还要叠加：分页的块粒度浪费（情境 5）、框架的保留缓冲、activation 暂存、以及非满载的碎片**——所以真机跑出来通常比理论值更吃紧，而不是更宽松。「128GB」这个数字之所以刚好等于某台机器的内存，是原始素材刻意挑的整数巧合，**请把它当成「同一个数量级」的示意，不要当成某机型的实测上限**。你该带走的是**方法**：拿你自己模型的 L/H/D 代进公式，乘上你的目标并发与上下文，先算出 KV Cache 的天花板，再决定买多大显存、开多大 batch——**在动手部署前，先用这把尺量一遍，是专业和裸奔的分界**。

> 🔍 高端点评 — 为什么「最大并发」是被 KV Cache、而不是被算力决定的
> 这个计算器最反直觉的推论是：**一台推理服务器能同时服务多少用户，上限通常不是它的算力（FLOPS），而是它的显存能装下多少条 KV 尾巴。** 这彻底改写了容量规划的思路。传统 CPU 服务你会算「QPS × 每请求 CPU 时间」；但 LLM 推理服务，你要先算「显存总量 ÷ 每条 sequence 的 KV Cache 峰值」——这个商，才是你的并发上限（vLLM 启动参数 `--max-num-seqs` 与 `--gpu-memory-utilization` 调的就是这件事）。于是一连串工程决策全被这把尺重新定义：**为什么长上下文服务这么贵？** 因为 ctx_len 直接线性放大每条 KV，128K 上下文的单请求 KV 可能就是 8K 的十六倍，能塞的并发数同比例暴跌。**为什么 GQA/MQA/MLA 是显学？** 因为它们砍 H，等于直接放大你的并发上限（情境 2）。**为什么 PagedAttention 是革命？** 因为它消灭了「为最大长度预留」的浪费，把实际能塞的 KV 条数逼近物理极限（情境 5）。**把这四件事连起来你会发现，现代 LLM 推理引擎的全部努力，都是在同一个约束下做文章：显存装得下多少 KV，你就服务得了多少人。** 算力，反而是 Decode 阶段最不缺的东西。

## 情境 5　工程演进史：从 Naive 预分配到 PagedAttention 的虚拟内存革命

**背景**：知道了 KV Cache 会吃爆显存，接下来的问题是怎么**把这有限的显存用到极致**。这条路开源社群走了三代，从一个浪费到令人发指的 Naive 方案，演进到一个直接抄操作系统的天才设计——而抄的那份作业，是你在计算机组成原理课上学过、却没想到会用在这里的**虚拟内存分页**。

**第一代，Naive 方案（早期 PyTorch 朴素实作）。** 做法简单粗暴：为每个请求，**预先开辟一块连续的、大小等于 `Max_Seq_Len` 的显存**。模型支持 4K 上下文，那就一律先霸占 4K 的空间，不管你实际会生成多少。这带来两种致命碎片：

- **内部碎片（Internal Fragmentation）**：用户其实只输出了 100 个 token，但你为他预留的 4K 空间里，剩下 3900 个 token 的位置被**强行占住、别人用不了**，纯浪费。
- **外部碎片（External Fragmentation）**：显存总量明明还够，但因为要求「连续」的大块，零碎的空闲凑不成一块连续区，**明明有空间却无法初始化新请求**。

实测下来，Naive 方案的显存有效利用率常常只有两三成——七成的显存在替「万一用户写很长」的假设空转。

**第二代，vLLM 的 PagedAttention（虚拟内存革命）。** vLLM 团队的洞察堪称经典：**KV Cache 的碎片问题，和操作系统四十年前面对的内存碎片问题，是同一个问题。** 而 OS 早就解决了——用**分页（Paging）**。于是 PagedAttention 把 KV Cache **离散地存进一堆固定大小的物理块（Physical Block）**里，这些块在显存 HBM 里可以**散落各处、不要求连续**：

```
  逻辑视角 (每个 sequence 以为自己的 KV 是连续的):
    Logical Blocks:  [ Block 0 ] -> [ Block 1 ] -> [ Block 2 ]
                         │              │              │
  Block Table (映射表):  │              │              │
                         ▼              ▼              ▼
    Physical Blocks: [ Frame 42 ]   [ Frame 7 ]   [ Frame 99 ]
                     (GPU HBM 里物理上散落各处，但逻辑上连续)
```

每个物理块固定装 16 或 32 个 token。一个 `BlockManager` 管着全部空闲块的池子，一个 `BlockTable` 维护「逻辑块 → 物理块」的映射——这完全就是 OS 的页表（page table）。新请求来了就从池子抓空闲块，生成到哪、长到哪、就抓到哪，**用多少抓多少，内部碎片几乎归零**；块大小固定且可任意散布，**外部碎片也消失**。显存利用率被逼到近乎 100%。

```python
# vLLM PagedAttention 内核架构（源码逻辑伪代码）
class PhysicalTokenBlock:
    def __init__(self, block_number, block_size):
        self.block_number = block_number  # GPU 显存块的实际物理索引
        self.block_size = block_size      # 通常 16 或 32 个 token
        self.ref_count = 0                # 引用计数 → 支持 Block 级 Copy-on-Write

class BlockAllocator:
    def __init__(self, num_gpu_blocks, block_size):
        # 启动时把全部可用显存切成固定大小的块，做成空闲池
        self.free_blocks = [PhysicalTokenBlock(i, block_size)
                            for i in range(num_gpu_blocks)]

class BlockTable:
    def __init__(self):
        self.mapping = {}  # 逻辑块索引 → 物理块 (这就是「页表」)
    def append_token(self, logical_block_idx, block):
        self.mapping[logical_block_idx] = block
```

但分页带来一个新麻烦：KV 在显存里不再连续了，标准的 `torch.cat` 之类连续内存操作就用不了。vLLM 的解法是**自己手写 CUDA / Triton kernel**，在算注意力时**通过 `block_table` 动态寻址**，去那些散落的物理块里逐块捞 K、V：

```
  // vLLM CUDA Kernel 寻址内核伪逻辑
  __global__ void paged_attention_kernel(
      float* out, const float* q, const float* k_cache, const float* v_cache,
      const int* block_table, const int* context_lens, int block_size) {
      int seq_id = blockIdx.y;
      int token_idx = threadIdx.x;
      // ① 用逻辑 token 索引查 block_table，找出物理块编号
      int logical_block_idx = token_idx / block_size;
      int physical_block_num =
          block_table[seq_id * max_blocks_per_seq + logical_block_idx];
      // ② 算出该 token 在显存里的绝对偏移地址
      int block_offset = token_idx % block_size;
      float* k_ptr = k_cache
          + physical_block_num * block_size * num_heads * head_dim + ...;
      // ③ 在散落的物理块上运行常规 Attention 计算
  }
```

那个 `ref_count`（引用计数）还顺手解锁了一个漂亮特性：**Copy-on-Write 的块共享**。并行采样（一个 prompt 同时生成多条候选）或 beam search 时，多条串行共享同一段前缀的物理块，只在真正要写入分歧时才拷贝——这又是 OS 的 `fork()` 写时拷贝，搬到了 KV Cache 上。

> 🔍 高端点评 — 为什么「抄 OS」是 LLM 推理引擎最深的护城河
> PagedAttention 表面是一个显存管理技巧，但它真正示范的，是**系统工程里最高级的一种能力：在一个全新领域里，认出一个被彻底解决过的老问题，然后把成熟解法整套搬过来。** vLLM 之所以能在 2023 年一炮而红、成为事实标准，不是因为它发明了新数学，而是因为它的作者看着 KV Cache 的碎片，脑中浮现的是「这不就是 1960 年代的虚拟内存吗」——然后把页表、分页、写时拷贝、甚至「换页到 CPU 内存」（显存不够时把冷的 KV 块 swap 到主内存，正是 OS 的 swap）整套照搬。**今天围绕 KV Cache 的整个生态，几乎都是 OS 概念的 LLM 化身：** PagedAttention 是分页，prefix caching 是去重，RadixTree（下一节）是前缀树索引，KV 量化是压缩，KV offloading 是 swap，连 NVIDIA 的 **FlashInfer**、Hugging Face 的 **TGI**、SGLang 的块管理，骨子里都在玩同一套内存层级的把戏。**这给蒸馏者一个极实际的启示：当你想优化你那台 DGX Spark 的本地推理，别急着去 GitHub 找新框架——先问自己「操作系统当年是怎么解这个问题的」。** 计算机科学六十年的内存管理智能，全是你现成的军火库；只会等别人发明新 kernel 的人，永远慢一个身位。

## 情境 6　服务端的 RadixTree 与算子级的 FlashAttention：把内存还给你的 DGX Spark

**背景**：PagedAttention 解决了「单机显存怎么用满」，但还有两个层次的优化没讲——**服务端如何让不同用户的请求互相复用前缀**（这对应情境 1 的 Prompt Cache），以及**最底层的算子如何不把内存带宽浪费掉**。这两者一个在最上层、一个在最底层，却共同回答了本章开头的问题：怎么把那一百多 GB 还给你。

**服务端：SGLang 的 RadixTree（基数树）前缀缓存。** 情境 1 讲过服务层 Prompt Cache 要复用相同前缀，但问题是——成千上万个用户，前缀各不相同又部分重叠，怎么高效地「千人千面地共享公共前缀」？SGLang 框架的答案是 **Radix Tree（基数树）**。它把每个用户输入的 prompt 转成 token ID 数组，树的**每个节点存一段 token 串行 + 那段对应的 KV Cache 物理指针**。公共前缀自动汇聚在靠根的枝干上，被所有命中它的请求复用：

```
  三个请求共享同一句 system prompt：
    请求1: "You are a helpful assistant. Please tell me a joke."
    请求2: "You are a helpful assistant. Please write Python code."
    请求3: "You are a helpful assistant. Translate this to Chinese."

                        [ Root ]
                           │
        ( tokens of "You are a helpful assistant. " )  ← 命中公共前缀！KV 只算一次
                           │
            ┌──────────────┼──────────────┐
            │              │              │
      ["Please tell   ["Please write  ["Translate
       me a joke."]    Python code."]   to Chinese."]   ← 各自分叉的尾巴
```

那句反复出现的 system prompt，它的 KV 状态**全树只算一次、所有请求共享**——这正是情境 1「稳定前缀放最前面」黄金法则在服务端的物理兑现。当显存吃紧要腾空间，RadixTree 触发 **LRU（最近最少使用）驱逐**，但它驱逐得很聪明：**先斩叶子节点**（已结束的长对话尾部、最不可能再被命中的分支），**保留根节点和公共枝干**（system prompt 这种被命中几率最高的部分）。先斩叶、后保根——这个策略本身就是对「前缀越靠根、复用价值越高」的精准利用。

**算子级：FlashAttention 与 FlashDecoding。** 光有分页和前缀树还不够，最底层的注意力算子如果写得蠢，照样浪费内存带宽。回到情境 2 的内核矛盾——Decode 是访存受限的，瓶颈在 HBM 搬运。

- **FlashAttention** 攻击的是「中间矩阵写回 HBM」这个浪费。朴素实作会把那个 N×N 的注意力矩阵**完整算出来、写回慢速的 HBM、再读回来**做 softmax——光是这一来一回就烧掉巨量带宽。FlashAttention 用 **Tiling（分块）** 把计算切成能塞进芯片上**高速 SRAM** 的小块，并用 **online softmax**（边扫边增量更新 softmax 的归一化分子分母）的数学技巧，**全程不把完整的 N×N 矩阵写回 HBM**。结果是显存读写量降低一个数量级——它没让 GPU 算得更快，是让 GPU **少搬了一个数量级的数据**，这恰好正中访存受限的要害。

- **FlashDecoding** 专治情境 2 的 Decode 阶段、且专治**超长上下文**。Decode 时 batch 与头数往往不足以喂满 GPU 的全部计算单元，若历史 KV 又极长（比如 128K 上下文），那一条长 KV 沿串行维度的串行扫描会让大量 SM（流多处理器）闲置。FlashDecoding 的关键是**沿 KV 的串行维度（而非 batch/头维度）再切一刀**：把**那条超长的 KV 切成多段，并行分配到 GPU 上多个计算单元（SM）**各算各的局部注意力与局部 softmax 统计量，最后做一次 **Reduction（规约）**、用各段的 softmax 分母重新加权合并成正确结果。它把「一条长尾巴的串行访存」变成「多段并行访存」，正是在「batch 小、上下文长」这个传统并行维度喂不满 GPU 的死角，把利用率重新拉起来。

**把这一切接回你的 DGX Spark。** 情境 1 你那台机器爆内存，现在你手上有了一整套还内存的工具：用 **vLLM / SGLang** 启动，靠 **PagedAttention** 消灭碎片把显存利用率逼到 100%（情境 5）；靠 **RadixTree prefix caching** 让多个会话共享那段固定的 system prompt，省掉重复 Prefill；靠 **FlashAttention / FlashDecoding** 把访存受限的 Decode 阶段的内存带宽榨干；再配合第 5 章后处理的 **KV Cache 量化（FP8）** 把公式里的精度字节从 2 砍到 1。这四层叠起来，原本一爆就 OOM 的 128GB，能撑住的并发与上下文可以翻好几倍——**这就是「把那一百多 GB 还回来」的具体工程路径**。

```python
# 把内存还给 DGX Spark：vLLM 启动参数的物理含义
# vllm serve <your-distilled-model> \
#   --gpu-memory-utilization 0.90   # 显存用到九成 (留余量防碎片，情境4)
#   --max-num-seqs 64               # 并发上限 = 显存 ÷ 每条KV峰值 (情境4高端点评)
#   --max-model-len 8192            # 上下文上限，直接决定每条KV尾巴多长
#   --enable-prefix-caching         # 开 RadixTree 前缀缓存 (情境6)
#   --kv-cache-dtype fp8            # KV量化，公式第二个2→1，长文内存腰斩
#   --enable-chunked-prefill        # Prefill分块，与Decode交织提吞吐
```

> 💡 君之一席话
> 与君一席话：**这一章从头到尾，没有一个技术是在「让模型更聪明」——KV Cache、PagedAttention、RadixTree、FlashAttention，全是在「让同样的聪明，用更少的内存跑起来」。大模型工程真正的战场，从来不在算力的巅峰，而在内存的深渊。** 你回头看这六个情境的暗线：情境 2 揭示瓶颈是访存不是算力，情境 3 用一个 concat 把算力省成线性、代价是内存常驻，情境 4 算出那个内存会爆到吓人，情境 5 用分页把内存用满，情境 6 用前缀树和算子把内存带宽榨干——**整章是一条「围绕内存做文章」的单一主线**。这呼应了第 3 章 DGX Spark 那个 128GB 统一内存的杀手级价值，也预告了一个残酷的现实：**当你把一个强模型蒸到本地、量化到能跑，真正决定它『能不能服务真实流量』的，不是它有多聪明，是它在并发压力下的内存曲线稳不稳。** 业余者炫耀「我本地能跑 70B」，专业者追问「你本地能在多大并发、多长上下文下，把 70B 的 KV 稳住不 OOM」——前者比的是峰值，后者比的是在内存深渊边上走钢索的功力。**模型的智商是天花板，内存的管理是地板；而一个系统能不能用，从来是地板说了算。**

## 本章小结

- **两大缓存分类**：① 推理层 **KV Cache**（住显存、由 vLLM/TensorRT-LLM 管、把自回归解码从 O(N²) 降到 O(N)、一次请求生命周期）；② 服务层 **Prompt / Context Cache**（住服务端、跨请求复用相同前缀的 KV 状态、Claude 用 `cache_control` 显式锚定、命中后输入 token 大幅折扣）。黄金法则：**稳定静态内容放最前、动态高变内容放最后**，因为两者本质都是前缀匹配，一个流浪的时间戳或多余的 `\n` 就能让整条缓存作废。
- **访存受限是物理底层**：Prefill 是 Compute-Bound（张量打包、吃满 Tensor Core），Decode 是 Memory-Bound（每步只 1 token、GPU 大部分时间在 HBM→SRAM 搬运）。**KV Cache 不是消灭瓶颈，是把瓶颈从算力搬到内存**——这是全章的物理基础，也是 GQA/MQA/MLA 拼命砍 KV 头数的根本动机。
- **数学推导**：无缓存每步重算历史 K/V → Σ O(t) = **O(T²)**；有缓存只算新 token 并 `concat` 进旧矩阵 → 单步 O(1)、总 **O(T)**。降维的魔法只用到一个朴素的拼接，背后是「历史不会改变，所以不必重算」的 memoization 母题。
- **显存计算器**：`Size_per_token = 2(K+V) × 2(FP16字节) × L × H(KV头) × D`。Llama 3-8B（L=32, KV Heads=8, D=128）= **128 KB/token**；batch=128 × 8K ctx ≈ **128 GB**，而权重才 16GB。**瓶颈是 KV Cache 不是权重**，最大并发由显存能装多少条 KV 尾巴决定，不由算力决定。
- **工程演进**：Naive 连续预分配 → 内/外碎片把利用率打到两三成；vLLM **PagedAttention** 抄 OS 虚拟内存分页，KV 离散存进固定大小物理块、`BlockManager`/`BlockTable` 做逻辑→物理映射、自订 CUDA/Triton kernel 经 `block_table` 动态寻址、`ref_count` 解锁写时拷贝块共享，利用率逼近 100%。整个 KV 生态都是 OS 概念的 LLM 化身（分页/去重/前缀树/压缩/swap）。
- **服务端 + 算子级**：SGLang **RadixTree** 每节点存一段 token + KV 指针、公共前缀（system prompt）命中复用、LRU 驱逐先斩叶子保根枝；**FlashAttention** 用 SRAM tiling + online softmax 不把 N×N 矩阵写回 HBM；**FlashDecoding** 把超长 KV 切段并发于多 stream 再 reduction。四层叠加（PagedAttention + prefix caching + Flash 算子 + KV 量化）就是把那消失的一百多 GB 还给 DGX Spark 的具体路径。

KV Cache 算清楚了，分页也分了，前缀树也建了——可这一切都还在「一台机器」的尺度上。当请求量从几条涨到每秒几十万条、当缓存要跨越成百上千张 GPU 共享、当你得在「命中率」和「一致性」之间做分布式的取舍，单机的这套physics 就不够用了。下一章，我们把缓存从一台机器放大到一整个数据中心，看超大规模下的缓存架构，以及它如何收束进那座为攻防而生的、二十个方向的军火库。
