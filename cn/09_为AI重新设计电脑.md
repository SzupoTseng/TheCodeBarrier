# 第九篇　AI 原生

# 第9章　为 AI 重新设计电脑：从像素点击到语意状态总线

> 硅谷，某 AI 基础设施公司的深夜办公室。你刚把上一章那套双环架构调到能用——弹道鼠标、Delta-Frame、SIGSTOP 中断，一应俱全。它确实流畅了。但你盯着屏幕上那个正在「看截屏、想、点击」的 Agent，忽然冒出一个让你脊背发凉的念头：**你花了整整一年，教一个硅基智能去学一件事——怎么像一个人类那样，用一根虚拟手指，去戳一块本来就是「画给人类眼睛看」的玻璃。** 那块玻璃上的每一个像素，都是电脑先把自己内部清清楚楚的「结构化状态」，辛辛苦苦渲染成一张图，再由你的 Agent 用几千个 Vision Token、花一秒钟，把那张图「逆向」回它本来就有的结构。你做的不是让 AI 变聪明，你做的是给一场巨大的、荒谬的信息熵浪费，盖了一层精美的补偿架构。窗外是整个硅谷的灯火，你关掉 demo，问了自己一个比「怎么点得更准」深得多的问题：**电脑，为什么还非得长成给人操作的样子？**

> ⚠️ 本章的素材来源与真伪纪律
> 本章集成的原始素材，是一段以「Google / Gemini 工程口吻」写成的、**高度前瞻（forward-looking）**的架构论述。它把「为 AI 重新设计电脑」这件还在发生的事，描述得像已经发生。本章的纪律分两层：凡涉及**真实、可查证的研究与开放标准**——State Space Models / Mamba、MCP（Model Context Protocol）、Set-of-Mark、SAM 2、CogAgent、ScreenAI、Mind2Web、OSWorld、WebArena、VisualWebArena、Ferret-UI、Constitutional AI——本书**照常平实陈述**（皆可在 arXiv / GitHub / 官方博客查证，见附录 F）；凡涉及**具体产品名、版本、发表会时程、未公开的内部架构名**（如 Google I/O 2026、Aluminium OS、Antigravity、Project Jarvis、AppFunctions、Hydra Memory Layer 等），一律以 `> ⚠️ 真伪提醒` 标出，当成「对行业走向的合理推演」而非既成事实。**本章是全书最思辨、最不确定的一章——请带着这把尺读。**

## 情境 1　范式冲突：你在教 AI 玩一场它本不必玩的解码游戏

**背景**：今天地球上几乎每一台电脑，从操作系统内核到应用程序接口，都创建在一个从未被质疑的前提上——**它的最终用户，是一个有两只眼睛、两只手、会被颜色和动画吸引注意力的人类**。窗口、图标、鼠标指针、下拉列表、红色的错误弹窗——这整套「图形用户接口（GUI）」的发明，是 1970 年代全录帕罗奥图研究中心（Xerox PARC）为了解决一个非常具体的问题：**让人类不必背指令、不必懂底层，用「看见即可操作」的方式驾驭电脑**。它是人类认知的伟大让步，是电脑向人类的妥协。

而现在，我们做了一件诡异的事：我们把一个**没有眼睛、没有手、不会被动画吸引、本身就能直接读懂结构化数据**的智能，硬塞进这套为人类肉身设计的妥协里，然后惊叹它「居然学会了点鼠标」。

这套流程的荒谬，要把它的信息熵帐算清楚才看得见：

```
人类使用电脑 (GUI 设计的初衷):
  [App 内部结构化状态] ──渲染──► [像素图] ──人眼──► [人脑语意理解] ──手──► [鼠标事件]
       (语意,低熵)         (高熵)        (人类光学)      (还原语意)        (动作)

AI 用 Computer Use 操作电脑 (今天的做法):
  [App 内部结构化状态] ──渲染──► [像素图] ──截屏──► [VLM 重新感知] ──推理──► [座标点击]
       ★本来就是语意★      ★白白渲染★   ★白白传输★  ★花几千token把★    ★再退化回★
                                                    ★图逆推回语意★      ★模糊的座标★
```

看出那个荒谬的循环了吗？**App 一开始手里握着的，就是「按钮叫 submit_btn、现在 enabled、座标在这」这种完美的结构化语意**。它把这份语意「渲染」成一张对人眼友善、对机器极不友善的高熵位图。然后 AI 接过这张图，用一个几十亿参数的视觉模型、烧掉几千个 Vision Token、花掉秒级的推理时间，**费尽力气把那份本来就存在、刚刚才被丢弃的语意，重新「猜」回来**——而且猜回来的还是个带误差的 `(x, y)` 座标，远不如原始的 `submit_btn` 精确。

这不是技术不够好，这是**范式错了**。上一章那套精巧的双环、Delta-Frame、弹道鼠标，本质上全都是在这个错误范式上打的补丁——它们让「渲染再逆向」这条冤枉路走得更快、更省，但**没有一个动作，质疑过这条路本身存不存在的必要**。

| 维度 | 「为人类设计」的电脑 | 「为 AI 设计」的电脑（本章命题） |
|---|---|---|
| 接口本质 | 像素位图（给眼睛） | 结构化语意流（给模型） |
| App 暴露什么 | 渲染后的视觉 | 它本来就有的内部状态与可调用动作 |
| AI 要做的事 | 把图逆向回语意，再退化成座标 | 直接读语意、直接调用动作 |
| 主要成本 | Vision Token + 渲染 + 视觉推理延迟 | 一次结构化读取（近乎免费） |
| 失败模式 | 像素漂移、骨架屏、DPI 歪斜（第 8 章二十关大半） | 大半根本不存在 |
| 模拟 | 让会读盲文的人，先把书印成油墨图再用摄影机拍下来认字 | 直接把盲文递到他手里 |

> 💡 君之一席话
> 与君一席话：**当一个聪明的系统在一件事上显得特别吃力，第一个该问的不是「怎么让它更努力」，而是「它是不是正在被迫做一件本就不该由它做的事」——Computer Use 的全部精巧，都是在为一个范式错误付利息。** 你看第 8 章那二十道难关——像素漂移、DPI 自适应、骨架屏欺骗、无限滚动失忆、自订控件语意断层——把它们排在一起会发现一件惊人的事：**它们几乎全部，是「逼 AI 用人眼读人类接口」这个前提的直接后果，而不是 AI 本身的缺陷。** 一个能直接读到 `submit_btn` 的 Agent，根本没有「像素漂移」这回事，因为它从不处理像素。我们花了行业级的智力去解这二十道难关，却很少有人退一步问：**这二十道难关，有多少是真的难题，有多少只是我们选错了战场才凭空长出来的？** 业余工程师在给定的范式里把问题解到极致，顶尖工程师会先审判范式本身值不值得待——**因为在错误的范式里，再漂亮的优化，天花板都是别人地板的高度。**

> 🔍 高端点评 — 为什么 ScreenAI 的「双流」已经是这场范式冲突的和解信号
> 真实研究界其实早就嗅到了这股不对劲。**ScreenAI**（Google，2024）这个 VLM 最关键的设计，不是它多会「看」截屏，而是它被逼着**从纯像素里反推出一份结构化的「屏幕结构描述」（screen schema）——每个 UI 组件的类型、座标、文本、彼此的层级关系**，并以这份结构当作训练与输出的内核表征（它的卖点之一，恰恰是不依赖 App 的 view hierarchy／无障碍树，纯从画面学出结构）。它的潜台词非常清楚：**把画面只当位图去分类是会错的，得逼模型在脑中重建出 App 本来就有的那层结构化语意，用结构去校正视觉的幻觉。** 同样地，**Set-of-Mark（SoM）**（Microsoft，2023）的整套思路——先用 **SAM 2**（Meta，2024）把可点元素分割出来、编号、叠在图上，让模型输出「点 7 号」而不是「点 (x,y)」——本质上是**把连续的像素座标问题,偷偷降维成离散的语意选择问题**。**Code-as-Action / CodeAct**（UIUC，2024）走得更远：干脆让 LLM 直接输出可运行的 JS/Python 当动作，绕过视觉座标。把这三条线连起来看，方向是一致的：**整个研究前沿，都在用各种办法「往语意层靠、往像素层退」。** 它们都还困在「接口是给人看的」这个既成事实里——所以只能在 App 外面、事后地、用侦测器去「重建」语意。本章要讲的命题比它们激进一个量级：**与其在事后重建语意，不如让 App 从一开始就别把语意丢掉。** SoM 和 ScreenAI 是这场范式战争里的和解派，本章情境 3 讲的，是革命派。

## 情境 2　「逃离像素」的二十个前沿方向：一条从「点得更准」到「根本不用点」的光谱

**背景**：研究界对「Computer Use 太蠢」的回应，不是一个方向，而是二十多个同时开火的前沿方向。把它们随手列成清单没有意义——它们真正的结构，是一条**光谱（spectrum）**：一端是「**承认接口是像素，想办法把像素操作做得更好**」，另一端是「**否定接口是像素，让 AI 根本不碰像素**」。看清这条光谱上每个方向站在哪，比记住每个名字重要一百倍——因为它告诉你**这个行业正在往哪个方向迁徙**。

下面把二十个方向按「离像素多远」分成六群，从左（最恋栈像素）到右（最彻底逃离）排列。标 ✅ 者为真实可查证的论文/项目/标准。

**第一群　视觉定位：把「座标回归」降维成「语意选择」（仍在像素层，但开始逃）**

| # | 方向 | 它在逃离像素的哪一步 | 出处 |
|---|---|---|---|
| 1 | **Set-of-Mark（SoM）** | 预先框出可点元素并编号，输出「选 7 号」而非座标 | Microsoft 2023 ✅ |
| 2 | **SAM 2** | 通用影像/视频分割，当 SoM 的前置 UI 组件侦测器 | Meta 2024 ✅ |
| 3 | **Ferret-UI** | 行动设备 UI 组件分类/OCR/定位的小模型 | Apple 2024 ✅ |
| 4 | **[0,1000] 座标范式** | 把座标归一到分辨率无关的整数区间，抹掉 DPI 歪斜 | CogAgent/Qwen-VL 通用法 ✅ |

**第二群　屏幕理解：让模型同时读「图」和「结构」（一只脚已踏进语意层）**

| # | 方向 | 它在逃离像素的哪一步 | 出处 |
|---|---|---|---|
| 5 | **CogAgent** | 18B 专为 GUI 微调的 VLM，高/低解析双信道 | THUDM, CVPR 2024 ✅ |
| 6 | **ScreenAI** | 从纯像素反推出结构化「屏幕描述」(screen schema)，以结构校正视觉 | Google 2024 ✅ |
| 7 | **Accessibility Tree 双流对齐** | 视觉幻觉时，靠无障碍树的语意把座标拉回来 | WebArena／行动 Agent 观测法 ✅ |
| 8 | **Location Tokens** | 用特殊 token 直接表示边界框，视觉定位 token 化 | Pix2Seq/Ferret ✅ |

**第三群　网页语意：直接操作 DOM / 输出代码，绕过视觉（基本离开像素）**

| # | 方向 | 它在逃离像素的哪一步 | 出处 |
|---|---|---|---|
| 9 | **Mind2Web** | 137 个真实网站的通用操作数据集，以 DOM 元素为动作空间 | OSU NLP 2023 ✅ |
| 10 | **Code-as-Action / CodeAct** | LLM 直接输出可运行 JS/Python 当动作，免疫视觉歪斜 | UIUC 2024 ✅ |
| 11 | **BrowserUse** | 抽出网页 DOM 可交互元素喂给 LLM、驱动真实浏览器的开源代理 | GitHub 开源 ✅ |
| 12 | **Skyvern** | VLM 驱动的浏览器自动化，免写死 XPath/选择器 | GitHub 开源 ✅ |
| 13 | **WebVoyager** | 端到端多模态 Web Agent（截屏+SoM），亦是评测集 | 论文 2024 ✅ |

**第四群　评测基准：定义「什么叫做到了」（没有它，逃离像素只是口号）**

| # | 方向 | 它逼出了什么真相 | 出处 |
|---|---|---|---|
| 14 | **OSWorld** | 真实 OS 级端到端操作（Office/终端/GIMP），业界公认最难 | 369 任务 2024 ✅ |
| 15 | **WebArena** | 自托管网站（电商/论坛/Git/CMS）的功能性任务 | 812 任务 CMU 2023 ✅ |
| 16 | **VisualWebArena** | WebArena 视觉强化版，逼模型理解图像内容 | 2024 ✅ |
| 17 | **AndroidWorld** | Android 真机动态任务，含参数化奖励 | 116 任务 Google 2024 ✅ |

**第五群　协定 / 语意层：让 App 主动把「能做什么」说清楚（彻底不碰像素）**

| # | 方向 | 它在逃离像素的哪一步 | 出处 |
|---|---|---|---|
| 18 | **MCP（Model Context Protocol）** | 开放标准，让模型以协定直接调用工具/读数据，**根本不渲染** | Anthropic 2024 ✅ |
| 19 | **Constitutional AI** | 一套「宪法」做自我批判/对齐，引申为 OS 动作拦截护栏 | Anthropic ✅ |

**第六群　底层架构：换掉模型的「脑干」，让长串行记忆变便宜（逃离的物理前提）**

| # | 方向 | 它在逃离像素的哪一步 | 出处 |
|---|---|---|---|
| 20 | **State Space Models / Mamba** | 线性复杂度串行模型，长时操作记忆的 Transformer 替代 | Gu & Dao 2023 ✅ |

**这条光谱在说什么**：从左到右，「像素的份量」单调递减，「语意的份量」单调递增。第一群还在跟像素的座标误差搏斗；到第五群的 **MCP**，App 已经完全不渲染了，模型直接调用 `submit()`；到第六群的 **Mamba**，连模型本身的「内存成本」都被重构，好让它能扛住一个跨越几百步、几小时的长任务而不失焦（直接对应第 8 章第 11 关「超长串行上下文失焦」）。**整个行业的重心，正在以肉眼可见的速度从光谱左端往右端滑。** 你今天投入的每一分精力，押在光谱哪一端，决定了三年后它是资产还是遗产。

> 🔍 高端点评 — 为什么 OSWorld 的「最难」恰恰证明了「像素范式」已到顶
> 把第四群的评测基准单独拎出来看，会读到一个残酷的信号。**OSWorld**（369 个真实 OS 任务）发布时，当时最强的多模态模型得分**惨不忍睹（个位数到一二十分区间，相对人类的 70%+ 是断崖）**——而它测的，正是「纯靠看截屏点鼠标」这条最恋栈像素的路。与此同时，第三群里那些**直接操作 DOM / 输出代码**的方法（CodeAct、BrowserUse），在 **WebArena** 这类能拿到结构化网页的场景里，成功率明显高出一截。这个对比不是偶然：**凡是还能拿到结构化语意的地方（网页有 DOM、行动 App 有 Accessibility 树），绕过像素的方法就赢；凡是只剩像素的地方（OSWorld 里的原生桌面 App、GIMP、终端机截屏），所有方法一起跪。** 这恰恰反过来证明了情境 1 的论点——**Computer Use 之所以在 OSWorld 上这么难，根本原因不是模型笨，是那些桌面 App 太自私：它们只肯吐像素、不肯吐语意。** 解法不在于把视觉模型再堆大十倍去硬啃像素（那是在光谱左端撞墙），而在于**逼 App 把语意吐出来**——这就把问题从「AI 研究」推到了「操作系统与 App 架构」的领域。OSWorld 的高墙，不是给模型团队的考题，是给 OS 设计者的战书。

## 情境 3　AI 原生操作系统的五个构件：当电脑第一次为「没有眼睛的用户」而设计

**背景**：如果接受了「电脑不该逼 AI 看像素」这个命题，那么一台**为 AI 原生（AI-native）设计**的电脑，会长成什么样？它不会是「现在的 OS + 一个更好的截屏工具」，而是从**接口哲学**上重做的东西。把散落在各前沿方向里的线索收拢，一台 AI 原生 OS 至少需要五个构件。注意：**这五件事的技术原语大多已经存在**（共享内存、无头渲染、IDL、MCP、容器编排都是成熟工程），真正新的，是**把它们组装成「以语意而非像素为第一接口」的整体**。

**构件一：语意状态总线（Semantic State Bus）——把「屏幕」换成「状态流」**

这是整台 AI 原生 OS 的心脏。在传统 OS 里，App 与外界沟通的「接口」是它渲染到显示缓冲区（framebuffer）的那张图。在 AI 原生 OS 里，每个 App 对外暴露的不是图，而是**一条持续广播自身结构化状态的事件总线**：

```
传统 OS:
  App ──► [Framebuffer 像素] ──► 显示器(给人) / 截屏(给AI去逆向)

AI 原生 OS (Semantic State Bus):
  App ──► [语意状态流] ──► {
            current_view: "login_form",
            elements: [
              {id:"user_field", type:"input", value:"", focused:true},
              {id:"submit_btn", type:"button", enabled:false}
            ],
            events: [{type:"value_changed", target:"user_field"}]
          }
        └─► AI 直接订阅, 零视觉推理, 状态变化即增量推送
```

这直接呼应第 8 章情境 3 的 Delta-Frame，但更彻底：Delta-Frame 还得靠光流和 OS 事件去**猜**「哪里变了」；语意状态总线里，App **主动、精确地告诉你**「`user_field` 的 value 变了」。它把第 8 章那条辛苦压缩出来的 Event Token Stream，从「AI 在外面逆向重建」升级成「App 在里面原生提供」。**第 8 章是在范式外打补丁，这里是换范式。**

**构件二：零拷贝内存映射（Zero-Copy Memory Mapping）——让「读状态」近乎免费**

语意状态总线要跑得快，状态的传递就不能每次都串行化、拷贝、反串行化。AI 原生 OS 把 App 的状态结构直接**内存映射（mmap）**到 Agent 可读的共享内存区，Agent 读 App 状态像读自己的变量一样，**零拷贝（zero-copy）**。这正是第 3 章 DGX Spark 的 **UMA 统一内存**、第 5 章 Apache Arrow 零拷贝缓冲、第 8 章 Mac UMA 截屏零拷贝——同一个物理智能在 OS 接口层的again体现：**当 CPU 截的、GPU 推的、Agent 读的是同一块内存的同一份数据，整条感知管道的延迟就从「秒」塌缩到「微秒」。**

**构件三：无头优先 + 反向渲染（Headless-First + Reverse Render）——人类接口降级成可选配件**

最反直觉的一件事：AI 原生 OS 里，**渲染像素变成「可选的、按需的、降级的」功能**。App 缺省**无头（headless）**运行——它只维护和广播语意状态，根本不画图，因为它的主要用户（Agent）不需要看。只有当一个**人类**碰巧要介入时（接管、审计、debug），OS 才**反向渲染（Reverse Render）**：把那一刻的语意状态，临时画成一张人能看的图。

```
传统:  状态 ──(总是)渲染──► 像素 ──► (人看 / AI 逆向)
AI原生: 状态 ──► AI 直接读
            └──(仅当人类介入时, 按需)──► 反向渲染成像素给人看
```

这是整个范式的彻底翻转：**过去「人是一等公民、AI 是用截屏偷看的二等公民」；现在「AI 是一等公民、人类接口是按需生成的兼容层」。** 像素，从「唯一的接口」降级成「给人类看的特殊输出格式」。

**构件四：LLM-Ready API / Agent Manifest——让 App 自己声明「我能做什么」（MCP 的角色）**

光有状态（能读）还不够，Agent 还得能**动作**。AI 原生 OS 要求每个 App 附带一份**Agent Manifest（代理清单）**：用机器可读的 IDL（接口定义语言）声明「我提供哪些动作、每个动作要什么参数、有什么前置条件、会造成什么副作用」。Agent 不再「移动鼠标到 submit 按钮再点击」，而是直接调用 `app.submit(user, pass)`。

**这正是 MCP（Model Context Protocol，Anthropic 2024）已经在做的事的 OS 级推广。** MCP 是一个真实、开放的标准，让模型以统一协定调用外部工具、读取数据源——它本质上就是「Agent Manifest」在工具/数据层的实作。把 MCP 的思路从「外部工具」扩展到「OS 里每一个 App」，就得到了构件四：**App 不再被动地被 AI 看，而是主动地向 AI 自我介绍「我是什么、我能干什么」。** 而动作的安全护栏，则由 **Constitutional AI**（Anthropic）式的原则拦截层负责——任何高风险动作（删库、转帐、发信）在运行前先过一道「宪法」审查（直接挡住第 8 章第 9、20 关的副作用与问责难题）。

**构件五：大规模多代理并行（Massive Multi-Agent Parallelism）——当 App 无头，一万个 Agent 可以同时跑**

一旦 App 无头、状态零拷贝、动作走 API，一个惊人的可能性打开了：**你不再受限于「一个屏幕、一个鼠标、一次一个动作」**。传统 GUI 是天生串行化的——一块屏幕同时只能显示一个焦点，一个鼠标同时只能点一个地方。但无头的语意 App 没有这个物理限制：**同一个 App 实例（或一万个 fork）可以被一万个 Agent 同时、并行地读状态、发动作**，像一万个数据库连接打同一个 DB。这把第 8 章第 16 关（沙箱性能权衡，每个 Agent 一个 Firecracker MicroVM 去跑高解析渲染）的成本结构整个推翻——**无头的 App 不渲染，一个 MicroVM 能塞下的并行 Agent 数量，是渲染版的几十上百倍。**

| 构件 | 一句话 | 替代了传统 OS 的什么 | 对应的真实原语 / 章节 |
|---|---|---|---|
| 语意状态总线 | App 广播结构化状态，不广播像素 | Framebuffer / 截屏 | MCP 资源、ScreenAI 屏幕结构描述、第 8 章 Delta-Frame |
| 零拷贝内存映射 | Agent 读 App 状态零拷贝 | 串行化拷贝的 IPC | UMA、Apache Arrow（第 3、5 章）|
| 无头优先 + 反向渲染 | 缺省不画图，仅人类介入时按需渲染 | 「总是渲染」的显示管线 | Headless 浏览器、Playwright |
| LLM-Ready API / Agent Manifest | App 自声明可调用动作 | 鼠标/键盘事件注入 | **MCP**、IDL、CodeAct |
| 大规模多代理并行 | 一万 Agent 并行读写无头 App | 「一屏幕一鼠标」的串行瓶颈 | 容器编排、第 8 章第 16 关反面 |

> 💡 君之一席话
> 与君一席话：**真正的范式革命，从来不是「把旧东西做得更好」，而是「把旧东西降级成一个可选的兼容层」——当你能让「屏幕」变成 AI 原生 OS 里一个按需生成的配件，你就不是在优化 Computer Use，你是在让它变得不必要。** 看这五个构件的共同骨架：它们每一个，都是把「人类在场」这个曾经的硬假设，从「系统的中心」挪到「系统的边缘」。语意总线让 App 不必为人眼渲染，无头优先让渲染变成例外，Agent Manifest 让动作不必经过为人手设计的鼠标——**整套设计的灵魂，是承认一件很多人不愿承认的事：在一台机器与机器对话的电脑里，人类是访客，不是房东。** 这话听起来冰冷，但它其实是最诚实的工程：**过去四十年，我们强迫机器戴上「人类面具」来跟另一台机器沟通，浪费掉的算力、延迟、与精度，全是这个面具的税。** 摘下面具，不是要把人类赶走——反向渲染那个构件恰恰证明，人类随时能被请回来看一眼——而是承认「机器跟机器之间，本该用机器的语言」。能设计出「让人类优雅退场、又能随时优雅返场」的系统的工程师，比那些毕生在优化「人类面具怎么戴得更舒服」的工程师，看得远一个时代。

> 🔍 高端点评 — MCP 为什么是这五个构件里唯一「已经落地」的那块拼图
> 五个构件里，四个还停留在「合理推演」，唯独**构件四（Agent Manifest）有一个真实、开放、已被广泛采用的实作：MCP（Model Context Protocol，Anthropic 2024）**。值得把它为什么重要说透。在 MCP 之前，每接一个工具/数据源给模型，都要手写一套客制的胶水（function calling 的 schema、认证、错误处理），N 个模型 × M 个工具是 N×M 的工程灾难。MCP 做的事极其朴素却极其关键：**它定义了一套标准协定，让「工具/数据源」用统一的方式自我描述（提供哪些 resource、哪些 tool、哪些 prompt），让「模型主机」用统一的方式发现和调用**——把 N×M 塌缩成 N+M。**这正是「App 主动向 AI 自我介绍」这件事的协定化。** 把 MCP 的心智模型放大：今天它连的是 GitHub、数据库、文件系统这些「外部工具」；如果操作系统把**每一个原生 App** 都包装成一个 MCP server，那构件一（语意总线=MCP resource 的订阅）、构件四（Agent Manifest=MCP tool 的声明）就同时实现了。**所以 MCP 不只是「又一个工具协定」——它是这场范式迁徙里，第一块已经砌好、且被整个行业接受的地基。** 你如果今天就想往光谱右端押注，学透 MCP 是成本最低、确定性最高的第一步——它是真实的、开放的、可立刻上手的，不像本章后面那些还飘在发表会 PPT 里的名字。

## 情境 4　一条可能的演进路线图（四阶段）—— 本章最思辨、请带最重的怀疑读

**背景**：前三个情境的「范式」与「构件」是站在真实研究地基上的推论。这一节不同——它试图**预测这场迁徙「怎么一步步发生」**，必然踏进大量「尚未发生 / 无法查证」的领域。原始素材在这里塞进了一连串听起来煞有介事的产品名与时程。**本书的处理是：把这条路线图当成「一种合理的演进假说」呈现，但对其中每一个具体产品名、版本号、发表会时程，全部挂上 ⚠️——它们是叙事道具，不是新闻。**

> ⚠️ 真伪提醒（总纲，适用于本节全部产品名）
> 本节出现的 **Google I/O 2026、Aluminium OS、Antigravity、Project Jarvis、AppFunctions / Android MCP、Hydra Memory Layer**，本书**无法核实其真实存在、规格或时程**，一律视为**情境推演**。本节唯一可平实陈述为真的，是 **Project Astra**（Google DeepMind 公开展示过的通用多模态助理原型）与 **MCP**（Anthropic 开放协定）。请把这四阶段读成「如果这场迁徙真的发生，它大概会长这样」，而不是「它已经这样了」。

**第一阶段：寄生期——在像素范式里，偷偷喂语意给模型**

最务实的起点，是不动现有 OS，只在 Agent 侧做事。模型仍然「看截屏」，但同时被喂入能拿到的一切结构化旁支信号（网页 DOM、行动 Accessibility 树）做交叉校正——这就是 **ScreenAI**、**SoM** 已经在做的（情境 2 第一、二群）。像素还是主接口，语意是偷偷塞的小抄。**这一阶段是真实的、正在发生的。**

> ⚠️ 真伪提醒
> 原始素材声称此阶段的标志性事件是「**Google I/O 2026 上 Gemini 3.5 Flash 内置 Computer Use，OSWorld 拿 78.4 分**」。具体分数、版本名、发表会时程均**未经查证**，视为虚构锚点。可确认为真的只是趋势方向：主流模型确实在朝「内置屏幕操作能力」演进，且 OSWorld 是真实的、公认最难的评测基准。

**第二阶段：共生期——OS 开一道「语意侧门」给 Agent**

下一步，是 OS 与 App 开始**主动**吐语意。最可能先发生在**有现成语意层的地方**：浏览器（DOM 天然存在）与行动平台（Accessibility 框架成熟）。App 开始声明「我有哪些动作可被程序调用」——这正是构件四 / MCP 的精神往 OS 渗透。像素接口仍在，但 Agent 已有一条「不必看图」的快车道。

> ⚠️ 真伪提醒
> 此阶段被原始素材具象成几个产品：**Project Jarvis**（号称 Chrome 内置的代理服务）、**AppFunctions / Android MCP**（号称 Android 让 App 声明可调用动作的框架）、**Antigravity**（号称云端 Agent Harness）。这些名称**本书无法核实**，视为情境道具。**底层命题是真的**——MCP 是真实开放协定，「让 App 声明动作给模型调用」是真实且正在推进的方向；但「某公司某产品在某发表会落地」一概不背书。

**第三阶段：原生期——出现「为 Agent 而生」的操作系统**

当语意侧门被走得够多，会出现一个质变：有人从头设计一个**缺省无头、语意总线为第一接口、像素为按需兼容层**的 OS（情境 3 的五构件合体）。App 在这个 OS 上原生地是「一个 MCP server + 一条状态总线」，渲染引擎变成可选组件。人类用户通过「反向渲染」按需取得画面。

> ⚠️ 真伪提醒
> 原始素材给这个 AI 原生 OS 起名 **Aluminium OS**（号称 AI 原生笔电系统），并配一个叫 **Hydra Memory Layer** 的内存架构（号称支撑零拷贝状态映射与大规模并行）。**这两个名字都查无公开出处，本书视为纯虚构**。其下的工程命题（无头优先、零拷贝状态映射、多代理并行）是情境 3 已论证的合理推演，但「Aluminium OS / Hydra」本身请当小说人物看待。

**第四阶段：通用期——Agent 不再「用某台电脑」，而是「调度整个语意网络」**

终局，是个人设备上的助理（**Project Astra** 式的通用多模态代理是这个方向的真实原型）不再被绑在「一台电脑的一块屏幕」上，而是同时调度云端、本地、行动设备上一大片无头语意 App，并行地完成跨设备任务。「屏幕」这个概念，对 Agent 而言彻底消失；它只在你——人类——想看一眼的时候，被反向渲染出来。

> ⚠️ 真伪提醒
> **Project Astra 是真实的**——Google DeepMind 公开展示过这个「能即时看、听、记忆并对话」的通用助理原型。但「Astra 在第四阶段调度一整片无头语意 App 网络」是**本章的推演**，不是 Astra 已公开的能力。真实的 Astra 目前仍主要处于「即时多模态理解 + 对话」的展示形态；把它放进第四阶段，是借它当「通用代理」这个方向的真实代表，**而非宣称它已具备本节描述的调度能力**。

```
阶段一 寄生   阶段二 共生      阶段三 原生         阶段四 通用
像素为主      语意侧门打开     语意总线为第一接口   屏幕概念消失
+偷塞语意 ──► App声明动作 ──► 无头OS+反向渲染 ──► 跨设备语意网络调度
(真实,进行中) (MCP方向,真实) (五构件合体,推演)   (Astra方向,推演)
   ⚠️I/O2026     ⚠️Jarvis        ⚠️Aluminium OS      Astra(真)+调度(推演)
   ⚠️78.4分      ⚠️AppFunctions  ⚠️Hydra Memory
                ⚠️Antigravity
```

> 💡 君之一席话
> 与君一席话：**读任何「路线图」，要学会把它劈成两半——「方向」和「站名」。方向往往是对的（语意终将压倒像素），站名（哪个产品、哪场发表会、哪个分数）几乎全是错的，或至少是不可信的。把信仰放在方向上，把怀疑放在站名上，你就既不会错过浪潮，也不会被某张 PPT 骗进坑里。** 这一节我特意把每个产品名都钉上 ⚠️，不是因为它们一定假，而是因为**一个专业工程师对「具体宣称」的缺省态度，就该是「未经证实」**——尤其当这些宣称来自一段「以厂商口吻写成的前瞻论述」时。新手读到「Aluminium OS、78.4 分、I/O 2026」会兴奋地记下来当论据，老手读到的第一反应是「出处呢？我能复现这个分数吗？这个 OS 我能下载吗？」——**能不能在兴奋的叙事里，本能地分离出『哪些是我能验证的、哪些是别人要我相信的』，这条线，划开了会思考的工程师和会复述的传声筒。** 全书反复敲的那把「真伪尺」，在这一章——这个充满诱人未来名词的一章——是最该握紧的。

## 情境 5　哲学收束：当电脑终于为 AI 而生，「点击像素」会变成一门传真机手艺

**背景**：把前四个情境连起来看，会浮现一个关于「Computer Use 这门技艺的命运」的判断。它有点残忍，但对任何正在这个领域投入时间的人，都极其重要。

回想全书的弧线。第 8 章，我们用尽心思——双环解耦、Delta-Frame、弹道鼠标、SIGSTOP 中断——把 Agent「点像素」的功夫，磨成了流畅的肌肉记忆。那一章的成就是真实的、了不起的。但这一章逼我们面对一个更冷的事实：**那身精湛的肌肉记忆，是为一个正在退场的范式练的。**

打个比方。今天，地球上还有一些人，懂得怎么操作一台传真机——怎么放纸、怎么拨号、怎么听那串拨接的杂音判断有没有接通。这曾经是一门**必要的、需要熟练的办公室技艺**。今天它不是没用了——某些政府机关、某些医院还在用传真——但它已经从「人人必备的内核技能」，降格成「一个在特定角落才用得上的、兼容旧世界的边缘手艺」。一个年轻人不会传真机，没人觉得他不会用电脑；一个资深职员精通传真机，也没人觉得他特别厉害。**传真机技艺没有消失，它只是被降级成了一层「兼容层」。**

**「Computer Use——让 AI 看截屏、点像素」这门技艺，命运大概如此。**

```
                  Computer Use 技艺的生命周期
  今天 ───────────────────────────────────────► 未来
  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐
  │ 内核竞争力     │  │ 通用基础能力   │  │ 边缘兼容层            │
  │ 谁做得好       │→ │ 人人都会,      │→ │ 只在「碰到没有语意接口 │
  │ 谁领先         │  │ 不再是差异化   │  │ 的旧 App」时才激活     │
  │ (第8章的世界)  │  │               │  │ = AI 的「传真机手艺」  │
  └──────────────┘  └──────────────┘  └──────────────────────┘
       像素范式            过渡期              语意范式为主
```

当电脑终于为 AI 原生而生——App 广播语意、声明动作、缺省无头——AI 操作电脑的**主干道**，将是读语意总线、调用 Agent Manifest（MCP）。而「看截屏点像素」这套第 8 章的精湛功夫，会退到一个专门的角落：**专门用来对付那些永远不会升级成语意接口的历史遗留系统**——银行三十年的内核终端、工厂的老 MES、政府那些跑在 Windows 上、混杂五代 UI 框架的报表程序（正是第 8 章情境 6 结尾说的「Windows 高熵战场」）。

这带来两个都成立、却方向相反的结论，必须同时握住：

- **其一，别把毕生押在像素上。** 如果你以为「Computer Use = 把截屏点击做到极致」就是这个行业的终局，你押错了范式。光谱（情境 2）正在往语意端滑，地基（MCP）已经砌好。把学习重心放在语意层、协定层、Agent 架构上，而不是无止境地优化像素定位的最后 2 个像素。
- **其二，别把像素手艺完全扔掉。** 传真机到今天还在某些角落运转——同理，**那些永不升级的历史系统会活得比我们想像的久得多**，而它们恰恰是企业自动化最有钱、最刚需的战场。第 8 章那身功夫不会白费，它会变成「兼容层专家」的稀缺价值——就像今天还懂 COBOL 的人，反而因为稀缺而值钱。

> 💡 君之一席话
> 与君一席话：**一个技术人最珍贵的能力，不是把当下的技艺练到顶尖，而是清醒地知道「我此刻练的这门技艺，在范式的时间轴上站在哪里」——是朝阳、是正午、还是夕阳。同一份精湛，押在朝阳上是复利，押在夕阳上是沉没成本。** 这不是要你别练像素手艺——恰恰相反，第 8 章那身功夫真实、深刻、且在「兼容层」这个角落会长久值钱。它是要你**练的时候心里有数**：你练的是一门「终将降级为兼容层」的技艺，所以你既要练好它（因为兼容层的活计永远有人付钱），又不能被它困住（因为主干道已经改道）。**最高级的职业判断，是能同时持有两个看似矛盾的真相：『这门手艺正在过时』和『这门手艺仍然值钱』——并据此做出『精通它、但同时把更多筹码押向取代它的那个范式』的下注。** 传真机师傅里混得最好的那一个，一定不是传真机按得最快的，而是那个一边精通传真机、一边早早学会了电子邮件的人。在 AI 操作电脑这件事上，你要当的，就是那个人。

> 🔍 高端点评 — 为什么「兼容层」不是失败，而是任何成熟范式迁徙的标准终态
> 把「Computer Use 降级为兼容层」说成「Computer Use 失败了」，是个常见的误读。看计算机史就懂：**没有任何一次范式迁徙，是靠「彻底消灭旧范式」完成的，全都是靠「把旧范式封装成一层稳定的兼容层」完成的。** x86 处理器内部早就是 RISC 微指令了，但它对外永远仿真一套 CISC 指令集——CISC 没死，它变成了兼容层。浏览器渲染引擎重写了无数遍，但永远保留着对二十年前烂 HTML 的兼容模式——旧 web 没死，它变成了兼容层。云端跑的是容器与微服务，但里面跑着无数个被封装起来的单体旧应用——单体没死，它变成了兼容层。**Computer Use 的「看截屏点像素」，注定也是这个结局：它不会被消灭，它会被一个更高层的语意范式（MCP / 语意总线）包起来，成为「当语意接口缺席时自动降级激活」的底层保险。** 而这恰恰是第 8 章那套双环、Delta-Frame、弹道鼠标的**真正归宿与长期价值**——它们不是要在主干道上跟语意范式竞争（那场仗它们会输），它们是要成为**那层永远可靠、永远兜底的兼容基座**。一个只在 demo 里点得漂亮的 Computer Use，三年后一文不值；一个被工程化成「企业旧系统最后一哩自动化保险」的 Computer Use，会像今天的 COBOL 维护工一样，安静地、长久地、高薪地活着。**认清自己的技艺终将成为兼容层，不是悲观，是看懂了历史之后的踏实——因为兼容层，是这个行业里最不容易被裁掉的位置。**

## 本章小结

- **范式冲突（情境 1）**：今天的电脑是为人眼/人手设计的，逼 AI 看截屏点像素，是一场巨大的信息熵浪费——App 把本来握在手里的结构化语意渲染成高熵像素图，AI 再花几千 Vision Token 把它逆向回语意，还退化成带误差的座标。第 8 章二十道难关，大半是这个范式错误的直接后果，而非 AI 的缺陷。
- **逃离像素的光谱（情境 2）**：二十个前沿方向构成一条从「点得更准」到「根本不用点」的光谱——视觉定位（SoM/SAM2/Ferret-UI）→ 屏幕理解（CogAgent/ScreenAI）→ 网页语意（Mind2Web/CodeAct/BrowserUse/Skyvern）→ 评测（OSWorld/WebArena/VisualWebArena）→ 协定语意层（**MCP**/Constitutional AI）→ 底层架构（**Mamba/SSM**）。行业重心正肉眼可见地从像素端滑向语意端。OSWorld 的「最难」恰恰证明像素范式已到顶。
- **AI 原生 OS 五构件（情境 3）**：① 语意状态总线（广播结构化状态，非像素）；② 零拷贝内存映射（UMA/Arrow 同源）；③ 无头优先 + 反向渲染（像素降级为按需的人类兼容层）；④ LLM-Ready API / Agent Manifest（**MCP 的 OS 级推广**，五构件中唯一已落地的地基）；⑤ 大规模多代理并行（无头 App 让万级 Agent 并行读写）。灵魂：在机器对机器的电脑里，人类是访客，不是房东。
- **四阶段路线图（情境 4，最思辨）**：寄生（偷塞语意，真实进行中）→ 共生（App 声明动作，MCP 方向真实）→ 原生（无头 OS，推演）→ 通用（跨设备语意网络调度，Astra 方向）。**所有具体产品名（I/O 2026、Aluminium OS、Antigravity、Project Jarvis、AppFunctions、Hydra Memory Layer）一律 ⚠️ 未经查证**；唯 Project Astra（原型）与 MCP（协定）为真。读路线图：信方向、疑站名。
- **哲学收束（情境 5）**：当电脑为 AI 原生而生，「看截屏点像素」会像今天的传真机手艺——不消失，但降级成「对付永不升级的历史遗留系统」的兼容层。两个并存的真相：别把毕生押在像素上（主干道已改道语意），也别把像素手艺扔掉（兼容层的活计长久且高薪，如同 COBOL 维护工）。兼容层不是失败，是一切成熟范式迁徙的标准终态。

第二乐章到此落幕，它走完了自己的一条弧：第 7 章让你造出的智能第一次「看懂并动手」，第 8 章把这双手磨成肌肉记忆，第 9 章则拉远镜头、质疑这门技艺所站的范式本身。本章与第 7–8 章出现的每一个真实模型、框架、评测、协定（SoM、SAM2、CogAgent、ScreenAI、MCP、OSWorld、Mamba……），都收进了**附录 F「代理操作工具与论文速查」**，并把所有「未核实的产品名」单独钉在真伪校准区，随时可查。

但第二乐章留下了一个它自己回答不了的问题，而这个问题，正是第三乐章的入口。

我们从第 7 章一路把那个本地代理叫做「**廉价**的眼和手」——廉价，是它能当器官、能贴身、能一刻不停睁着眼的全部前提。可是「廉价」这个词，在 demo 里是真的，在规模上却是一张会跳票的支票。当你把这双眼手从一台机器上的一个 Agent，放大到一座数据中心里同时服务成千上万人的万级 Agent——每一个都拖着几万 token 的屏幕历史、每一个都要求毫秒级的反应——你会发现帐在一个最意想不到的地方爆掉：**不是算力，是内存。** 那个又小又便宜的模型，权重也许才 16GB，但它们的 KV Cache 会在某个清晨悄悄吃光你那台 DGX Spark 的 128GB，然后整个集群因为 OOM 而瘫痪。

第三乐章，我们就去打这场最隐形、也最致命的仗——**内存与缓存的战争**。翻到第 10 章，先从「为什么大模型的瓶颈根本不是算力」这个反直觉的物理事实讲起。
